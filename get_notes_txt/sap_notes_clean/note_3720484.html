Symptom

Often, you’ll find the row count in the target table is much higher than in the source table.

Reasons and Prerequisites

When you’ve ruled out replication latency as the cause of these disruptions.
Tables in ECC often use CHAR as the primary key, but user input may be NULL, a single space, or multiple spaces.
These differences are hard to spot visually, yet they can cause errors for UPDATE and DELETE operations based on the primary key in SAP HANA.
because SAP HANA does not automatically trim extra spaces.
To standardize the data, we introduced the following option to automatically trim data from the ECC tables so SQL comparisons are consistent.

Trim the training spaces

Set this parameter to True to trim training spaces from strings for both primary key and non-primary key columns. The default value is False.

For example, “A 1” is trimmed to “ A 1”, and “ ” is trimmed to “”.

Note:
For transparent tables, only fully empty space is trimmed.

# Verifying the missing (UPDATE/DELETE) operation**

## For row count mismatch according to DELETE operation

If such table is not a big table, you can perform the following sql to check who the row is missing according to the primary key:

For example:

hana sql
SELECT y.*
FROM "your_A522_target_table" y
WHERE NOT EXISTS (
SELECT 1
FROM "your_A522_virtual_table" x
WHERE x."MANDT" = y."MANDT" and x."KAPPL" = y."KAPPL" and x."KSCHL" = y."KSCHL" and x."MATNR" = y."MATNR" and x."KFRST" = y."KFRST" and x."DATBI" = y."DATBI"
);
```

## For row data mismatch according to UPDATE operation

Please execute the following check to verify whatever any primary key field contains **blank values** (spaces but not truly empty):

sql

SELECT your_primary_key_list
FROM your_virtual_table_on_A522
WHERE
(TRIM(your_primarykey1) =''AND your_primarykey1 <>'')
OR (TRIM(your_primarykey2) =''AND your_primarykey2 <>'')
OR ...
```

Also, You check whether the remote source is using pk_only(primary key only), but capture_before_after_images is not set to true?

If there are updates to the primary key columns (including column like MANDT,KAPPL etc), the DPAgent will not capture these changes, which can lead to data inconsistency in the target system.
If you confirm these data mismatches are used by spaces in the primary key.
Please consider setting the remote source parameter “Trim the training spaces” to true to avoid such issues after suspend the replication.

If you change it from **false → true**,

you must:

1.**Empty the target table**,

2.**Reload target table** from the source,
3.**Restart replication**.

This ensures consistent key handling without trailing-space mismatches.
