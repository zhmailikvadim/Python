Symptom

An issue was identified during IDoc processing, where the generated TE_IN XML was missing attributes 3, 4, and 11.

Other Terms

SETI, TXINTF

Reason and Prerequisites

After the upgrade to S/4, the code flow changed. The include program /TXINTF/DEFINE_CALCTYPE_C is missing or is no longer used in the same way as in ECC.

In standard include program LMEPOF1T, subroutine MEPO_BADI_ME_DEFINE_CALCTYPE, a new subroutine was introduced after the upgrade. This new subroutine is executed as part of the BAdI processing and is used to call function module /TXINTF/DEFINE_CALCTYPE_C.
The new subroutine, MEPO_BADI_MM_PUR_S4_PO_DEFINE, performs a mapping of PO data into intermediate structures:

  • purchaseorder_old/new
  • poitem_old/new

This mapping follows a limited field-mapping approach, which means:

  • Only selected fields are mapped and populated.
  • Fields that are not included in the mapping remain initial (blank/zero) in both the “old” and “new” structures.

Due to this limited mapping, the comparison logic in function module /TXINTF/DEFINE_CALCTYPE_C no longer works in the same way as in ECC.

  • Both the new and old PO header/item structures contain the same data at the time of comparison.
  • This occurs because the limited field mapping has already taken place earlier in the new S/4 standard flow.

As a result:

  • Some fields that are expected to differ are not populated, or remain initial, in the mapped structures.
  • The comparison detects no difference, and therefore the calctype is not assigned.

Solution

An implicit enhancement was added at the end of subroutine MEPO_BADI_ME_DEFINE_CALCTYPE to return the Include Program: /TXINTF/DEFINE_CALCTYPE_C, and the BAdI implementation /TXINTF/PO_DEF_CNDNCTRL was deactivated to allow function module /TXINTF/DEFINE_CALCTYPE_C to use the full EKKO and EKPO structures for comparison logic when assigning the calculation type.
