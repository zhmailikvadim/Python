Symptom

You have implemented IAS originally using the API version 1 (ODATA).

As part of your implementation, one of the fields you are pulling from SuccessFactors and mapping in IAS is the User's userId.

Now you are performing the migration from Odata to SCIM (V1 to V2) and you have noticed that userId field is not returning and you also can't enable it to return.

Environment

SuccessFactors

IAS using api version 2 (SCIM APIs)

Reproducing the Issue

  1. Have an implementation of IAS using API version 1
  2. In your IAS implementation have a customization reading userId from User entity
  3. Start the migration from V1(Odata) to V2(SCIM)
  4. Notice that userID no longer appears to be mapped

Cause

Expected Behavior

Resolution

The UserId field was not originally meant to be returned neither in API V1 on IAS meaning that if you had this field returning, this means that you have customized your IAS mappings.

Although this was possible to achieve in Odata API, in SCIM API it is not possible to customize the query to return fields which are not documented in the SCIM metadata here.

This means that userId can't be returned.

If you believe such field should be supported, then please feel free to use the influence tool by reviewing the document in 'See Also'.

Potential Workaround

If in your implementation, the personIdExternal is the same as the userId, then you can use the field employeeNumber from the SCIM API payload.
