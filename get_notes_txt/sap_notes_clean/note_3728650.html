Symptom

An activity cannot be processed because it is blocked within the FSM system. This may be observed in areas such as the Planning Board or the FSM mobile application, where the activity appears unavailable, restricted, or cannot be updated.

Environment

  • SAP Field Service Management (FSM)
  • Proaxia Connector (PACG)
  • ECC, S/4 HANA

Reproducing the Issue

Actual Behavior

  • The activity remains blocked on the FSM side (e.g., Planning Board or FSM mobile application).
  • The following error message is displayed:
    • “This activity is blocked and cannot be edited until synchronization with the ERP system is completed.”
  • The activity remains blocked for an extended period and does not become editable as expected, even after synchronization should have been completed.

Expected Behavior

After the change is completed and data is successfully exchanged with ECC or S/4 HANA, the activity is automatically unblocked and becomes editable in FSM.

Cause

  • Inbound IDocs in ECC/S4 are not processed successfully and remain in a pending or error state.
  • Outbound IDocs are stuck in the queue and are neither processed nor confirmed.
  • As a result, communication between FSM and ECC/S4 is disrupted, preventing successful data synchronization and leaving the activity in a locked state.

Resolution

1. Verify Block Details in FSM

  • In FSM, check the details of the blocked object. refer also to 3444543 - Root Cause of BLOCKED objects in SAP Field Service Management - SAP for Me 

  • Identify:

    • The object causing the block

    • The date and time when the block was set

This information helps determine whether the issue originates from inbound or outbound communication.

 

2. Check Incoming Communication (Inbound IDocs)

If the blocked object is caused by incoming communication:

  1. In ECC/S4, check inbound IDocs using:

    • Transaction /PACG/ECM_COCKPIT2

    • Transaction WE02

  2. Filter IDocs from the relevant date with status:

    • 64 (ready to be processed)

    • 51 (error)

  3. Identify IDocs related to the blocked object and process them.

  4. If processing fails:

    • Review the application log in status 51

    • Resolve the underlying error

    • Reprocess the IDoc

Once successfully processed, an outbound message should be generated and the object in FSM should be unblocked.

 

3. If Inbound Processing Is Successful

If all inbound IDocs are processed successfully and the object remains blocked, the issue is likely related to outbound communication.

 

4. Check Outgoing Communication (Outbound IDocs)

  • Any FSM object synchronized with ECC/S4 (see supported objects list in  /PACG/ECM_TRIG_SUPO - Supported object send - FSM Cloud Connector - Confluence ) requires a corresponding outbound IDoc to release the block.

  • In most cases, inbound and outbound messages form pairs, for example:

    • /PACG/ECM_SERVICECALLIN ↔ /PACG/ECM_SERVICECALL

    • /PACG/ECM_ACTIVITYIN ↔ /PACG/ECM_ACTIVITYOUT

  • Some inbound messages are confirmed via the generic message:

    • /PACG/ECM_GENERIC

 

5. Validate /PACG/ECM_GENERIC IDoc

  1. In the application log of a successfully processed IDoc (status 53), locate the reference to the /PACG/ECM_GENERIC IDoc.

  2. Copy the IDoc number and check its status in WE02.

 

6. Resolve IDoc Status Issues

Take action based on the IDoc status:

  • Status 03 or 18

    • Change status to 30 using program RC1_IDOC_SET_STATUS (transaction SE38)

    • Reprocess via BD87

  • Status 40

    • Check the detailed error in:

      • /PACG/ECM_COCKPIT or /PACG/ECM_COCKPIT2 ( Outbound idocs statuses - FSM Cloud Connector - Confluence )

    • Attempt to reprocess the IDoc

    • If the issue persists, escalate to the support team

  • Status 41

    • Change status to 30 (via SE38 – RC1_IDOC_SET_STATUS)

    • Reprocess using BD87
