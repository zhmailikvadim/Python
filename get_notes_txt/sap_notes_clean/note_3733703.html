Symptom

Repartitioning an already partitioned table causes all resulting partitions to be placed on a single NUMA node.

When a table is initially unpartitioned, the first partitioning operation distributes the partitions evenly across the NUMA nodes of the host.

However, any subsequent repartitioning, whether changing the number of partitions or switching the partitioning type (hash, round‑robin, or range), results in all partitions being moved to a single NUMA node.

You can verify the NUMA placement of each partition by checking the NUMA_NODE_INDEX column in the M_CS_COLUMNS monitoring view.

Other Terms

Partition, NUMA

Reason and Prerequisites

Issue number 356198

Reason:

Due to a programming error in SAP HANA, repartitioning operations lack NUMA awareness, causing all partitions to consolidate on a single NUMA node.

Affected Releases:

  • SAP HANA 2:
    • Revisions of SPS00 - SPS06
    • Revisions <= 079.09 (SPS07)
    • Revisions <= 089.01 (SPS08)

Prerequisites:

  • A table is already partitioned.
  • The host has 2 or more NUMA nodes.

Solution

Apply one of the following SAP HANA Revisions:

  • SAP HANA 2:
    • Revisions of SPS07: SAP is currently working on a permanent solution to this issue. This SAP Note will be updated as soon as a revision containing the solution is available
    • Revisions of SPS08: SAP is currently working on a permanent solution to this issue. This SAP Note will be updated as soon as a revision containing the solution is available

Workaround:

  • Post repartitioning, unload and reload the table from memory.
    1. UNLOAD <SCHEMA>.<TABLE>
    2. LOAD <SCHEMA>.<TABLE> ALL
  • After the reload, check the NUMA distribution of partitions using M_CS_COLUMNS
