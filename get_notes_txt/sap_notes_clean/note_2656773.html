Symptom

After upgrading from Redwood CPS 8 to BPA 9.0.11 or higher, some jobs are not triggered as per schedule.

Environment

SAP Business Process Automation by Redwood version 9.0.11.x or higher

Cause

In BPA 9.0.11 and higher, Runtime Parameter default expressions are no longer evaluated as Process Definition Owner by default.

Previously, the default expressions of runtime parameters were evaluated using a session of the process definition owner; this is changed in BPA 9.0.11 and higher. In 9.0.11 and higher, the default expressions will be evaluated as the process owner, the user submitting the process. This means that the process owner will need permissions to all objects used by the expression. You set the session subject to the process definition owner by specifying JobDefinitionOwner in the Evaluate Runtime Parameters With User field on the Options tab of the process definition and chain definition editors; the technical name of the field is EvaluationOwnerType.

Before 9.0.11, processes/jobs used to be resubmitted in an internal session, this cause differences in parameter evaluation between initial submit and resubmit. From 9.0.11, process resubmits are done in a Process Owner session. This has impact on jobs/proceses initially submitted by users that are disabled or have their roles/privileges revoked as they do not have the correct privileges anymore to resubmit the job/process.

Resolution

You can identify processes (jobs) effected by this issue by running the following queries in the BPA support servlet: http://<hostname>:<port>/redwood/support

Look for Scheduled Jobs of disabled Subjects/Owners:

select j.* from Job j, Subject s where j.ParentJob is null and j.Status in ('S', 'R', 'H', 'W') and j.OwnerSubject = s.UniqueId and s.Disabled = 't'

Look for users that own jobs but have no roles anymore:


select s.* from Subject s where exists (select 1 from Job j where (j.Status in ('S','R','W','O','a','q','V','I','N','8','6','Q','B','o','c','G','4','H','-') or (j.Status in ('C', 'E', 'U', 'K') and exists (select 1 from ActionWork a where a.ActionWorkType = 'G' and a.RelatedObjectId = j.JobGroup) ) ) and j.SubmitFrame is not null and j.OwnerSubject = s.UniqueId) and   not exists (select 1 from SubjectRoleGrant g where g.GranteeSubject = s.UniqueId)

 Solution:

  1. Resubmit the processes found in the output of the queries above with a new user.
  2. Ensure that the process owner has all privileges required to scheduleprocesses. The required privileges can be found in the BPA admin guide.
  3. Update the Evaluate Runtime Parameters With User filed on the processdefinitions to specify JobDefinitionOwner. The processes will then use that user's privileges when scheduling processes.
