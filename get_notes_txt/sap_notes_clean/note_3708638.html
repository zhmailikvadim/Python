Symptom

accounts.search REST API requests error with "403048 Api rate limit exceeded" when including the parameter openCursor=true

Environment

  • SAP Customer Data Cloud
  • REST API

Reproducing the Issue

When sending a relatively high volume of request to SAP Customer Data Cloud (CDC)'s accounts.search REST API endpoint with the parameter openCursor=true, error "403048 Api rate limit exceeded" responses are returned.

Cause

This is most likely due to a sub-optimal implementation flow. As previously announced by CDC on the documented Changes Requiring Your Attention page:

  • Title: Limit the Amount of Concurrent accounts.search Open Cursor Requests to the Platform
  • Effective On: Mar 31, 2023

"On March 31, 2023, we will limit the amount of concurrent accounts.search cursors requests that can be opened to a given apikey within a 10-minute window to 1000.

Who Will Be Affected:Customers that issue accounts.search requests with openCursor=true in high-rate flows such as login or registration.

Impact:Calls that cross this threshold will return with a rate limit error code. IdentitySync jobs will not be affected by the change.

Why: In the interest of improved platform behavior: cursor searches incur additional cost to the platform as the results need to be cached and retained for extract for the cursor life span. Opening many concurrent cursor queries which are not iterated can cause unnecessary strain on the platform and lead to search performance degradation.

Action Required:Ensure that when searching for a small number of search results such as lookup by email or a specific unique data field, refrain from passing the openCursor parameter. In case of large result search queries, create a cursor and iterate through it instead of issuing multiple concurrent search queries with individual cursors."

Resolution

When batching through multiple "pages" of results using the accounts.search REST API ensure that only the first request contains the parameters:

  • query
  • openCursor=true

All subsequent requests for the next batch(es) of results should include only the cursorId parameter retrieved from the nextCursorId parameter in the response of the preceding request.
