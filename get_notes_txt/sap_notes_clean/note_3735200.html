Symptom

When refreshing some Web Intelligence reports, get error:  

[Simba][BigQuery] (131) Unable to authenticate with Google BigQuery Storage API. Check your account permissions. For information about this error, please refer to SAP Knowledge Base Article 2054721 on the SAP Support Portal. (IES 10901).

Environment

  • SAP BusinessObjects Business Intelligence Platform 2025 (BI 2025)
  • Google Big Query
  • ODBC connection
  • Web Intelligence (WebI)
  • Information Design Tool (IDT)

Reproducing the Issue

  1. Set up an ODBC connection to Google Big Query in the ODBC Administrator
  2. Create a connection and universe in IDT that uses this connection
  3. Refresh a WebI report that connects to this universe

If you limit the amount of data that returns, then you do not get the error.

As a test, you can add "limit=1000" to the end of the SQL query in the webi report to see if the data returns

Cause

The odbc connection is not correctly configured for the amount of data that is returned.

Resolution

In the ODBC Administrator, edit the DSN.  For a particular customer, the following settings resolved the error:

  • Enable High-Throughput API
  • Set Rows per block set to 10,000 (This was reduced)
  • Minimum Query Result size set to 100,000,000 (This was increased)

Keywords

GBQ, odbc

Environment

  • SAP S/4HANA
  • SAP Master Data Governance

Reproducing the Issue

  1. Go to NWBC.
  2. On the role selection screen, select any custom role maintained in the menu.
  3. Observe that the system dumps with a 500 Internal Server Error and the above authorization-related message.

Cause

Incorrect configurations. 

Resolution

  1. Verify role assignment: In PFCG, ensure the required custom role(s) are assigned to the user.
  2. Verify cockpit/service availability: Ensure the corresponding NWBC cockpit exists and the related ICF services are active in SICF.
  3. Align composite role menus: If using composite roles, import the Menu from the included single roles to the composite role so that the menu structure is present and consistent (especially after an upgrade).
  4. Compare with standard: Review and compare the custom roles with the standard MDG roles. If needed, copy the standard MDG roles and adapt authorization values to align with the intended UI behavior.
  5. Re-test: Log on to NWBC again and select the custom role to confirm the 500 error is resolved.
  6. Optional (display topic): If technical names appear instead of descriptions in composite role menus after menu import, refer to the KBA below to adjust the display setting.
  • Technical names displayed instead of descriptions in composite role menus after menu import (0003670084)
