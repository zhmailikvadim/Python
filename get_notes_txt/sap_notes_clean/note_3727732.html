Symptom

As described in SAP ONE Support Launchpad Notification of Legal Change Unified monthly employer report 2026 (JMHZ)

The functionality delivered in this SAP Note is currently under development. It is intended exclusively for customers registered as JMHZ pilot testers, or for the preview of features that may become available for productive use in the future.

These updates are provided:

  • REGZEC (DM - data mining, CS - compose submission, PF - process feedback, SO - special operations, INIT - import of IDs to IT3457)

    • DM: All employees who weren't registered in ONZ are evaluated as relevant employees for registering via REGZEC with the real hire date

    • DM: Issue with reading org. structure via PLOGI attribute (newly released functionality in the previous SAP Note) has been resolved

  • MH (DM - data mining, CS - compose submission, WB - workbench)

    • Logical check 248 reported the existence of non‑zero elements on an employment relationship other than the primary one (PPV) in the DME section.
      Most source wage types for the DME section are stored in payroll results under the main personnel number, and are therefore correctly reported on the primary employment relationship (PPV). However, the source wage types for attributes 10289, 10416, 10417, 10308 can occur in the results for secondary personnel numbers. If that happened, these attributes were incorrectly populated for secondary PPVs as well. Going forward, the values of these wage types are transferred to the primary PPV.

    • Bug in the XML editor of transaction PC00_M18_JHM_WB_ADMI
      After editing and saving an XML, its content remained in memory. If the user then opened another XML and chose not to save it (Cancel), the content of the previous XML still in memory was used and saved incorrectly.

    • Place of work (10229, 10230, 10231) – New parameter for an alternative subtype of Infotype 0006
      The attributes are taken primarily from Infotype 6, subtype CZMV. It is now possible to set a different subtype using a parameter in table T7CZRE:

      • PAR_AREA = JHM_VYP_PLACE_OF_WORK
      • PAR_NAME = W_ADDRESS
      • PAR_VALUE = <subtype>

    • Place of work (10229, 10230, 10231) – New method for determining place of work from organizational assignment

It is now possible to set the algorithm for determining the place of work using a parameter in table T7CZRE:

      • PAR_AREA = JHM_VYP_PLACE_OF_WORK
      • PAR_NAME = METHOD_ORG
      • PAR_VALUE = 2

The first part is analogous to the processing in REGZEC:

      • The program reads table T5T1L with key TXTYP = CZMV and uses the values in fields ANOWN and VAOWN to read table T536A.
      • From table T536A it determines the country (key S) and the municipality (key W).

It then retrieves the municipality name from table T5T54 using only two fields: Sign (ZNAK) and Municipality code (OBEC), i.e., without using the Region (BAND) field. Therefore, it is assumed that each municipality belongs to only one region. If table T5T54 contains duplicates, the algorithm will select one of the matching records and report the situation as a warning in the log.

    • Enhancement of Method 2 for determining attribute 10259 Statutory working time

      • If the value of Statutory (10259) is lower than the value of Agreed (10260), 10259 is set to 10260.
      • For reduced working hours, Statutory (10259) has so far been calculated from Agreed (10260) by grossing up for the reduction in Infotype 7 and/or the reduction in the work schedule plan itself (T508A‑M2STD).
        It is now possible to avoid this calculation by specifying a reference work schedule in table T7CZPPD, field REFSC.
        The program goes through the work schedules it finds for the given period in WPBP (field SCHKZ) and looks them up in table T7CZPPD.
        • If it finds at least one record in T7CZPPD with a filled reference schedule (REFSC), the program sums the shift lengths from T552A based on work schedules that are either in WPBP (if no REFSC is found for the given SCHKZ) or in T7CZPPD (if a non‑empty REFSC exists in T7CZPPD).
        • If no record is found in T7CZPPD, or it is found but the REFSC field is empty, the existing algorithm applies. That is, the change is backward compatible and no configuration is needed to retain the current behavior.

    • WB: fix in XML editor to prevent overwrite if editor was closed without confirmation

    • DZMH: added handling for new known "nonsense-protocol content errors".

    • Introduced several substates for B2A final status to better hint users about overall state:

      • in case that protocol was not yet ready and therefore opened (i.e. status 80) rows are not handled, status will be marked as RED ERROR (previously green, as response was received, for example "Unprocessed") which confused user.

      • Status OK will be only if all forms in the submissions are accepted without any error (warnings are considered as OK).
      • Status OK - Partially accepted will be if any form in the submission has an error
      • Notice: Status is relevant for each submission (Correlation ID), not to the general state of the Monthly statement itself (GUID Podani).
