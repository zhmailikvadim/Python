Symptom

Other Terms

Alphanumeric CNPJ; eSocial 2.5; Historical data; CNPJ format; Law 14849/2024; Transparent tables; Data elements;

Reason and Prerequisites

Law Nº 14,849/2024 establishes that CNPJ can contain alphanumeric characters starting from July 1st, 2026. However, the listed ABAP Objects above are historical and will not receive any Alphanumeric CNPJ.

Technical Background:

The data elements listed above use fixed-length NUMC fields that were designed specifically for numeric CNPJ format. The Alphanumeric CNPJ format requires different field properties and validation rules that are incompatible with the existing data element definitions and their associated structures.

IMPORTANT: The places where these data elements are used were changed to new data elements in CHAR format. The current system uses updated data elements that support Alphanumeric CNPJ format, while the historical objects listed in this note remain unchanged for data integrity purposes.

The transparent tables listed above were specifically designed for eSocial version 2.5 functionality, which used a structured table-based format for data storage.

With the evolution to eSocial version S-1.0, the system architecture changed to use JSON format for data storage and transmission. As a result, these tables became obsolete for new eSocial events and are now used exclusively to store historical information from older eSocial events that were processed using the previous table-based approach.

Since these tables contain only historical data from eSocial version 2.5 and earlier, which were created before the introduction of Alphanumeric CNPJ format, they will never receive records containing Alphanumeric CNPJ values. Therefore, updating these tables to support the new CNPJ format is technically unnecessary.

Solution

The data elements and transparent tables listed in the Symptom section remain unchanged regarding Alphanumeric CNPJ support. These tables serve as historical repositories for eSocial events processed under version 2.5 and earlier versions, which exclusively used numeric CNPJ format.

Current eSocial functionality utilizes JSON-based data structures that are handled by different components of the system, which are being updated to support Alphanumeric CNPJ format through separate SAP Notes.

This approach ensures:

  • Historical data integrity is maintained
  • System performance is optimized by avoiding unnecessary updates to obsolete structures
  • Development resources are focused on active system components

Recommendation

If you have custom code that uses any of the data elements or transparent tables listed in the Symptom section and you need to support Alphanumeric CNPJ format, you must adapt your custom code to use different objects. These tables will not support the new CNPJ format, so any custom developments relying on them for current or future eSocial processing should be updated to use the current eSocial JSON-based architecture or other appropriate system components that support Alphanumeric CNPJ.

Review your custom developments and identify any dependencies on these obsolete objects to ensure compatibility with the new CNPJ format requirements.
