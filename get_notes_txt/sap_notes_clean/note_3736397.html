Symptom

  • The error message "M7120 - You have no authorization for this transaction in plant &1" is raised when posting the Goods Issue via transaction VL02N.
  • To avoid granting plant authorizations to the user, the following KBA is considered and the check indicator for authorization object M_MSEG_WWE is set to "Do not check" in transactions SU24 or SU22:
    • KBA 2985553 - VL02N/Vl06O :Authorization check M_MSEG_WWE for receiving plant (SAP ERP & SAP S/4 HANA)
  • With this configuration, the authorization check is successfully skipped when posting the Goods Issue directly in VL02N.
  • However, the same error occurs when reporting the "Departure" event in a Freight Order within Transportation Management. This action tries to automatically post the Goods Issue in the related delivery.
  • In transaction SU53, the authorization failure is identified in function module /SCMTMS/LI_POST_GM_RFC for authorization object M_MSEG_WWE.
  • Why is the authorization check not skipped in this scenario, and how can this be handled?

Environment

SAP Transportation Management embedded in S/4HANA 1709 and subsequent releases

Reproducing the Issue

  1. Open the Fiori Launchpad
  2. Create or edit a Freight Order
  3. Perform the required planning/execution steps
  4. In the Execution tab, report the "Departure" event
  5. The system try to automatically post the Goods Issue in the delivery
  6. Check the Document Errors tab in the Freight Order
  7. Error message "M7120 - You have no authorization for this transaction in plant &1" is displayed

Cause

  • This is standard system behavior.
  • The approach described in KBA 2985553 applies only to Goods Issue postings executed directly via the online transaction VL02N.
  • The system differentiates between online transaction processing (e.g. VL02N) and background/RFC processing (e.g. external calls). The skip of the authorization check does not apply to background executions.
  • Therefore, the system still enforce the authorization check for M_MSEG_WWE when the Goods Issue is posted by RFC users, external systems and other external calls (e.g. BAdIs, reports, APIs...), including those triggered via the TM RFC function module during the Departure event.

Resolution

  • Currently, there is no standard approach to bypass the authorization check for authorization object M_MSEG_WWE when posting the Goods Issue from background or other RFC external calls.
  • In that case, the recommended standard approach is to grant the required authorizations for both the sending and receiving plants. This is required because movement type 638 is executed in the sending plant, and this subsequently triggers movement type 107 in the receiving plant, and the authorization object M_MSEG_WWE is checked for these movement types.
