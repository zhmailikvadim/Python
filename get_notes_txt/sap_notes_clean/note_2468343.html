Environment

  • SAP HANA 1.0
  • SAP HANA 2.0

Reproducing the Issue

Attempt to start a recovery with backup files which are not all from the same backup or have been overwritten due to prefix not being distinct.

Cause

This issue can be caused if a few backups have been taken but the "Backup Prefix" has not be changed.

That results in overwriting the taken backup file every time.

So by a restore the catalog has a lot of entries with different backup ids, but only the last backup, as file, is available.

By running the command (as <sid>adm)
hdbbackupcheck -v <path to backup image>
on each file of the backup, the output will show the backup id of each and you can confirm that one or more has a different backup id than the others, so does not belong to the same backup 

Resolution

SAP HANA Administration Guide for SAP HANA Platform
Section
Naming Conventions for Data Backups

For file-based backups, it is strongly recommended that you use a unique prefix for each data backup name. For example, a timestamp.

If you use the same prefixes, it is recommended that you replicate a data backup to a new destination as soon as the backup is created. Otherwise, an existing complete data backup with the same name will be overwritten by the next data backup.

If you have encountered this issue and want to cancel out of the restore situation, if the persistence has not yet been written to disk, you can remove the recoverInstance.sem file from $DIR_INSTANCE/work

With this removed, the instance will not be locked by the failed recovery and as long as it had not started writing the persistence from the backup files, the instance will come up in the state it was before recovery had been initiated.

If it had started to write the persistence, then the instance will fail to start, and you will need to perform a successful recovery from a set of backup files which have the same backup id.

In the case of a system copy to another instance:

If the source system is still available, take a new full backup of tenant using a unique prefix and to a new separate directory for this particular backup instead of the place regularly used to take regular full backups.

Then transfer them to a unique, newly created directory on target and then perform the recovery.

This will ensure that all the files that make up the backup image belong to the same backup id.
