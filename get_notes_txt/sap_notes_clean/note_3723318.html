Symptom

  • The 'sp_rename' command can rename a table even if there is an open transaction against it.
  • After renaming the table, the transaction remains open, but the table is no longer accessible under its original name.
  • A warning is displayed stating that changing an object or column name could break existing stored procedures, cached statements, or other compiled objects.

Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental.

Environment

SAP Adaptive Server Enterprise 16.0 and later

Reproducing the Issue

  1. Run 'select * from table1' to retrieve table data, which returns rows.
  2. Start a transaction using 'begin tran'.
  3. Check transaction count with 'select @@trancount', which returns 1.
  4. Insert a new value into the table using 'insert into table1 values(300)'.
  5. In a separate session, run 'sp_rename table1, t200' to rename the table. A message confirms the object name has been changed, along with a warning about potential impacts on stored procedures, cached statements, or other compiled objects.
  6. Attempt to query the original table name 'select * from t2', which results in an error message stating that the table cannot be found.

Resolution

  • A table can be renamed while an open transaction exists on it because 'sp_rename' acts only on the metadata, rather than locking the physical data rows themselves.
  • The 'sp_rename' command acquires an exclusive lock on the system tables (specifically sysobjects) to update the table name, not on the data pages of the table being renamed.

Note: Active transactions (INSERT/UPDATE/DELETE) hold locks on data pages or rows, whereas 'sp_rename' does not require a full table lock to change the name, allowing it to proceed independently of active data manipulation.
