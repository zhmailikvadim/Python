Symptom

  • After upgrading SAP Commerce Cloud to 2211-jdk21.x (Framework Update: Java JDK 21 and Spring 6.2), Single Sign-On to Backoffice returns HTTP 500 Internal Server Error.
  • The same SSO configuration works on the previous (JDK 17–based) version.
  • Logs/trace show failures similar to:
    • "java.lang.IllegalArgumentException: Failed to evaluate expression 'IS_AUTHENTICATED_FULLY'"
    • "org.springframework.expression.spel.SpelEvaluationException: Property or field 'IS_AUTHENTICATED_FULLY' cannot be found on object of type 'org.springframework.security.web.access.expression.WebSecurityExpressionRoot'"

Environment

SAP Commerce Cloud

Reproducing the Issue

  1. Upgrade an existing SAP Commerce Cloud 2211 (JDK 17) to 2211-jdk21.x without fully adapting OAuth and Spring Security configurations.
  2. Attempt Backoffice SSO.
  3. Observe HTTP 500 response code returned by Backoffice and exceptions in application logs referencing failure to evaluate 'IS_AUTHENTICATED_FULLY'.

Cause

  • With the framework update to JDK 21 and Spring 6.2, SAP Commerce Cloud adopts a newer Spring Security version and a new OAuth resource server implementation.
  • Previous configuration patterns using access-voter keywords (for example, IS_AUTHENTICATED_FULLY, role and scope voters) are deprecated and no longer evaluated as before.
  • The new configuration requires Spring EL expressions to be enabled and used in intercept-url  rules, and scope checks must be performed using Spring Security methods (for example, hasAuthority('SCOPE_xxx')), not via the removed oauth2 extension utilities.
  • If these follow-up manual activities are not completed, Spring fails to evaluate legacy access attributes such as IS_AUTHENTICATED_FULLY, leading to authorization errors and HTTP 500 responses during SSO flows.

Resolution

Product changes to be aware of

  • As part of the 2211-jdk21.x update (Spring 6.2), the OAuth resource server and Spring Security configuration model changed. Access voters are deprecated; expression-based access control is required.
  • This affects Backoffice and other web modules secured via Spring Security.

Actions

  1. Move to the new resource server

    • Review and adapt your resource server configuration according to the Moving to the New Resource Server section in the SAP Commerce 2211-jdk21 framework update documentation.
    • Ensure your extensions’ dependency tree does not reintroduce legacy configs.

  2. Enable Spring EL expressions and authorization manager

    • In the relevant Spring security XML, add:
      • use-authorization-manager="true"
      • use-expressions="true"
    • Example:
      <http pattern="/**" use-authorization-manager="true" use-expressions="true">
          <intercept-url pattern="/**" access="isFullyAuthenticated() and hasAuthority('SCOPE_READ') and hasRole('ROLE_BUSINESSADMINGROUP')"/>
          ...
      </http>

  3. Replace legacy access voters with EL equivalents

    • Replace IS_AUTHENTICATED_FULLY with isFullyAuthenticated().
    • Replace role checks (for example, ROLE_BUSINESSADMINGROUP) with hasRole('ROLE_BUSINESSADMINGROUP').
    • Replace scope checks (for example, SCOPE_READ) with hasAuthority('SCOPE_READ').

  4. Replace removed oauth2 utilities

    • If using @oauthSecurityChecker.hasScope(authentication,'basic') from de.hybris.platform.oauth2.util.OAuth2SecurityChecker (removed with the old oauth2 extension), replace with:
      • Previous:
        <intercept-url pattern="/**" method="GET" access="@oauthSecurityChecker.hasScope(authentication,'basic')"/>
      • Updated:
        <intercept-url pattern="/**" method="GET" access="hasAuthority('SCOPE_basic')"/>

  5. Review SAML/SSO security chains

    • Verify that SAML filters and Backoffice endpoints are covered by the updated http security configuration that uses expression-based access rules.
    • Ensure no legacy XML fragments re-enable voters or bypass the updated security chain.

  6. Validate and test

    • Clear caches and perform new deployment.
    • Test SSO again against Backoffice and verify no IS_AUTHENTICATED_FULLY or voter-related exceptions appear.
    • Review logs to confirm successful evaluation of expressions such as isFullyAuthenticated(), hasAuthority(...), and hasRole(...).

Notes

  • The change in behavior is by design with the Spring 6.2 migration. Systems remaining on JDK 17 may still work with legacy access voters, but after upgrading to 2211-jdk21.x, the new configuration is mandatory.
  • If SSO still fails after applying the above:
    • Search for any remaining legacy access="IS_AUTHENTICATED_FULLY, ROLE_..., SCOPE_..." attributes and convert them to EL expressions.
    • Ensure there are no duplicate or conflicting http security configurations shadowing the Backoffice context.
    • Validate that all custom security beans are compatible with Spring Security 6 and do not reference removed classes from the old oauth2 extension.
