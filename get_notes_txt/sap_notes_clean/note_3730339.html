Symptom

You have a POB whose price has been altered by invoices/credit memos/debit memos, and no further invoices are expected. You process an order RAI, which doesn't introduce any change in quantity or price. The price of the POB is reset to what it was in the original order, which is unexpected. The price of the POB will never go back to the invoiced amount, because no further invoices are coming. That's incorrect because invoices are the most important source of truth for POB price.

Other Terms

FINAL_INVOICE, FINAL_INVOICE_IND, rejection, cancellation, finalization, delivery-related billing, OCM

Reason and Prerequisites

Prerequisite: The original SD item was not fully delivered. The item was also rejected, but the rejection couldn't be processed because in OCM, you cannot process rejections where fulfilled quantity is less than invoiced quantity.

Reason: At order RAI creation, the previous coding checked if SD item flag LFSTA (delivery status) was set to C, which means that the item was fully delivered and fully invoiced. If it wasn't, the coding used to set order RAI flag FINAL_INVOICE_IND to C (clear), which would clear the POB's FINAL_INVOICE flag during RAI processing. Clearing this flag would incorrectly make RAR think that further invoices were coming, so it would reset the POB price. Especially when INVOICE_EFFECT_TYPE is set to F. The LFSTA flag however cannot be used in cases where the SD item is rejected, because then it would be set to B.

Note: SD item flag LFSTA is checked only for items with delivery-related billing, used by many event-based POBs. For order-related billing, used by many time-based POBs, the FKSAA flag (order-related billing status) is checked.

Solution

This note replaces replaces the check of LFSTA flag (delivery status) with a check of LFGSA flag (overall delivery status), which is a more reliable measure if the item's invoicing status. This will ensure that the POB's FINAL_INVOICE flag will remain set and the POB's price will not change.
