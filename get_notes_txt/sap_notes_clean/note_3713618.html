Symptom

  • User‑defined fields (UDFs) that originate for example from SAP ECC and are configured in SAP Field Service Management (FSM) are not evaluated correctly on FSM Mobile (iOS) when used in Screen Configuration visibility or validation rules.
  • Although the UDFs appear in the FSM Web application, the iOS client does not trigger Screen Config conditions (for example: visible‑if rules or JavaScript validation expressions).
  • This behavior typically occurs when the UDF has never been transferred from SAP ECC to FSM via IDoc—meaning the FSM system and mobile clients have no complete field definition available.

Environment

SAP Field Service Management

Reproducing the Issue

  1. Define a UDF in SAP ECC that is mapped to an FSM object (e.g., Activity).
  2. Configure a Screen Configuration rule for this UDF in FSM Admin (visibility, validation, etc.).
  3. Do not send the UDF via IDoc to FSM (e.g., no outbound Activity IDoc containing the UDF).
  4. Log into the FSM Mobile App (iOS) and open an Activity that should contain the UDF.

Expected:

  • The Screen Configuration rule should evaluate correctly, and the field should appear or validate according to the defined logic.

Actual:

  • The iOS mobile client does not evaluate the Screen Configuration rule.
  • The UDF behaves as if it does not exist in the Activity's data model.

Cause

  • FSM Mobile clients—especially iOS—require a complete and explicit field definition in the underlying FSM object model.
  • This field definition is only created when SAP ECC sends the UDF at least once in an outbound IDoc, even if the value is empty.
  • If ECC never sends the UDF structure (ID, data type, segment), the FSM Cloud does not generate the full object model.
  • As a result, mobile clients cannot recognize or evaluate Screen Configuration rules referring to that UDF.
  • UDF definitions and data types must be transferred to FSM via IDoc segments before they become part of the mobile data model.

Resolution

To ensure mobile clients recognize and evaluate UDF‑based Screen Config rules:

  1. Send the UDF at least once from ECC via an outbound Activity (or Equipment) IDoc, even if the value is empty.

    • The segment must contain the UDF ID, data type, and structure.
    • Empty values are acceptable; the structure itself must be present.
  2. In ECC, verify the UDF is included under the correct Cloud Message Type (e.g., ACTIVITY) in the connector configuration.

  3. After the initial IDoc is sent and processed by FSM:

    • Synchronize the FSM iOS Mobile App.
    • The UDF is now part of the mobile object model.
    • Screen Configuration visibility or validation rules start functioning as expected.

  4. Optional:
    Because mobile behavior differs by platform, validate the result on Android/Windows if necessary.
