Symptom

On having bitmask 128 set for parameter gw/reg_no_conn_info, proxy info ACL should be maintained in the destination gateway. However, when performing a Connection Test on an RFC destination related to a registered program in the transaction SM59, the following output is raised: 

Connection Error

Error when opening an RFC connection (CPIC-CALL: 'ThSAPOCMINIT', communication

ERROR: You cannot use the gateway as a proxy for this connection
...

RETURN CODE: 750

Even with proxy info ACL file maintained, mentioned error message persists. 

Reproducing the issue with gateway trace in the destination side set with trace level 2, following message is written for issued connection test:

GwIPrxyCheck: no matching entry found (SOURCE=<client hostname>(<client IP address>):33<instance number> -> DEST=%%RFCSERVER%%(0.0.0.0):<RFC destination name>) 

Cause

If this connection should be allowed, a proxy info file must be created or updated at the RFC Gateway where the program is registered, including a rule that allow this connection.

As it shows in the gateway trace, the destination IP address checked is "0.0.0.0". This means that in the destination side the "localhost" hostname interface is being used to identify itself. So, even with proxy info file maintained with "DEST=" set to the destination external IP address, the proxy info check fails because it compares its localhost interface with the content of "DEST=" field.

Resolution

To resolve this issue "localhost" hostname can be used in "DEST=" field of its respective proxy info file rule. For example:



P SOURCE=<client-host> DEST=<destination server>,localhost
