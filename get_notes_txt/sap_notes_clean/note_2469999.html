Symptom

During the execution of XCLA CL_XCLA_ACDOCA you encounter one of the following symptoms which are displayed in the log file.

A. Errors leading to a RED log entry or a runtime error, which always need a reprocessing/re-execution of the XCLA:

1. Message FINS_ACDOC_DB070: Package too large (read SAP note 2469999)

2. The XCLA terminates with an out-of-memory error

3. The system reports a DBSQL_TABLE_UNKNOWN error for VXCLAACDOCA

4. Message PU475: Error in parallel XCLA execution

5. Message PU470: Execution of XCLA CL_XCLA_ACDOCA canceled due to error

6. Message PU471: Execution of package xxx of XCLA CL_XCLA_ACDOCA canceled

7. Message SHDB_PFW302: Dispatcher Timeout: 'PackSel xxx' waited 900 sec for free WP

8. Message UPGBA028: Database object does not exist on the database

9. Message PU461: "CL_XCLA_ACDOCA" was already executed in a previous step

B. Info messages which should be checked individually and normally do not need any reprocessing/re-execution:

  • FINS_ACDOC_DB067: Project not found: "100"/"CS-00005"
  • FINS_ACDOC_DB059: Not all project IDs have been assigned (read SAP note 2469999))
  • FINS_ACDOC_DB061: Not all partner project IDs have been assigned (read SAP note 2469999))
  • FINS_ACDOC_DB068: WBS element not found: 100/PZ016697122
  • FINS_ACDOC_DB060: Not all partner WBS elements have been assigned (read SAP note 2469999)
  • FINS_ACDOC_DB055: Asset class not updated for "100/0001/20094-2"
  • FINS_ACDOC_DB069: Not all asset classes have been assigned (read SAP note 2469999)

Other Terms

CL_XCLA_ACDOCA, DC00000000, FINS_ACDOC_DB 070, BELNR, DOCLN, DC00000000, VXCLAACDOCA, DBSQL_TABLE_UNKNOWN, FINS_ACDOC_DB 055, FINS_ACDOC_DB069, FINS_ACDOC_DB 069, FINS_ACDOC_DB 076

Reason and Prerequisites

In most cases the root cause for symptoms 1 -3 is an inefficient packaging by standard packaging field ACDOCA-BELNR which leads to too large packages for document numbers like e.g. DC00000000 (from CO migration) which can have several million lines.

Solution

What does CL_XCLA_ACDOCA do?

CL_XCLA_ACDOCA fills fields which have been newly added to ACDOCA in a certain release:

  • S4CORE 102
    • Fill PS_PSP_PNR (internal WBS element) from PS_POSID (external WBS element) or OBJNR (CO object number)
    • Fill CO_MEGBTR/CO_MEFBTR/CO_MEINH from VMSL/VMFSL/RVUNIT
  • S4CORE 103
    • Fill ANLKL/KTOGR from ANLA-ANLKL/ANLA-KTOGR
    • Fill PANL1/PANL2 from related ACDOCA-ANLN1/ANLN2 entries
    • Fill PS_PRJ_PNR (internal Project) from PS_PSPID (external project)
    • PPS_PSP_PNR (internal partner WBS element) from PPS_POSID (external partner WBS element) or PAROB1 (CO partner object number)
    • Fill PPS_PRJ_PNR (internal partner project) from PPS_PSPID (external partner project)

During this massively parallelized processing, errors or problems may occur which must be checked in the XCLA log file (which can be found via SE09 in the transport log of the request containing the XCLA CL_XCLA_ACDOCA or via execution of the program RCTS_DISP_AIM_LOG with parameter OBJNAME = CL_XCLA_ACDOCA). Important info: If the XCLA log can't be found, please check this also on all other application servers (via transaction SM51)!

Please note that only XCLA executions which lead to a RED log entry in the log file (pointing to a severe error) or a runtime error always need a reprocessing/re-execution of the XCLA.

If there has been no red log entry or runtime error, there is most probably no need for reprocessing/re-execution of the XCLA.

Another quite good method to find out whether there were errors during the execution (which need reprocessing/re-execution of the XCLA) is to search for the string 'error' in the log.

A. ERRORS leading to a RED log entry or a runtime error which needs reprocessing/re-execution:


1. Message FINS_ACDOC_DB070 'Package too large (read SAP note 2469999)':

1.1 Please implement the following notes in the TARGET release in case they are not contained in your system yet:

- 2586975: Out Of Memory Error / package too large / timeout for CL_XCLA_ACDOCA (only relevant for S4CORE102)

- 2667721 - CL_XCLA_ACDOCA with new packaging (only relevant for S4CORE102)

- 2995926 - XCLA CL_XCLA_ACDOCA: Message FINS_ACDOC_DB070: Package too large  (relevant for S4CORE102 - S4CORE105 SP00)

- 2949279 - PFW: Workaround for failed AUTOCOMMIT OFF (only relevant for SAP_BASIS754 - SAP_BASIS755)

- 2990959 - PFW: Improve Collector SQL-Performance (only relevant for SAP_BASIS754 - SAP_BASIS755)

Info:

If possible, please implement these notes in the target release/system before the XCLA CL_XCLA_ACDOCA is executed (by automatically incorporating these notes implementations e.g. from previous test systems into the SUM procedure) to avoid problems during the upgrade.

Some info about implementing notes before/during the system update can be found in note 2207944: How to implement a SAP note during System update using SUM (ABAP only)

In case of any questions/problems regarding the note implementation during the system update please contact SAP support on component BC-UPG-TLS-TLA.

1.2 Then do the following depending on the target release:

1.2.1 Releases S4CORE102 - S4CORE103:

- Make sure that the only remaining error (which can not be ignored) is

2PEPU471 Execution of package "?" of XCLA "CL_XCLA_ACDOCA" canceled

A2PEFINS_ACDOC_DB 070 Package too large (read SAP note 2469999)

Note that as long as there are any other errors (which can not be ignored) in addition, the following actions may cause additional errors as the processing of outstanding regular documents turns into an inefficient direction. Typical errors which can be normally be ignored are e.g. FINS_ACDOC_DB 055 Asset class not updated for "100/0001/*-*" or FINS_ACDOC_DB 069 Not all asset classes have been assigned (read SAP note 2469999 - more info see below.

- Add the following entry in T811FLAGS via SE16 in all clients where data for ACDOCA exist:

TAB: CL_XCLA_ACDOCA

FIELD: PACK_FIELD

VALMIN: DOCLN

By creating this entry in T811FLAGS, you force the system in the second run to build packages on ACDOCA-DOCLN (instead of standard field ACDOCA-BELNR) which will process the ACDOCA data which could not be processed in the first run (where the standard packaging by ACDOCA-BELNR was done).

- Repeat the failed SUM phase. This will process the very large documents besides the "normal" ones.

- If the alternative packaging by ACDOCA-DOCLN should not help, please change the entry in T811FLAGS via SE16 in all clients where data for ACDOCA exist:

TAB: CL_XCLA_ACDOCA

FIELD: PACK_FIELD

VALMIN: TIMESTAMP

Via this entry in T811FLAGS, you force the system in the third run to build packages on ACDOCA-TIMESTAMP which will process the ACDOCA data which could not be processed in the previous runs.

- Repeat the failed SUM phase. This will process the very large documents which could not be processed in the previous runs.

- If the alternative packaging by ACDOCA-TIMESTAMP should not help either, please create the following entry in T811FLAGS via SE16 in all clients where data for ACDOCA exist:

TAB: CL_XCLA_ACDOCA

FIELD: PACK_MAX_SIZE

VALMIN: 100000000

Via this entry in T811FLAGS, the system allows packages up to 100 million records to process the ACDOCA data which could not be processed in the previous runs.

- Repeat the failed SUM phase. This will process the very large remaining documents which could not be processed in the previous runs.

- Once everything is ok, please remove the T811FLAGS entries mentioned above again, so that in a future upgrade to a higher release the standard packaging by ACDOCA-BELNR can be done as a first step.

1.2.2 Release S4CORE104:

By newly implementing note 2995926 the problem might already be solved by starting the XCLA CL_XCLA_ACDOCA once more (especially in case you have a huge ACDOCA which is partitioned, see e.g. ST04).

So please simply repeat the failed SUM phase.

If the problem should not be solved, please proceed as described under 1.2.1 Releases S4CORE102 - S4CORE103.

1.2.3 Release S4CORE105 and higher:

By newly implementing note 2995926 the problem should already be solved by starting the XCLA CL_XCLA_ACDOCA once more, because the changes in S4CORE105 will lead to a much better (multidimensional) packaging.

So please simply repeat the failed SUM phase.

see solution 1.

4. Message PU475: Error in parallel XCLA execution

This message only gives you the general info that there was an error. Please look for messages mentioned in this note before or after this message which might give you the real root cause for the error.

Examples of messages before or after message PU475: Error in parallel XCLA execution:

a)   PU519: XCLA 71.131: 71130 rows processed,   rows modified on DB (rollback)

and some lines later:

UPGBA031: Exception raised at CX_SHDB_PFW_EXCEPTION=========CP, CX_SHDB_PFW_EXCEPTION=========CM002

These error messages are normally displayed because there are duplicate keys in ACDOCA (for key fields RCLNT, RLDNR, RBUKRS, GJAHR, BELNR, DOCLN) what should never be the case.

To check and correct this problem, please do the following:

Execute program FINS_XCLA_ACDOCA_DUPLICATES in 'Check' mode to check for ACDOCA duplicates.

Possible root causes for duplicates are e.g.

  • A previous XCLA run of a previous upgrade created corrupt records due to a bug in the XCLA framework - more info can be found in note 2661437
  • The ACDOCA compression run created duplicates due to an error described in note 3232266 (records can be identified via ACDOCA-SGTXT which contains 'Compression*')
  • Corrupt data have been imported/copied from another system/client

Possible workaround in case program FINS_XCLA_ACDOCA_DUPLICATES should fail due to a memory error (because of a very high data volume in ACDOCA):

You can also use the following SQL statement (by replacing 'xxxx' for each GJAHR) to check for duplicates:

SELECT * FROM ( SELECT RCLNT , RLDNR , RBUKRS, GJAHR ,BELNR, DOCLN, COUNT(*) AS COUNT FROM ACDOCA GROUP BY RCLNT, RLDNR, RBUKRS, GJAHR, BELNR, DOCLN )
WHERE GJAHR = 'xxxx' and COUNT <> '1'

If corrupt records are found by the check program FINS_XCLA_ACDOCA_DUPLICATES, please execute it in 'Repair' mode in background (it may run for a while) to delete the corrupt records.

Once the repair run has finished and deleted the corrupt records, please continue with the upgrade and restart XCLA CL_XCLA_ACDOCA again.

If the XCLA fails again, there must still be ACDOCA duplicates in the system, which are normally caused by erroneous customer or correction programs or incorrect client/system copies/setup.

Since in the SUM phase it is not so easy to find and correct these duplicates, we recommend to skip these errors for XCLA CL_XCLA_ACDOCA and proceed with the upgrade.

In case you need the data to be corrected after the successful upgrade (please check whether this is really necessary for this system) please execute program FINS_ACDOC_UTIL_BELNR_UNIQUE in each client and come back to us with the results.

We will then try to correct the duplicates according pilot note 2666674 and after successful correction restart XCLA CL_XCLA_ACDOCA manually as described below.

In any case you should execute check programs FINS_XCLA_ACDOCA_DUPLICATES in 'Check' mode and FINS_ACDOC_UTIL_BELNR_UNIQUE (must be executed in each client) before upgrading your next system (with productive data) to check whether there are also duplicates which need to be corrected before the upgrade.

7. Message SHDB_PFW302: Dispatcher Timeout: 'PackSel xxx' waited 900 sec for free WP

This error occurs if for one specific PFW package (PFW: HANA Performance FrameWork) no free work process could be found within the given time.

Please check whether there are DB triggers (e.g. for SLT) defined on ACDOCA causing performance problems. If yes, please disable them (see also section 'Further important recommendations' below).

Please reprocess/re-execute the XCLA.

If the problem should occur again, there seems to be a general performance problem with HANA/PFW.

In this case the root cause for the performance problems must be analyzed by component BC-DB-HDB-PFW (or HAN-DB-ENG).

8. Message UPGBA028: Database object does not exist on the database

This error occurs in case of problems with the enqueue server for which the root cause could not be found yet.

Please reprocess/re-execute the XCLA.

If the problem should occur again, the issue must be analyzed by component BC-DB-HDB-PFW.

9. Message PU461: "CL_XCLA_ACDOCA" was already executed in a previous step

If this error message is erroneously displayed (because actually the XCLA was not executed yet), please implement PFW note 3532300.

B. INFO messages which should be checked individually and normally do not need any reprocessing/re-execution:

The following info messages are displayed if some specific ACDOCA fields could not be filled, in most cases due to missing master data. Normally this is no severe problem since mostly those ACDOCA fields with missing master data are e.g. outdated data and not needed anymore.

You can check the master data for these ACDOCA fields at least randomly to make sure that there is no major master data problem in your system.

In the worst case those ACDOCA records concerned may not appear in reporting.

  • FINS_ACDOC_DB067: Project not found: "100"/"CS-00005"

This info message is displayed when in project master table PROJ in client "100" for the (external) field PROJ-PSPID "CS-00005" no corresponding (internal) field PROJ-PSPNR could be found and therefore the corresponding (internal) ACDOCA fields PS_PRJ_PNR and/or PPS_PRJ_PNR (see next 2 messages) could not be filled. Most likely the project has been removed or renamed.

  • FINS_ACDOC_DB059: Not all project IDs have been assigned (read SAP note 2469999)

Due to a missing entry in project master table PROJ (see message FINS_ACDOC_DB067 above), the (internal) field ACDOCA-PS_PRJ_PNR could not be filled from the (external) field ACDOCA-PS_PSPID.

  • FINS_ACDOC_DB061: Not all partner project IDs have been assigned (read SAP note 2469999))

Due to a missing entry in project master table PROJ (see message FINS_ACDOC_DB067 above), the (internal) field ACDOCA-PPS_PRJ_PNR could not be filled from the (external) field ACDOCA-PPS_PSPID.

  • FINS_ACDOC_DB068: WBS element not found: "100"/"PZ016697122"

If note 3055196 is not implemented yet in your system, please implement it and restart/re-execute the XCLA, then the message should not be displayed anymore.

If it still should be displayed:

This info message is displayed when in project master table PRPS in client "100" for the (external) field PRPS-POSID "PZ016697122" no corresponding (internal) field PRPS-PSPNR could be found and therefore the corresponding (internal) field ACDOCA-PPS_PSP_PNR (see next message) could not be filled from the (external) field ACDOCA-PPS_POSID. Most likely the WBS element has been removed or renamed.

  • FINS_ACDOC_DB060: Not all partner WBS elements have been assigned (read SAP note 2469999))

If note 3055196 is not implemented yet in your system, please implement it and restart/re-execute the XCLA, then the message should not be displayed anymore.

If it still should be displayed:

Due to a missing entry in project master table PRPS (see message FINS_ACDOC_DB068 above), the (internal) field ACDOCA-PPS_PSP_PNR could not be filled from the (external) field ACDOCA-PPS_POSID.

  • Message FINS_ACDOC_DB055: Asset class not updated for "100/0001/20094-2" (and FINS_ACDOC_DB069: Not all asset classes have been assigned (read SAP note 2469999)) 

This means that in table ACDOCA there are records in client 100 for ACDOCA-RBUKRS "0001", ACDOCA-ANLN1 "20094" and ACDOCA-ANLN2 "2" where no corresponding record in master table ANLA exists and therefore field ACDOCA-ANLKL could not be filled.

Please note that messages like FINS_ACDOC_DB055: Asset class not updated for "100/0001/*-0" concerning migrated records with ACDOCA-ANLN1 = '*' can be ignored because these records are absolutely no problem - note 3052241 prevents the system from displaying these unnecessary messages.

If you want to have the master data for these messages checked in detail, please contact the following components:

  • Projects, WBS elements: Component PS-ST-OPR
  • Assets: Component FI-AA-AA

C. Further important recommendations:

  • Please read SAP Note 2351294 - S/4HANA System Conversion / Upgrade: Measures to reduce technical downtime
  • Try to repartition your ACDOCA as soon as an opportunity arises, especially before upcoming migrations/upgrades - more info can be found in note 2289491.

  • Please check for basis releases >= SAP_BASIS 755 whether the coding changes of standard note 3227915 are contained in your system, so that partition striping is active for all PFW applications. If not, please implement them to activate partition striping.

  • To avoid massive performance problems, please check whether there are DB triggers (e.g. for SLT or customer defined triggers) defined on ACDOCA (via SQL command 'SELECT * FROM TRIGGERS WHERE SUBJECT_TABLE_NAME = 'ACDOCA'), which may lead ta longer runtime when updating/modifying ACDOCA records. If yes, please disable these DB triggers (see e.g. note 2755741).

  • In case there should be problems or errors in the SUM phase with XCLA CL_XCLA_ACDOCA, you can also start program RCTSXCLA for CL_XCLA_ACDOCA via transaction SE38 manually in the background after the relevant notes above have been implemented and the actions mentioned above have been performed. Please note that in this case the XCLA log file name to be analyzed can then be found at the very end of the batch job log, e.g. something like e.g. /usr/sap/trans_TST_S4H/ECC/log/XCLA_CL_XCLA_ACDOCA.TST. The content of this XCLA log file name can be displayed with program RSPUTPRT by copying the log file name to the selection parameter 'Full File Name (w/ Directory)'. Please note that the log file contains the logs of all RCTSXCLA executions - therefore please make sure that you analyze only the actual log entries and not e.g. the logs with the problems of the previous executions which have already been solved in the meantime. To analyze only the correct relevant log entries please expand the log file between the following log entries on the first display level which contain the date and time in concatenated form:
    • Parallel execution of XCLA conversions started: 20210414160118
    • Parallel execution of XCLA conversions ended: 20210415012903
  • By starting RCTSXCLA for CL_XCLA_ACDOCA you can "validate", whether the notes/actions mentioned above have helped. Although RCTSXCLA for CL_XCLA_ACDOCA performs database updates on ACDOCA, the failed SUM phase for XCLA CL_XCLA_ACDOCA should be repeated which will be much faster since the ACDOCA data have already been processed before by RCTSXCLA for CL_XCLA_ACDOCA.
  • In case you skipped the errors of CL_XCLA_ACDOCA during the SUM phase, you will have to start program RCTSXCLA for CL_XCLA_ACDOCA via transaction SE38 manually in the background after the upgrade has finished successfully. Please note that in this case the XCLA log file name to be analyzed can be found at the very end of the batch job log, e.g. something like e.g. /usr/sap/trans_TST_S4H/ECC/log/XCLA_CL_XCLA_ACDOCA.TST. Before starting program RCTSXCLA for CL_XCLA_ACDOCA manually after the upgrade, you must create the following uncritical entry in table T811FLAGS via transaction SE16 to make the XCLA CL_XCLA_ACDOCA "think" that the current release is still the original source release (e.g. 102), otherwise the XCLA CL_XCLA_ACDOCA will do nothing:
    • TAB        XCLA_ACDOCA
    • FIELD     SOURCE_RELEASE
    • VALMIN   102 (your original source release)

Without this entry, the XCLA will do nothing - in this case there will be the following messages in the log:

    • SHDB_PFW 507 Split of DB object "VXCLAACDOCA" results no packages
    • SHDB_PFW 503 PackageStore build with "0" Packages from "TableSplitter"
  • When starting RCTSXCLA for CL_XCLA_ACDOCA via transaction SE38 manually please make sure that the user has enough authorizations (e.g. for authorization object S_DEVELOP) for DDIC and DB tables activation, because during the execution of CL_XCLA_ACDOCA also DDIC views are generated on the fly.

  • CL_XCLA_ACDOCA can be executed multiple times as often as needed

  • With every new execution the XCLA only processes those ACDOCA records which have not been successfully processed by the previous executions

  • Please delete the above T811FLAGS entry again after all ACDOCA records have been successfully processed

D. Analysis of unconverted ACDOCA records  

The following SQL statement (which can be executed in the SQL editor in transaction DB02 or ST04) gives the number of unconverted ACDOCA records:

select count ( * ) from acdoca where

( ( ( ( PS_PSPID <> '' AND  PS_PRJ_PNR = '00000000' )

OR ( ( PPS_POSID <> '' AND PPS_PSP_PNR = '00000000' )

OR ( PPS_PSPID <> '' AND PPS_PRJ_PNR = '00000000' ) ) )

OR ( ( ANLN1 <> '' AND ANLKL = '' )

OR ( MIG_SOURCE = '' AND (ANLN1 <> '' AND ( PANL1 = '' AND ( BSTAT = '' AND ( AWTYP <> 'AS91' AND (VORGN = 'AUUA' OR VORGN = 'AUUZ' OR VORGN = 'UMAB' OR VORGN = 'UMZU' OR VORGN = 'UMAI' OR VORGN = 'UMZI' ) ) ) ) ) ) ) )

OR ( ( PS_POSID <> '' AND PS_PSP_PNR = '00000000' )

OR ( ( OBJNR <> '' AND RVUNIT <> '' ) AND CO_MEINH = '' AND BTTYPE <> 'EBDP' and BTTYPE <> 'OSDP' and KTOSL <> 'BSX' ) ) )

The following SQL statement gives some detailed information about the unconverted ACDOCA records:

select

rldnr, rbukrs, gjahr, belnr, docln, PS_POSID, PS_PSP_PNR, OBJNR, vmsl, vmfsl, RVUNIT, co_megbtr, co_mefbtr, CO_MEINH, PS_PSPID, PS_PRJ_PNR, PPS_POSID, PPS_PSP_PNR, PPS_PSPID, PPS_PRJ_PNR,

ANLN1, ANLN2, ANLKL, KTOGR, PANL1, PANL2, fiscyearper, bttype, BSTAT, AWTYP, VORGN, MIG_SOURCE, TIMESTAMP

from acdoca where

( ( ( ( PS_PSPID <> '' AND PS_PRJ_PNR = '00000000' )

OR ( ( PPS_POSID <> '' AND PPS_PSP_PNR = '00000000' )

OR ( PPS_PSPID <> '' AND PPS_PRJ_PNR = '00000000' ) ) )

OR ( ( ANLN1 <> '' AND ANLKL = '' )

OR ( MIG_SOURCE = '' AND (ANLN1 <> '' AND ( PANL1 = '' AND ( BSTAT = '' AND ( AWTYP <> 'AS91' AND (VORGN = 'AUUA' OR VORGN = 'AUUZ' OR VORGN = 'UMAB' OR VORGN = 'UMZU' OR VORGN = 'UMAI' OR VORGN = 'UMZI' ) ) ) ) ) ) ) )

OR ( ( PS_POSID <> '' AND PS_PSP_PNR = '00000000' )

OR ( ( OBJNR <> '' AND RVUNIT <> '' ) AND CO_MEINH = '' AND BTTYPE <> 'EBDP' and BTTYPE <> 'OSDP' and KTOSL <> 'BSX' ) ) )

order by TIMESTAMP DESC
