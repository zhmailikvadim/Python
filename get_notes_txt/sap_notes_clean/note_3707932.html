Symptom

  1. The values of partner functions, like "Developer" and "Tester" in the search results for change documents don't match the values on the documents themselves.
  2. The values of the partner functions is correct when using document specific searches, like Normal Change searches, but not when using Change Document search

Environment

SAP Solution Manager 7.2

Reproducing the Issue

  1. Execute SM_CRM
  2. Perform a change document search
  3. Check the values for a partner function, like "Developer"
  4. Open the change document in question
  5. The name under "Developer" is different

Cause

The configuration of the business partner functions, or the fields is incorrect.

Resolution

Point 1:

It is not enough to change field labels in order to link partner functions. A link exists in the back end and the corresponding fields must be used to accurately represent the connection.

It must be defined in SM34 > AIC_SETTINGS2 > Choose your transaction > Specify Partner Function Display. 

Pay attention to the column "Order".

Here we see that for SMMJ the partner function "Developer" is given order number 01.

We can check in the search results using the "Show Configurable Areas" button and selecting the results section. Then click the button "Show Technical Details".

We can see the number at the end of the field name is "01", which matches the order defined in AIC_SETTINGS2. This is correct.

Next we can check the change document itself by clicking the "Show Configurable Areas" button again and choosing the "Details" section. Once the screen is open, select the field you wish to check, click "Show Field Properties" and "More" > "Show Technical Details".

We can see that here the field name also ends with "01". This is correct.

If the technical field name on the change document, or in the table for the search results does not have a number that matches the order number in AIC_SETTINGS2, it can cause a mismatch with the display of the data. 

Note: If you don't see the "Show Configurable Areas" button, navigate to "Personalize" -> "Personalize Settings" -> Check the box "Enable configuration mode"

Point 2:

When using the generic "Change Documents" search we first read all the transaction types and then we sort these process types into descending order so that custom transaction types (Z* and Y*) would come at the top (in general).

After that we read BPs from each process type until we have read 10. The top 10 BPs will be mapped with the column names of the search result list. So if you have several process types with different BPs with different order of display then you should have process type specific UI configuration for the search query view because that will filter accordingly all the BPs in the search query view.

When doing the search selecting a specific transaction type then we consider such process type and read those corresponding BPs and map with STRUCT./AICRM/BP_NO_01 to STRUCT./AICRM/BP_NO_10 (column fields of Search Result List).
