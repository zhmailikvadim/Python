Symptom

When performing an activity in the system, it dumps with both SQL_CAUGHT_RABAX and TSV_TNEW_PAGE_ALLOC_FAILED.

Environment

All ABAP-based SAP systems below:

  • SAP ERP
  • SAP NetWeaver
  • SAP S/4HANA onPremise/Cloud environments

Cause

The root cause is a massive internal table build-up inside the SADL query executor. This is not a generic memory shortage you can fix by tweaking profile parameters. The TSV_TNEW_PAGE_ALLOC_FAILED happens first; the SQL_CAUGHT_RABAX is just the wrapper because CL_SADL_SQL_EXECUTOR caught the ABAP memory error.

  • Check both the “Error analysis” and “How to correct” section:




    • For example, when checking the sections, the following values are observed:
      • Allocated rows: 35,113,021
      • Row width: 510
      • Session memory quota: 18000000000
      • As per KBA 2180736, a quick calculation to find out how much data is being consumed would be the following: Allocated rows × Row width ≈ Data Consumed
      • Using the values from the example, the calculation would be: 35,113,021× 510 ≈ 17,907,640,710 (17.9 GB)
      • This means that one internal table alone consumed almost the entire session quota (17.9 GB out of the 18 GB). That is why the allocation of the next rows failed. This is the classic "WP quota reached" scenario described in KBA 2180736.

  • Specifically, for SADL cases, it is needed to check which class/method calls the program CL_SADL_SQL_EXECUTOR > SELECT into @et_data. Checking the “Source Code Extract, you can find the SADL query:



    • The following variables from “Selected Variables” section can be checked, based on the above query:
      • <clauses>-LIMIT:
        In almost every modern ABAP SELECT (and especially in SADL/CDS/OData queries), LIMIT (or sometimes called UP TO n ROWS) controls how many rows maximum the database should return. LIMIT = 0 means no restriction. The database is allowed to return all matching rows (the entire result set).
      • <clauses>-OFFSET:
        OFFSET tells the database to skip the first N rows and start returning from row N+1. OFFSET = 0 means start from the very first row (no skipping).
      • <clauses>-ORDER_BY:
        ORDER_BY defines in which sequence the rows should be sorted before they are returned. When it is empty, it means that the database returns rows in whatever physical/index order it finds convenient (usually not sorted at all). This is important because some code paths in SADL behave differently when there is no ORDER_BY (especially regarding OFFSET + LIMIT logic).
      • <clauses>-WHERE:
        The WHERE condition determines which rows are selected from the database table/CDS view. To avoid such dumps, it is needed to ensure the WHERE condition is not empty or almost empty, otherwise, too many rows will be returned.

Resolution

  1. Go through the suggestions below:

    • Add proper paging (PACKAGE SIZE in the SELECT/SADL query):
      Instead of loading all rows in one go, the database returns data in smaller chunks (e.g., 5,000 ~ 20,000 rows per package). This ensures the internal table never grows beyond one package size at a time and the memory consumption stays constant and low (e.g. 5,000 rows × 510 ≈ 2.5 MB per iteration, instead of the 17.9 GB from our example addressed in the “Cause” section).
      • Let SADL handle the PACKAGE SIZE automatically, which can be obtained by ensuring the query clauses include a reasonable <clauses>-LIMIT; OR
      • Manually add PACKAGE SIZE in custom SELECT logic.
    • Use a stricter filter:
      Ensure you are using a strong WHERE clause so it does not return too many rows or an entire table at once. Adding mandatory filters reduces the result to thousands or fewer rows, this way the internal table will be small and the system will not receive memory dumps.
      • Mandatory filters (via UI or service design). On the example we are using in this KBA, a filter for the last 30 days could have been used in the WHERE clause, as this would have avoided too much data being retrieved.
    • Implement a limit in the OData service:
    • Even if the client forgets $top, the backend enforces a maximum number of rows (e.g., 5,000 or 10,000). This is a safety net against "select all" misuse and prevents memory dumps.

  2. If none of the above options are feasible or solve the issue, the class/method responsible for calling CL_SADL_SQL_EXECUTOR==========CP should analyze further.
    • To check the caller class/method, refer to the “Active Calls/Events” section on the dump. The class/method that comes before the SADL classes is the caller method.



    • On the above example, the method _GET_ENTITY_SET from class CL_APS_OM_GWS_RUNTIME (owned by BC-SRV-FP-OM application component) is the caller of SADL programs. The owners of this class/method must be involved. In order to identify the application component responsible for a certain class/method, refer to KBA 1887985.
