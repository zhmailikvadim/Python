Symptom

You may notice that Security Audit Log (SAL) files in your ABAP system are only available for a short time—usually about 2 days. After that, you can’t find older logs in SM20 or AL11.

 

Environment

Enterprise Cloud Services ( SAP's Private Cloud )

Reproducing the Issue

You are not able to get events in SM20 or do not find audit files older than certain period in AL11.

Cause

On older systems (installed before September 2025 or running older support packages), the audit logs were saved to the file system instead of the database.
Those systems had an automatic cleanup process that moved/deleted old SAL files from the folder $DIR_AUDIT.
 
The standard for cloud is to store SAL logs in a database.

Resolution

To ensure audit logs are kept longer, you need to switch the storage method from File System to Database.

You can do this in transaction RSAU_CONFIG.
For step‑by‑step guidance, see this document:
<https://d.dam.sap.com/a/t1W2psW/ABAP_SAL_Audit_loggingf_-_Private_Cloud.pdf?rc=10&inline=true&doi=SAP1271700>

Need Older Files Restored?

If you recently switched to database storage, SAP can restore your older audit files one time as an exception.
These restored files will be placed back into $DIR_AUDIT, and you may need extra file system space.

How to Use the Restored Logs

You have two choices:

  1. Read them directly

    • Open them using your usual SAL transactions, or
    • Use report RSAU_READ_LOG

  2. Load them into the database for long‑term retention

    • Use report RSAU_LOAD_FILES
    • Important: Fix known issues first by implementing SAP Notes 3644711, 3650449, 3658972, and 3413125
If you’re unsure or need assistance, simply contact your private cloud representative, who will guide you through the process.
