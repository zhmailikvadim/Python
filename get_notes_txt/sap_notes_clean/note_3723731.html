Symptom

This SAP Note provides enhancements to the existing Grouping Reason 'UKLE' that is required to be used by all customers that use the Multiple Employment (ME) Payroll solution in the United Kingdom (UK). The enhancements are optional until you experiences any issues with the existing Grouping Reason 'UKLE'.

The enhancements are as follows;

1) The 'UKLE' grouping values used by payroll areas (ABKRS) are generated at runtime (as they were previously), but are now stored in a database table (T5G_UKLE_GRPVL). The rules for defining the Grouping Value are as follows;

a) Payroll Areas that are not payroll relevant use an empty grouping value.

b) Payroll Areas that have no entry in T5G51 use an empty grouping value.

c) Any employees that are not live yet on the ME Payroll Solution (i.e. Feature 08MEL provides a Go Live date in the future) use an empty grouping value.

d) All other Payroll Areas are given an entry between 0001 and 9999, but all Payroll Areas that have the same Period Parameters, Tax District and Tax Reference will use the same grouping value.

Activating the Enhancements to Grouping Reason 'UKLE'

System Constants (T5GPBS_ALL)

ME  UKLE  01.01.1800   31.12.9999                X = 'UKLE' Enh. (Note 3723731)

Non-Productive Systems Only

User Parameter: HR_GB_ME_UKLE                  HRGB: 'UKLE' Enhancements (Note 3723731)

  • Set the user parameter  to X to test in a non-productive system (for just that user).
  • Set the system constant to X to test in a non-productive system (for all users).
  • Set the system constant to X to go live in your productive system (for all users).

Implementation

1) Activate the enhancements (see 'Activation' above)

2) Customise your system (see 'Limitations' below)

3) Check that your ME data is consistent by running the ME Check program; RPLMESG1.

4) Run the grouping value update programs; RPUGRPG0_GRPVL_0065, RPUGRPG0_GRPVL_0069 and RPUGRPG0_GRPVL_0121.

5) Run the Data Sharing Fix program; RPUFIXG0_UKLE

6) Check that Data Sharing is now consistent by running the International Data Sharing program in test mode; RPUFIXDS (taking care not to process infotype 0065, 0069, 0121)

7) Test your various business scenarios that use Data Sharing; Master Data, E-Filing, Payroll, Reporting.

Limitations (for GB Infotypes)

The enhancements to the Grouping Reason 'UKLE' are based on the following rules for 'Data Sharing' for GB infotypes (see list below), but from a "general" perspective, customers that previously used the Infotype Copy (i.e. T582A-CPRFN = ‘X’) should now use XMOL. But with the CAVEAT that, if the customer is implementing Employee Central Payroll (ECP), and the data is 'Replicated' from Employee Central (EC), then they should generally use XNON. If the customers were not previously using the Infotype Copy (i.e. T582A-CPRFN = ‘  ’) should now use XNON.
