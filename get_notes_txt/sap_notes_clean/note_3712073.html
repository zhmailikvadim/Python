Symptom

  • After performing a System Copy with PCA (Post Copy Automation), by the very nature of this automation, the entirety of the Fiori Catalog should be carried over, and why are we missing Fiori Custom Z BC Catalogs?
  • Is there a System Copy or Import Best Practice Guide?

Environment

SAP S/4HANA

Reproducing the Issue

  1. Execute System Copy with PCA:

    • Navigate to SAP Landscape Management (LaMa) or initiate system copy via SWPM
    • Select source system (S4P) and target system (S4Q)
    • Choose PCA-enabled system copy with standard task list (SAP_BASIS_COPY_REFRESH)
    • Complete system copy operation

  2. Verify Missing Catalogs:

    • Log into target system (S4Q) client 100
    • Access Fiori Launchpad via transaction /UI2/FLP or browser URL
    • Navigate to custom catalog groups (ZBC*)
    • Observe empty or missing tile content

  3. Backend Verification:

    • Execute transaction SE16 → Table /UI2/C_CAT
    • Search for entries with ID = ZBC* (catalog headers should exist)
    • Execute transaction SE16 → Table /UI2/C_CHIP
    • Search for chip definitions linked to ZBC* catalogs (entries missing or incomplete)

  4. Expected vs Actual Results:

    • Expected: Custom catalogs display with all tiles and functionality
    • Actual: Catalog structure exists but tiles are missing or non-functional

Cause

  • Cause 1:  The loss of custom Fiori catalogs (ZBC* or any Z Business Catalogs) after PCA system copy typically results from incomplete metadata preservation during client copy operations. While PCA handles most post-copy automation, Fiori catalog metadata spans both customizing (Type C) and application data (Type A) classifications, creating potential gaps when using SAP_CUST (SAP Custom) profiles. The issue manifests as "shell" catalogs existing in header tables but missing content definitions, rendering tiles non-functional. 
  • Cause 2:  Incomplete metadata preservation during PCA system copy operations. Fiori catalog metadata spans both customizing (Type C) and application data (Type A) classifications, creating gaps when using SAP_CUST (SAP Custom) profiles during client copy operations.
  • Cause 3 (Secondary Contributing Factors): 
    • Missing dependent objects (CHIPS and Page Fragments) that are classified as Type A (Application Data) but required for catalog functionality
    • Incomplete table coverage in PCA task lists for Fiori-specific metadata tables
    • System reference inconsistencies after logical system name conversion
  • Cause 4: Data Inconsistency Issues
    • Catalog "shells" preserved in header tables (/UI2/C_CAT) but content definitions missing from chip tables (/UI2/C_CHIP)
    • Cross-client table dependencies not properly handled during client copy operations
  • Cause 5: PCA Limitation Details:
    • Standard PCA task lists may not include all Fiori metadata tables in preservation whitelist
    • Cache synchronization timing issues between catalog metadata and application index
  • Cause 6: The loss of catalog content (tiles/CHIPs) while headers remain is often due to SAP classifications:
    • Table Type 'C' (Customizing): e.g., /UI2/C_CAT. Usually included in standard PCA/Client Copy profiles.
    • Table Type 'A' (Application): e.g., /UI2/C_CHIP. Often excluded by standard "SAP_CUST" profiles used by System Copy PCA.
    • The results:  The system copies the catalog 'names' but wipes the "content," leading to the empty shell symptoms.
  • Cause 7: Architecture Migration (S/4HANA 2023+): In S/4HANA 2023, the physical storage for Fiori catalog master data migrated from /UI2/C_* to the SUI_TM_MM_* (Software User Interface) tables. If a system refresh does not properly include new tables, or if the SAP_FIORI_FOUNDATION_S4 task list was not run in the target client to generate them, catalogs will appear missing or empty.
  • Cause 8: Depending on the Table Type, the System Copy either copies or ignores it.  See Table Matrix below.
    Delivery ClassDescriptionPCA / Client Copy (SAP_CUST) BehaviorImpact on Fiori Catalogs
    Type CCustomizingCopied automatically.Headers (e.g., SUI_TM_MM_CAT) move fine.
    Type WWorkbenchExcluded (usually moved via Transport).Critical metadata for app descriptors often sits here.
    Type LLocalExcluded (Client-specific/Temporary).Often used for generated caches; if missing, UI looks empty.
    Type G/ESAP ProtectedExcluded (Usually Read-only).Standard SAP delivered catalogs.

      Step 3: Follow the Mandatory Steps.  This is due to if you try to transport and the required tables does not exist, the transport will just ignore it.

      • Mandatory Action: SAP_FIORI_FOUNDATION_S4 Task List
          • Before importing Fiori Transport (SCC1N or STMS) into 2023+ system, you must run task list SAP_FIORI_FOUNDATION_S4 via STC01
            • Why: This generates the physical SUI_TM_MM_* tables.
            • Verification: Before transport, check SE11 or SE16 for table SUI_TM_MM_APP. If it is "inactive" or "Not Found", the transport will fail to populate the Fiori content.
            • The SUI_* table structures are generated and active in the DDIC
            • Special Note: The Fiori Foundation components are correctly registered for the 2023 version. This task list is frequently updated by SAP to handle naming conventions in the future FPS releases (faster than this SAP KBA document gets updated). 
            • Special Note 2: Running SAP_FIORI_FOUNDATION_S4 via STC01 is required because it initializes the Delivery Class L (Local) tables and structures. Without this, even if you transport Type W or Type C data, the Fiori framework cannot "read" it because the local persistence layer hasn't been generated in the new client.

      • System Architecture Verification

        • Check if your system is using embedded Fiori deployment (standard in 2023)

      Step 4: Review the Source and Target Tables

      • Table Discovery Process

        • Use SE16 with search pattern: /UI2/PB* or SUI* (See mapping above).
        • The Fiori Launchpad Content uses a SUI framework (Software User Interface).  The data flow is as follows:
          • Example: SUI_TM_MM_CAT (Physical Storage Table) --> View X (Internal View) --> /UI2/FLP_CAT_DATA (CDS View Layer used by UI). 
            • Verification: To check if data actually exist in the target client, the team must query the Physical Storage Table (SUI_TM_MM_CAT or SUI_TM_MM_APP) . Querying the CDS layer may show stale data from the cache, even if the physical table is empty. 
            • In the example Table above: SUI_TM_MM_APP - It is of Type W (Workbench), please check the Cause 8 (above) to see how System Copy  typically handles W type of data.

      Step 5:  Review some of  SAP's Frequently Asked Questions (FAQ):

      • Q: Why do I see empty shell catalogs in SUI_TM_MM_CAT but no tiles?
        • A: The header data exists, but the tile content (SUI_TM_MM_APP) was likely excluded from the client copy because it is classified as Type W (Workbench). The team must import the missing content via Transport or redo the client copy with a profile that includes Type W data.
      • Q: I used PCA, why did it still fail?
        • A: PCA whitelists often do not include the new SUI_* tables by default. You must manually add these tables to the PCA preservation whitelist for 2023 refreshes.
      • Q: I populated SUI_TM_MM_CAT via transport, but I still see issues in Fiori Launchpad.
        • A: Verify that the Fiori Foundation was correctly initialized in the target client. Run task list: SAP_FIORI_FOUNDATION_S4 via transaction STC01 to ensure the SUI framework is functional. Check the "Display Log" section to review each status of the "Activate Fiori Foundation" steps to ensure the SUI_TM_MM_* tables were generated without errors. 
      • Q: Why did SCC1N succeed but my catalogs are still empty?
        • A: SCC1N successfully move the Type C (Customizing/Headers), but bypassed the Type A (Application/Tiles) or Type W (Workbench). You must ensure your transport includes the R3TR UIAD (App Descriptor) and R3TR UIAC (Catalogs) objects specifically. Use /UI2/FLP_TAB_BROWSER to confirm the content tables are populated.
        • Note: UIAD corresponds to the physical table SUI_TM_MM_APP. This will help users/Basis to bridge "Transport Object" to the "Database Table."

      Step 6:  Learn about what tools are available for the administrator/Basis team.

      Official Verification Tool:
      Use Transaction: /UI2/FLP_TAB_BROWSER (Overview of DB Tables). This is a SAP Advance tool used by SAP Backend team. This is a SAP overview for all tables persisting catalogs, groups, spaces, and pages. Use this to verify content existence before and after transport.



      Step 7:  Get ready to document the Source System VS the Target System and Follow the Process below.

      I) Process 1 (Prior to System Copy): Analysis of Table from originating System and Client:

      1. May need to do a full table analysis prior to System Copy (Analyzing the Fiori Important Tables):
        • SUI_TM_MM_APP - Launchpad App Descriptor Items (UIAD)
        • /UI2/PB_C_CHIP - Definitions for Tiles and Target Mappings
        • and others (per Table Matrix Above - check criticality)

          - Check all the tables for details and their Table Type (L, W, A, C and others - See legend above)
          - Execute transaction SE16 → Table SUI_TM_MM_CAT
          - Search for entries with CATALOG_ID = ZBC* (catalog headers should exist)
          - Execute transaction SE16 → Table SUI_TM_MM_APP
          - Search for chip definitions linked to ZBC* catalogs (entries missing or incomplete)
          - Verify chip instances and their associations

      2. May try the Transaction code SOBJ (Pre-Transport Audit):
        • The goal of this audit is to determine the Delivery class and Object Type of the Fiori metadata tables. This dictates how the SAP Change and Transport System (CTS) treats the data.
          1. Go to Transaction SOBJ.
          2. Click "Display" (Ignore the "not modifiable" warning as we are only auditing).
          3. Click the "Position" button and enter the table name found in /UI2/FLP_TAB_BROWSER (e.g., SUI_TM_MM_APP or WDY_CONF_USERT).
          4. Double-click the entry to view the Object Multi-Selection details.
        • Why look for these results?
          • The team needs to Audit the following:
          •  Delivery Class. 
            • If 'W' (Workbench): SCC1N will not move this data because it is considered a cross-client development object. It requires a Workbench Request and a full STMS import.
          • Category
            • If 'CUST': This is a standard Customizing. If 'APPL': This is Application Data (Type A) and is likely excluded by the SAP_CUST PCA Profile. 
          • Object Type 
            • Look for 'TABU' (Table Content). This confirms the transport is looking for specific rows in the database. 
          • Current Client
            • If this flag is uncollapsed, the data is client-dependent. If it is collapsed/grayed out, the data is cross-client and SCC1N will have no effect.
        • Example Problematic Scenario:
          • A user used an SCC1N from Client 100 to 200 something is missing?
            • The Conflict: The Catalog Header (SUI_TM_MM_CAT) is often Type C (Customizing), so SCC1N moves it.
            • The Failure: The actual Tile Content (SUI_TM_MM_APP) or Web Dynpro Config is often Type W (Workbench) or Type L (Local).
            • Conclusion: SCC1N is a Customizing Tool. When it encounters a Type W table during the import, it simply skips the data without throwing an error. This is why a user may see "Success" in SCC3 but "Empty" in the Fiori Launchpad. 
        • If the SOBJ audit confirms the table is Type W or Type L:
          • Do not use SCC1N.
          • Ensure the objects are captured in a Workbench Request.
          • Use Transaction STMS to "Import Request" into the target client.
          • For Type L Tables: After import, The user must run the "Precision Repair" tool -> /UI2/SYNC_PBC to activate local metadata in the target client (Ex: 200).
            • /UI2/SYNC_PBC definition:  A tool to perform individual item synchronization within the UI2 Cache Infrastructure.  It supports selective synchronization operations for specific Fiori Launchpad artifacts. Facilitates targeted cache invalidation for specific business objects or user-dependent data.  This is used when specific Fiori Launchpad content requires immediate synchronization OR for troubleshooting cache inconsistencies for individual items OR for targeted updates after configuration changes affecting single artifacts OR when bulk synchronization is not appropriate or efficient.
      3. May need to do an Extended Metadata Table analysis for the following Tables:
        • WDY_CONF_USERT (WebDynpro Personalization)
        • /UI2/VC_CAT_AL (Catalog Aliases)

        • /UI2/INTEROP (Interoperability Settings)
        • /UI2/C_GROUPS (Group Definitions)
        • SUI_TM_MM_CAT
        • SUI_TM_MM_APP

      7: Target System Preparation:

      1. Backup current catalog configuration using Transaction code SCC8.
      2. Clear existing incomplete catalog entries.
      3. Ensure sufficient import queue capacity.

      8:  Follow the Import Process Workflow:

      1. Import transport in dependency order (IWSV -> WDCC -> UIAC -> UIAD)  <-- very important per Checklist Mapping Table above. 
      2. Execute post-import activities immediately:

        ---Sample Checklist-----

        1. Clear HTTP Server cache -  Tcode = SMICM → HTTP Plugin → Server Cache → Invalidate Globally 
        2. Clear Metadata Cache - Report/Tcode =  /IWFND/CACHE_CLEANUP 
        3. Clear Metadata Cache (backend) - Report/Tcode =  /IWBEP/CACHE_CLEANUP 
        4. Invalidate Global UI2 caches - Report/Tcode =  /UI2/INVALIDATE_GLOBAL_CACHES 
        5. Recalculate Application Index - Report/Tcode =  /UI5/APP_INDEX_CALCULATE 
        6. Synchronize Chip cache - Report/Tcode =  /UI2/CHIP_SYNCHRONIZE_CACHE 
        7. If needed: /UI2/PAGE_CACHE_RESET
        8. If needed: /UI2/PAGE_CACHE_CREATE (For Spaces/Pages)
        9. If needed: Invalidate Client Cache - Report/Tcode =  /UI2/INVALIDATE_CLIENT_CACHES 
        10. If needed: /UI2/PAGE_CACHE_SYNCHRONIZE
        11. Note: /UI2/SYNC_PBC - Surgical solution (If the 10 cache clearing above is limited, do not attempt to redo all 10, try this surgical key option). 

      Note:  The above step 1 to 10 comprehensive Cache Refresh Check

      Step 8:  Advanced Transport Identification (Recommended for Complex Landscape)
