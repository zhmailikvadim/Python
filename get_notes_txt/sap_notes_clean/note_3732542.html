Symptom

In S/4HANA Cloud Public Edition, the buffering is switched-on for number range object RV_BELEG 'Number Ranges for SD Documents' with type of buffering 'Parallel Buffering' and buffer size = 1.  Additionally, the Italian Solution is switched-on for billing documents.

This FAQ provide details regarding the reason and consequences of this behavior.

 “Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental.“

Environment

SAP S/4HANA Cloud Public Edition 

Resolution

This KBA explains the buffering for Billing RV_BELEG in S/4 HANA Cloud Public.

  1. Question: Why was the buffering switched-on in S/4HANA Cloud Public Edition? 

    Answer: Without buffering performance can decrease in case several postings are done in parallel: If these parallel posting processes need to draw the next sales document number from the same number range interval, delays can occur due to database locks on the entry in table NRIV: The first posting locks the entry until the posting is finished, the second posting has to wait until this lock was released. Only after that the second posting can draw its next number. Buffering was switched-on in standard for the following reasons:

    - Performance is a key feature in S/4HANA Cloud Public, the standard software must guarantee excellent performance in standard. In addition to buffering also the Italian Solution was switched-on in standard for billing documents: The Italian Solution guarantees a chronologically ascending sequence of document numbers despite buffering is active. For more details see questions [2] and [4].

  2. Question: How does parallel buffering work? 

    Answer: Parallel buffering is switched-on in S/4HANA Cloud Public. The 'trick' of buffering number ranges with type 'Parallel Buffering' is that each ABAP work process draws a number from table NRIV and can release the lock immediately again because it persists this number in a separate table NRIVSHADOW! No delays due to locks occur any more - and the buffer is persisted and cannot get lost when the server crashes or similar: With parallel buffering there are in principle not more gaps in number range intervals than without buffering. The buffer size is set = 1 because each ABAP work process reserves for his process the amount of numbers that are defined as buffer size: Example: Assume the buffer size would be = 100. Then you do two postings: The first posting gets number 1000100. But the next posting will most likely get number 1000200 which might be surprising. The gap of 100 occurs because the two postings happend on different ABAP work processes. Each process has reserved a package of 100 numbers in 'his' buffer. This kind of gaps is only temporary: If the next posting with the same technical ABAP work process ID takes place, then the number 1000101 will be drawn. To avoid such confusing temporary gaps, the buffer size was set = 1. Also in the SAP Help Portal there is a section for Parallel Buffering. Parallel buffering was invented in 2003 with note 599157.

  3. Question: How can buffer size = 1 increase the performance? 

    Answer: During postings the immediate release of the lock of table NRIV is the reason why even the minimal buffer size of 1 dramatically reduces the probability of delays caused by NRIV locks. This way the performance is improved. See also answer to the above question 'How does parallel buffering work'.

  4. Question: How does the Italian Solution work?

    Answer: With parallel buffering, the numbers in the buffer cannot get lost because this buffer is persisted in table NRIVSHADOW. This way gaps in numbers are avoided as far as possible. But without the Italian Solution there is a drawback: Assume a posting draws a number and then, before the posting finishes, an error occurs that leads to a ROLLBACK WORK: The number remains in the buffer - and will be reused later as soon as the next posting with the same ABAP work process ID happens. But since the user cannot influence which ABAP work process a posting uses, it can take quite some time until this number is reused, for example it can take several days. Postings on other ABAP work processes will likely occur in between. As a consequence the numbers (journal entry IDs) of the postings will be not chronologically ascending any more. This contradicts legal requirements in some countries, for example in Italy. To overcome this drawback, in order to fullfull also the requirement to have a chronologically ascending sequence the Italian Solution was developed: 
    The Italian Solution makes sure that in case of a ROLLBACK WORK the numbers in the buffer table NRIVSHADOW are not reused in a later posting. Instead, in case of a ROLLBACK WORK the entry of this table is moved from table NRIVSHADOW into table NRIV_DOCU: The next posting will draw a new number from table NRIV! This way the chronological ordering of numbers is achieved. But since the number that was moved to NRIV_DOCU is not used any more, a gap in the numbering now occurs: The table NRIV_DOCU serves as explanation for this gap in numbering. The Italian Solution is described in detail in note 840901.

  5. Question: In my country it is legally required to have gapless document numbers in SD billing. How does this fit to buffering?

    Answer: Gaps in numbering can always occur, even if buffering is switched-off: Gaps in numbering can occur in case the update process.
    With parallel buffering and Italian Solution switched-on some additional gaps can occur in case ROLLBACK WORK statements happen which is regarded as a seldom event.
     
  6. Question: Does the Italian Solution guarantee sequentially ascending numbering of document numbers?

    Answer: The note 840901 mentions that with the Italian Solution deviations of the ascending ordering can occur on a very small timescale (typically one second) for example in case there are parallel posting processes running. That's why the Italian Solution just claims to provide a pseudo-sequential numbering. Such deviations of the ascending ordering do not violate the legal regulations for example in Italy as long as it is guaranteed that these deviations do not occur at the border between two days: Within the same day non-ascending numbering is allowed: As a consequence in Italy for example you must make sure that long running posting jobs will finish before 23:59 o'clock.

  7. Question: How to check number gaps related information in the system to meet audit requirements?

    Answer:  It's possible to identify these number range gaps by performing the following steps in the system:

    1. Open the configuration activity: Define Number Ranges for Billing Documents (SSCUI ID 103284).
    2. Enter the number range prefix you want to check. (Leave this field blank if the target numbers have no prefix.) Then choose the Display Interval button.

                       

          3. From the menu, go to Interval → Non-Assigned Numbers.
                       
          4. The results will show the non-assigned numbers, along with the creation user name and timestamp. The results can be exported to Excel. And you can also use the “Buffer From” number as a filter to view the required data.
