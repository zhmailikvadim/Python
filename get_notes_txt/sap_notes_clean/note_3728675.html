Symptom

An error occurs when a user approves multiple service sheets from SAP Ariba Buying solution, resulting in the message: "User already processing Purchase Order "


Environment

SAP Ariba Buying
SAP S4Hana
SAP ECC

Cause

The error occurs when multiple Service Entry Sheets (SES) with the same purchase order are sent for processing, leading to document lock in the ERP. 

Resolution

In order to resolve this issue, please perform the below steps:

SES parameters "ENTRYSHEET_CREATE" & "ENABLE_SES_ITERATION"  need to be enabled under SPRO as per this documentation Maintaining Parameters for Service Entry Sheet | SAP Help Portal . Both of these parameters needs to be set carefully so that all incoming Service Entry Sheets (SES) can be processed within the available time window and avoid the locking mechanism in SAP ERP.

To explain this in a simple way, let’s take an example. If the timeout is set to 2 minutes and there are 5 SES documents, and each one takes about 30 seconds to process, the system will handle them one after another:

1st SES → 30 seconds
2nd SES → 60 seconds
3rd SES → 90 seconds
4th SES → 120 seconds

At this point, the 2-minute limit is reached. So when the 5th SES tries to process, it may fail with an error like:

“User already processing Purchase Order” or a "Timeout Error" 

This happens because of how SAP handles data consistency. When an SES is being processed, the system places a lock on the Purchase Order to prevent parallel updates. Since all SES are linked to the same Purchase Order (PO), they cannot be processed at the same time. If too many requests are queued and the timeout is exceeded, the later ones may run into this lock and fail.

What this means in practice?

1. SES processing is sequential per Purchase Order, not parallel
2. The timeout should be set based on:
3. How long one SES takes to process
4. How many SES are expected at once
5. Overall system performance

If there are many SES approvals happening together, lock conflicts and delays are expected

What can be done?

Increase the timeout so it comfortably covers the total processing time
Improve system performance so each SES is processed faster
Make sure related parameters (like SES iteration settings) are properly tuned for higher volumes

Conclusion :-

This is expected system behavior and can vary from one system to another. The right setup usually comes from a bit of testing—balancing timeout values and system performance so that SES processing completes smoothly without running into locking issues.
