Symptom

The Data Obsoletion feature in SAP FSM is designed only to flag outdated or completed objects as obsolete so they are excluded from mobile synchronization.

Key points:

  • Obsoleted objects remain stored in the company database.
  • They remain visible in web applications and APIs.
  • Obsoletion does not delete data and therefore does not free up database storage.

To reduce the actual database size, Data Retention (soft/hard deletion) must be used instead of or in addition to obsoletion.

Environment

SAP Field Service Management

Reproducing the Issue

  1. Log in to the FSM Admin Portal
  2. Select a Company
  3. Navigate to Company Settings
  4. Search for the SAP.FSM.Mobile.Sync.DaysSinceLastChangeToMarkObsolete company setting
  5. Add or edit the setting, enter the desired number of days, and save the change.
  6. Allow the scheduled obsoletion job to run
  7. Review the overall database size in monitoring or admin tools

Actual Result:
Database size does not decrease because objects marked as obsolete are still stored in the cloud database

Expected Result:
Users may expect a reduction in database size, but Data Obsoletion only affects mobile synchronization and does not delete any data

Cause

Data Obsoletion in SAP Field Service Management only marks objects as obsolete so they are excluded from mobile synchronization. It does not delete objects from the company database and does not reduce stored data volume. Objects marked obsolete remain in the cloud database and continue to appear in web applications and other endpoints, because obsoletion affects only mobile sync behavior and not data storage.

To actually remove data and reduce database size, Retention Policies (soft or hard deletion) must be configured. Soft deletion only hides data from applications but still keeps it stored, while hard deletion permanently removes it.

Resolution

To reduce data volume and ensure that outdated or irrelevant objects are actually removed from the system, configure Data Retention instead of relying on Data Obsoletion.

  1. Configure Retention Policies under Admin → Company → Data Retention to define when objects should be soft deleted or hard deleted.

  2. Soft‑deleted objects are hidden in applications but still stored in the database.
    Hard‑deleted objects are permanently removed, which is required to reduce database size.

  3. Use CoreSQL / Query API to identify large object types, attachments, or high-volume tables that should be targeted with retention rules (e.g SELECT COUNT(a) FROM Attachment a;) This helps ensure the system stays within recommended data volume limits.

  4. Review attachment data, as attachments often represent a large portion of storage.
    Attachments are automatically deleted when their parent object is deleted via retention.

  5. Clarify functional expectations: Data Obsoletion is intended only to reduce mobile synchronization load, not to delete data or reduce overall database size.
    Objects marked as obsolete remain stored in the cloud and remain visible in web apps and APIs.

  6. Optional: Adjust synchronization rules to reduce the amount of mobile data being downloaded, which can complement but not replace retention‑based cleanup.
