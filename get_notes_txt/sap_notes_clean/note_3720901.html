Symptom

This article covers the required migration steps for SAP Advanced Workflow on Hyperscalers SFTP service. To provide with a more secure and reliable platform, we are moving the service to a new SFTP solution.  

This upgrade provides specific security enhancements, including:  

  • Enhanced file verification and integrity checks 
  • Support for MDC signatures on PGP-encrypted files 
  • Expanded compatibility with modern PGP and SSH key formats 

Please note that this transition will require modifications to your current implementation. Detailed instructions are provided in the resolution section of this KB.

Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental.

Environment

  • SAP Sales Performance Management
  • Advanced Workflow on Hyperscalers

Resolution

This guide provides the required steps to connect to the new SFTP service after your migration is complete. To ensure a smooth transition, this article explain the process into three phases: Planning, Execution, and Verification. 

Phase 1: Pre-Cutover Planning & Inventory

Before the migration date, it is critical to identify all systems and processes that interact with the current SFTP service. 

     1. Identify All Connecting Systems: 

    • Create an inventory of all applications, scripts, or third-party tools that send files to or retrieve files from the Incentive Management SFTP site. 

     2. Check for Special Cases (Critical): 

    • Workflow Scripts/Custom Parameters: Are there any Workflow scripts or Custom Parameters that reference the SFTP hostname? These will need to be updated.

Integration Folder: Check the SFTP directory for a folder named Integration. This folder is being deprecated, and it is needed to take manual action if it exists. After the migration, the folder and the content will be removed.

      3. Perform SFTP Site Housekeeping (Recommended):  

    • To ensure a faster and smoother migration experience, we highly recommend reviewing the files currently on your SFTP site and performing some light housekeeping.  
    • Action: Consider deleting any files that are no longer needed, such as old data load files that have been successfully processed, outdated extracts, or temporary test files. Reducing the number of files to be moved will directly contribute to a more efficient migration.  

Phase 2: Post-Migration Execution 

After the scheduled maintenance window for the environment (SBOX or PROD) is complete, perform the following actions. 

      4. Update Your System Connection Details: 

    • Using the inventory from Phase 1, update each system with the new connection details provided in the migration announcement.

Port: 9093

Username: Customer tenant id/moniker adding the SAP product as a suffix. For example, if your tenant id is “mockTenant” then the username will be “mockTenant_XDL” for XDL, “mockTenant_DSP” for Datasphere, and “mockTenant_WFL” for Workflow. 

SSH Keys (User Authentication): Existing keys will be migrated. For new SSH keys, please contact our global Support Team via the SAP for Me portal to push keys into SFTP. 
(The new SFTP solution supports modern key formats like RSA (2048-bit or stronger) and ECDSA. The minimum length for SSH keys is 2028b.)

NOTE: Your network team may need to update firewall rules or IP allow-lists to permit connections to the new hostname. Be sure to coordinate with them. 

      5. Download Inbound PGP Encryption Key (If Applicable): 

    • This step is mandatory if you send PGP-encrypted files to SAP. The old key will no longer work. 
    • Action: In the XDL UI, navigate to Configuration > File Transfer Settings. Click Download Public Key. 
    • Provide this new public key to all internal or external processes that encrypt files before sending them to you.
    • Why is this necessary? This ensures that file decryption is handled securely on the new platform. 

      6. Manually Relocate Integration Folder Files (If Applicable): 

    • If you have identified an Integration folder in Phase 1, you must manually move the files to their new, dedicated product folders. The Integration folder is no longer supported. 
    • Workflow has its own connection details on the new platform. Please refer to the screenshot below, which illustrates the folder structure. For further guidance, consult the specific documentation for guidance.  

 

 Phase 3: Verification & Testing 

Once the execution steps are complete, verify that everything is working as expected. 

      7. Perform a Connection Test: 

    • Please see Help documentation for performing connection tests in Workflow
      https://help.sap.com/docs/Advanced_Workflow/b7cb13f8b6a04157a722ad9ddfde4a28/72a847e27c231014a804993ce4041860.html?version=2602

      8. Conduct End-to-End File Transfer Tests: 

    • Using Help documentation on 7, please test transferring files to an from SFTP/Workflow.

Congratulations! Once these tests are successful, the migration for that environment is complete. 

What to Do If Encountering an Issue 

In case of any trouble happens during this process, please do not hesitate to contact our global Support Team via the SAP for Me portal. They are fully briefed on this migration and ready to assist you. 

Frequently Asked Questions (FAQ) 

Why is this SFTP migration necessary? 
The migration to our new platform is required to provide enhanced security, improved reliability, and support for modern encryption standards. Specific benefits include advanced file verification, MDC signature support for PGP-encrypted files, and compatibility with a wider range of key formats. 

Will there be any downtime during the migration? 
Yes. The SFTP service will be unavailable during the scheduled maintenance window for your environment. Please ensure that any automated jobs or scripts are paused during this time to avoid failed transfers. 

What connection details do I need to update? 
Update the SFTP hostname, port, and username in all systems and scripts that connect to the SFTP site. The new username is the tenant ID with the suffix "_XDL" (e.g., gxxx_XDL). 

Do I need to update my firewall or network settings? 
Yes, this is a common point of failure. If your organization restricts outbound traffic, your network team must update firewall rules to allow connections to the new SFTP hostname on port 9093. 

What happens to the existing SSH keys? 
All of the existing SSH keys will be migrated to the new platform. However, as a best practice, test the connection for each user to confirm their key works as expected post-migration. 

Do I need to generate new PGP keys? 
Yes, this is mandatory if you send PGP-encrypted files to us (inbound files). After the migration, it is mandatory to generate a new PGP key from the XDL UI and use its public key for all future encrypted uploads. Public PGP keys uploaded through the XDL UI for encrypting outbound files will be moved to the new SFTP site as part of the migration.  

What should I do if I have a folder named “integration” on my SFTP site? 
The "integration" folder is being deprecated and will no longer be supported. You must manually move any files from this folder to the new, appropriate product-specific folders after the migration. 

How can I verify that the migration was successful for me? 
After updating the connection details, perform two key tests: first, a basic connection test to confirm the log in, and second, an end-to-end test transferirng sample inbound and outbound files to confirm the processes are working correctly. 

What should I do if I encounter an issue after the migration? 
Please contact our global support team immediately via the SAP for Me portal. To help us resolve the issue quickly, please provide specific details about the error and the steps already taken. 

Is there a risk of data loss during this migration? 
No files will be deleted by the migration process. All of the data will be moved to the new platform. However, files that reside in the deprecated “Integration” folder will require manual action from you to ensure they are in the correct location for your other processes. 

Is there an automatic rollback option if my testing fails? 
No, there is no self-service or automated rollback available to customers. If you encounter a critical issue after migrating, you must contact SAP Support immediately for assistance.
