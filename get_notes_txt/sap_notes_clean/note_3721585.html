• It is being done to provide enhanced security, improved reliability, and support for modern encryption standards.  
• Passwords will no longer be used for authentication for SFTP access.  Going forward, all access will use key-based authentication.

Resolution

Your Implementation Checklist: A Step-by-Step Guide

Phase 1: Pre-Cutover Planning & Inventory 

Before the migration date, it is critical to identify all systems and processes that interact with the current SAP SFTP service.

1. Copy each of your current FTP Definitions that use the above source FTP URLs and append the name and ID with -old to preserve your current information.

2. Identify All Connecting Systems:

    • Create an inventory of all applications, scripts, or third-party tools and users that send files to or retrieve files from or move files within the SAP Agent Performance Management or SAP Agent Lifecycle Management SAP SFTP site.
    • Identify any new whitelist / ip allow changes that will be necessary for your security team to deploy for the new solution to work. Your internal network team should be engaged to help identify where existing whitelist entries may exist and need to be updated for the new non-production and production solutions.
    • Each connection identified will need a key pair generated for this migration. Work with your IT / Network team to determine how these keys should be generated.

3. Check for Special Cases (Critical):

    • Workflow/Onboarding Scripts: Do you use any Workflow scripts that reference the SAP SFTP hostname or IP address? These will need to be updated.
    • Agent Compliance: If you use Agent Compliance, your implementation will need to be updated.

4. Perform SAP SFTP Site Housekeeping (Recommended):

    • To ensure a faster and smoother migration experience, we highly recommend reviewing the files currently on your SAP SFTP site and performing some light housekeeping.
    • Action: Consider deleting any files that are no longer needed, such as old data load files that have been successfully processed, outdated extracts, or temporary test files. Reducing the number of files to be moved will directly contribute to a more efficient migration.

5. Set up Process Notifications in APM/ALM

    • Notifications should be set for all imports
    • Notifications should be set for all extracts

These notifications will alert you to errors that may be encountered in using the new GoAnywhere SFTP solution

See Process Notification | SAP Help Portal for more information regarding this process. for more information regarding this process.

If you can identify and generate the keys (4096 bits) needed for your future access before the dates of the data set up of GoAnywhere and the data copies, April 4 (Non-Prod) and June 6 (Prod) , you can follow Appendix 3 to preload your keys PRIOR to the April 4 and June 6 dates.

Phase 2: SAP GoAnywhere Provisioning 

All activities with the SAP Provisioning of GoAnywhere will occur in maintenance windows. Activities that will occur will include the following:

1. The GoAnywhere environments for prod and sandbox will be provisioned using the userid currently used for the SAP SFTP servers

2. Data in the current SAP SFTP folders will be copied from the existing SAP SFTP environment to the GoAnywhere SFTP spaces based on the following schedule (image of the timeline is also below):

Non-Production - April 4, 2026

· Because we cannot guarantee what is considered TEST versus PRODUCTION, ALL FOLDERs (including empty folders) and ALL FILEs that are in your SAP SFTP location will be copied to the Non-Production GoAnywhere SFTP location.

· Your organization will need to connect to the Non-Production GoAnywhere SFTP location and clean out any files and folders that are not Non-Production related.

Production - June 6, 2026

· Because we cannot guarantee what is considered TEST versus PRODUCTION, ALL FOLDERs (including empty folders) and ALL FILEs that are in your SAP SFTP location will be copied to the Production GoAnywhere SFTP location.

· Your organization will need to connect to the Production GoAnywhere SFTP location and clean out any files and folders that are not Production related.

Note: This copy will occur on the date identified. We are asking you to migrate your processing after that, but we are not demanding a completion date, provided it is before August 1, 2026. If additional files are loaded to the current SAP SFTP site following the copy, you will need to re-upload those files to the new GoAnywhere SFTP solution for processing.

Note: Once connections are set up to the new environments, any unwanted data can be deleted.

Phase 3: Post-Provisioning Execution 

Once your environment (SBOX or PROD) is set up, perform the following actions.

Make sure you have copied your current FTP definitions as outlined in the Phase 1 portion of this document.

Update Your System Connection Details (FTP Servers) Within Your APM/ALM environment: 

  1. Reopen your current FTP Server definitions
  2. Modify the URL to use the correct new URL in place of the old one.  Only change the URL portion, not the folder path.
  3. Make sure FTP Transfer Type is set to SFTP – FTP over SSH.
  4. Make sure the Port selected is 22
  5. Make sure SFTP Authorization Type is set to Key.
  6. Select Save
  7. After you save the record, select the Generate Key item from the menu on the lower right of the screen.  (You MUST choose a key length of 4096)
  8. Select the Download Public Key item from the menu on the lower right of the screen.
  9. Select OpenSSH from the Public Key Format drop down, enter a File Name to identify the FTP definition, and select Download Public Key dialog button.
  10. Repeat for each FTP Server connection you have that points to the current SAP SFTP targets.
  11. Once all definitions are updated, open a support ticket (P4), attach all of the public key files you downloaded and get SAP to place the public keys in the GoAnywhere vault for APM or ALM, depending on your product.
  12. Once the keys are loaded in the vault, for each definition above,
    • Choose Test to test the FTP connection and ensure that the application can connect based on the information you've defined. If the test fails, please reach out to your partner / SAP Services / whomever is helping with this transition for assistance. If there is still an issue, please open a support ticket.
    • Choose Activate to change the FTP Server's status to Active.

Update Your System Connection Details Identified in your Phase 1 Inventory:

1. Each application, script, third-party tool or user that sends files to, retrieves files from, or moves files within the SAP SFTP solution will need to be updated based on the new connection details provided in the migration announcement. For each connection to and from your environment, you will need to:

2. Generate a private/public key pair (select a key length of 4096 bits)

3. Provide the public key via a support ticket to SAP Support to get the public key added to the appropriate key store for the GoAnywhere SFTP solution.

    • The support ticket needs to be a P4
    • The support ticket should contain all public keys if possible to limit the number of tickets

4. Modify the connection definition to use the new URL and the generated keys to connect

Note: Steps 2 & 3 are applicable for any keys that were not uploaded in the bulk process outlined in Appendix 3.

Heads Up: Your network team may need to update firewall rules, or IP allow-lists to permit connections to the new hostname. Be sure to coordinate with them.

If applicable, coordinate the updates in your Workflow.

  1. This will require assistance from whomever manages your workflow environment.
  2. Any script referencing one of the IP addresses or URLs listed above will need to be updated to the environment specific URL for the new environment.
  3. A guide to Workflow / Onboarding scripting can be found here.

If applicable, coordinate the updates to Agent Compliance 

  1. This will require assistance from whomever manages your compliance configuration. 
  2. A reference for setting up the ftp connections in Agent Compliance can be found here.

Phase 4: Verification & Testing 

Once the execution steps are complete, you must verify that everything is working as expected.

Perform a Connection Test: 

  1. Use your preferred SFTP client to perform a basic connection test with the new hostname, new username, new port, and an SSH key to confirm that you can log in successfully. Test this for all required users. 

Conduct End-to-End File Transfer Tests:

  1. Inbound Test: Send a sample inbound file (e.g., a data load file) and confirm it is processed correctly by the APM/ALM system. 
  2. Outbound Test: Trigger a process that generates an outbound file (e.g., an extract) and confirm you can successfully retrieve it from the GoAnywhere SFTP site. 
  3. Full Test: Run all imports from a normal processing cycle, along with all payout and extract processing to ensure all files are generated and accessible.

Congratulations! Once these tests are successful, your migration for that environment is complete.

What to Do If You Encounter an Issue 

If you have any trouble during this process, please do not hesitate to contact our global support team via the SAP for Me portal. They are fully briefed on this migration and ready to assist you.

Appendix 1 - Frequently Asked Questions (FAQ) 

Why is this SFTP migration necessary? 
Migration to our new platform is required to provide enhanced security, improved reliability, and support for modern encryption standards.  

Can a user id / password combination still be used for connections? 
No. We are moving to a more secure key-based authentication approach.

How do I set up a key pair for secure SFTP? 
Please refer to Setting Up SFTP Client | SAP Help Portal for more information.

What connection details do I need to update? 
You must update the SFTP hostname, verify the port, and username in all systems and scripts that connect to the SAP SFTP site. Your username will be the same as the one in use for the old SAP SFTP solution that is being phased out. All connections will also need to be updated to use key based authentication, as passwords will no longer be supported.

Do I need to update my firewall or network settings? 
Yes, this is a common point of failure. If your organization restricts outbound traffic, your network team must update firewall rules to allow connections to the new GoAnywhere SFTP hostname on port 22.

How can I verify that the migration was successful for me? 
After updating your connection details, you should perform two key tests: first, a basic connection test to confirm you can log in, and second, an end-to-end test where you transfer sample inbound and outbound files to confirm your processes are working correctly.

What should I do if I encounter an issue after the migration? 
Please contact our global support team immediately via the SAP for Me portal. To help us resolve your issue quickly, please provide specific details about the error you are seeing and the steps you have already taken.

Is there a risk of data loss during this migration? 
No files will be deleted by the migration process. All your data will be copied to the new platform. However, files that are loaded or generated to the SAP SFTP site after the copy occurs will not be present in the new GoAnywhere SFTP solution.

Is there an automatic rollback option if my tests fail? 
No, there is no self-service or automated rollback available to customers. If you encounter a critical issue after migrating, you must revert to the old ftp server using your original connection specifics to go back. This will be a temporary solution though, as the existing SAP SFTP servers will be toggled to read only on 8/1/2026. 

Appendix 2 - Set Up a New GoAnywhere SFTP Connection

1. In the Portal menu, navigate to (Administrator) > Processing Configuration and choose FTP Server Search.

2. Click on the (+).

3. On the FTP Server Detail screen, enter a unique value for the FTP Server Id field.

4. Enter a unique value for the Full Name field.

The value is used to describe the FTP server. It defaults to the FTP Server Id.

5. Enter the URL.

Indicates the hostname and path to the FTP site, for example, localhost/path/to/files/.

6. In the Port field enter “22”

7. Make sure SFTP Authorization Type is set to Key.

8. Select Save.

9. After you save the record, select the Generate Key item from the menu on the lower right of the screen.  (You MUST choose a key length of 4096)

10. Select the Download Public Key item from the menu on the lower right of the screen.

11. Select OpenSSH from the Public Key Format drop down, enter a File Name to identify the FTP definition, and select Download Public Key dialog button.

12. Open a support ticket (P4), attach the public key file you downloaded and get SAP to place the public keys in the GoAnywhere vault for APM or ALM, depending on your product.

13. Once the keys are loaded in the vault, for each definition above,

    • Choose Test to test the FTP connection and ensure that the application can connect based on the information you've defined. If the test fails, please reach out to your partner / SAP Services / whomever is helping with this transition for assistance. If there is still an issue, please open a support ticket.
    • Choose Activate to change the FTP Server's status to Active.

Appendix 3 - Public Key Bulk Upload

Uploading Customer Public Keys for Individual SFTP Access

Purpose

Customers must upload their public keys to enable individual SFTP access.  Keys should be generated with 4096 bits.

Folder Structure

Public keys must be uploaded to the following folders:

  • For Production - /Prod/PUBLIC_KEYS

  • For Non-Production / Sandbox - /Test/PUBLIC_KEYS

 

Naming Convention

The public key file must follow the naming convention below:

  • For keys being used by applications or scripting a name like the following
    • facets.pub
    • retrieveextracts.pub
  • For keys for individuals, use the following:
    • Useremailid.pub (user.name@corp.com)

Deadline for Uploading Public Keys

Keys must be loaded into the folders for bulk processing no later than:

  • For Non-Production / Sandbox – April 3, 2026.
  • For Production – June 5, 2026.

After the Deadline or If no Access

If customers fail to upload their public keys by the deadline, or if they do not have access to upload the keys, they must contact Support, after April 3 for Non-Production / Sandbox or June 5 for Production, for assistance.  If tickets are needed, please keep the following mind:

  • The support ticket needs to be a P4
  • The support ticket should contain all public keys if possible to limit the number of tickets
