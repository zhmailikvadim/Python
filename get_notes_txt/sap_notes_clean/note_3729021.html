Symptom

Transactions may erroneously get ignored and not applied to the secondary ASE server in an HADR environment when the primary ASE server is shutdown unclean and brought online again.
This leads to inconsistent data on the secondary (standby) site.

Other Terms

CR 831866, SAP Adaptive Server Enterprise, SAP ASE , HADR, SAP Replication Server, SAP SRS,

Reason and Prerequisites

Your database is SAP ASE version :

  • SAP ASE 16.0 SP03 (any PL)
  • SAP ASE 16.0 SP04 PL00 - PL06 HF1 and PL08

SAP ASE version 16.0 SP04 PL07, SP04 PL07HF1 and  SAP ASE 16.1 (any PL) are not affected.

You are using SAP Replication Server (SAP SRS) to replicate transactions to an HADR site (Always on). ASE was brought down unclean either through a power failure , an ASE crash or the command shutdown with nowait. After this unclean shutdown of the primary ASE server no failover to the secondary site is performed. The primary server is restarted again as primary HADR server.

At reboot after the unclean shutdown the primary ASE server determines the timestamp of the last replicated log record from an internal structure that is preserved to disk. After an unclean shutdown of ASE this timestamp value may be out of sync with the real timestamp value of the last log record replicated.  SAP SRS may then erroneously ignore some transactions that should get replayed.

The issue does not arise if a failover to site 2 happens after the unclean shutdown.

Solution

Install an ASE version with a fix for CR831866. 
A fix is planed for SAP ASE 16.0 SP04 PL09 and higher. The issue is also fixed in 16.1 SP00 (any PL)

Until a fix for CR 831866 has been installed in the system:
Rematerialize the database on the secondary HADR server when the primary ASE server was rebooted after an unclean shutdown.
Do not perform a failover to site 2 before the rematerialization has happend.
