Symptom

 You seek information on how to configure the Intel CPU-idle time management on your SAP HANA servers.

Other Terms

 HANA, C-state, intel_idle, Red Hat, SUSE

Reason and Prerequisites

Modern CPU cores can enter idle states, when they are not busy, to save energy. Energy saving is achieved by lowering frequency, reducing voltage, or shutting down parts of the core. Depending on the CPU's generation, different idle states are available. The higher the idle state, the more energy can be saved. That comes at the expense of a higher wakeup latency.

The default Linux idle time management driver on hardware with Intel processors is "intel_idle".

Solution

As SAP HANA benefits from low latency idle states, a recommended balance can be achieved by allowing the intel_idle driver to make use of C-states with a wakeup latency of up to 70 microseconds (µs).
The Linux distributors' SAP-specific helper tools assess the available C-states and their corresponding wakeup latencies. Based on this assessment, they choose the maximum (i.e. deepest) idle state according to this guidance.
Please see the distributors' documentation for more details:

  • SAP Note 3427561 on Red Hat systems for the 'sap-hana' tuned profile
  • SAP Note 1275776 for 'saptune' on SUSE systems

More background information

The 'cpupower' tool helps to display the available idle states and their corresponding wakeup latency, like in the following, shortened example:

# cpupower idle-info
CPUidle driver: intel_idle
[..]
Number of idle states: 4
Available idle states: POLL C1 C1E C6
POLL:
[..]
Latency: 0
C1:
Latency: 1
[..]
C1E:
Latency: 2
[..]
C6:
Latency: 290
[..]

In this example, the deepest idle state to be utilized should be set to 'C1E' - which corresponds to index #2 in the list of available states. This state should be selected because it represents the deepest C-state that still conforms to the recommended wakeup latency of up to 70 microseconds.

It is recommended to configure the available CPU idle states by using the tools provided by your OS distributor, as described above.

To manually limit the available CPU idle states at OS level, you can use the 'cpupower' tool:

# cpupower idle-set --disable-by-latency 70

This disables the usage of C-states with more than 70 microseconds (in this example deeper than C1E, index #2, until the next OS reboot.


To manually make these settings persistent across OS reboots, you can append the following to the Linux kernel command line:

intel_idle.max_cstate = <index>
intel_idle.max_cstate = 2

After a reboot, the usage of C-states deeper than C1E (index #2) will be disabled in this example.


Please note: The number of available CPU idle states may already be limited at the hardware or hypervisor level, making them invisible on OS level. For more information, please consult the respective vendor's or technology's documentation. For example:

  • 1656250 - AWS 
  • 2993054 - Azure 
  • 2100566 - Lenovo 
  • 2015392 - VMware 

The loaded CPUidle driver should be 'intel_idle'. Which can be checked as follows:

# cat /sys/devices/system/cpu/cpuidle/current_driver
intel_idle
