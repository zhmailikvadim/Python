Symptom

The sequenced files transferred from the primary server to the backup server appear to be transferred, and then applied, in no particular order.
In some cases this results in a discrepancy between the master data on the primary and backup servers. For example, if the full PLU data is applied after the cross-reference data, then the cross-reference data will be wiped-out.
This symptom can be also observed when synchronizing the product data between the primary server and a local PLU database at the register.

The files are expected to be processed in the same order in which they were generated on a primary server.

Other Terms

PLU, backup server, local PLU, product data synchronization, data synchronization mechanism, TRACKPLUUPDATES, TRACKDOWNLOADUPDATES, MAX_SEQ_TXN_UPDATES

Reason and Prerequisites

This issue occurs in POS 2.3 SP8 and in the earlier releases.

The backup server is updated by applying the sequenced files that are transferred from the primary server. These sequenced files (for example - plutxn.001, plutxn.002, ...) originate from the corresponding ascii files downloaded from a head office. They are saved in the 'txnsave' folder and are transferred from there to the backup server, or to the registers with a local PLU database, during the data synchronization process.

Solution

Fixed in POS 2.3 SP9.

Corrected the sorting function, so that the files are processed in a correct order.
