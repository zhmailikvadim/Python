Symptom

You have received a notification email advising of the switch in API Management Runtime Root Certificate from DigiCert Global Root G2 to DigiCert TLS RSA4096 Root G5.

Environment

API Management (NEO)

Cause

Cause:
As part of our ongoing commitment to security and trust, SAP is updating the Root Certificate Authority (CA) used for certificates in runtime environments from DigiCert Global Root G2 to DigiCert TLS RSA4096 Root G5.

Impact:
All inbound HTTPS traffic to API proxies is potentially impacted. Client systems that do not trust the DigiCert TLS RSA4096 Root G5 Root certificate will fail to establish a secure connection. 

Resolution

Action Required:

To ensure a smooth transition and avoid service disruption, client systems must trust the new DigiCert TLS RSA4096 Root G5 root CA alongside the current DigiCert Global Root G2 (required for the transition phase).

  • When the client system is SAP BTP–based:
    No action is required. BTP will handle the certificate import.
  • When the client system is SAP on-premise or non-SAP software:
    Customers must follow the relevant product documentation to update the Root CA in the client system trust store.
    Download Link: DigiCert TLS RSA4096 Root G5

Note: If you are using two-way SSL (mTLS) for connectivity to SAP API Management virtual hosts, you must verify that your client certificates include the Client Authentication Extended Key Usage (EKU) attribute. Certificates that do not include this attribute may be rejected during the TLS handshake.

Note on mTLS and External Consumers

When connecting to SAP API Management virtual hosts using Two-Way SSL (mTLS), your client certificate must include the Client Authentication Extended Key Usage (EKU).
Certificates that do not contain the mandatory Client Authentication EKU will be rejected during the TLS handshake. As a result, API requests will fail.
 
Communication to External Clients and Partners
All external partners, third-party clients, and end customers consuming your API proxies must meet these requirements.
Please ensure this information is clearly communicated to all relevant parties.
 
Action Required
Inform all clients, consumers, and end customers to:
•    Trust the DigiCert TLS RSA4096 Root G5 root CA.
•    Use client certificates (for mTLS) that include the mandatory Client Authentication EKU.
 
Non-compliance with these requirements will result in service disruption.

3. Who is the target audience for this email?
This email is intended for Business Managers and Integration Administrators, as they should be aware of the potential impact on all inbound integrations scenarios.

4. Why is this change happening?
SAP API Management is introducing the DigiCert TLS RSA4096 Root G5 root certificate for runtime environments as part of an ongoing effort to strengthen security standards. Root certificates are foundational to establishing secure, encrypted communications between systems, and updating them is a standard industry practice.

This change is happening because:

  • Enhanced Security: The DigiCert TLS RSA4096 Root G5 certificate uses a 4096-bit RSA key, providing stronger encryption and greater resistance against modern cyber threats.
  • Lifecycle Management: Older root certificates are gradually being deprecated or expiring. Moving to a newer root ensures continued trust and compatibility.
  • Compliance and Standards: The update aligns SAP with the latest security and compliance requirements, including those from browsers, operating systems, and industry bodies.
  • Future-Proofing: Implementing this root certificate ensures SAP services remain secure and trusted in modern and future IT environments.

5. Which certificate is being changed?
API Management Runtime Server Certificate chain. A new server certificate is issued by DigiCert DigiCert TLS RSA4096 Root G5.

6. Can the old Root Certificate "DigiCert Global Root G2" and the new Root Certificate "DigiCert TLS RSA4096 Root G5" co-exist in my client system?
Yes, both certificates can and should co-exist in your client system as Trust Anchors to ensure smooth transition.

7. Do we need to trust Intermediate CA / Leaf certificate on our client systems and receiver systems?
No, SAP recommends that customers trust only the Root Certificate.

8. How can I verify whether my client system is secure after this change or already trusts the DigiCert TLS RSA4096 Root G5?
To verify this, initiate a connection from your client system to the following URL: https://certtest.apimanagement.hana.ondemand.com/test. If you receive an HTTP status code and a JSON response, your system is configured correctly. No authentication is required for this process.

9. Which PSE in STRUST should we update?
This question can only be answered by your system administrator, as the usage of PSE is known and handled exclusively by customers.
PSE is specified in the SM59 destination and the Logical Port. In a nutshell, Anonymous PSE is for Basic Authentication, while Standard PSE is used for Client Certificate Authentication.
If you require SAP assistance in assessing your PSE usage, please refer to FAQ, point 2, for SAP Professional Services.

A simpler (and more straightforward) approach would be:

  • If a PSE has DigiCert Global Root G2 but does not contain DigiCert TLS RSA4096 Root G5, import DigiCert TLS RSA4096 Root G5.
  • If a PSE does not contain DigiCert Global Root G2, no action is likely required.

10. Are Custom Domains (non-SAP default domains) impacted by this change?
No. This migration only affects the default domain virtual hosts which use default certificate chain provided by SAP (e.g., URLs ending in *.apimanagement.*.hana.ondemand.com).
Custom domains (e.g., api.yourcompany.com) use certificates managed and provided by the customer. These certificates have their own independent trust chain and do not rely on the SAP-managed DigiCert Root certificate.

11. Is there an impact on browser access to Design-time?
Design-time endpoints (API Portal and Developer Hub) will be migrated to a different Root Certificate Authority, DigiCert Global Root G3. For more information, please refer to KBA 3705965.

12. Is there an impact on the Cloud Connector scenario? 
Impact is not anticipated. The DigiCert Global Root G3 root certificate (used for SAP BTP Neo endpoints for Cloud Connector connectivity) is widely included in standard JVM distributions. As long as the Cloud Connector is running on a supported JVM version, the certificate should already be present. For legacy or highly restricted environments, we recommend a manual verification of the trust store.

13. Why is the Client Authentication Extended Key Usage (EKU) attribute required in the client certificate used for mTLS?
In a Mutual TLS (mTLS) handshake, the Client Authentication Extended Key Usage (EKU) is a mandatory cryptographic extension that validates the intended purpose of the certificate.
If the certificate does not include the Client Authentication EKU, it cannot be used for client-side authentication. In such cases, SAP API Management virtual hosts will reject the connection during the TLS handshake.
This requirement ensures that only certificates explicitly issued for client authentication are accepted for API proxy calls. It prevents the unauthorized or incorrect use of certificates that are intended for other purposes (for example, certificates issued solely for Server Authentication).

14. How can I verify whether my client certificate includes the "Client Authentication" EKU?
To verify whether the Client Authentication Extended Key Usage (EKU) is present in your client certificate, use one of the following methods:

Method 1: Using OpenSSL (Command Line)

Run the following command:
openssl x509 -in <your_certificate>.crt -text -noout
 
Validation:
In the output, locate the X509v3 Extended Key Usage section. It must explicitly include:
•    TLS Web Client Authentication or
•    Client Authentication (OID: 1.3.6.1.5.5.7.3.2)
If this value is not present, the certificate cannot be used for mTLS client authentication.

Method 2: Using Windows Certificate Viewer

  1. Open the certificate file (.crt).
  2. Navigate to the Details tab.
  3. Locate the Enhanced Key Usage field.

Validation: The field must list Client Authentication (1.3.6.1.5.5.7.3.2)

15. Does this change impact external partners or consumers?
Yes. If your SAP API Management proxies are consumed by external partners, third-party clients, or end customers using mTLS, they are also subject to these requirements.
 
All consuming clients must ensure that:
•    Their client certificates include the mandatory Client Authentication Extended Key Usage (EKU).
•    Their systems trust the required Root CA.
 
Failure to comply with these requirements will result in failed TLS handshakes and service disruption.
 
Recommendation:
Customers are strongly advised to proactively communicate these requirements to all external partners and API consumers to ensure continued service availability.

16. How can I get notified of changes related to API Management?
SAP periodically sends out automated emails with updates about your API Management service. To receive these emails, end-users must subscribe. For more information on the subscription process, please refer to the following resources:

17. Who should subscribe to the automated emails?
The decision to subscribe to automated emails is up to each customer. It is recommended that Administrators subscribe to stay informed. End users and developers may also find these emails beneficial. Additionally, business leaders should be aware of major changes to ensure proper planning and testing are implemented.

18. How to identify whether a tenant is Production or Non-Production?
Open API Portal > Navigate to Settings tab. Under the Account section, the Account Type field indicates if the tenant is Production or Non-Production.
Note: To access the Settings page, the APIManagement.Selfservice.Administrator role must be assigned to you. For more information about user roles refer Assign Users to Roles.
