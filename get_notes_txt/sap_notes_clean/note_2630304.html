Symptom

You are using "db6util ... -lock2key" to record lock waits and to resolve a lock name to key values of a table. This does not work if the target table of the row lock is a column-organized table.

Older db6util versions may return an SQL error in this case.

Other Terms

DB6 UDB LUW RID RID_BIT TSNID BLU Acceleration

Reason and Prerequisites

On row-oriented tables, the table function MON_FORMAT_LOCK_NAME can be used to retrieve the TABSCHEMA, TABNAME, DATA_PARTITION_ID, PAGEID, and ROWID from a given row lock name.  This information can be used to calculate the RID of the locked row. After this, a "SELECT ... FROM TABSCHEMA.TABNAME WHERE RID( TABNAME ) = ?" statement can be used to retrieve the locked key values.

The same does not work on column-organized tables. On a column-organized table, MON_FORMAT_LOCK_NAME only returns the TABSCHEMA, TABNAME, and TSNID from a given row lock name. Currently, there is no way to use the TSNID in a SELECT to retrieve the locked key values.

Solution

To avoid an SQL error, use a current version of db6util:

db6util PL 23

patch text: "DB6: row locks on BLU tables cannot be translated to key values"
