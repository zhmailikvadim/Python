Symptom

Resolution

Issue


Each user's ShipTo address in Ariba is being set to their site's SAP Business Network ID (ANID). This, also translates to each PunchOutSetupRequest (POSR) being sent from Ariba to the supplier when initiating a punchout session. The customer expects an actual ShipTo address to be set for the user and sent in the POSR.

Cause


When accessing Ariba Catalogs from Oracle, this is default behavior. An example Uniform Resource Locater (URL) for accessing Ariba Catalogs through Oracle is as follows:

https://s1.ariba.com/Buyer/Main/ad/transformPOSR/ariba.content.transform.TransformListener?ANURL=
https://service.ariba.com/service/transaction/cxml.asp&Groups=xxxxxxxxxx&ShipTo=ORA_ADD_100&PreferredSu
pplierDomain=xxxxxxxxxx&SelfRedirect=FALSE

The URL contains a ShipTo value of ORA_ADD_100, however the system sets the user's ShipTo value to their site's ANID.

Solution


This is controlled by a variable entitled OverrideShipTo in the URL sent from Oracle to Ariba. If the value of the OverrideShipTo parameter is set to False, the ShipTo address is not replaced with the ANID of the external procurement system. If the value of the parameter is set to True, the ShipTo address is replaced with the ANID of the external procurement system. The default value of the OverrideShipTo parameter is True; therefore, if it is not included, the value is seen as True.

By setting the OverrideShipTo value in the URL to False, the customer is able to specify a ShipTo value for these users.

https://s1.ariba.com/Buyer/Main/ad/transformPOSR/ariba.content.transform.TransformListener?ANURL=
https://service.ariba.com/service/transaction/cxml.asp&Groups=xxxxxxxxxx&ShipTo=ORA_ADD_100&PreferredSu
pplierDomain=xxxxxxxxxx&SelfRedirect=FALSE&OverrideShipTo=FALSE
