Symptom

We can use Payment Terms to trigger different Price Lists on Sales Orders desktop client. When we change the payment method and say ''Yes'' to the message prompted, the selected price list of Payment method is used and the application changes the price. But it is not working at Service Layer.

Reproducing the issue

Prerequisites:

1. Creating two payment terms and assigning distinct price lists to each (for example, Tyrkey Company DB).

For example: - Payment Terms code: paymentterm1;

Price list: pricelist1 - Payment Terms code: paymentterm2; Price list: pricelist2

2. Setting the Business Partner's default payment term to paymentterm1, where the associated price list is pricelist1.

3. Configuring items to have different prices depending on the price list.

For example:

  • pricelist1: 1000TRY 
  • pricelist2: 1500TRY

 

Scenario:

Adding a sales order and setting the payment terms to paymentterm2 (overriding the default payment term) via DI API or service layer.

Documents oDoc = oCompany.GetBusinessObject(BoObjectTypes.oOrders);

            oDoc.CardCode = "C001";

            oDoc.DocDueDate = DateTime.Today;

            oDoc.PaymentGroupCode = 6;

            oDoc.Lines.SetCurrentLine(0);

            oDoc.Lines.ItemCode = "I001";

            oDoc.Lines.Quantity = 1;

            int n = oDoc.Add();

This issue can be reproduced with Service layer:

POST https://cl017-i2:50000/b1s/v1/Orders

{

    "CardCode": "C001",

    "DocDueDate": "2026-01-08",

    "PaymentGroupCode": 6,

    "DocumentLines": [

 

{             "LineNum": 0,             "ItemCode": "I001",             "Quantity": 1         }

    ]

}

Expected behavior:

The price and price list correspond to the payment term specified in the DI API code or service layer request.

Actual behavior:

The Payment Term is correctly updated to paymentterm2 as specified in the code. However, the price list applied to the item line defaults to pricelist1 instead of pricelist2, which is assigned to paymentterm2.
