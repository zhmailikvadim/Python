Symptom

With 2408 response & supply planning introduced a lot grouping id to logical bundle multiple stock transfer requisition of the same product and date stemming from the same minimal lot size into one stock transfer order. This attribute is filled in application jobs in conjunction with characteristic based planning. First the lot group id was only an additional attribute in context of CBP, but with 2502 when combining CBP and shelf life in shelf life 2.0, it can also used for shelf life only customer.

On backend however there is no field to use this. This must be added by custom extension mechnismen in the backend. In addition with shelf life two additional fields are filled by IBP called expiry date and minimal required shelf life on input side. Also those fields have no strict equivalent in S/4HANA where only the minimal remaining shelf life field on output side being the interval between expiry date and availibitly data exist. So at least the minimal required shelf life need custom extensions fields to be stored on the  STR or STO and retrieved when converting or sending back to IBP.

Other Terms

CBP, multi-item, STR, STO, minimal lot size

Reason and Prerequisites

Some shelf life fields are missing on S/4HANA or  ERP side.

The root to have multiple stock transfer requisition of the same product and date originating from the same minimal lot size might be the usage of characteristic based planning or shelf life 2.0 which is available in a harmonized planning area by default and also  on flexible model by setting the global configuration parameter ENABLE_SHELF_LIFE_PRODUCT_LEVEL to true.

The material in the backend need to have the indicator "automatic purchase order allowed" be set in the tab "Purchasing" in case you want to also do step 4 below.

(transaction MM02, field KAUTB in MARC)

You have followed note 3110007 to bring the backend system to the latest state which includes the code changes of note 3435854.

Solution

With 2408 SAP provides in the real time back integration of stock transfer requisitions from IBP response and supply to the backend the additional lot field called CIF_LOT_SIZE_GROUP_ID. You can also see the field in the enhancement spot/BAdI CIF_PO_MAP_IBP in table CT_OUTPUT_IBP with the line item structure being of type CIFPUOROUT_IBP for method CHANGE_ORDER_INBOUND. In addition, with 2602, we also added the field EXPIRY_DATE_TIME for the expiration date and the minimal required shelf life on input side .

In order to use this field the conversion process and collect multiple stock transfer requisitions in one stock transfer order, following steps need to be done:

  1. Extend ERP Structures and UI to include the lot grouping id, expiry date and minimal required shelf life as custom fields.
  2. Implement BAdI CIF_PO_MAP_IBP.
  3. Implement CIF Custom Exit to fill lot grouping id, expiry date and minimal required shelf life in CIF Inbound.
  4. Implement BADI to consider lot grouping id when converting STR to STO’s.
  5. Sending back shelf life fields expiry date and minimal required shelf life to IBP

Then transaction ME59n can be used to convert STR’s to STO’s and bundle them on the same lot field.

In the following we detail out the steps above:

  1. Extend ERP Structures and UI to include lot grouping id

    Use Standard extension mechanism (app Custom Fields) to add lot grouping id, expiry date and minimal required shelf life  to ERP Purchase Requisition Item
    Relevant Business Context: MM_PURREQN_ITEM
    Enable Business Scenario for GUI

    In older ERP systems, where the new custom fields do not yet exist, you can extend the EBAN table via the append structure to the EBAN table.

  2. Implement BAdI CIF_PO_MAP_IBP in inbound method IF_CHANGE_PO_IBP~CHANGE_ORDER_INBOUND

    An example implementation is shown here below. It stores the from IBP coming CIF_LOT_SIZE_GROUP_ID, EXPIRY_DATE_TIME and SHLF_MIN fields to a buffer:
        DATA(lo_lot_size_buffer) = zibp_cl_lot_size_buffer=>get_instance( ).
        lo_lot_size_buffer->set_lot_size_group_id( ct_output_ibp ).
        lo_lot_size_buffer->set_min_required_sl( ct_input_ibp ).
    The used custom class can be found in the attachment

  3. Implement CIF Custom Exit to fill lot grouping id in CIF Inbound

    Exit transfers lot grouping id from IBP STR to ERP Purchase Requisition (into extended custom field)
    EXIT CIFPUR02 and FM EXIT_SAPLMEPI_005
    An example coding in include ZXCIFU23 would be using again the buffer class in the attachment:

    DATA(lo_lotsize_buffer) = zibp_cl_lot_size_buffer=>get_instance( ).
    DATA(lt_doc_lotsize) = lo_lotsize_buffer->get_lot_size_group_id( ).
    DATA(lt_min_required_sl) = lo_lotsize_buffer->get_min_required_sl( ).

    " if order was buffered during CIF inbound -> substitute lot size group value with the required one
    ASSIGN lt_doc_lotsize[ ordid = cp_eban-ordid
                           schedid = cp_eban-schedid ] TO FIELD-SYMBOL(<ls_doc_lotsize>).
    IF sy-subrc = 0.
      cp_eban-lot_size_group_id = <ls_doc_lotsize>-cif_lot_group_id.
      cp_eban-expiry_date_time = <ls_doc_lotsize>-expiry_date_time.
    ENDIF.

    ASSIGN lt_min_required_sl[ ordid = cp_eban-ordid
                             schedid = cp_eban-schedid ] TO FIELD-SYMBOL(<ls_min_required_sl>).
    IF sy-subrc = 0.
       cp_eban-shlf_min = <ls_min_required_sl>-shlf_min.
    ENDIF.


  4. Implement BADI to consider lot grouping id when converting STR to STO’s

    Standard BADI MM_PUR_AUTOPO can be used to assure that STR’s with the same lot grouping id are grouped as multiple items in one STO.
    An example is attached here as ZIBP_CL_PUR_AUTOPO_LOTSIZE.ABAP.  It is based on the Implementation Example Class CL_MM_PUR_AUTOPO_BADI_EXAMPLE and adopted to the needs for the lot size grouping.

  5.  Sending back shelf life fields to IBP

    This step contained as step 2 in BAdI CIF_PO_MAP_IBP but no the mirrowing method on outbound IF_CHANGE_PO_IBP~CHANGE_ORDER_OUTBOUND.
    Read the attributes you want to send back from the persitency (e.g. directly from EBAN if you use append fields) or from the released key user extensiblity API I_PURCHASEREQUISITIONITEMAPI01
    Consider here in addition note 3626634 and the business parameter  EXPIRY_DATE_HANDLING to eiher determine the expiry date on IBP side from your own stored expiry date or from the min_SHLF on output/input.
