Symptom

When consuming OData APIs in SAP Intelligent Agriculture, expanding deep or multiple navigation properties may fail with errors such as:

"error": {
  "code": "400",
  "message": "Navigation property \"characteristics\" is not defined in \"CropZoneOData.Areas\""
}

The objects shown here are only examples, similar issues may occur with other entity types.

 
 

Environment

SAP Intelligent Agriculture

Cause

This occurs when a request uses excessive navigation properties or attempts to $expand across multiple, unrelated objects.

These patterns can lead to:

  • Security risks, such as indirect access paths similar to Broken Object Level Authorization (BOLA). 
  • Performance degradation, because large or deep $expand chains consume significant resources. 
  • Over‑exposure of associations, which goes against platform recommendations favoring use‑case‑oriented and least‑privilege service designs.

This behavior is expected by design to ensure secure, predictable, and high‑performing API consumption

Resolution

When encountering this error — or when designing expansions — please follow these recommended best practices:

1. Avoid multiple or deep $expand chains across different objects
Expanding several unrelated entities in one request increases complexity, may trigger authorization conflicts, and can produce errors or degraded performance.

2. Expand only from the object you are directly working with
Begin the $expand from the primary entity of your use case and avoid jumping across several distinct entity hierarchies. Multi‑level navigation is allowed by OData, but each additional hop increases security and stability risks. 

3. Do not merge navigations from different domain areas
If two data domains are not directly related in the same context, they should not be combined within one $expand. This avoids exposing unrelated data and prevents indirect access paths.

4. Use multiple smaller requests instead of one large expanded request
If you need information from different entities, request them separately. This approach is more stable, avoids timeouts, and is aligned with OData performance guidance
