Symptom

  • No error is initiated in the ERP system for delivery date tolerance defined for Ship Notice (Inbound Delivery) based on the purchase order.
  • When a supplier creates a Ship Notice in SAP Business Network with a delivery date exceeding the defined tolerance, no error occurs in the ERP system.
  • For Order Confirmation, when the delivery date exceeds the tolerance, an error is identified in WE02.
  • Error is visible in WE02 for ORDRSP when tolerance is exceeded; no error is logged for DESADV.

Environment

  • Product: SAP Business Network Commerce Automation
  • Supply Chain Collaboration - Fulfillment

Reproducing the Issue

  1. Define delivery date tolerance for Ship Notice (Inbound Delivery) in the ERP system based on the purchase order.
  2. Create a Ship Notice in SAP Business Network with a delivery date that exceeds the defined tolerance.
  3. Observe that no error is initiated in the ERP system.

Cause

  • Ship Notices (ASN) are processed via DESADV and create an inbound delivery (for example, using IDOC_INPUT_DESADV/BAPI_INB_DELIVERY_CREATE). In standard processing, purchase order delivery date tolerance checks are not enforced during ASN inbound delivery creation, so no error is produced even when the date exceeds tolerance.

Resolution

  1. Note that delivery date tolerance checks are executed during Order Confirmation (for example, ORDRSP) IDoc validation against the purchase order, but not during Ship Notice (DESADV) inbound delivery creation.
  2. If enforcement of delivery date tolerance is required for Ship Notices, implement a custom enhancement in inbound delivery processing to validate the ASN delivery date against the purchase order tolerance.
  3. Use an appropriate enhancement point, such as BAdI LE_SHP_DELIVERY_PROC or a relevant user exit during DESADV/inbound delivery creation, to compare the ASN date with the purchase order tolerance and raise an error or return status accordingly.
  4. Test in a non-production environment by sending a DESADV with a delivery date outside the tolerance and verify that the IDoc/inbound delivery processing reflects the expected error handling.
