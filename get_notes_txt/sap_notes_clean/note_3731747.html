Symptom

You have a very large backlog for some receiver channel(s) which has been stopped for some time. Starting the channel, your PI system experiences performance or memory issues. Checking on DB level, table XI_AF_ADM_MSG_STOP has a very large number of entries.

Other Terms

OOM, XI_AF_ADM_MSG_STOP, channel backlog

Reason and Prerequisites

There is an automatic procedure to resend messages for stopped receiver channel once they are up. An issue with the program logic causes a very large amount of data to be loaded from DB in memory causing memory or performance issues.

Solution

The issue is resolved with source code correction. An automatic mechanism is implemented to prevent memory or performance issues. With the current change, only up to 30,000 messages per channel can be resent automatically at once, when the receiver channel is up. An AFW scheduler job is added to clean the DB table.

You need to apply the patch of this note.

With it two new properties are added to XPI Service: AF Core:

 - administration.stoppedChannelsDataReorgInterval - defines the time period in milliseconds for running the job to delete the entries in table XI_AF_ADM_MSG_STOP. This job can only delete all the content or does nothing.

 - administration.stoppedChannelsDeleteAllData - set this property to true to have the job clean up the table content.

In all cases when there are more than 30,000 messages per channel, the automatic mechanism of resending messages for stopped receiver channels will break and no messages will be resent. You need to use an MS Restart Job to resend those messages.

To clean up the table's content in those cases, set the property administration.stoppedChannelsDeleteAllData to true. With the next data reorganization job run or any channel with failed messages restart, all entries (for all channels) will be cleared on DB level.

Hint: To avoid issues, stop the sender channels in order to eliminate any incoming messages to the PI system, before stopping the receiver channels.
