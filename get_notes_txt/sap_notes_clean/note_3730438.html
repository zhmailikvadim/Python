Symptom

  • It is not clear why system message "Corrections to previous months are not reported in declaration files" (Message No. /ECRS/BL_TD034) is given.
  • It is not clear how the system determines Aggregated Correction Items in Intrastat.

Environment

  • International Trade (ITR)
  • Intrastat
  • SAP S/4HANA
  • SAP Global Trade Services, edition for HANA

Reproducing the Issue

  1. In transaction /ECRS/RP_EDIT, users may see button 'Aggregated Items' above line items.
  2. When selecting a relevant item and clicking on 'Create File' button, a popup with system message "Corrections to previous months are not reported in declaration files" (Message No. /ECRS/BL_TD034) is given.
  3. If user selects the Aggregated Items button, on the next screen there is also button 'Aggregated Correction Items' in the top menu bar. 

Cause

  • In cases where there are corrections (such as credit memos, debit memos, or cancellations) to an original invoice in a subsequent month, the system creates correction items which are displayed in Aggregated Correction Items.
  • As an example: 

    An invoice is posted in April and reported in the April dispatch declaration. If a credit memo is posted in May, it will appear in the “Correction items” section of the May dispatch declaration.

    Correction items are included in the declaration file only for certain countries (i.e. Italy and Slovakia).

    For Austria for example, they are not part of the declaration file. Therefore, users are advised to check official guidelines to determine whether corrections should be reported through alternative means (such as adjusting a previous declaration in the government portal or notifying authorities via email), or if they can be ignored.

  • The popup with system message "Corrections to previous months are not reported in declaration files" appears for all “Providers of Information” where correction items exist for a previous month in the Intrastat declaration, and the declaration country is not Italy or Slovakia.

Resolution

This is standard system behavior as explained above.
