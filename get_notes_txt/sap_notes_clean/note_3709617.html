Symptom

A customer is implementing MFA using Risk-Based Authentication (RBA) in SAP Customer Data Cloud and wants to ensure that email verification is performed before allowing MFA (phone number) enrollment.

The customer observes the following behaviour:

  • If a user’s email address is already verified

  • When MFA (Assurance Level 20) is required

  • The user is taken directly to phone number registration

  • No additional email OTP verification is requested before MFA enrollment

The customer asks whether it is possible to:

  • Force email verification again before MFA enrollment

Environment

  • SAP Customer Data Cloud (CDC)

  • Risk-Based Authentication (RBA)

  • Email Identity Verification

  • MFA (Phone / SMS / Authenticator App)

Reproducing the Issue

  1. Email verification in CDC is a one-time account state, not a verification step that is automatically repeated.

  2. Once an email address is verified, CDC considers email ownership confirmed and does not re-trigger email OTP verification during subsequent actions such as MFA enrollment.

  3. RBA is not a sequential workflow engine:

    • RBA evaluates risk and enforces a single required assurance level

    • If multiple RBA rules apply, CDC enforces only the highest assurance level

    • Lower levels (for example, Level 10) are implicitly satisfied when Level 20 is enforced

As a result:

  • Two RBA rules cannot be chained to force “verify email first, then enroll MFA”

  • If MFA (Level 20) is required and the email is already verified, the user proceeds directly to MFA enrollment

Cause

This behaviour is expected and aligned with CDC design.

  • If isVerified = true for the user’s email:

    • CDC does not prompt for email OTP again

    • MFA enrollment is allowed directly when Level 20 is required

  • RBA cannot be used to re-verify email ownership once the email is already verified

Resolution

To prevent MFA enrollment after password-only authentication and mitigate MFA enrollment hijacking scenarios, SAP recommends the following approach:

1. Enforce email verification via flow logic (not RBA)

  • Require email verification during onboarding or first login

  • Use Screen Sets and account state checks (for example, isVerified) to block access to MFA enrollment screens if the email is not verified

  • Email verification should be treated as a precondition, not an RBA step
