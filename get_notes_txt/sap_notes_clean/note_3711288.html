Symptom

You are processing (or reprocessing) RAIs of a contract which contains soft deleted POBs. The contract is also in error status, and the POBs were soft deleted before this error status was set. Upon processing, you receive dump GETWA_NOT_ASSIGNED.

Other Terms

CCM, soft deleted, GETWA_NOT_ASSIGNED, error status

Reason and Prerequisites

Reason: A new functionality was introduced in note 3544351, which corrects a previous error where contracts would spontaneously reopen upon reprocessing, just because they contained a soft deleted POB. The code of that note checks whether the soft deleted POB has also been soft deleted before the processing, which means that there is no change and no need to reopen the contract. To do this, the system checks old POB data from the DB. Old data of this POB should exist, because you cannot create a new soft deleted POB during processing. However, there is a special edge case where these DB data exist, but are ignored because the contract is in error. In this case, the system doesn't load POB data from table FARR_D_POB, but from table FARR_D_POB_CORRT (correct POB data before contract error). Data of this POB might however be missing there, because the system doesn't create CORRT records for soft deleted POBs. The missing POB data causes the dump. This note makes sure, that there is no dump in this situation, and the contract still doesn't needlessly reopen.

Solution

 Please implement this SAP note.
