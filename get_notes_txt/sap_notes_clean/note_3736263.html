Symptom

You want to enable an SAP HANA SQLDBC or ODBC trace but are unsure of what tracing category or level to use.

Other Terms

client trace, hdbsqldbc_cons, hdbodbc_cons, SQLDBC trace, ODBC trace

Reason and Prerequisites

The SAP HANA client is installed on your system.

Solution

Many levels of SQLDBC/ODBC tracing can be very verbose which can significantly affect performance. Additionally, verbose levels risk filling the file system or quickly wrapping size limited traces (likely losing the desired information). The following recommendations give a place to start and generally avoid the most verbose levels.  Always remember to disable tracing once you have reproduced the condition.

Scenario (reason you want tracing) Production or Test System Suggested Trace Levels (for hdbsqldbc_cons or hdbodbc_cons)

A connect timeout or connect error. For example, the "Connect failed ..." -10709 error.

An unexpected "Connection down ..." -10807 error (for example, when silent reconnect is expected). Note that the "Connection down ..." error typically indicates that the server or network infrastructure dropped the connection, and SQLDBC tracing may not provide much more information beyond the error text itself.

The application is hung/blocked in the middle of an operation that may be making database requests, but there is no HANA database operation in progress.

Avoid in production systems unless it can be reproduced within a few minutes.

This may have a noticeable performance impact (perhaps in the range of 5-50% slower) while the application is accessing the database.

TRACE API ON

TRACE PACKET 0

TRACE SIZE 20M

TRACE FLUSH ON

The application is crashing and you are unable to get a crash call stack or core dump. You think it might be crashing in the the client driver and want to determine where in the client driver it is crashing.

Note that getting a crash call stack or core dump is preferrable to this tracing. If trying to use tracing to narrow down where the client is crashing, you must use TRACE FLUSH ON (otherwise the most important part of the trace will not be written). This level of tracing is very verbose and will significantly impact performance.

Must not be used in production systems since this will significantly affect application performance.

If you have no alternative but to try this in a production system, ensure the system owner knows that performance will be impacted (the application could run several times slower when accessing the database), and ensure you have permission from the system owner.

If possible double check that the trace file(s) includes the condition (eg SQL statement or error).

Notes:

  1. "TRACE ONLY ON ERROR" can result in tracing after the error occurs not appearing in the trace file. The buffering nature of TRACE ONLY ON ERROR reduces the performance impact of tracing but delays when the tracing is written to the file. If possible, turn off tracing and close the application after reproducing the error to ensure all of the tracing data is written. If closing the application is not possible, the trace will still contain what happened leading up to the error, which is normally the most important part.
  2. "TRACE SHOW TIMESTAMPS ON" requires the version 2.25 or higher client.
  3. For a basic description for the trace categories and levels see the Client Interface Programming Reference.
