502 Bad Gateway (nginx)

 When trying to access a manager URL, an error page with 502 is shown. The proxy (here nginx) is misconfigured and blocks traffic to the application server. First, make sure that application server and proxy can communicate with each other (ping, telnet). Then validate correct traffic forwarding and correct SSL certificates used.

java.security.KeyManagementException: problem accessing trust store After opening some content via Library button in manager, you should be able to press the Translation button. This process sends a translation request to a SEN cloud server, and receives the translation to display. For an onprem customer, the request resulted in 500 error, where the logs showed the message written.  The customer initially installed Oracle JDK, whose "cacerts" file is in JKS format, and later switched to SapMachine JDK 17. When using the old Oracle "cacerts" or a custom JKS truststore, SapMachine 17’s default PKCS12 assumption caused it to fail parsing the file, resulting in the ASN.1 PKCSParsingException. When installing SapMachine JDK initially, it also comes with a cacerts in JKS format, even tho the JDK17 uses PKCS12 as default. But because its in java home default path ($JAVA_HOME/lib/security/cacerts) it uses a fallback procedure to check the trustStoreType and is then able to read JKS.

Specifying -Djavax.net.ssl.trustStoreType=JKS forced the correct format, resolving the truststore loading error. Else its recommended to use a new VM and only install SapMachine17 JDK using default settings.

 
javax.security.auth.login.LoginException: No CallbackHandler available to garner authentication information from the user When trying to save a config on the LDAP Import page, an error popup is shown together with this log message.

Inside the manager settings, the keytab directory C:\keytab\ and login domain "sen" was configured, but during the LDAP import keytab upload, a wrong keytab file was selected.

The upload and saving the config works fine when selecting the correct keytab file that corresponds to the defined service principal, having the keytab file in an empty folder. The folder must NOT be the one registered in manager settings (C:\keytab\), BUT the exact SAME keytab file that was uploaded must be inside this folder as well!! Else the upload fails with the mentioned logs!

First, open Administration → Server Settings → Security and Password → SSO → and then specify the folder of the keytab directory (e.g. C:\keytab\). Then restart the tomcat.

Then go to Administration → User Import → LDAP → Add New or edit existing, upload keytab file and click save. Should work if keytab file and principle match!

Manager-manager.log: javax.naming.CommunicationException: simple bind failed test.ad:636
Tomcat9-stderr: sun.security.validator.ValidatorException: PKIX path building failed: sun.security.provider.certpath.SunCertPathBuilderException: unable to find valid certification path to requested target		 When trying to save an LDAP import parameter, an error pops up and the logs show this message.  When the tomcat tries to establish a trusted connection to an active directory server, it needs to validate the certificate provided by the ad server. This only works if the tomcat explicitly trusts all root and intermediate certificates which have signed the AD server certificate. 

To solve this issue, all root and intermediate certificates which have signed the AD server cert or are included in its key chain must be imported into the JDK trust store (C:\Program Files\SapMachine\JDK\17\lib\security\cacerts). The certificates can be pulled using openssl command:
openssl s_client -connect TEST.AD:636 -showcerts

BUT CAREFUL: Often the root cert is not printed, so keep in mind to also find the root CA cert somewhere.

For the import, use following example command:
keytool -importcert -trustcacerts -keystore "C:\Program Files\SapMachine\JDK\17\lib\security\cacerts" -storepass changeit -alias ad-root -file d:\Company_Root_CA_1.crt
and import all certificates one at a time!
After tomcat restart, the issue will be resolved.

To validate if the bind user is valid, try this command on a linux machine:
ldapwhoami -vvv -H ldaps://TEST.AD -D "user@test.ad" -x -W

To add debugging level into tomcat9-sterr.log add following Java params:
-Djavax.net.debug=ssl,handshake
-Dcom.sun.jndi.ldap.trace.ber=true
No assertions found in response. After SSO login, error page shows up with:
No assertions found in response. <aws-sso-t4>		 On the manager side, the setting "Create user on SSO login" is deactivated. Now if an existing user is having the correct user name, but is missing the email or is holding a different email than in AD, the SSO login user can not log in!

Log into the manager using admin, modify the existing user to have correct username and email attribute like in AD, and then login should work, assuming that SAML SSO is configured correctly!

If First Name and Last Name are empty, the SAML user will fill it or overwrite it. But email and username must match!

com.sap.db.jdbc.exceptions.SQLIntegrityConstraintViolationExceptionSapDB: [301]: unique constraint violated: "SENTABLE"."UPDATEDNFORLDAPIMPORTEDIDENTITIES": line 27 col 1 (at pos 842): Table(ACTOR), Index(UNIQUE_ACTOR_DN) with error: unique constraint violation in self check for table SEN::SENTABLE:ACTORen, constraint='DN',

A client reported a problem with LDAP import where the import did not finish successfully, and listed errors where visible in log files. The LDAP configuration was saved and selected successfully, but after 8hrs of running the import, it did not show a summary table.

Another client reported only the Unsupported signature error inside the logs and wanted to get rid of it.

The main problem found here was that the client tried to import 45k users every night on their prod system and on their dev system, and at some point the prod failed. It was found that the client had 40k inactive users which caused duplicate entries after some import, because a user was deleted in their AD and then recreated, but not deleted in the manager.

After importing a license key the error popped up inside the logs.

The solution was to simply do Server Housekeeping inside the manager and delete all inactive history and user data. After that, the import worked again. Additionally, its possible to search for duplicated user/roles/ous entries on DB level like this:

select auth_user, count(auth_user) as usercount, a.ou_id, parent.name as parent_ou from actor as a inner join ws_user on ws_user_id = a.identity_id inner join actor as parent on parent.identity_id = a.ou_id group by auth_user, a.ou_id, parent.name having count(auth_user) > 1

and then rename the inactive user with:

update ws_user set auth_user = '<new_name>' where ws_user_id = 28919

update actor set dn = 'U=<new_name>,OU=Name1,OU=Name2,OU=Name3,OU=Name4,OU=domain,OU=root' where identity_id = 28919

Remove the license key from database with: “delete from LICENSE;"

The manager will still have full functionalities without license, but error disappeared.
