Symptom

Historical means to protect system-internal communication between the different ABAP server components were:

  • Protecting the network segment where the components reside ("firewalling")
  • Protecting access to different components with access control lists (ACLs)

Netweaver Kernel 742+ software components that can use Secure Internal Server Communication:

  • (BC-CST-STS) sapstartsrv and clients to sapstartsrv (sapcontrol, icman)
  • (BC-CST-MS) AS ABAP Message Server (on the "internal" message server port), and specific Message Server clients (disp+work, sapevt)
  • (BC-CST-GW) AS ABAP Gateway (on the "internal" gateway port), and specific Gateway clients (disp+work)
  • (BC-CST-EQ) AS ABAP Enqueue Server and Enqueue Server clients (disp+work, enqt)
  • (BC-CTS-TMS) AS ABAP tp & sapevt

Other Terms

SystemPKI, system/secure_communication

Solution

Beginning with ABAP Release 7.40 SP8 (component SAP_BASIS 740 SP8) and minimal kernel version 742, it is possible to make Netweaver AS ABAP kernel components use TLS-protected communication to other backend server components of the same ABAP system <SID>, using certificates created by a system-internal private PKI (systempki) to authenticate each other.

Important Restriction: 

Only "pure" ABAP systems are supported (i.e. neither NetWeaver DualStack ABAP+Java systems, nor Netweaver AS Java standalone systems).

Known Issues:

  • Custom profile parameter values for profile parameters ssl/ssl_lib, sec/libsapsecu and ssf/ssfapi_lib may result in multiple seperate instances of CommonCryptoLib getting loaded, which will break the use of MemoryPSEs by systemPKI-protected communication.
    You have to remove all custom profile parameter values for parameters ssl/ssl_lib, sec/libsapsecu, and ssf/ssfapi_lib from all profiles, and then restart all Instances (all sapstartsrv services of all instances, ASCS & ERS instances, alll AppServer instances, in that sequence order).
    Pretty much all errors related to an inability to access #_MemPSE_# are caused by custom values for profile parameters ssl/ssl_lib, ssf/ssfapi_lib and sec/libsapsecu -- and can be resolved by removing all custom values and restarting everything.
    btw. the same incorrect custom profile parameters also break SUM (SUM is erroneously running "sapcontrol" with -systempki command line parameter and SUM seems lacking a fallback to traditional operation).
  • incorrect value for the OS-level setting of the environment variable "SECUDIR" for the <sid>adm (Unix) or SAPService<SID> (Microsoft Windows) causes sapstartsrv to compute an incorrect location for storing the "sap_system_pki_instance.pse" file.
    For Netweaver AS ABAP and S/4HANA, the value of the environment variable "SECUDIR" must either be undefined at the OS-level (such as in Shell login scripts), or SECUDIR must match "$(DIR_INSTANCE)$(DIR_SEP)sec".  Unfortunately, the existing
    sapstartsrv process does not evaluate "SETENV_XX" parameters in the profile for its own process environment.
  • The tool "enqt" requires Kernel patch from SAP Note 2778519 (749 patchno 710 from 03-May-2019,  753 patchno 416 from 26-Apr-2019)
    to become compatible with system/secure_communication = ON
  • Transport Management System (TMS) has a few issues with setting system/secure_communication = ON because it is running the tool "sapevt" with incorrect command line parameters.
    The ABAP correction from SAP Note 2785030 (26-Apr-2019) is supposed to solve this TMS problem.
  • If you try starting an NW 740+ Dual-Stack ABAP&Java or NW 740+ Java standalone system with profile parameter system/secure_communication=ON, then AS Java will become unable to communicate with the message server and enqueue server, and AS Java will fail to start.
  • If you erroneously add profile parameter system/secure_communication=ON to the profile of a old Netweaver 70x/71x/72x/73x system, then it will not have any effect on that old Netweaver system itself, because this profile parameter is unrecognized by any of the old Kernel components.  But 742+ versions of SUM (the SAP Upgrade Manager) will fail, if the parameter system/secure_communication=ON is present in the profile of an old system, because the newer kernel components included with SUM will falsely assume, and attempt to use TLS with SystemPKI to communicate with kernel components of the running old Netweaver system -- and fail poorly.
  • Setting the profile parameter INSTANCE_NAME = ERS$(SAPSYSTEM) for the instance of the enque replication server version 1 (ERS1) may cause fault while initializing the SSL library. The ERS1 developer trace dev_enrepserver points to wrong path loading the libsapcrypto library:
    [Thr 139890775922464] = SSL Initialization    platform tag=(linuxx86_64_gcc43)
    ...
    *** ERROR => DlLoadLib()==DLENOACCESS - dlopen("/usr/sap/DS1/ERS35/exe/libsapcrypto.so") FAILED
      "/usr/sap/DS1/ERS35/exe/libsapcrypto.so: cannot open shared object file: No such file or directory"  [dlux.c       550]
    [Thr 139890775922464] *** ERROR => secussl_LoadLibrary(): Unable to load "/usr/sap/DS1/ERS35/exe/libsapcrypto.so" [ssslsecu.c   609]
    [Thr 139890775922464] *** ERROR => Loading of SSL library failed -- NO SSL available!
    The value of the $(SAPSYSTEM) parameter is replaced with the value of the command line option "NR=<ASCS instance number>" at runtime for ERS1 process. Therefore, the parameter substitution should not be used for the parameter INSTANCE_NAME in ERS1 instance profile. Set the parameter to string representation of the instance name and the instance number, for example INSTANCE_NAME = ERS05.
  • old versions of SUM, and in particular older versions of SUM for Java contain a defect, and SUM might abort with a non-sensical error message buried somewhere in its Logfiles:
    "The following problem has occurred during step execution: com.sap.sdt.util.diag.DiagException: SUM has detected that the SystemPKI is supported by your system. To continue, you have to configure it as described in SAP Note 2200230"
    There exist *NO* user-configurable parts in SystemPKI, and it is unclear why SUM for Java is even looking for SystemPKI.
  • lightheartedly configuring TLS profile parameter values for enabling TLSv1.3 for an old Netweaver Kernel (older than 754 STACK 500) will also fatally break "Secure Internal Server Communication", in addition to breaking all outgoing and incoming TLS-protected communication via ICM with TLSv1.3 capable communication peers.
    this applies to Kernels 742, 745, 749, 753, 773, 781, 785, 789  plus all early Kernel versions of 754, 777 and 793.
    See SAP Note 3318423 for Information on Kernels for S/4HANA and Netweaver that have support for the substantially backwards-incompatible protocol version TLSv1.3.

 

How to Enable use of Secure Internal System Communication (by message server, enqueue server and gateway/gwrd):

  1. Stop all instances of the system (including e.g. ASCS, ERS) by using e.g. SAP MMC:  "Stop..." context menu on the system node or Command line tool: "sapcontrol ... -function StopSystem".
  2. Add the following profile parameter to the default profile DEFAULT.PFL of the system: system/secure_communication = ON
    (with a text editor at the OS level in directory $(DIR_PROFILE), since the system is down, and ABAP tx RZ10 not available at this point...)
  3. Restart of Start Services of all instances of the system by using e.g. SAP MMC: "All Tasks->Restart Service" context menu for each instance
    Command line tool: "sapcontrol ... -function RestartService".
  4. Start the system using e.g. SAP MMC: "Start..." context menu on the system node or Command line tool: "sapcontrol ... -function StartSystem".
  5. Verify that all instances are displayed "green" in SAP MMC or "sapcontrol ... -function GetSystemInstanceList".
  6. In case you are normally maintaining profile DEFAULT.PFL through ABAP tx RZ10, you should now update DEFAULT.PFL through tx RZ10 to contain the same value assignment system/secure_communication = ON,  which you added in step (2) with a text editor to file $(DIR_PROFILE)$(DIR_SEP)DEFAULT.PFL at the filesystem level, while your system was down/stopped.

A sufficiently recent version of sapstartsrv will automatically create a systemPKI and an instance-specific PSE (sap_system_pki_instance.pse) and allowing use of systemPKI-protected communication with sapstartsrv, even without the profile parameter system/secure_communication being defined.   All server components of the ABAP system are participating that PKI and receive an instance specific certificate with private key that they can use to authenticate and to encrypt their system internal communication with other backend server components of the same Netweaver AS ABAP system (same <SID>).  The profile parameter setting system/secure_communication = ON cause the message server and the enqueue server to require use of TLS to protect all communication on the internal message server communication port and enqueue server communication port.

  • The certificates are automatically renewed.
  • The root PSE file is stored in the secure store.
  • Instance PSEs (sap_system_pki_instance.pse) are cached in the filesystem, where they are encrypted with a strong random PIN that is memorized in the secure store as well.  

After starting a Netweaver AS ABAP 740+ system with parameter value system/secure_communication=ON, system-internal communication with certain backend server components (Message Server, Enqueue Server and Gateway/gwrd) of a Netweaver AS ABAP system (same <SID>) requires use of SSL/TLS with systemPKI certificates for encryption and authentication on their internal communication port.  Connection attempts without SSL/TLS & systemPKI will then fail (and vice versa).

To Disable use of Secure Internal System Communication (by message server, enqueue server and gateway/gwrd) you have to use the same steps 1.-5. as above, only removing parameter value "system/secure_communication=ON" from DEFAULT.PFL in step (2), instead of adding it.

 

BEWARE: You must *not* add, remove or change the parameter system/secure_communication within profile files underneath a live Netweaver system.  If you do, this may cause short-running Netweaver Kernel components such as "sapevt" use a different value for system/secure_communication as the value which the long-running message server process is using, resulting in a fatal communication failure between the message server client within "sapevt" and the message server.  Tools from the correction and transport system (BC-CTS-TMS) such as "tp" will be adversely affected if communication with the message server is no longer possible as a result of switching or adding/removing the profile parameter system/secure_communication=ON to profile files of a live running system.

Security Considerations

The usage of the secure internal server communication can replace use of ACLs for server ports of (purely) internal communication, which exist for some server components. See SAP Note 1495075.

Access Control Lists (ACLs) for external communication server ports, such as the RFC Gateway sec_info and reg_info are still required and used as they continue to control the external access to the system. See SAP Note 1408081.

 

Impact

The usage of SSL for secure internal server communication results in a higher CPU consumption and a modest performance impact.

External components that directly access internal ports of server components will no longer be able to connect to the system.
For the SAP tool SAPEVT, please check SAP Note 2000417.

AS Java instances that want to perform RFC communication w/ logon-balancing to an AS ABAP system *MUST* use the external message server port to talk to the AS ABAP message server.  If AS Java is erroneously configured to access the internal message server port of an AS ABAP system, then enabling secure internal server communication on the ABAP system will cause the AS Java RFC communication to fail.  The AS Java configuration must be fixed to access the external message server port of the AS ABAP system.   This AS Java configuration error is occasionally seen with Solution Manager 7.2 systems, a product based on Netweaver 740 AS ABAP and includes a seperate AS Java instance.
