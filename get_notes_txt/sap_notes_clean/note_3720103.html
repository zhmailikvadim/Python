Symptom

The SOA service "Request Utilities General Data Replication as Bulk" (documentation, technical name UtilitiesGeneralDataBulkRequest_Out) is used to replicate consumption related information per business partner for each utilities contract out of SAP S/4HANA Utilities to systems like SAP Emarsys and SAP Customer Data Platform. This SOA service is designed for full (initial) and manual replication, but does not support a specific mechanism to replicate updated data after utilities billing has created new billing documents. There is no utilities process that creates any trigger for this SOA service. This note describes how the general data can be kept updated in the target systems after the initial replication.

Solution

SOA service UtilitiesGeneralDataBulkRequest_Out is enabled to be executed in the Data Replication Framework in SAP S/4HANA via a replication model with outbound implementation ISU_GNRL (Utilities General Data). The execution of the replication for this replication model can be started via transaction DRFOUT. Different replication modus are available. After the initial replication, you can perform a manual replication at any time. To minimize the volume of replicated data, you should apply manual filter criteria so that only new and changed data are replicated. When new billing documents are created, updates for the related business partner should be replicated. Among the manual filter criteria, there is the bill creation date. An efficient approach consists in executing manual replication in DRFOUT once per month for a selection based on a date range from the first to the last day of the previous month for the bill creation date. You must ensure that all dates are at least considered once in the selection so that all relevant changes are replicated.

Usually, the new and changed data are not needed in realtime for the processes in the target systems. However, the frequency of manual replication can be reduced to weekly or even daily execution. The date ranges for the bill creation date should be then adjusted accordingly to avoid replication of the same data multiple times.
