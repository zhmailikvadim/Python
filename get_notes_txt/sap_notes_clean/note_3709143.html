Symptom

  • The system allows over-confirmation of Sales Orders (same material/plant combination) in parallel sessions of VA01/VA02, leading to incorrect ATP results and unreliable delivery commitments.
  • Negative cumulated ATP quantities are observed after removing confirmed quantities in VA02 without saving.
  • The field ACENQ in the table TMVF is empty, and the expectation is that the system should consider TMVFP-VFPSP (material lock).

Environment

  • SAP S/4HANA
  • Availability check
  • Available-to-Promise

Reproducing the Issue

  1. Create Sales Order XXXX and the confirmed quantity = 10;
  2. Change Sales Order XXXX from 10 to 6 in VA02 but do not save.
  3. Create Sales Order YYYY with confirmed quantity = 6 and save.
  4. Exit Sales Order XXXX without saving.
  5. Check CO09. Negative cumulated ATP quantity is observed.

Cause

The system automatically sets TMVF-ACENQ = 'X' during ATP checks in SAP S/4HANA releases, which is the expected standard behavior. This leads to over-confirmation and negative ATP quantities in parallel sessions.

Resolution

During the ATP check in the sales order, the field TMVF-ACENQ is automatically set to 'X', as shown in the debugging trace below:

15    FORM  TMVF_SELECT SAPLV03V    LV03VFT1 31
14    FUNCTION RV_REQUIREMENT_GET   SAPLV03V    LV03VU02 47
13    FORM  BEDARF_LESEN   SAPFV45V    FV45VF0B_BEDARF_LESEN   308
...
*  In S4/Hana only individual requiremens and Qty-Loc
  tmvf-sumau = chara.
  tmvf-sumlf = chara.
  tmvf-acenq = charx.

This behavior is expected in SAP S/4HANA releases, where the system automatically sets ACENQ = 'X'. In standard SAP, this field can no longer be modified, as explained in SAP KBA 2468512.

There is SAP Note 3027225, which introduces the following logic:

  IF tmvf-ADV_ATP IS NOT INITIAL.    " 3027225
    tmvf-sumau = chara.
    tmvf-sumlf = chara.
    tmvf-acenq = charx.
  ENDIF.                             " 3027225

With this change, if TMVF-ADV_ATP (advanced ATP) is not active, the system would no longer automatically set TMVF-ACENQ to 'X'.

However, SAP Note 3027225 was later identified as incorrect. As a result, SAP development team released another SAP Note that removes this logic:

3453131 – SAP system blocks material and plant

In conclusion, the correct and expected standard behavior is that the system always sets TMVF-ACENQ = 'X' for SAP S/4HANA Releases. If you would like to prevent this behavior, you would need to implement SAP Note 3027225 only, understanding that it deviates from the corrected standard behavior.
