Symptom

In the infotype Electronic Data Exchange (0700) with the subtype DBPV, existing data records cannot be delimited in terms of time. The system only allows you to change the existing data record, whereby the start and end dates must not be changed.

This leads to the following problem:

An employee is registered using the DaBPV notification procedure. The confirmation in the infotype Electronic Data Exchange (0700), subtype DBPV, contains correct data (e.g. B. Parent Property: No, Number of Children: 0, Payroll-Relevant: Yes). Children that cannot be reported using the DaBPV notification procedure (e.g. B. foster children). These are maintained in the infotype Family Members/Beneficiaries (0021) and the parent property is overridden using the SI attribute 31 in the infotype Social Insurance DE (0013).

To take these children into account in payroll to reduce contributions, the indicator Payroll-Relevant must be set to "Not Relevant for Payroll" in infotype 0700, subtype DBPV. Since a time-based delimitation is not possible, the change to the existing record would trigger unwanted retroactive accounting over the entire period.

In addition, the use of the buttons Use Data or Do Not Use Data when processing DaBPV responses so that all existing data records in the infotype Electronic Data Exchange (0700), subtype DBPV, are overwritten. In this case, too, unwanted retroactive accounting is triggered over the entire period, even if the difference in the number of children or parent property occurs only at a later point in time.

Other Terms

Social insurance, foster children, DaBPV, CI children

Solution

Implement the attached correction instructions or import the specified support package.

After the implementation, it is possible to copy data records in the infotype Electronic Data Exchange (0700), subtype DBPV. You can now delimit an existing record and change the Relevant for Payroll indicator as of a specific key date without triggering retroactive accounting for the entire period. However, it is only possible to copy the infotype record within the time limits of the original record. For example, bringing forward the start date is not allowed.

Post-Implementation Procedure:

  1. Call the infotype Electronic Data Exchange (0700), subtype DBPV, for the affected personnel number.
  2. Copy the existing data record and delimit it to the required start date.
  3. In the new data record, change the Relevant for Payroll indicator from selected to not selected (or vice versa).
  4. Save the data record.

In addition, the functions of the Use Data and Do Not Use Data pushbuttons have been adjusted. If data of the DaBPV procedure already exists in the infotype Electronic Data Exchange (0700), the system adjusts only entries that are after the date for which differences in the number of children or parent property occur for the first time. As a result, longer retroactive accounting is avoided.

Example:

A proactive confirmation from the BZSt is to be checked in the administrator list:

StartEndNumber of Children (Notification)Number of Children (IT 0700)Relevant for Billing
01.04.2025 31.07.2025 1 1 Yes
01.08.2025 30.09.2025 1 1 No
01.10.2025 31.01.2050 2 1 No
01.02.2050 31.12.9999 0 0 No

The number of children from the notification differs from the number of children saved in the infotype Electronic Data Exchange (0700) for the first time as of October 1, 2025.

If the administrator chooses the Do Not Use Data pushbutton, only the data records as of October 1, 2025 are saved with the deactivated Relevant for Payroll indicator. The records before this date (01.04.2025 – 30.09.2025) remain unchanged. Retroactive accounting over the entire period is therefore avoided.

In addition, after the implementation, it is possible to create and edit data records in the infotype Electronic Data Exchange (0700), subtype DBPV, using batch input.

After you import the HR Support Package or implement the correction instructions, you do not have to perform any further follow-on activities.
