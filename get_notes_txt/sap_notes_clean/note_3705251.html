Symptom

As of the SAP Commerce 2211.47 / 2211-jdk21.5 patch release, deprecated password hashing encoders were removed. Customers who upgraded without first rehashing existing user passwords are experiencing post-upgrade login failures.

The password reset functionality will fail due to trying to read the old hash. This is documented as known issue in the release note:

Upgrading to 2211.47

Upgrading to 2211-jdk21.5

Cause

It was initially introduced by Improved Security of Password Handling feature, rolled out in version 2211.28. With the release of 2211.47 and 2211-jdk21.5, the Deletion of Deprecated Password Hashing Algorithms was announced.

There is password.encoding.auto.update.enabled property to aid in automatic password rehashing. Once the property is set to true, user passwords are automatically rehashed upon login. 

Resolution

It is recommended to bypass 2211.47 / 2211-jdk21.5 (due to the password reset issue), and update to 2211.48 / 2211-jdk21.6 directly. Please refer to the release note of 2211.48 and 2211-jdk21.6.

Pre-upgrade:

  • Please make sure password.encoding.auto.update.enabled is set to True in your production environment as early as possible to allow a grace period for users - including storefront users and backend users - to log in and rehash passwords, before updating your system.

You may run the below sql query in your HAC to confirm the status of the password re-hashing:

select p_name, p_uid, p_passwordencoding from users where [your_condition];

Particularly the critical users like admin and anonymous.

(if the passwordencoding column is shown as argon2, it means it's rehashed successfully)

It is also suggested to set toggle.disallowTaeEncryptionFallback.enabled=true to prevent security breaches that may occur if the system accepts potentially compromised plain text values for TAE-encrypted attributes or uses the plaintext-based initialization in creation of system users: administrator and anonymous.

Post-upgrade:

If there are remaining users who haven't logged in before the update.

  • Storefront(customer-type) users: can use the “Forget password” process to reset the password.
  • Backend(employee-type) users: the password needs to be manually reset by the administrator role. There are no self-service options available.
  • Admin user can use the Generating New Passwords in Cloud Portal by Customer System Administrator (CSA) role.
