Requirements Gathering and System Design

The fields that are part of the standard integration, and where they map to, are documented in the SuccessFactors / WorkForce implementation guide here

Appendix A covers the default field mappings from the XML fields to the fields inside of WT&A. It is important to note that this interface can be used for non-EC systems as well, just be mindful of the field mapping. There is no decoder for the SOAP Import. The default mappings from the guide are applied first. Then, afterwords, you can use the PPI_E2G_HR_IMPORT PPI Employee Import Configuration policy to configure mapping additional generic fields or doing additional configuration.
The policy mapping screen still applies all of its mappings.
It is also possible to do multiple assignments with this interface.
Logic around effective dating from the CSV import equally applies to the web services import.
The E2G_importEmployees section of the Web Services Specs guide also has useful information.

There is a wealth of information in that section as well, including how to generate XML that will produce new hires, re-hires, changes, and terminations of employees.
If integrating with the WFS Suite and standard or universal scheduling, make sure that if the TA_SCHEDULING field is false, that the ASSIGNMENT.SCHEDULE_TEMPLATE field is forced to be EMPTY or null. This will ensure the schedule template engine in WorkForce T&A does not create schedules.

The SAP Integration with EC is a full historic import. It will only send employees with changes. What this means is that the full history of an employee is sent each time that employee has any change at all in their data. This setup allows the employee import to import rates as a retro trigger field. This has to be configured and is not default behavior. If not using SAP, this import can be configured to be partial history.

The import supports split assignments, but updates cannot be made to employees or assignments from before the latest split date.


If you need to default a language for employees - set up a locale policy using the policy mapping screen.

Please read the File based Employee Import guide for more details – and review the linked articles.

Instructions for creating user only employees with this import are in this article: SOAP/SuccessFactors User Only Import
 

Schedule Cycle Integration

Schedule templates can be assigned to assignments by the built in employee import support on the policy mapping screen. For assigning schedule cycles to employees it is required to use a separate script running on a regular schedule to assign them.  This can either be from a file or from data fields that are imported as part of the employee import.  For more information, see the article on Schedule and Cycle Assignment Imports  

NOTE: If using this method, the script should be run on a regular schedule, not after each employee import. It assigns cycles to all employees so it could cause performance issues if the customer sends rapid fire employee changes one after another.



Configuration

The Web services specs guide, as mentioned in the previous section covers how to add custom fields. In general the configuration all takes place within the PPI_E2G_HR_IMPORT PPI Employee Import Configuration policy.

The SOAP/EC import has not been transitioned to using the PPI Data Connector policy, so at this time the E2G_HR_IMPORT_CONFIG Configuration Javascript library is still required to set up the data connector options.

If you need to default a language for employees - set up a locale policy using the policy mapping screen.

There is no way at this time to get the full batch job log emailed to an email address. There are 2 possible workarounds. 

1. Using a script in the Post Import hook to email the information available in that part of the script. This will allow an email address to get the number of errors/warnings and the number of employees processed, even if the full log is not sent.  See below:
//Distributed Libraries
includeDistributedPolicy("EMAIL_API");
//Set email properties
var emailFormat = 'PLAIN_TEXT';


  var emailRecipient = ""; // set recipient name
  var emailSenderName = 'WorkForce Time and Attendance';
  var emailSenderEmail = ''; //set email address that workforce is to use when sending the mail
  var emailSubject = 'Employee Import Results'; // can do things like add a WFSDateTime here if required. This is the email subject line for the information emails.
  //Set email body
  var stats = transferStatus.aggregatedStats;
  var emailBody = "Employees with validation errors: " + stats.employeesWithValidationErrors + "\n" +
                  "Employees with system errors: " + stats.employeesWithSystemErrors + "\n" +
                  "Employees with warnings: " + stats.employeesWithWarnings + "\n" +
                  "New employees: " + stats.newEmployees + "\n" +
                  "Discarded employees: " + stats.discardedEmployees + "\n" +
                  "Updated employees: " + stats.updatedEmployees + "\n" +
				  "Any Additional Error Info: " + stats.additionalErrorMessage + "\n" +
                  "Total number of employees processed: " + stats.totalEmployees;
  //Send email
  // could have this only send emails if errors or warnings, if desired
  Email.createAndSendEmail({
                   recipients: Email.createRecipientList(emailRecipient, "TO"),
                   subject: emailSubject,
                   body: emailBody,
                   format: emailFormat,
                   senderName: emailSenderName,
                   senderEmailAddress: emailSenderEmail
  });
2.  Using the E2G_getJobStatus SOAP web service. This will allow pulling the job log information. The E2g_importEmployee service that is used for SuccessFactors and other SOAP clients will return a job log when it is called. The E2G_getJobStatus service can be used to retrieve the job logs or a more abbreviated information with just the number of errors and the job status. For more details please review the WTA Web Services Specs Guide

 

Split Assignments

If you are configuring the SuccessFactors import or any other full historic import and you have multiple different policy profile groups/split assignments - you will need to wrap any assignment-specific mapping logic in the source-to-stage hook in a check against the asgnmtRecordWillBeProcessed flag, and to only execute that logic when that flag is true. 

 

if(asgnmtRecordWillBeProcessed ) {
// mapping logic that uses assignment data
asgnmtStage.POLICY_PROFILE = policyMappingApi.evaluate("POLICY_PROFILE", employeeStage, asgnmtStage);
}

 


The asgnmtRecordWillBeProcessed flag allows the script logic to check whether or not the assignment record is a dummy record or an actual  record, and to allow logic that requires certain mappings to have been applied to only be executed against the actual data and not the dummy data

When a split assignment exists already, the import filters out any incoming source data that is effective on or before the split date of the most recent split for the assignment. However, the employee data doesn't get filtered out, and potentially still needs to be evaluated. Since both employee and assignment data are provided in the source-to-stage mapping hook, this means that dummy assignment data needs to be placed in scope that doesn't actually get written out but still allows operations to be performed against it to avoid causing catastrophic mapping script issues

However, that dummy data is dummy data, so it doesn't have anything actually mapped to it. Adjusting the policy mapping screen may be necessary in order to ensure that a policy profile exists for these employees to be mapped to. 

If you have a customer that may make retroactive changes to pay frequencies (ie, they need retroactive split assignments). You should know that this is not supported by WorkForce Time and Attendance. One possible solution is to:

  1. Create a new custom field in SF Employee Central to hold the WFS policy profile. This field should be able to be manually changed and not have any affect on rules or calculations within EC. This field will need to be updated when employee records change if the employee is to be moved into a new policy profile.
     

  2. Use the policy mapping screen in WT&A to only do a simple 1-1 mapping from the policy profile field in SF Employee Central to WT&A.

  3. Create a business rule in SF Employee Central to elicit a warning if someone changes their pay frequency in a prior period.

  4. If someone is realized to change in a prior period, ensure that their policy profile for that prior period is manually set to their old one to allow importing of the retroactive data

  5. For that retroactive period(s), manually make pay adjustments




This will allow the employee data to continue to flow from SF to WFS without interruption. 

Frequency for the Packaged Integration with SAP

Because this is a web service endpoint exposed by WT&A, the frequency is not controlled by the T&A system. However, the scheduling recommendations still apply. Best practice is to send data once an hour, and only the deltas at that frequency.  A full load can be sent as a 'true up' on a weekly basis. 

 

Testing

It is extremely important to test this interface with a reasonable data volume. You will want to test to ensure that you get realistic data volumes from SAP. If you have concurrent employee or contingent workers, those should be part of testing as well. They should be terminated and re-hired as part of testing.

Go-Live and Support

Go live is similar to the file based import. Ensure that your go-live date and any exceptions are set up properly. The GLRB and Enablement Services will want to see a similar number of employees in the TEST environment as will be in the system on go-live.
