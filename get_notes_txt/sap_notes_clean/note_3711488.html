Symptom

  • By performing a schedule to File System or Program object is noticed that NTLM is being used.
    • In the Wireshark traces, by searching for "ntlmssp" it can be seen that the NTLMSSP_AUTH is being used.
      • To get the above traces, perform a Wireshark on the machine that is running the AdaptiveJobServer.
  • When auditing NTLM information at the windows level - it may be possible to see NTLM authentication happening
  • EventID 4776 seen in the Windows Event Viewer tracing

Environment

  • SAP BusinessObjects Business Intelligence Platform 4.x / 2025

Reproducing the Issue

  1. Schedule a report or object to the File System or a Program Object, and it fails.
  2. NTLM is currently disabled in the server.
  3. In the Wireshark traces, it was seen that NTLM is being used instead of Kerberos.

Cause

When the system attempts to delegate credentials to the OS, it uses the LogonUserW API. The root cause of the malfunction is an incomplete username format at the BI level:

  • For Kerberos (Negotiate) to function, the domain name must be explicitly provided.
  • If the username is provided without a domain (e.g., just USERNAME), the API receives a NULL domain parameter.
  • Per Microsoft documentation, passing a NULL domain forces the system to default to NTLM.
    • https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-logonuserw

Resolution

To solve this behavior when scheduling to a File System or Program Object, you must ensure the account is configured to use the full domain format.

In the Destination settings of the schedule, update the User Name field to include the domain:
DOMAIN\USERNAME

Adding the DOMAIN\ prefix allows the system to use the Negotiate/Kerberos protocol. This correctly delegates the logon session, providing the permissions required to write to the file system or run the program.

This is the only requirement from the BI side, it is up to the OS what protocol is used afterwards, so long as the username is provided in DOMAIN\USERNAME format then the OS will attempt Kerberos authentication, if that fails it will fall back to NTLM. If after providing the correct username format NTLM authentication is still being seen - it must be investigated by the OS administrators.
