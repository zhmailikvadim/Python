Symptom

We are using OCM and table ACDOCA have fields RA_CONTRACT_ID and RA_POB_ID, however there is nothing getting populated when we check AWTYP = FARA (postings from RAR)

Environment

Revenue Accounting and Reporting

Reproducing the Issue

  1. Execute SE16N transaction to check table ACDOCA .
  2. The fields RA_CONTRACT_ID  and RA_POB_ID are not populated.

Cause

Check if using Aggregated posting mode

Resolution

It is working as designed.

The fields RA_CONTRACT_ID and RA_POB_ID in ACDOCA are not always populated. They are utilized only in the case of so-called Direct Posting. Please understand that for OCM contracts, SAP offers two posting modes:

•    Aggregated Posting: This is the original way of posting. The data is prepared in the RAR sub-ledger table FARR_D_POSTING. When you execute posting programs A, B, and C, the data is posted into FI/GL. The resulting accounting document aggregates FARR_D_POSTING data, but CONTRACT_ID and POB_ID are not part of the aggregation dimensions. Therefore, data from many contracts/POBs results in a single ACDOCA line, leaving those fields empty.

Please find more details in this help: https://help.sap.com/docs/SAP_S4HANA_ON-PREMISE/f63dd39a28bb4b90adbf9e608aff58ea/ff466e55e4fd7173e10000000a44538d.html


•    Direct Posting: This is a newer posting mode. It must be explicitly turned on in customizing (Transaction IMG -> Revenue Accounting -> Revenue Accounting Contracts -> Define Posting Mode for Company Code). Postings are made directly into ACDOCA when a specific event happens (e.g., fulfillment, invoice correction, etc.) without the need to run all three programs (A, B, C). Because these postings are done at the contract/POB granularity, the RA_CONTRACT_ID and RA_POB_ID fields are populated.

Please find more details in this help: https://help.sap.com/docs/SAP_S4HANA_ON-PREMISE/f63dd39a28bb4b90adbf9e608aff58ea/3f4af84dcf0f452aa0804e04c588b689.html

Notes :

  •  It is important to know that if you decide to switch to this mode, it will only apply to contracts created after the switch. The posting mode is locked into the contract at the time of creation and is used for the entire contract lifetime, regardless of subsequent customizing changes.

  •  Posting at the contract/POB granularity hugely increases the number of document lines and accounting documents. Therefore, we do not recommend using it in systems with a large data footprint (e.g., hundreds of thousands of contracts) due to performance reasons. So, if you decide to switch on, then you should consider also this aspect.
