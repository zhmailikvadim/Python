Symptom

During a tenant or database restore using BACKINT with the AWS Backint Agent, the recovery starts normally and backup.log shows data recovery activity, but the restore does not complete and no database error is raised.

Typical backup.log sequence:

  • RECOVERY state of service: ... RecoveryExecuteDataRecoveryInProgress
  • services (for example indexserver or other database services) enter recovery phase
  • RECOVERY state of service: ... RecoveryLogInitializationStarted
  • afterwards no further progress entries appear

The database remains in recovery state and the restore appears “hung”.
SAP HANA database traces may not contain a direct database error because the database is waiting for the external BACKINT provider.

In comparison, restores performed from a source system located in the same AWS account complete successfully.

Environment

SAP HANA DB Version 2.0

Cause

The behavior is typically not related to SAP HANA recovery logic but to the external BACKINT storage layer.

When SAP HANA performs a restore using BACKINT, the database depends on the external backup storage provider. In environments using the AWS Backint Agent, the database waits for the storage layer to deliver data from Amazon S3.

If the source backups are stored in a different AWS account, insufficient cross-account permissions (for example missing S3 GetObject and/or ListBucket permissions, missing role trust, or missing KMS access) can prevent the AWS Backint agent from retrieving the backup objects.

In this situation the storage request does not fail immediately at database level. Instead, SAP HANA remains waiting for the external component, which results in a restore that stays active but makes no progress and produces no database error.

This situation can lead to scenarios where restores from the same AWS account succeed while restores from a different AWS account remain waiting for data retrieval.

Resolution

Verify and correct the AWS Backint and S3 configuration for cross-account restore.

  1. In the source account (where backups are stored):

    • Grant read access to the target system IAM role in the S3 bucket policy

    • Ensure permissions include at least s3:GetObject and s3:ListBucket

    • If encryption is used, allow KMS decryption permissions

  2. In the target account:

    • Verify IAM role trust relationship (AssumeRole) with the source account

    • Confirm the role used by the HANA host is the one referenced in the bucket policy

  3. Verify AWS SAP integration components on the HANA host:

    • systemctl status aws-dataprovider

    • review logs in /var/log/aws-dataprovider

    • rpm -qa | grep aws-sap

    • confirm Backint version with hdbbackint -v

  4. After permissions and services are corrected, repeat the restore operation.

If required, involve the AWS team or the Backint provider to validate S3 policies and IAM roles.

Keywords
