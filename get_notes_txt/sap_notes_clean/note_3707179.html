Symptom

This article outlines the accessibility improvements implemented on the career site during the 1H 2026 release. These updates aim to enhance user experience and ensure compliance with accessibility standards. 

Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental

Environment

  • SAP SuccessFactors Recruiting Management
  • SAP SuccessFactors Recruiting Marketing

Resolution

Cookie Banner Issues:

  • The buttons in the footer of the Cookies Consent Manager modal should be accessible with a keyboard (RMK-37003):
  • The Close button on cookie banner should have translations in all languages (RMK-37008):


Candidate Profile Issues:

  • Buttons that expand and collapse all partitions have an aria-expanded attribute that doesn't belong to buttons (RCM-158591):
  • When hitting the space bar on "Remove" for a second background element entry, the focus jumps to another field in the first background element. Focus instead should be moved to the next focusable element (RCM-158546):
  • The Remove and Add options in these background elements are read aloud on a screen reader in English even if the language is different so if language is in French, then the screen reader should also read these buttons in French (RCM-156556):
  • Making a mistake, e.g. entering an invalid email address, displays an error but the error points to field ID instead of field label. The error should point to field label instead of field ID for clearer instructions (RCM-157087):
  • When withdrawing an application, screen readers relay irrelevant information to end users (RCM-158612)
  • When uploading a resume or other document on candidate profile, there is a pop-up with options, which are not complaint for accessibility as the focus indicator is not meeting the standard 3:1 contrast so this will be fixed (RCM-158742):
  • The drop-down arrow on picklist fields contrast is increased to 17px (RCM-158867):
  • Unique labels need to be added to #skipContent span or the #rcmJobApplicationCtr div (RCM-158897)

Create Account Page/Quick Apply Issues:

  • The aria-pressed attribute has been added to the button that displays the password, which tells you whether the button has been pressed or not but because the button has a different name when pressed (Hide vs. Show), this mainly causes confusion. Therefore the aria-pressed should be removed (RCM-158571):
  • On the create account page/quick apply page, there is a radio button group "Make My Profile Visible to:" where the aria-required attribute has been added to the group element (role="group") with which that attribute is not allowed. The fix is to change the role="group" attribute to role="radiogroup", which is what aria-required is supported (RCM-158658):
  • A role="menuitem" attribute has been added to the options in the "Country/Region of Residence" drop-down menu but these should be removed (RCM-157943):
  • Screen Readers do not accurately read pre-screening questions. If there are two questions with 2 options each, the screen reader will indicate that there are 4 options - it will not tell that there is 2 checkboxes for the first question and another 2 checkboxes for the second question (RCM-158638):
  • Instruction/heading fields not marked as headings (RCM-158773):
  • When accepting DPCS statement, user should get feedback from screen reader that action was successful (RCM-158623):
  • An error message should be added next to the Data Privacy statement link to announce to reader if they have not selected a country: RCM-158933

Job Posting Page Issues:

  • Ensure the visual main heading of job posting is programmatically marked as the primary heading (<h1>): RMK-37075.

Unified Data Model Issues:

  • Role "application" is to be removed from the code to restore normal screen reader behaviour (RMK-37036):

  • On the mobile career site, the accessible names of some of the buttons are in English in all languages but they should be in all languages e.g. when the career site language is French but the accessible names are still in English (RMK-37009):

  • For the same buttons above, the screen reader only reads out the first character of each button but the full accessible name should be read out (RMK-37010)

  • The aria-live attribute has been added to the entire search results area in two different elements, which causes the entire area to be read (from Sort By to the end of the pagination) with a screen reader always. Screen reader announcements should be concise and relevant, informing the user what changed (e.g., "12 results found"). Only the necessary feedback (such as updated result count) should be announced, not the entire content area (RMK-37049)

  • Issue with sort controls on UDM on mobile (RMK-37002):
    • Sorting options should be fully operable using the keyboard.
    • Screen readers should announce the accessible name of each sorting option clearly.
    • The Close button should display a visible focus indicator when focused.
