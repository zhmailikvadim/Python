Symptom

  • Error log reports: "Current process infected with signal 11 (SIGSEGV)" in module 'pchk_hash_search' during concurrent DBCC CHECKSTORAGE.
  • Issue observed after upgrade to ASE 16.1.
  • A typical log sequence includes:

    Current process infected with signal 11 (SIGSEGV) at 0x0x0000000001556116 (pchk_hash_search+0x96)

    with stack functions
    pchk_hash_searc
    pchk_page_row
    pchk_ndx_page
    pchk_page
    pchk_all_pages
    pchk_scan_work
    pchk_scan_main

    Error: 9970, Severity: 20, State: 1
    DBCC cannot update the finish time in dbcc_operation_log table for this operation(opid = '13') of database 'data_govcorp'. This can be patched by executing sp_dbcc_patch_finishtime.

    Address 0x0x000000000107ac08 (prPCHK_INDEX_CHECK_IND+0x93)
    Current process (0x77d03bf) infected with signal 11 (SIGSEGV)
    SQL Text: insert dbcc_operation_log(dbid, opid, optype, suid, start, seq, id) values(16, 13, 2, 1, getdate(), null, null)
    curcmd = 317 (DBCC_CMD) program = isql

    with stack functions
    prPCHK_INDEX_CHECK_IND
    pchk_get_all_rids
    checkstorage

Environment

  • SAP Adaptive Server Enterprise(ASE) 16.1

Reproducing the Issue

  1. Use ASE 16.1 SP00 PL01.
  2. Run multiple DBCC CHECKSTORAGE commands on different databases concurrently.
  3. Observe error log entries reporting "Current process infected with signal 11 (SIGSEGV)" in pchk_hash_search.
  4. Run DBCC CHECKSTORAGE for a single database; the issue does not reproduce.

Cause

The issue has been addressed under SAP CR 831894. Its description reads, "The message, "Current process infected with signal 11 (SIGSEGV)", in the module 'pchk_hash_search' together with a stack trace may be reported in the SAP ASE error log when multiple DBCC CHECKSTORAGE commands are executed concurrently."

Resolution

  1. There is currently no target Fix Pack or Service Pack scheduled for this issue. It is being reviewed by our Product Group.
  2. SAP CR 831894 is scheduled to be addressed in a future SAP ASE Service Pack. Please mark this KBA as a Favorite to be automatically notified when the CR is fixed. 
  3. As a temporary workaround to avoid the SIGSEGV, do not run multiple DBCC CHECKSTORAGE commands concurrently; run them sequentially or reduce concurrency/parallelism.
  4. If a SIGSEGV occurs, enable dump on conditions and configure a shared memory dump for signal 11, then collect and provide the dump for analysis:
  • sp_configure 'dump on conditions', 1
  • go
  • sp_shmdumpconfig "add", signal, 11, 1, "/path/to/your_dump_dir", null
  • go

The solution of this issue is currently under investigation. Please follow this KBA for updates.
