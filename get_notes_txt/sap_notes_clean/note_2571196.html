Symptom

  • Your SAP Enterprise Inventory and Service-Level Optimization (EIS) run fails.
  • The EIS Application Log contains the following error: There are stocking points that have no parent node.
  • You see the following message(s) in the EIS Alert Log: A stocking point does not have an incoming supply path or has multiple incoming supply paths from process points and other nodes. Environment

SAP Enterprise Inventory and Service-Level Optimization (EIS)

Cause

STG_SOURCING records are either incorrectly defined or missing for the item and location specified in the Alert Log.

Resolution

  1. Verify that STG_SOURCING data exists for the item ID and location ID reported in the Alert log in the Stocking Point field.
  2. Check if the sourcing records for the item ID and location ID have been rejected. 
    1. To do this, you can either review the reject_log table entries for the inbound_gateway_queue_key or download the "Data Loading Errors" published report from the UI. Note that the published report is generated as a .csv file. It may be best to open this with a tool like Notepad rather than Excel.  The report can be large and Excel may not display all of the data.
    2. Check for an error description that states "STG_SOURCING_DATES ARE OUTSIDE THE DATA HORIZON." for the item ID and location ID.  If this is found, the sourcing record was rejected and no supply path will exist for this data in SOPS_GATEWAY or SOPS_MIPO.  This rejection occurs when the dates in the STG_SOURCING table are incorrect. 
  3. Validate the STG_SOURCING dates.
    1. Verify that the from_date is earlier than the max valid date for your data horizon.
      The max valid date can be determined by running the following query in SOPS_GATEWAY:
      select max(valid_date) from stg_valid_dates;
    2. Verify that the to_date is greater than or equal to the base period.  The base period can be found on the Manage Scenario screen in the UI or the Inbound Gateway Queue. IMPORTANT: If the STG_SOURCING.TO_DATE shows as '99' in SQLPlus, view the 4 digit date to ensure it is not set to 1999.  If it is 1999, you must update it to a valid date.
