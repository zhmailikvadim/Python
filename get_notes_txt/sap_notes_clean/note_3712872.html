Symptom

  • Trusted Authentication (eg SAML, Query String, TrustedVintela) may fail with the following error:
    Account information not recognized: Enterprise authentication could not log you on. Please make sure your logon information is correct. (FWB 00008)
  • In the CMS logs, one of the below errors are seen:
     
    HTTP client error: 28: Timeout was reached
    HTTP client error: 60: SSL peer certificate or SSH remote key was not OK
    HTTP client error: 35: SSL connect error 
    HTTP client error: 58: Problem with the local SSL certificate
    HTTP client error: 56: Failure when receiving data from the peer

  • Sometimes, the secret may be pushed with a non-200 HTTP error code (Eg from high level CMS logs)
    (TrustedAuthMgr.cpp:186) TrustedAuthMgr: secret pushed to https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet: code 403

Environment

  • SAP BusinessObjects Business Intelligence (BI) Platform 4.3 SP04 Patch 1000+, 4.3 SP05+, 2025
  • Windows
  • Linux / Unix 

Reproducing the Issue

  1. Run a Trusted Authentication workflow
  2. Observe it fails with the errors mentioned in the symptoms

Cause

The Trusted URL - either configured in the global.properties (trusted.auth.url) or set automatically by the web application server is not reachable from the CMS host

In the new Trusted Authentication, the CMS sends the secret via HTTPS back to the web application server. If this fails, Trusted Authentication fails

Resolution

There are multiple requirements:

  • The HTTPS URL configured in global.properties (trusted.auth.url) must be reachable from the CMS host
  • The HTTPS certificate issuer configured for this URL must be trusted by the CMS in the cacerts.pem
  • The cacerts.pem must be readable by the CMS
  • The HTTPS certificate configured for this URL must match the hostname being accessed (either in the CN or SAN)
  • The HTTPS certificate must be in date / valid - not expired

Testing CURL outside of BI:

CURL can be used independently of the CMS / BI, by using the following command on the same machine as the CMS is located
Windows (command line):
curl -v -X POST --ssl-no-revoke  --cacert <PATHTO>/SAP BusinessObjects Enterprise XI 4.0/win64_x64/config/cacerts.pem https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet 

Linux:
curl -v -X POST --cacert <PATHTO>/sap_bobj/enterprise_xi40/linux_x64/config/cacerts.pem https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet 

Note: If a proxy is configured, eg when HTTPS_PROXY environment variable is set at the OS level, then add -x <proxyhost>:<proxyport> to the command eg:
curl -v -X POST -x proxy:3128 --ssl-no-revoke --cacert <PATHTO>/SAP BusinessObjects Enterprise XI 4.0/win64_x64/config/cacerts.pem 

Specific errors, causes, solutions

HTTP client error: 28: Timeout was reached

Cause: The Trusted URL - either configured in the global.properties (trusted.auth.url) or set automatically by the web application server is not reachable from the CMS host
Solution:

  1. If a load balancer is in use, it's recommended to upgrade to the fixed versions from:
    3646222 - Trusted Authentication not working with load balancer
  2. Ensure Tomcats/webapp HTTPS url is configured in the <INSTALLDIR>\tomcat\webapps\BOE\WEB-INF\config\custom\global.properties file:
    eg:
    trusted.auth.url=https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet 
    Note: This must be the direct URL for the tomcat or web application server - NOT a load balancer
  3. Ensure the URL set here is configured as trusted in CMC -> Applications -> Authorized URL configuration
  4. Try to access this URL outside of BI from the CMS machine, eg via a browser or CURL directly:
    https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet
    If it works, a HTTP 405 will be seen - this is normal
    The following command can also be used from the CMS host (using the information from the URL)
    telnet <HOST> <PORT>
  5. If a proxy is required to access the URL from the CMS then this can be configured using the OS level environment variable:
    HTTPS_PROXY
    The SIA must be restarted after setting this
  6. If the telnet / browser tests fail, the issue must be investigated with the responsible network team

HTTP client error: 60: SSL peer certificate or SSH remote key was not OK
HTTP client error: 35: SSL connect error

HTTP client error: 56: Failure when receiving data from the peer

Cause: When performing the HTTP request to the trusted auth URL https://<HOST>:<PORT>/BOE/CMC/TrustedAuthSetTokenServlet, a non-200 return code is sent back. 
In some environments (such as ECS), this can be caused because the HTTPS_PROXY environment variable is configured (so CURL/CMS uses it), but the proxy configured does not allow the access to the specific URL configured (returns HTTP 403)
Solution: Exclude the URLs which should not be forwarded to the proxy
In Windows

  1. Add a new system environment variable at the OS level (on the CMS nodes):
    NO_PROXY ".domain.name,localhost,127.0.0.1"
    Where .domain.name is the domain name of the URL configured for trusted authentication
  2. Restart the SIA

In Linux

  1. Add a new system environment variable in the profile of the user that starts the SIA:
    ex for csh:
    setenv NO_PROXY ".domain.name,localhost,127.0.0.1" 
    ex for bash:
    export NO_PROXY=".domain.name,localhost,127.0.0.1" 
    Where .domain.name is the domain name of the URL configured for trusted authentication
  2. Restart the SIA
