Symptom

When updating HR Mini Master employee data in S/4HANA through an integration (such as from Workday via middleware or interface), the interface currently transmits all infotypes (e.g., PA0001, PA0002, PA0006) regardless of whether data in those infotypes has been changed.
Customers want to only send and update infotypes with actual data changes (for example, only send PA0006 when an employee’s address changes) and seek clarification on:

  • Whether selective infotype updates are supported (i.e., sending only changed infotypes)
  • Whether there are standard SAP APIs or BAPIs to update only selected infotypes in HR Mini Master
  • Whether other data or infotypes not included in the payload will be lost or unintentionally cleared in HR Mini Master or in the connected Business Partner record

Environment

  • SAP S/4HANA On-Premise
  • SAP S/4HANA Private Cloud Edition

Reproducing the Issue

  1. Set up interface to update HR Mini Master data in S/4HANA (e.g., via middleware integration from Workday or another system).
  2. Configure the interface to send only one infotype record (e.g., PA0006 Address) with changed data, omitting others (e.g., PA0001, PA0002) in the transmission.
  3. Observe the outcome in S/4HANA after the iDoc is processed:
    • Confirm whether only the sent infotype is updated.
    • Check if any omitted infotypes are unintentionally cleared or affected.

Cause

The HRMD_A ALE iDoc process in SAP S/4HANA is designed to allow for selective infotype updates. Only the infotypes and subtypes sent in the iDoc are processed and updated; infotypes not included in the payload remain unchanged in the HR Mini Master.

Resolution

  1. Use the class CL_HRPA_MASTERDATA_BL with the method of interface IF_HRPA_MASTERDATA_BL for updating employees with delta changes.
  2. For initial creation of employee records, send info types 0000, 0001, 0002, 0003, and 0006 (subtype 1) to create the employee’s HR Mini Master and Business Partner.
  3. For ongoing updates, send only the info types that have changed. For example, when the address changes, send only info type 0006 with the new data.
  4. Suppress segments and fields for info types with no changes in the middleware so they are not transmitted at all.
  5. Avoid sending segments with empty fields for unchanged data, as blanks can be interpreted as overwrite or deletion.
  6. Maintain the distribution model (BD64) and partner profiles (WE20) for HRMD_A ALE iDocs.
  7. Activate change pointers for delta processing by enabling BD61/BD50 and scheduling RBDMIDOC, so only changed info types are collected and sent.
