Symptom

Monitoring tools (for example Avantra) or cluster logs report a failover or takeover in an SAP HANA high-availability environment.
Pacemaker logs contain messages similar to:

Choosing rsc_SAPHana_<SID>_<Instance>:0 (Master) on <hostname> for promotion

However:

  • SAP HANA System Replication state remains unchanged
  • Primary and secondary hosts remain unchanged in SAP HANA monitoring tools (for example SAP HANA Cockpit).
  • Cluster resource roles remain unchanged 
  • Cluster status still shows the same master and slave nodes

Administrators are unsure whether a real HANA failover occurred.

Environment

SAP HANA DB Version 2.0

Reproducing the Issue

  1. Monitor the SAP HANA cluster solution using the monitoring system.
  2. Observe that the monitoring system reports a failover from Node A to Node B.
  3. Check the roles and replication status using SAP HANA monitoring tools (for example SAP HANA Cockpit).
  4. Review the logs indicating "takeover occurred to Master."

Cause

The message originates from the Pacemaker scheduler for a promotable SAPHanaSR resource.
It reflects an internal cluster scheduling decision and does not indicate a database role change.

This cluster promotion confirmation is not an SAP HANA System Replication takeover and does not trigger a database role switch.

A SAP HANA takeover results in a change of the System Replication roles (primary/secondary).

Resolution

No corrective action is required.

Verify the actual SAP HANA System Replication status using:

  • cluster status (crm_mon)
  • SAP HANA System Replication state
  • M_SYSTEM_REPLICATION view 

If the primary and secondary roles are unchanged, the database has not performed a takeover.

Do not initiate manual failover actions based solely on Pacemaker promotion log entries.

A real HANA takeover is typically visible in:

  • Pacemaker resource role change
  • SAP HANA System Replication state
  • SAP HANA monitoring tools (for example SAP HANA Cockpit or SQL monitoring views)
