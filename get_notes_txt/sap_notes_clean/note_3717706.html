Symptom

When using multiple OAuth client IDs for different login pages which uses the same public endpoint, the OAuth client ID remains the same during the login flow, causing the login process to fail.

Environment

  • Platform: SAP Commerce Cloud
  • Version: JDK21.1 or higher
  • SAP Composable Storefront

Reproducing the Issue

1. Configure two separate custom login entry points in the storefront and ensure both login entry points are served from same storefront endpoint.

For example:

  • One intended for standard users

  • One intended for marketplace 

2. Setup the different OAuth Client for one single public endpoint for SAP commerce Composable Storefront. Refer SAP Help Portal - Configuring OAuthClients page for setting up as shown below. ie) One for standard login page which would be mobile_android_public and other for marketplace page mobile_android_marketplace

3. Now login to standard login endpoint and observe that the login completes successfully 

4. But accessing the secondary login page (eg: marketplace) fails as it redirects to the standard login flow as it does not retain the intended OAuth Client which can be seen from below screenshot 

Error logs show 302 redirect status when reproducing the issue :

"origin":"access-log","timeMillis":1771427853115,"thread":"hybrisHTTP22","contextMap":{"remoteHost":"10.245.1.34","remoteUser":null,"requestLine":"GET /authorizationserver/oauth/authorize?response_type=code&client_id=mobile_android_marketplace&<Primary-Public endpoint> &scope=&code_challenge=N7KD6lGtIZibGn8zj4WrOaZgF38cDApN6EiQw0AhWhA&code_challenge_method=S256&continue HTTP/1.1","statusCode":302,"bytesSent":0,"processMillis":21448019,"commitMillis":21,"connectionStatus":"+"}} 

Cause

Spartacus does not support multiple OAuth client IDs out-of-the-box for the same URL, leading the client ID to remain the same during the login flow, causing the login process to fail.

Resolution

    Composable Storefront on SAP Commerce Cloud does not natively support using multiple OAuth client IDs within a single storefront build; the only standard approach is to create separate storefront builds, each configured with its own client ID and typically hosted on different endpoints.
