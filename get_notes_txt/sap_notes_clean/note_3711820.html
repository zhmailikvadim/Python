Symptom

During repeated ATP checks in VA22 for quotations, schedule lines that represent referenced quantities may shift between runs. Although ATP results themselves remain stable, the non‑ATP‑checked schedule lines (those restored from reference processing) may appear differently ordered or split after each check. The issue appears in scenarios where multiple schedule lines correspond to previously converted referenced quantities. No error message is displayed; the system behaves according to standard logic.

Other Terms

ATP, referenced quantity, VA22, schedule line restoration, STVARV, MERGE_SL_BY_REF, SAPFV45V, FV45VFMV_MVERF_AUFBAUEN_WMENG

Reason and Prerequisites

Standard logic in SAPFV45V / MVERF_AUFBAUEN_WMENG attempts to reconstruct referenced schedule lines after ATP. This reconstruction consumes the referenced quantity across existing lines and may result in variation of restored schedule lines if their distribution changes across iterations. The system does not enforce that the full referenced quantity is assigned to the first schedule line unless an optional merge switch is enabled. The observed shifting is therefore expected behavior, not a defect.

Solution

A correction instruction is provided to introduce an optional system switch to stabilize referenced‑quantity merging. This switch allows customers to force the entire referenced quantity to be merged into the first schedule line, ensuring stable results across repeated ATP checks.
To enable this optional behavior:

A) System/Client level

  1. Open transaction STVARV.
  2. Create a new parameter named MERGE_SL_BY_REF.
  3. Set its value to ‘X’ and save.

B) User level

  1. Open the desired user in transaction SU01
  2. Create a new user parameter named MERGE_SL_BY_REF.
  3. Set its value to ‘X’ and save.

 

Other Terms

DATREF_NOT_ASSIGNED, GET_CONTAINER_SINGLE_EQUI_CONT, GET_EQUZ_STARTTIME, EQUZ Time Slices, Container Allocation, Logical Installation (LOGIKNR), ELOC, CL_EEWA_RCI_COMMON.

Reason and Prerequisites

The dump occurs when the program tries to dereference an unassigned data reference (LEQUZ2).The reference variable LEQUZ2 in the GET_CONTAINER_SINGLE_EQUI_CONT method should point to a legitimate entry in the internal table LEQUZS (EQUZ time slices).
