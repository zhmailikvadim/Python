Symptom

When executing the SAP-delivered recovery validation tool
hdbrecovercheck.py checkPointInTimeReachable,
the tool terminates with the error:

“hdbrecovercheck.py checkPointInTimeReachable got unexpected error – list index out of range”.

The termination occurs during analysis of backup-catalog metadata. No recovery operation is started and no data restore is executed.

Environment

SAP HANA DB Version 2.0

Cause

The SAP HANA backup catalog contains snapshot-based backup entries (ENTRY_TYPE_NAME = data snapshot) in addition to classical data backups (file-based or Backint-based) and log backups.

Snapshot backups are represented as metadata entries in the backup catalog but are not part of the classical data-backup and log-backup recovery chain evaluated by hdbrecovercheck.py.
When such entries are present, the tool may encounter a catalog structure that it does not process correctly and terminates with a runtime error during catalog evaluation.

This behavior is limited to the recovery validation tool and does not, by itself, indicate a limitation of SAP HANA backup or recovery functionality.

Resolution

This issue concerns the behavior of the recovery validation tool hdbrecovercheck.py and can be addressed by aligning the backup-catalog content with the recovery scenario that is being validated.

Proceed as follows:

  1. Determine the scope of the recovery validation
    Clarify which backup types are intended to be validated by hdbrecovercheck.py.
    The tool evaluates the reachability of a classical point-in-time recovery based on a data-backup baseline and the corresponding log-backup chain. 
    Snapshot-based backups are not part of the classical recovery chain evaluated by hdbrecovercheck.py.
  2. Align the backup catalog with the validation scope
    If snapshot-based backups are not required for the intended recovery validation, snapshot entries can be temporarily removed from the SAP HANA backup catalog and the validation command can then be executed again.
    Removing entries from the backup catalog is a supported administrative operation and affects catalog metadata only. It does not, by itself, delete physical backup data.
  3. Validate Backint-based data backups consistently
    If the recovery validation is intended for data backups created via Backint, ensure that hdbrecovercheck.py is executed with the correct Backint parameter file as configured for the database. This allows the tool to consistently enumerate Backint-managed data backups during validation.
  4. Clarification of supported configuration
    SAP HANA allows data backups and log backups to be configured independently, including different destination types (for example, Backint for data backups and filesystem for log backups). This configuration is supported and does not represent a recovery limitation. It is therefore not the root cause of the observed tool behavior.
  5. Scope limitation
    This KBA describes a limitation in the behavior of the recovery validation tool only. Actual recoverability of the database depends on the availability of a valid data-backup baseline and a continuous log-backup chain.
