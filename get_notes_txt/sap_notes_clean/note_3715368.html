Symptom

  • Reference products functionality does not support chained references/multiple levels of refence Products.
  • When a new product location references a product location that itself has a reference, the forecast for the new product location is not executed.
  • The exception /DMF/UDF_BUSINESS exception 011 (Reference product &1 location &4 for product &3 location &2 is not used)" is generated.

Environment

Customer Activity Repository application bundle

Cause

Reference products functionality does not support chained references/multiple levels of refence Products, this is a known limitation in S4. 

For example, Product "A" can be configured to reference "C" and "B" can be configured to reference "C", but configurations such as "A" referencing "B" and "B" referencing "C" are not supported.

In the example the absence of forecast results occur because the outlined rule is not applied to the product location A. 

Since A is referenced to B, and B itself is a placeholder product, the requirement for having its own TS_PS entries is violated. 
Consequently, during forecasting, A's own PAR values cannot be located by the system. Instead, the PAR values from either Reference Product B or the hierarchical prior (HPR) are attempted to be used.
(Further descent to Reference Product C through Reference Product B is not performed.)

Requirements for Modelling and Forecasting:

In order to use placeholder product (PHP) or a reference product the following conditions apply:

  • the used placeholder product (PHP) or the used reference product must have at least one time series entry for the selected Time Series Source and for the selected multichannel.
  • a placeholder product field PROD_STATUS_CD is set to 03 (PHP) in table /DMF/PROD has to be defined REF_PRODUCT_ID of table /DMF/PHP is not empty and active PHP_STATUS_CD of table /DMF/PHP is not set to Deleted.
  • for a reference product, the fields CONS_REF_PROD_ID and CONS_REF_LOC_ID of table /DMF/PRODLOC have to be set to a product location, can be found in the table /DMF/PRODLOC by referencing PROD_ID and LOC_ID.

Requirements for Forecasting:

  • If own model parameters are present in table /DMF/UMD_PAR after a modeling run (given if the product location has own demand data)), these are used.
  • If a reference product is defined and has model parameters, these are used.
  • Else if hierarchical priors are present in table /DMF/UMD_PRI after a HPR run, these are used.

Otherwise, the forecast for the product location will not be executed.

Resolution

To resolve this issue, the assignment of references should be adjusted so that chains are not formed.
