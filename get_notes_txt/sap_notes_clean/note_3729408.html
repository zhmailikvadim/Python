Symptom

When processing an inbound IDoc of type HRMD_A, the IDoc status shows as successful, but no employee data is posted to the HR infotype tables (e.g., PA0000, PA0001, etc.). 

Environment

  • SAP ERP
  • SAP S/4HANA on-premise
  • SAP S/4HANA private cloud edition

Cause

The IDoc has an incorrect segment structure. The following inconsistencies in the IDoc content can lead to this behavior:

  1. Missing multiple E1PITYP segments: The main segment E1PLOGI should contain one E1PITYP segment for each infotype being transmitted. For example, if the IDoc includes data for infotypes 0000, 0001, 0002, 0003, and 0006, there must be five separate E1PITYP segments under E1PLOGI, each containing the corresponding infotype data segment (E1P0000, E1P0001, E1P0002, etc.). If all infotype data segments are placed under a single E1PITYP segment, the data will not be posted correctly.

  2. Missing INFTY field in E1PITYP segment: Each E1PITYP segment must have the field INFTY populated with the corresponding infotype number (e.g., INFTY = 0002 for infotype 2). If this field is not filled, the system cannot determine which infotype the data belongs to, and the data will not be posted.

These issues typically occur when the IDoc is created manually or generated by an external non-SAP system that does not follow the correct HRMD_A IDoc structure.

Resolution

Review and correct the structure of the inbound IDoc to ensure it conforms to the HRMD_A IDoc type definition:

  1. Create one E1PITYP segment per infotype: Under the main segment E1PLOGI, ensure that each infotype has its own dedicated E1PITYP segment. Do not group multiple infotype data segments under a single E1PITYP segment.

  2. Populate the INFTY field: In each E1PITYP segment, fill the field INFTY with the corresponding infotype number (e.g., 0000, 0001, 0002).

  3. Correct segment hierarchy example:

    • E1PLOGI
      • E1PITYP (INFTY = 0000)
        • E1P0000
      • E1PITYP (INFTY = 0001)
        • E1P0001
      • E1PITYP (INFTY = 0002)
        • E1P0002

If the IDoc is generated by an external system, adjust the mapping or logic in the sending system to produce the correct IDoc structure. You can use a correctly structured IDoc in your system (e.g., one generated via standard HR ALE distribution) as a reference for the expected format.
