Symptom

  • Miro postings using standard BAPI BAPI_INCOMINGINVOICE_CREATE1 are creating self-billing invoices in the eDocument cockpit for Poland.
  • Self invoices are being generated for MIRO postings, which is unintended. The desired behavior is for self invoices to be generated via MRRL only.

Environment

  • SAP ERP
  • SAP S/4HANA
  • SAP Document and Reporting Compliance Service
  • SAP Document and Reporting Compliance Poland

Reproducing the Issue

  1. Post invoices via MIRO.
  2. Observe that self-billing invoices are created in the eDocument cockpit for Poland.

Cause

The behavior is expected if the system is configured to generate Poland KSeF self‑billing eDocuments from Invoice Verification. Both MIRO and postings via BAPI_INCOMINGINVOICE_CREATE1 (used by VIM to create MM/IV documents) use the same Invoice Verification flow, so they will trigger the same eDocument creation if the self‑billing settings are active. 

Resolution

Notice that this only a suggestion, it is not a standard recommendation from SAP and you should discuss this with your ABAP and Consulting Team. In case you have further questions or issues, we cannot assist with it.

Use the EDOC_ADAPTOR BAdI to override standard creation logic with method IS_RELEVANT. This lets you suppress eDocument creation for MIRO while keeping it for MRRL, even though both use the same source type (INV_VERIF).

1) Configuration and implementation steps:

  • Keep Poland KSeF self-billing customizing active for INV_VERIF so MRRL can create eDocuments:
    • Activate INV_VERIF for your company code (EDOCOMPANYACTIV).
    • Assign PL_KSEFSB/PL_KSEFSBC as needed and keep the Poland KSeF interfaces and mapping in place (per your existing setup).
  • Create BAdI implementation for EDOC_ADAPTOR:
    • Transaction SE18, BAdI EDOC_ADAPTOR.
    • Create a new implementation (for example Z_PL_KSEF_IV_FILTER).
    • Maintain BAdI filters for Poland: Country = PL and Generic Filter = KSEF.
  • Implement method IS_RELEVANT:
    • Determine the origin of the LIV/MM invoice document (MRRL vs MIRO).
    • Return abap_true only for MRRL-originated self-billing; return abap_false for MIRO-originated postings so no eDocument instance is created.
    • Check the posting transaction code of the associated FI document (BKPF-TCODE = ‘MRRL’ vs ‘MIRO’).
    • Or check an ERS/MRRL indicator available on your LIV document header/items in MM-IV (use whichever is most reliable in your system).
    • Purpose: Decide at runtime whether an eDocument should be created for a given source document, regardless of Customizing.
    • Logic idea for source type INV_VERIF:
    • Typical determination options:
  • Optional: Implement CHANGE_EDOCUMENT_TYPE if you prefer to redirect MIRO to a different (non-self-billing) eDocument type instead of suppressing creation. IS_RELEVANT is the cleanest way to stop creation.
  • Transport and test:
    • Post an MRRL run: verify a self-billing eDocument is created and can be submitted to KSeF.
    • Post a MIRO invoice: verify no eDocument is created in the cockpit.

2) Example skeleton for IS_RELEVANT

  • Pseudocode outline (adapt to your system’s available keys in the method):
    • Read the associated FI document for the LIV document and check BKPF-TCODE.
    • If BKPF-TCODE = ‘MRRL’, set cv_is_relevant = abap_true.
    • If BKPF-TCODE = ‘MIRO’, set cv_is_relevant = abap_false.
    • If source_type = INV_VERIF:
    • Else: leave relevance unchanged or apply standard logic.

3) Notes

  • Ensure your EDOC_ADAPTOR implementation uses filters Country = PL and Generic = KSEF so it only affects Poland KSeF processing.
  • You can combine EDOC_ADAPTOR with EDOC_CUSTOM_FIELDS if you need to carry custom markers into the eDocument for auditing, but this is not required to suppress MIRO creation.
