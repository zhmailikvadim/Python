Symptom

A goods movement is posted for an inspection lot, where the related material document has to be linked to the inspection lot (written into table QAMB). This action generates a queue on ERP side with prefix QIMD* (FM QPLEXT_MATDOC_GET). The queue gets stuck in SMQ2 and in ST22 a dump exists related to the queue:

Category               ABAP programming error
Runtime Errors         MESSAGE_TYPE_X
ABAP: Program          SAPLARFC
Application Component  BC-MID-RFC

The dump's "Error Analysis" section contains the information about the queue:
"Short text of the error message:
    Inbound queue name QIMD******************** cannot be used for classic q RFC"

Environment

SAP Extended Warehouse Management

SAP S/4HANA

Reproducing the Issue

A goods receipt (IOT4), posting change (IOT5) or a usage decision is posted for a lot origin 17 inspection lot.

QIMD* queue will get stuck in SMQ2 on ERP side and ST22 dump appears related to the stuck queue:

Runtime Errors         MESSAGE_TYPE_X
ABAP: Program          SAPLARFC

Cause

There is a feature with qRFC that if an out-inbound qRFC call (a qRFC is sent to be triggered and processed on the receiver side) is made, and the used queue name is registered in "SBGRFCCONF", then that qRFC call would be converted to a bgRFC call on the receiver side. Thus, if the prefix maintained in SBGRFCCONF is only "Q", the dump occurs as a preventive measure for this case.

Resolution

Use a more specific queue name registration in SBGRFCCONF rather than just 'Q'. To do so:

  1. Go to transaction SBGRFCCONF
  2. Go to tab Define Inbound Dest-> Choose the relevant destination. The destination name is usually found under "L_DEST_NAME" in the dump in ST22
  3. Delete the "Q" prefix entry and create a new one that is a more specific queue prefix. It should also not match the queue's full name. For processing the QIMD* queues, it is advised to use prefix e.g. "QIMD".
