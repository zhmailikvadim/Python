Symptom

You notice that in an already established 3-tier System Replication scenario, after sr_register command was run on secondary, the previous registered DR site loses contact with secondary. It will no longer be shown in statistics like python script systemReplicationStatus.py or the site mapping in output of command 'hdbnsutil -sr_state'.

Same error may also seen if you have an established replication B->C and you try to directly register site B under site A to have a replication landscape A->B->C.

Following traces could be observed:
In DR server, following trace will be observed in nameserver trace:
[xxxxxxxx]{-x}[-x/-x] xxxx-xx-xx xx:xx:xx.xxxxxxx e sr_nameserver    SystemReplicationWatchDog.cpp(00073) : could not retrieve source site topology;

In Secondary server however, the trace depends on what command has been run:

  • Scenario 1:
    Only sr_register is run on secondary:
    Following trace can be observed in Secondary nameserver:
    [xxxxxxxx]{-x}[-x/-x] xxxx-xx-xx xx:xx:xx.xxxxxxx i sr_nameserver    DRRequestHandler.cpp(01175) : drCheckTopology: Not a source site, won't send topology to secondary.
  • Scenario 2:
    sr_enable is also run on secondary, after sr_register
    Following trace can be observed in Secondary nameserver:
    [xxxxxxxx]{-x}[-x/-x] xxxx-xx-xx xx:xx:xx.xxxxxxx e sr_dataaccess    DisasterRecoveryPrimaryImpl.cpp(03488) : Reject new replication connection from 10.180.2.102 because no registration information for siteId=3 found. Please check if the host from 10.180.2.102 is intended to be running and is properly unregistered or removed. If this it not the case, please unregister/remove the site/host from the landscape. To properly unregister a system replication secondary site with hdbnsutil please checkout SAP note 1945676.

Environment

SAP HANA, platform edition 2.0

Reproducing the Issue

  1. Set up the 3-tier replication
  2. On secondary site, run hdbnsutil -sr_register command.
  3. Check the status in primary with python script or hdbnsutil -sr_state and see the issue.

Cause

By running sr_register, the secondary is reset back to a pure secondary system who replicates from primary.

This means that the process will rewriting secondary's topology information, removing all the information regarding the sites replicate from the secondary site. Thus it won't know about DR site after sr_register.

Resolution

To make the replication work again, the DR has to go through the complete register process.

If hdbnsutil -sr_enable is not run on Secondary Site, run hdbnsutil -sr_enable on secondary first.

Then, run sr_reigster for all the DR site that lose replication with the registered Secondary site.

In case the replication is still broken, follow the 'Repairing a landscape that has a broken secondary system after wrong usage of hdbnsutil sr_unregister' section of note 1945676 to hdbnsutil -sr_cleanup --force and try again with sr_reigster
