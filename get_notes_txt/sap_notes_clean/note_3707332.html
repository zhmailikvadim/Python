Symptom

This SAP Note contains answers to frequently asked questions on how the data from the operation activities is transferred to the order.

Other Terms

SAPLMPE_OAC_RECG, CL_MPE_OAC_UR_EXEC, CORUPROC1,

AFRP0, MPE_OAC_UR

Background process, Update Request, Confirmation Request

Performance,

Cost Activities, Operation Confirmation

Solution

Please familiarize yourself with the online documentation How Actual Data from the Operation Activities is Transferred to the Order.
This provides detailed information and an explanation on how the order is updated via a background process which is decoupled from the operation activity execution itself.

Answers to additional/dedicated questions:

Question 1: Why is the order / operation not updated after having posted an action for an operation activity?

Answer:

Please check the online documentation How Actual Data from the Operation Activities is Transferred to the Order. The following sections are particularly relevant:

  • Exception: Situations in which OA Postings do Not Trigger Operation Postings
  • Troubleshooting

In addition, the background process is not triggered if it is clear from the posted action that no operation update is necessary, i.e., if neither the status nor the quantity of the operation changes, e.g.:

  • Actions "Pause" and "Resume Work" do not influence the operation's status.
  • Any action (with the exception of ‘scrap’) on an operation activity that is neither start-OA nor end-OA of the operation does not influence the operation's actual data.

Question 2: How is the user of the generated operation confirmation determined?

Answer:

The automatically generated operation confirmation typically comprises data from many actions that have been executed for the corresponding operation activities.

  • Those actions might have been executed by different users.
  • In addition, the decoupled background process with which the confirmation was posted might have been executed by a different user.

The operation confirmation stores two users (which are displayed in transaction CO14 “Display production Order Confirmation” on the tab “Administration”, for example):

  • Entered by User (field AFRU-ERNAM):
    • This is the user with which the confirmation is created e.g., the user with which report CORUPROC1 was executed to convert the confirmation request records into confirmations.
    • This user might not have participated in the execution of the operation activities at all.
  • Created By (field AFRU-EXNAM):
    • This is one user who participated in the execution of the operation activities of that operation.
    • Typically, this is the user who posted the very last action.

Question 3: Is there a way to influence the generated operation confirmations?

Answer:

Yes, there is a BAdI that can be used to adjust the generated confirmations according to the individual needs:

  • This is BAdI-Definition EX_MPE_EXEC_OPERATION_ACTIVITY of enhancement spot ES_MPE_EXEC_SHOP_FLOOR_ORDER.
  • In SAP’s standard IMG, this can be found via the path “Production -> Manufacturing for Production Engineering and Operations -> Business Add-Ins (BAdIs) -> Production Operations -> Order Confirmations -> BAdI: Adjustment of PP Confirmations”.
  • Please familiarize yourself with the corresponding IMG documentation, which explains which fields must not be adjusted, for example.

Question 4: How are the activities of the operation confirmation calculated?

Answer:

With the automatically generated operation confirmation, up to six actual activities can be posted (fields AFRU-ISM01 to AFRU-ISM06 with corresponding units AFRU-ILE01 to AFRU-ILE06).

  • Without making any adjustments, the actual activities are calculated “as planned”; i.e., the actual activities are calculated by the formulae, using the actual confirmed quantity, for example.
  • You can adjust this logic according to your needs via the BAdI EX_MPE_EXEC_OPERATION_ACTIVITY of enhancement spot ES_MPE_EXEC_SHOP_FLOOR_ORDER.
    In SAP’s standard IMG, this can be found via the path “Production -> Manufacturing for Production Engineering and Operations -> Business Add-Ins (BAdIs) -> Production Operations -> Order Confirmations -> BAdI: Adjustment of PP Confirmations”.
    • There is an example class CL_MPE_ADJ_PP_CONF that can be used as a copy template. It shows one way of using the actual durations of the corresponding actions as a basis for calculating the activities of the operation confirmation.
    • Note that a precondition for that logic is previous assignment of each activity to a record type group (setup, processing, teardown) in the Activities Overview table in the Costing tab in the work center master data.

Question 5: Is there any way to influence the background process to increase the system performance?

Answer:

The background process is run with default technical settings.

  • Standard Customizing can be used to optimize the background processes based on the specific circumstances in the current client by adjusting those default settings.
  • These Customizing settings don‘t change any business logic.
  • For detailed information on how to do so, check note 3660199 – Optimizing Performance of Background Execution.
