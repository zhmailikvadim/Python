Symptom

An issue occurs with a scenario such as the following.

1. Use the Netweaver configtool to set the number of server processes for the Netweaver instance to a value greater than 1 (i.e., 2).
2. Restart Netweaver.
3. Complete a number of SFCs to a Done status (i.e, 500).
4. Set the system rule Days Before Archiving SFC/SO to 0.
5. Use the Background Process activity to trigger immediate execution of Archiving (i.e., set to run every 1 minute).

Expected Result: Archiving is executed on 1 of the Java server processes.
Actual Result: Archiving may be triggered on multiple processes simultaneously, causing exceptions.

Other Terms

duplicate key, unique index

Reason and Prerequisites

The table column BACKGROUND_PROCESS.CURRENT_RUN_ID is used to block all subsequent execution until the value is cleared.
A race condition exists in which multiple processes can read the table column BACKGROUND_PROCESS.CURRENT_RUN_ID and then later try to update the value.

A temporary lock file is created to also block execution. This mechanism works when execution is triggered from the operating system via a batch script. However, for background processing the file is created in a different temp directory for each Java process.

Solution

Fixes have been delivered in the form of a code correction.
Changes were made to verify that the CURRENT_RUN_ID is null or represents a value previously set by the current Java process. The check is performed in the WHERE condition of the UPDATE statement.
The lock file mechanism is now bypassed for background processing.

Impact
Changes were made to modules that affect the execution of all background processes. Both Batch and background execution are affected.
