Symptom

In the Create Maintenance Request (F1511A) app, the Failure Effect value help may show entries not derived from the selected Technical Object’s catalog profile when opened immediately after changing Technical Object or Notification Type. During this window, lists can be filled based on the Notification Type instead of the technical object’s Catalog Profile. Release scope: In releases prior to OP2023 SP2, the behavior concerns Failure Mode Group only; as of OP2023 SP2, it affects Failure Mode and Failure Effect.

Other Terms

PM-FIO-WOC-MN; Value Help; SmartField; Failure Effect (MaintNotifEffectCode); Failure Mode (MaintNotificationCode); Failure Mode Group (MaintNotificationCodeGroup); Catalog Profile (CatalogProfile); Notification Type (NotificationType); side-effects; beforeSideEffectExecution; createWrSmartFieldFailureMode; createWrSmartFieldMaintNotifEffectCode; OP2023 SP2.

Reason and Prerequisites

The reason is a program error in synchronizing value‑help data during side‑effect processing. The UI allowed opening the Failure Mode/Effect value helps before dependent data was refreshed, so lists could reflect values from the Notification Type catalog instead of the Technical Object catalog profile.

Solution

Apply attached Correction Instruction via SNOTE. The fix sets the Failure Mode and Failure Effect fields to busy/disabled immediately after changes to Technical Object or Notification Type, and re‑enables them only after side‑effects complete so the value helps are populated from the correct Catalog Profile. No manual post‑implementation steps are required.
