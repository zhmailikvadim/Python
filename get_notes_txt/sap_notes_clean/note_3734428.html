Symptom

  1. Preserve existing tasks, comments, and assignments
  2. Avoid “Former User” entries
  3. Maintain user consistency after IAS changes

"Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental."

Environment

  • SAP Cloud ALM
  • SAP Cloud Identity Services – Identity Authentication (IAS) 

Reproducing the Issue

  1. An SAP Cloud ALM tenant is already in use
  2. Customer attempts to change or replace the connected IAS tenant

Cause

The ability to change the connected IAS tenant depends on the current IAS connection state of the SAP Cloud ALM tenant.

Resolution

  1. Log on to SAP Cloud ALM
  2. Navigate to User Management
  3. Identify whether the IAS connection is in: State A (modifiable) or State B (locked)

     

    Scenario 1: IAS in State A

    In this case, the following approach can be used:

    1. Configure new IAS as Corporate Identity Provider

    • Configure the new IAS tenant as a Corporate Identity Provider in the currently used IAS tenant
    • Refer to: Corporate Identity Providers (SAP Help Portal)

    2. Maintain consistent Subject Name Identifier (SNI)

    To avoid "Former Member" entries throughout SAP Cloud ALM, it is mandatory to keep the "Subject Name Identifier" stable. 

    • In the current IAS tenant:
      • Go to Applications → SAP Cloud ALM
      • Identify the Subject Name Identifier (SNI) (e.g. email)
    • Ensure the new IAS tenant:
      • Uses the same SNI
      • Sends exactly the same value, including casing

    Ensure that the same Subject Name Identifier is used for the Corporate IDP configuration, which you created under Scenario 1 - step 1.

    Check this on the side of the new IAS Tenant, under the application created for the currently used IAS tenant.

    Note that also the values need to be exactly the same. I. e., if you've been using "email" as subject name identifier and you have a user "john.doe@your-company.com", the new IAS tenant needs to send exactly the same value, with the same casing.

    3. Validate

    After configuration:

    • Log on via the current IAS tenant (acting as proxy)
    • Navigate to:
      • SAP BTP Cockpit → Subaccount → Security → Users

    Verify:

    • Only one user entry exists
    • The user ID remains unchanged

    If a duplicate user is created:

    • The SNI configuration is not aligned and must be corrected

    Scenario 2: IAS in State B

    In this case:

    • Changing the connected IAS tenant is not supported

    If a different IAS tenant must be used:

    • Delete the SAP BTP SP Cloud ALM Subaccount and recreate it, afterwards a different IAS tenant can be assigned
    • This action result in a new SP Cloud ALM subscription, however this will also result in complete data loss, consequently, all previously created configurations and project content must be recreated.

     

    Additional Information

    • To avoid “Former User” entries, maintaining a consistent Subject Name Identifier (SNI) across identity providers is essential
    • Differences in SNI or attribute values will lead to creation of new user identities
