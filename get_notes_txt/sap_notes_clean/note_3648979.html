Symptom

Error "maximum number of rows per table or partition reached" during the phase "EU_CLONE_MIG_OPTDMO_INI_RUN" occurs while copying the KONV table from the source to the target system via the Software Update Manager (SUM) through a Downtime-Optimized conversion (DoC).

A1EEIMP 008 DB code 2055

A1EEIMP 008 maximum number of rows per table or partition reached: 'SAPABAP1:_SYS_SPLIT_KONV~<Partition Number>'

Environment

SAP ERP 6.0

HANA 2.00.075 

Reproducing the Issue

Copy table KONV using Software Update Manager (SUM)

Cause

The KONV table is specially handled in SUM as the table KONV is replaced in target with PRCD_ELEMENTS - shadow field handling: the fields for target table PRCD_ELEMENTS are added as shadow fields and are automatically filled by R3load with an update statement converting the content of KONV fields to the shadow fields as defined by the application during initial copying of data as well as during replay of changes later on. It means that each row in KONV receives an insert and an update; both changes are part of the delta in target HDB.

When partitions of KONV in the source system are over 1 billion records, because of the special handling with two changes per row, the delta store for these partitions oversteps the 2 billion row limit. R3load is doing an explicit merge statement only after all rows are processed, which caused the dump to occur as no intermediate Delta Merges were being triggered either by the SUM process or the HANA database automatically.

Resolution

To avoid this error, split every partition in the KONV table that contains more than 1 billion records. SAP development support is evaluating why neither SUM nor SAP HANA automatically triggered the Delta Merge process to prevent the overflow error.
