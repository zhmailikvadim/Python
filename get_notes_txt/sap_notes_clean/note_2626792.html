Symptom

You created a Complete Data Backup with SAP HANA 2.0. You recovered SAP HANA Database using the SQL statement RECOVER DATA [FOR <database_name>]{...}. After the recovery finished successfully, there is a wrong timestamp displayed in backup.log file.

  • backup.log entry from complete data backup:

INFO    BACKUP   SAVE DATA started
INFO    BACKUP   to  backint: /usr/sap/SID/SYS/global/hdb/backint/DB_SID/backup_SID_YYYY_MM_DD_hh_mm_ss_databackup_0_1
INFO    BACKUP   to  backint: /usr/sap/SID/SYS/global/hdb/backint/DB_SID/backup_SID_YYYY_MM_DD_hh_mm_ss_databackup_2_1
INFO    BACKUP   SAVE DATA finished successfully

  • backup.log entry from recover data:

INFO    RECOVERY command: RECOVER DATA FOR SID USING BACKINT ('/usr/sap/SID/SYS/global/hdb/backint/DB_SID/backup_SID_YYYY_DD_MM_hh_mm_ss')  CLEAR LOG
INFO    RECOVERY using backup catalog <xxx> from file /usr/sap/<SID>/HDB/backup/log/DB_SID/log_backup_0_0_0_0.xxx
INFO    RECOVERY RECOVER DATA finished successfully, reached timestamp 0001-01-01T00:00:00+03:00, reached log position xxx

Other Terms

SAP HANA Recover, Recovery

Reason and Prerequisites

Reason:

Due to a programming error in SAP HANA, when recovering data without log replay, a wrong timestamp will be recorded in the backup.log file .

Prerequisites:

You are performing a RECOVER DATA as RECOVER DATA [FOR <database_name>]{...}.

Affected Releases:

  • SAP HANA 2.0 database:
    • Revisions of SPS00 - SPS01
    • Revisions <= 024.00 (SPS02)

Solution

Apply one of the following SAP HANA database Revisions:

  • SAP HANA 2.0 database:
    • Revisions >= 024.01 (SPS02)
    • or higher

  • or higher

Workaround:

Since the recover process itself is running without issues, you can ignore the wrong timestamp entry.
