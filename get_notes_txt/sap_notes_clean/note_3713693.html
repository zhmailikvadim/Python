Symptom

  • City marked as Placename in the reference data
  • Convert locality name
  • Preferred city name
  • Zip4 added for address but city name is still misspelled
  • City returned is not the expected city name

Environment

  • Data Quality Management for SAP
  • DQMmicroservice for location data engine

Cause

There are numerous cities in the United States that have preferred city names and other city names that are considered valid for a postcode.  The reference data that is received from the data provider has both entries in them, but some are marked as "placenames".  This placename marking indicates that this is an alternative city name for the postcode.  The input city placename is retained by default when it is used on input as it is a valid, per the referenced data, for the postcode.  

Resolution

If the placename is not the expected result, the configuration that is being used on the DQMm side, can be modified to return a field called "std_addr_locality_official" rather than "std_addr_locality_full" to the city field.   This will change the output city name to use the USPS preferred city name. 

This is done using the configuration UI editor in DQMm following the DQMm guide.

Note:  It is required when changing the default configuration to make new configuration based on the default one,  (i.e. SAP_BusinessSuite_Config1) and make the changes in the custom configuration. Once there is an new configuration name, with the expected changes made and saved, the table /FLDQ/DQAAS_CFG on the SAP system needs to be modified to use this new configuration name.
