Symptom

  • Clarification is needed regarding the timing of PAL Consumption storage in the database table PAQTYASSGMT during the availability check process with PAL functionality in sales orders. 

  • Uncertainty exists regarding the behavior of PAL Consumption storage in scenarios such as 'No Confirmed PAL Quantity', when a 'Reason for Rejection' is set or in case of 'Credit Limit Check Failure' 

  • This SAP KBA includes a clarification of  PAL consumption storage in the database table PAQTYASSGMT in the following scenarios:

    1. Scenario 1: Create Sales Order via VA01 or EDI

    2. Scenario 2: Change Sales Order via VA02 or BOP

    3. Special Scenario: 'Reason for Rejection'

    4. Special Scenario: 'Credit Limit Failure'

Environment

  • SAP S/4HANA 
  • Supply Chain Management 
  • Advanced Available to Promise (aATP)

Reproducing the Issue

  1. Create or modify a sales order using VA01, VA02, BOP, EDI or Interface
  2. Perform an aATP check during sales order creation or re-check aATP
  3. Verify PAL Consumption in the database table PAQTYASSGMT after "Save" 
  4. Verify PAL Consumption in the database table in special scenarios such as after setting "Reason for Rejection" or in case of "Credit Limit Check Failure"
  5. Observe the unclar behavior of PAL consumption updates in the database table PAQTYASSGMT

Cause


Uncertainty regarding the process of PAL Consumption updates in the database table PAQTYASSGMT during the availability check in sales orders processing for SAP S/4HANA advanced ATP. 

Resolution

    1. Scenario 1: Create Sales Order via VA01 or EDI

    PAL Check Results

    Expected update in PAQTYASSGMT on Save

    PAL Consumption > 0Confirmed Quantity > 0 will be taken over
    PAL Consumption = 0 or Constraint Status = ‘No Availability’Confirmed Quantity = 0 will be taken over
    No PAL relevance, Constraint Status = ‘Not Relevant’ or ‘List does not contain any value’No entry
      

    2. Scenario 2: Change Sales Order via VA02 or BOP

    PAL Check ResultsExpected update in PAQTYASSGMT on Save
    PAL Consumption changes but remains with a value > 0Existing PAQTYASSGMT Entries will be updated with new results
    PAL Consumption changes to a value = 0Existing PAQTYASSGMT Entries will be updated with new results
    No PAL relevance, Constraint Status = ‘Not Relevant’ or ‘List does not contain any value’ No PAL updateNo PAL update
      

    3. Special Scenario: 'Reason for Rejection'
     

    When a reason for rejection is setExpected update in PAQTYASSGMT on Save
    Not yet delivered PAL Consumption will get released If necessary existing entries are split
     Already delivered PAL Consumption remains unchanged
     Entry for not yet delivered and now rejected PAL Consumption  will be deleted from PAQTYASSGMT
      

    4. Special Scenario: 'Credit Limit Failure'

    When 'Credit Limit Failure' occursExpected update in PAQTYASSGMT on Save
    Determined confirmation will be reset during saving Possibly prior determined PAL Consumption are reset
     PAL Check results are handled, as if no check would have been executed
     No entry is written or updated in PAQTYASSGMT
