Symptom

What it is about:
For cases where UI Data Protection for SAP S/4HANA does not support masking or data blocking out of the box, UI Data Protection API can be used for data protection.

Examples of such scenarios include:

  • System to system communication cases where user interface is not involved. E.g. Masking in RFCs and WebServices
  • Case where required context is not available on the UI layer. E.g. certain old classicial reports 
  • Out of scope UI technology is being used by application. E.g. embedded HTML views

For these cases, the solution provides Data Protection APIs in the form of class methods that can be used to implement custom masking or blocking logic. However, these APIs are intended as a fallback option and should be used only when traditional or configuration-based masking approaches are not applicable or do not work directly.

Intention of this SAP Note:
This consulting SAP Note provides an overview of the UI Data Protection APIs and explains how to use these APIs for implementing masking or data blocking in scenarios where standard or configuration-based approach is not supported or cannot be applied due to missing context.

Other Terms

UI Masking, UI Data Protection Masking, UIDP 100, Data Protection APIs

Reason and Prerequisites

 Standard UI Data Protection masking or data blocking cannot be applied because the required framework support or runtime context for attribute-based masking is not available in certain scenarios.

Solution

Masking API:

Follow the steps below to call the Data Protection Masking API at an appropriate interception point.

  1. Identify a suitable interception point in the application flow where custom logic can be implemented. This interception point can be an existing BAdI, enhancement spot, user exit, or an implicit enhancement in the code.
  2. Ensure that the interception point is reached after the application’s business logic has been executed, so that the final data to be returned is available. Typical examples include the end of an RFC, a post-processing BAdI, or similar exit points. This approach ensures that masking is applied correctly to the output data without impacting the core business logic.
  3. Call the UI Data Protection Masking API at the identified interception point.
  4. Pass a data reference of the data object (table, structure, or variable) that needs to be masked to the API method.

Parameters:

Parameter Name Type Optional Use
IT_FIELD_MAP Importing X

Required only for table–field–based masking; not needed for data-element-based masking.
Maintain the table name–field name mapping for the fields to be masked. The field name should be from the structure or table that is passed in paramter CR_DATA. Table name could be of your choice.
IV_UNAME Importing X User ID for which the authorization check is performed.
CR_DATA Changing   Data reference to the table, structure, or variable to be masked.

Data Blocking API:
Follow the steps below to call the Data Blocking API at an appropriate interception point.

  1. Analyze the business transaction and identify a suitable enhancement option where the API can be invoked. This can be an existing BAdI, customer exit, user exit, or an implicit enhancement point.
  2. Determine the blocking requirement based on the use case:
    • Data Blocking:For example, blocking the execution or response of a web service or RFC call when a sensitive value is detected in the request payload
    • Data Suppression/Filtering:For example, suppressing selected records in the response of a web service or RFC-based system-to-system data transfer
  3. Call the Data Blocking API at the identified interception point by passing a data reference to the data object (table, structure, or variable) that needs to be blocked or suppressed.
  4. Evaluate the API response:
    • The API returns an indicator specifying whether blocking or Suppression/Filtering has been applied.
    • In case of authorization failure, a corresponding message is returned.
  5. Raise the returned message explicitly in the interception logic to stop processing or suppress data as required.

Parameters:

Parameter Name Type Optional Use
IT_FIELD_MAP Importing X

Required only for table–field–based blocking; not needed for data-element-based blocking.
Maintain the table name–field name mapping for the fields to be blocked.
IV_UNAME Importing X User name for which the authorization check is performed.
CR_DATA Changing   Data reference to the table, structure, or variable that needs to be blocked or suppressed.
EV_INDICATOR Exporting NA

Returns the action applied by the API:

  • ‘B’ – Data Blocking (structure or variable)
  • ‘S’ – Data Suppression/Filtering (table data)
ES_RETURN Exporting NA

Returns the message class and message number.
The developer must explicitly raise this message at the interception point.

Considerations: 

  1. The Data Protection APIs are intended to be used only as a fallback option and should be considered when traditional or configuration-based masking approaches are not applicable or do not work directly. These APIs are not a replacement for standard UI Data Protection configuration and should be used selectively based on the application scenario.
  2. In the case of the Masking API, numeric fields are masked by replacing the original values with ‘0’, as masking with ‘*’ is applicable only to character-based fields.
  3. Masking APIs  should be used only in case Reac scenarios as for update scenarios more complex handling will be required. For Eg. Replacing masked value with actual value before commiting them to data base.
