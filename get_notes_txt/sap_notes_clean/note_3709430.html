Symptom

Due to inconsistent data you may encounter dumps with the runtime error MESSAGE_TYPE_X in program SAPLMCEX when processing MCEX queues in a collective run. This occurs even though the underlying RFC queue technology can ensure consistency — specifically when the NONE destination uses the fast serializer.

Other Terms

MESSAGE_TYPE_X, SAPLMCEX, queue delta, MCEX*, RMBWV3*, MCEX_GEN_AND_CHECK_HASH, fast serializer

Reason and Prerequisites

Originally, RFC queues did not check whether stored parameter data matched the actual structures in the target system. This could produce "garbage" data, which is worse than receiving no data. Consequently, an application-level consistency check was added. If the current structures did not match the format of the data stored in the queue, the check terminated queue processing with the runtime error MESSAGE_TYPE_X.

However, recent improvements in RFC technology mean that queues whose destination uses the so‑called "fast serializer" can handle the most common mismatches between stored data and target structures. Therefore, the application-level check needlessly interrupts queue processing in these cases.

Solution

Apply the correction instructions.

The corrections lead to skipping the application-level consistency check when the relevant NONE destination uses the fast serializer, preventing unnecessary MESSAGE_TYPE_X terminations.
