Symptom

An issue occurs with a scenario such as the following.

1. Set the activity rule ALLOW_PARTIAL to YES for the NC_DATA_ENTRY plugin.
2. Use POD Maintenance to configure a button in the Work Center POD with the NC Data Entry plugin. Uncheck the Clears SFC flag.
3. Use Data Field Assignment Maintenance to define an NC data type with the DEFECT_COUNT field.
4. Use NC Code Maintenance to configure an NC code with the data type and the ALL disposition group.
5. Start an SFC with multiple quantity in the Work Center POD.
6. Enter a partial quantity value in the Quantity field.
7. Open the NC Data Entry plugin.
8. Enter the partial quantity in the Defect Count field and perform the SCRAP disposition.

Actual Result:
The operation is still selected in the Operation List. The SFC field is cleared, but it is still selected in the background since a Complete can be performed without re-selecting the SFC.

Expected Result:
The SFC, Work List, and Operation List models should be cleared when the SFC is no longer available in the work list. Otherwise, the SFC should remain selected.

Other Terms

Synchronized

Reason and Prerequisites

The SFC field is cleared but the underlying model still contains the SFC.
The Operation List is not updated when it has been replaced in the panel by another plugin.
The NC plugin tries to manage the model instead of delegating to the work list plugin.

Solution

Changes were made to remove the SFC from the model when it is no longer available in the work list.
Changes were made to clear the Operation List selections when reloading the plugin in the panel.
Changes were made to the NC plugin to defer model updates to the work list.

Impact
Changes were made to modules that affect:
- Dispositioning
- Loading and refreshing of the SFC field, Quantity field, Work List and Operation List after any POD transaction
- All PODs and Standalone NC Logging
