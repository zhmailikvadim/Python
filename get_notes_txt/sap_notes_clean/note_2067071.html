Symptom

IQ server could crash a stacktrace in certain conditions.  The stack trace inlucde the following functions:

Dump all thread stacks at st_server.cxx:1442 for PID: 20465

df_ConsoleMessages::Fetch(a_cursor_orientation, dfv_Row*)
df_ProcCall::FillBuffer(a_cursor_orientation, dfv_RowBuffer*)
dfo_ProcCall::DoFetch(a_cursor_orientation)
dfo_ProcCall::DoFirstFetch(a_cursor_orientation)
dfo_Base::Fetch(a_cursor_orientation)
dfo_MaterializePack::PopulateFully(a_cursor_orientation) 
dfo_Materialize::DoFirstFetch(a_cursor_orientation)


The complete stack trace is included in an attachment to this KBA.

Environment

  • SAP IQ 15.4
  • SAP IQ 16.0

Cause

  • The issue has been identified and logged under CR 758063
  • In rare circumstances 'sa_server_messages', which is an internal procedure, if called in such a way that it returns an empty result set then it can put the server in a state where further concurrent calls to 'sa_server_messages' will cause a server crash
  • Sybase Central by default executes 'sa_server_messages' at regular intervals as part of its server monitoring and logging setup
  • The way to verify whether a situation will satisfy these conditions is by checking the iqmsg and the srvlog file to see if we indeed had any concurrent calls. An example on what you can look for is below :

SQLAnywhere 16.0 SP13 Build 19 and above
