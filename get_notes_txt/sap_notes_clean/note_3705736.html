Environment

Resolution

Data Extraction Source for 0FI_GL_10 Delta Processing

DataSource Overview

  • DataSource: 0FI_GL_10 (General Ledger: Balances, Leading Ledger)
  • Module: Financial Accounting (FI-GL)
  • Data Type: Transaction data with delta extraction

Primary Source Tables

  • Main Table: FAGLFLEXT (New General Ledger: Totals Table)
  • Related Tables: ACDOCA (Universal Journal Entry Line Items)
  • Join Conditions: FAGLFLEXT records are derived from ACDOCA data through aggregation and compatibility views 

Delta Mechanism

  • Delta Type: AIED (After Images Marked for Deletion via Extractor)
  • Delta Field: TIMESTAMP field in the totals table (FAGLFLEXT)
  • Timestamp Logic: Uses timestamp-based selection to identify records changed since the last delta load, with a two-step procedure that selects changed records and their logically dependent records 

Extraction Logic

  1. Initial Selection: System uses TIMESTAMP field in FAGLFLEXT to select all records changed since last delta load
  2. Dependent Record Selection: All logically dependent records (e.g., credit records for debit records) are selected and transferred to BW
  3. Safety Interval Processing: Default safety interval of 1 hour (3600 seconds) ensures data consistency by extracting overlapping time intervals

Dependencies

  • Prerequisite Objects: New General Ledger Accounting must be activated in Customizing 
  • Related DataSources: 0FIGL_O10 (DataStore Object required for delta process) 

Technical Configuration

  • Extraction Method: Direct extraction from totals table using timestamp procedure
  • Safety Interval: 3600 seconds (1 hour) lower safety interval for real-time extraction capability 
  • Serialization: Serialization of data packet required due to after-image delta process 

The 0FI_GL_10 DataSource extracts delta data directly from the FAGLFLEXT totals table using timestamp-based selection. The extractor determines delta records during the delta update by identifying changed records since the last extraction, then selects all logically related records to ensure data consistency. This approach allows for real-time data availability in BW while maintaining data integrity through the safety interval mechanism.
