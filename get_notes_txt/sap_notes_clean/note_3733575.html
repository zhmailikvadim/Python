Symptom

Error occurred while executing the consolidated report scheduled task. 

During execution of Merge Consolidation or Realignment in Schedule Jobs for Consolidation Tasks, the job fails with an error similar to:

"Value of field YY1_* is not valid"

The task cannot be completed for the affected data.

Environment

SAP S/4HANA Cloud Public Edition

Reproducing the Issue

  1. Open the Schedule Jobs for Consolidation Tasks app.
  2. Set the required parameters.
  3. Open the failed job entry.
  4. In Log and Results, choose "Error".
  5. Observe the error: "Value of field YY1_* is not valid"

Cause

The issue is caused by inconsistent values in custom fields (YY1_*) stored in Universal Journal (ACDOCA).

All fields belonging to Universal Journal business contexts, including:

  • Coding Block
  • Journal Entry Item
  • Market Segment

They are automatically part of the rule context of Group Reporting Preparation Ledger (GRPL) and are evaluated during Realignment.

If any of these fields contain:

  • values that are not compliant with the field definition (for example, invalid format or value not included in a code list), or
  • values that are technically inconsistent with the initial value (for example, "empty" values that are not equal to the system initial value),

the system cannot process the affected records, and the Merge Consolidation / Realignment task fails.

Resolution

The issue is caused by inconsistent values in custom fields within Universal Journal data, which prevents the system from processing the affected records during Merge Consolidation / Realignment.

To resolve this issue, the data must be corrected. This can involve:

  • correcting the affected Universal Journal entries, or
  • correcting the corresponding data in Group Reporting Preparation Ledger (GRPL).

In cases where the inconsistency cannot be resolved via standard application processes, a backend correction may be required.

Please create a case under component FIN-CS-COR-DT-2CL so that development team can analyze and perform the necessary correction.

Remark:

  • For this issue the backend corrections are typically handled as a one-time fix.
  • If inconsistent values are created again in the future, the issue may reoccur.
