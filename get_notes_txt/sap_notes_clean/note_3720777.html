Symptom

When executing change pointer write operations, the system performs an authorization check for authorization object /ASADEV/RC (field /ASADEV/RF = DATA_PROC).

If the required authorization is missing, processing is terminated and no change pointers are written.

As a result, resending failed messages or events from operations executed without the required authorization is not possible, since no change pointers were created.

Other Terms

asanwee, asaaemee, change pointer, BDCP2, reprocessing, authority check

Reason and Prerequisites

As of Support Package 12 (SP12), an additional authorization control has been introduced for change pointer (CP) write operations.

The system performs a standard AUTHORITY-CHECK for object /ASADEV/RC.

If the authorization check fails, processing is terminated unless a global override is maintained in the system.

Solution

If CP write processing should be permitted independently of the standard authorization check, implement the SAP Note and perform the following manual steps:

1. Go to transaction n/ASADEV/ACI_PART.
    Create or update the entry with the following value:
     - ACI Global Constant = ACI_AUTH_GLOBAL_CP_WRITE

2. Go to transaction n/ASADEV/ACI_PAR.
    Create or update the entry with the following values:
     - ACI Global Constant = ACI_AUTH_GLOBAL_CP_WRITE
     - Field Value = X

If no global authorization is required, ensure that users are assigned the necessary authorization for object /ASADEV/RC.

Usually events are processed by the workflow user SAP_WFRT ot WF-BATCH with the assigned authorization role /ASADEV/ACI_JOB_ROLE, that includes the necessary authorization.

After maintaining the customizing entry, no further system changes are required.
