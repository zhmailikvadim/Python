Symptom

There are some questions about activation of SCMON & SUSG.

Resolution

1. Data Cleanup: Should we manually clean up the old SCMON data before activating SUSG/SCMON? 

Yes, it is recommended to clean the SCMON data (of course only if old data is not needed any longer) to don't have the old data in SUSG snapshots. This can be done in transaction SCMON and in the menu SCMON > Delete Data.
Then just open SUSG and activate it. This will activate SCMON as well.

2. What is the difference between the "aggregated data" stored by the SUSG daily job and the "Snapshot" created via T-code SUSG? Is the daily aggregated data stored in a buffer/memory and only finalized when a Snapshot is created, or are they two different storage types? 

Technically the SUSG is using the same data base tables for storing the snapshots that user can create via the transaction and for the "aggregated data".
The aggregated data will be updated in the nightly job: new usage will be added, already known usages will be updated (increase usage counter and set new last used date).
As soon as the user created a snapshot in the transaction, the aggregated data will be copied into the snapshot that user can see/download in the transaction.
The snapshot contains all data from the aggregated data up to the creation date.

3. Transaction SUSG aggregates usage data recorded by the ABAP Call Monitor (SCMON). For example, if SUSG activation is configured with an aggregation period from 02/03 to 02/18, and SCMON is set to keep 2 Daily Time Slices, will SUSG still be able to aggregate usage data for the full period (02/03 to 02/18) when creating a snapshot?

If SCMON time slices is set to 2, the first run will only collect the last two days. So when activating on 02/18, it will collect data from 02/17 and 02/18.
But since the data is aggregated into one single big "snapshot", it will contain all days after the 02/17. The usage of the previous day is added in every nightly run to the single big snapshot.

So, it depends: When SUSG is activated on 02/03, then the full period is aggregated into the snapshot. When SUSG is activated on 02/18, only the last to days are contained. 

4. Since SUSG runs at 02:30 AM to collect the previous day's data, what happens if SCMON data is missing for the period of 02/03–02/15? Can the SUSG snapshot still aggregate data for the entire window?

No, when there is no SCMON data, there is no way the SUSG can collect these days. SUSG is only aggregating the last day of SCMON in the nightly run.

5. Is it necessary to ensure that the "SCMON Daily Time Slices" count matches the "SUSG Snapshot" period?

No, running with two 2 days should be fine, as only the last day is aggregated in the nightly run. Only remark: In case there is something wrong in the nightly run (DB connection issues, timeout, memory issued, etc.) the SUSG will try to collect the missing day in the next night.
So if there is only 2 days of SCMON, it could happen that if the nightly job runs in issue that it cannot properly restore the day. 

6. How SUSG aggregation is working?

When SUSG aggregation is activated, all available SCMON time slices will be collected and aggregated into one "snapshot" when it runs for the first time. For all following SUSG runs, just the last day of SCMON data will be collected and aggregated into the usage snapshot.

7. What is the best practice for SCMON/SUSG settings?

It is recommended to start directly via the SUSG and not via the SCMON transaction. The SUSG will then independently activate and control the SCMON. SUSG switches on the SCMON with a record limit of 200 million and a deactivation date in 7 days. The deactivation date is postponed by the SUSG every night by one day into the future, so that both tools would run indefinitely unless they are manually deactivated at some point.

There are currently still some old data in the SCMON, so if a clean restart is needed, the data beforehand could be deleted via the SCMON transaction. Otherwise, the SUSG will also retrieve these old data.

The SUSG retrieves the SCMON data from the previous day every night.

The SCMON data contains time information (e.g., function module XY was called 5 times on Tuesday and 8 times on Wednesday).

The SUSG data contains no time information at all (user can see that the function module XY was called a total of 13 times since the tools were activated, but user no longer know on which day or in which month/year). This makes the SUSG suitable for holding data over longer periods of time.

SCMON timeslices could be set to 365, then SCMON will keep the daily data for 365 days. On day 366, the data from day 1 will be deleted and recording will continue; on day 367, the data from day 2 will be deleted, etc. That means data retention in SCMON is rolling.

By default, the SCMON timeslices are only 7 days; 365 is already quite a lot. As long as the system does not have an extremely large number of users and processes, 365 should still be manageable, but it would recommend keeping an eye on the data volume from time to time (table names SCMON_*). The SUSG has no problem with data volumes over long periods, since the omission of time information in the SUSG allows the data to be much more strongly aggregated/condensed. The SUSG also does not delete any data on its own / does not have rolling data retention.

The SCMON records all ABAP calls (method calls, function module calls, form calls, etc.) from all users in all processes in all clients on all servers. The system becomes about 5% slower for all users as a result.

The user who activates the tools is also used for the jobs (especially for the hourly SCMON data collection job); if the user loses authorizations or system access, the job will no longer run properly, which can cause follow-up problems.

It is to be expected that the number of records in the SUSG will rise sharply at first and then slow down as it reaches saturation. If the increase in records in the SUSG does not slow down after a few days but continues to grow rapidly, this may be due to HTTP calls with dynamic URLs. Details on this can be found in SAP Note 3423771.

The activation popup in SCMON asks for a deactivation time, not a start time. That was probably the main misunderstanding. If user activates via the SUSG (as recommended), then user does not need the popup in SCMON at all.
