Symptom

You have questions about the integration of SAP ERP with Ariba Network, using the Ariba Network Integration 1.0 for SAP Business Suite add-on.

Other Terms

Ariba Network Integration, cXML, SAP ERP add-on

Reason and Prerequisites

This SAP note collects FAQs and common issues.

Solution

FAQ Note for Buyers:

1. cXML OrderRequest_Out:

1.1 Deviant ordering address for vendor

Q: An SAP ERP purchase order is created that contains a different ordering address for the vendor. Why is this ordering address not included in the cXML OrderRequest as the VendorSiteID.

A: In the SAP standard, the SAP ERP add-on does not map the ordering address to the VendorSiteID. You have to add it using the BAdI method MAP_BUS2012_TO_ORDR_OUT. Since the <Identity> element has a generic xsd:any structure, the <IDReference> element has to be created manually. A sample implementation is attached to this note.

1.2 Ariba Network Add-on working with Ariba Downstream

Q: Do I have to take any action if Ariba Downstream is used to create purchase orders?

A: Yes, you do. You have to adapt the classification code mapping in a BAdI. The reason is that Ariba Downstream sends the technical classification code to SAP ERP. The SAP ERP add-on maps the human-readable name of the classification code to the Ariba Network, which then performs a check against the technical code used by Ariba Downstream. To ensure this mapping works, you have to adapt it using BAdI method MAP_BUS2012_TO_ORDR_OUT. A sample implementation is attached to this note.

Ariba Downstream does not support the human-readable code; this cannot be changed for compatibility reasons.

2. cXML ConfirmationRequest_In:

2.1 Confirmation Request rejection indicator

Q: In SAP ERP, a cXML ConfirmationRequest with type "reject" has been received and successfully processed, but the purchase order item does not reflect the rejection. What do I have to do?

A: If your SAP ERP system is on a support package level that is lower than listed in SAP Note 1890715, implement the note in your SAP ERP system.

3. cXML InvoiceDetailRequest_In:

3.1 Invoice number length limit

Q: A cXML InvoiceDetailRequest has been received in SAP ERP, but cannot be processed. It produces error ARBFND_FRW 040 ("Cannot move <IV ID> to data element XBLNR1 with maximum length 000016"). What is the issue?

A: Invoices with an invoice number longer than 16 characters cannot be processed in SAP ERP due to the fact that the field "Reference" (XBLNR) is restricted to this length.

3.2 Tax rate mapping

Q: The tax codes I see in SAP ERP do not match the entries I made in Customizing activity “Incoming Invoice: Map Tax Codes for Logistics Invoice Verification” to map cXML tax rates to SAP ERP tax codes. What is the issue?

A: The add-on takes the tax rate from the cXML file, strips away all trailing zeroes, and then compares the value to the values stored in the Customizing activity. This means that in the Customizing activity you must not enter values with trailing zeroes – even if your cXML contains a rate with trailing zeroes. An entry with trailing zeroes cannot be mapped. – Please check your Customizing settings.

=> Example: cXML tax rate: 16.00% => Enter in customizing for ‘Tax Rate in cXML’: 16

3.3 Invoicing party on invoice is different from invoicing party on referenced purchase order

Q: On the purchase order header, an invoicing party was maintained under 'Partners' that is different from the one specified in the incoming invoice from Ariba Network. Why is the invoicing party from the purchase order ignored when incoming invoices from Ariba Network are posted in SAP ERP, and the invoicing party from the Ariba Network invoice is used instead? This behavior is different from transaction MIRO, where the invoicing party is automatically populated with the value maintained in the purchase order header.

A: In transaction MIRO, users post invoices manually and can verify if the invoicing party taken from the purchase order is the right one to use. If not, the users can adjust the invoicing party.
Incoming invoices from Ariba Network, on the other hand, can be posted automatically. Using the invoicing party from the purchase order automatically could mean that it differs from the invoicing party requested by the supplier.

A similar situation would occur if an invoice from Ariba Network references several purchase orders at the same time. If different invoicing parties exist on these purchase orders, it is unclear which invoicing party should be used.

In the SAP standard, the invoicing party from the purchase order is therefore not used automatically. But you can implement BADI method IF_ARBERP_BADI_INB_MAP~INVC_INV_PARTY_DETERMINATION to determine the invoicing party so that it meets your requirements.

As of ARBERPI1 600 SP10, a sample implementation for this BADI method is provided.

3.4 Support for subsequent credits and debits

Q: Are subsequent credits and debits supported?

A: Currently, we do not provide support for subsequent credits and debits out of the box.

4. cXML InvoiceDetailRequest_Out:

4.1 CC-Invoice IDs

CC invoice IDs must be unique at a given point in time. Duplicated CC invoice IDs that can occur under certain circumstances (for example, buyer-initiated MM CC invoices created by ERS) can cause follow-up processes such as payments to fail.

=> SAP Note  2180426 is available to ensure that CC invoice IDs are not duplicated. Please refer to the note for a more detailed explanation of the changes.

5. cXML ServiceEntryRequest_In:

5.1 Account assignment for service lines:

Q: A cXML ServiceEntryRequest has been received in SAP ERP, but cannot be processed. It produces error SE518 (“No account assignment exists for service line 0000000010”). What is the issue?

A: The system could not find the account assignment information it needs for the service line. This can occur if the purchase order item which is referenced by the service entry sheet has the account assignment category ‘U’ (unknown). The service entry sheet has to be created manually in SAP ERP, and the account assignment category ‘U’ cannot be used.
