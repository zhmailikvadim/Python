Symptom

You have a subaccount in the Cloud Foundry environment of SAP Business Technology Platform. You want to programmatically access the XSUAA configuration using a technical OAuth 2.0 client. An example for this is an integration with SCIM for integrating an identity management system.

Other Terms

xsuaa SCIM apiaccess API role role collection trust

Reason and Prerequisites

API integration to the xsuaa configuration settings

Solution

The Authorization and Trust Management (XSUAA) service provides the apiaccess plan. The apiaccess service plan creates an OAuth 2.0 client when a service instance is created. Using service keys, even applications outside the SAP Business Technology Platform can access the APIs of the Authorization and Trust Management service.

Prequisites for using the apiaccess plan

The apiaccess plan creates OAuth 2.0 clients that include all scopes required to access the XSUAA APIs. The scopes are as follows:

  • xs_authorization.read
  • xs_authorization.write
  • xs_user.read
  • xs_user.write
  • xs_idp.read
  • xs_idp.write

The prerequisite for the service instance creation is that the corresponding platform user also owns these scopes. This is the case if a platform user is an administrator in the corresponding subaccount (in which the service instance should be created), meaning that this user has the role "User & Role Administrator". Check in the Subaccount view in the cockpit (<subaccount_level> -> Security -> Administrators).

Note that you do not need to pass any parameters (like the xs-security.json) during the service instance creation (any input is ignored). After the service instance creation, the OAuth 2.0 client's credentials can be retrieved using a service key. Furthermore, any services instance created by this plan cannot be subscribed.

1. Creating a service key using the cf command line tool

# Log in to the Cloud Foundry environment and navigate to a space in your subaccount.

cf target -o <cf_org> -s <cf_space>

# Create a service instance using the apiaccess plan.

cf create-service xsuaa apiaccess myaccess

# Create a service key to gain access to the credentials of the newly created OAuth 2.0 client.

cf create-service-key myaccess myaccess-sk

# Show the service key with the credentials.

cf service-key myaccess myaccess-sk

2. Retrieve an access token using the new OAuth 2.0 client

To retrieve a token containing the scopes mentioned above, send a POST request containing the URL and credentials from the binding to the xsuaa.

You need the values of clientid, clientsecret, and url from the service key to build the request to the xsuaa. Example:

curl -X POST \
https://my-subdomain.authentication.eu10.hana.ondemand.com/oauth/token \
-H 'Accept: application/json' \
-H 'Content-Type: application/x-www-form-urlencoded' \
--data 'client_id=sb-some-id!a1&client_secret=my-secret&grant_type=client_credentials&response_type=token'

Example response:

{

"access_token":"<token-value>",

"token_type":"bearer",

"expires_in":99999,

"scope":"xs_authorization.read, xs_authorization write, xs_user.read, xs_user.write, xs_idp.read, xs_idp.write",

"jti":"<some-value>"

}

You can use this access token to access endpoints using the apiurl + endpoint suffix contained in the service key. Example:

curl -X POST \
https://api.authentication.eu10.hana.ondemand.com/sap/rest/authorization/v2/apps/roles \
-H 'Authorization: bearer <token-value>' \
-H 'Content-Type: application/json' \
-d '<data required by endpoint>'

SCIM endpoints:

curl -X POST \
https://api.authentication.eu10.hana.ondemand.com/Users \
-H 'Authorization: bearer <token-value>' \
-H 'Content-Type: application/json' \
-d '<data required by endpoint>'

curl -X POST \
https://api.authentication.eu10.hana.ondemand.com/Groups \
-H 'Authorization: bearer <token-value>' \
-H 'Content-Type: application/json' \
-d '<data required by endpoint>'

Note: In the SCIM API, role collections are represented as SCIM groups. To manage role collection assignments to users, use the /Groups endpoint and update the list of assigned users accordingly.
