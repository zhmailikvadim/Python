Symptom

  • When populating the shadow table /TMF/D_INCONSTER (K255), generated K255 lines are duplicated.
  • Shadow data does not override existing data because key fields differ between shadow and source.
  • Fields COD_ITEM and DT_PROD are treated as key fields, preventing date changes via the shadow.
  • No specific error messages are mentioned.

Environment

Tax Management Framework Brazil

Reproducing the Issue

  1. Populate the shadow table /TMF/D_INCONSTER for register K255 with entries where key fields (for example, COD_ITEM and DT_PROD) differ from the original source data.
  2. Generate the EFD ICMS/IPI (SPED Fiscal) files including block K.
  3. Observe that register K255 lines are duplicated in the output.

Cause

The shadow table respects the same primary key logic as the underlying data. Entering shadow records with different values in mandatory key fields (such as COD_ITEM and DT_PROD) creates additional records rather than overriding the original. Redefining primary keys is not supported, and this behavior is by design.

Resolution

  1. Do not change primary key fields (for example, COD_ITEM and DT_PROD) using the shadow /TMF/D_INCONSTER, as overriding primary keys is not supported.
  2. Use the shadow only for permitted changes: quantity (QTD) and the replaced input code (COD_INS_SUBST), or to generate K255 for an item not posted.
  3. Ensure the period fields are filled correctly in the shadow, including “DT_INI” (period start) and “DT_CONS” (MKPF-BUDAT – document posting date), to generate the K255 correctly.
