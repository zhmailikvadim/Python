Symptom

One or more Solr replica pods restart unexpectedly with OOMKilled status.

The following behavior may be observed:

  • Only the leader node (for example: solr-0) remains stable.
  • Replica nodes (for example: solr-1, solr-2, solr-3) restart repeatedly.
  • High container memory utilization is observed on Solr pods.
  • The issue occurs during or shortly after Solr collection replication.
  • Kubernetes events or pod status indicate OOMKilled.

Environment

SAP Commerce Cloud

Cause

During Solr replication, the leader node replicates index data across all collections to the replica nodes.

This process can temporarily increase memory consumption inside the Solr containers. If the Java memory scale factor configured for Solr is too low, the container may not have sufficient available memory to handle the replication workload.

As a result, the replica pods exceed the configured memory limits and are terminated by Kubernetes with OOMKilled.

Resolution

Increase the Java memory scale factor for Solr to allow additional memory allocation for the JVM.

In SAP Commerce Cloud environments, Customers cannot change this parameter directly in production environments. For sandbox environments, scaling options are described in the Scaling Sandboxes with Consumption Credits documentation.

To increase the JVM scale factor in a production environment, a Support case must be opened with SAP requesting an increase of the Solr JVM scale factor.

After adjusting the configuration:

  • Solr pods stabilized.
  • Replication completed successfully.
  • Container memory usage decreased from approximately.

If similar symptoms occur, the JVM scale factor should be increased gradually until the Solr container memory usage stabilizes and replication completes without pod restarts.
