Symptom

You have received a notification mail with title "[Action Required] Client Certificate Authentication change for Inbound communication", advising the deprecation of Client Authentication EKU and its impact to Cloud Integration / API Management service.

The purpose of this KBA is to provide clarification on the actions required on customer side in relation to this.

Note: This article covers only Inbound scenarios to SAP Integration Suite / Cloud Integration / API Management service.
For Outbound scenarios, see SAP KBA 3716130 - [Cloud Integration] Change in Root Certification Authority for SAP Default Certificate.

Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental.

    Environment

    • Cloud Integration
    • API Management
    • SAP Integration Suite
    • SAP Business Technology Platform

    Cause

    "Client Authentication" EKU (Extended Key Usage) is a mandatory attribute of a Certificate, when it's intended for Client Certificate Authentication (a.k.a. 2-way SSL, mTLS).

    Starting May 2026, Certification Authorities (CAs) are phasing out the Client Authentication (ClientAuth) Extended Key Usage (EKU) from publicly trusted SSL/TLS certificates due to an industry-wide security initiative separating server and client certificate roles. 

    • Public CAs began excluding ClientAuth EKU by default starting September 2025.
    • From May 2026*, newly issued public certificates will no longer contain ClientAuth EKU. 
      *Some CAs are extending this deadline to 2027.

    You can find more context in the See Also section.

    Resolution

    Am I Impacted?

    This change can potentially impact Inbound communication to SAP Integration Suite/Cloud Integration endpoints, when Client Certificate Authentication (a.k.a. 2-way SSL, mTLS) is used.
    Inappropriate handling of the certificate may typically result in HTTP 401 errors in inbound SAP Integration Suite/Cloud Integration communications.

    Particularly for API Management endpoints, see below article for more details:
    SAP KBA 3725252 - Client Authentication EKU Deprecation - Impact to Inbound API Management scenarios

    Inbound mTLS Communication

    This potentially impacts the Inbound mTLS communication, when below conditions are all met:

    1. Inbound mTLS is used to call SAP Integration Suite endpoints.
    2. The Client Certificate is signed by a Public, non-SAP CA (e.g., DigiCert), listed in SAP Note 2801396 - SAP Global Trust List.
    3. The Client Certificate is renewed since May 2026.

    Existing Client Certificate signed by Public CAs prior than May 2026 will continue to work until their next renewal.

    Action Required 

    Three possible approaches, in case you are within the impact of this change:

    1. Use Client Certificate signed by [SAP Cloud Root CA] . SAP Cloud Root CA is not restricted by public CA EKU policies and supports ClientAuth EKU.
      Specifically for China Data Centers CN20 and CN40: [CDC Cloud Root CA] 
      • For CPI CF customers who do not require the CSR process, below SAP Blog can be followed:
        SAP Blog: Client Certificate Authentication in SAP Cloud Integration: Configuration and Best Practices

      • If otherwise (i.e., any exposure of Private Key is not tolerated), you can get your keypair signed by SAP Cloud Root CA / CDC Cloud Root CA using SAP BTP Destination service. 
        See SAP Help - Manage Certificates Issued by the SAP Cloud Root CA
        For step-by-step guidance, see SAP Blog - SAP Integration Suite - Client Authentication EKU deprecation

    2. Use Private PKI.
      i.e., to get your Key Pair signed by a private (3rd party) CA that could guarantee the Client Authentication EKU.

      For Cloud Integration, SAP BTP Custom Domain service is mandatory, so that Private CAs could be allowed for inbound mTLS communication.
      For API Management:
      1. Upload the private CA Root Certificate to the tenant-specific trust store in SAP API Management.
        For NEO: SAP Help - Manage Certificates
        For CF: SAP Help - Manage Certificates
      2. Configure mTLS for your Virtual Host
        For NEO: Collect information mentioned in Requesting for Two-Way SSL Certificate, and open a Support Case under OPU-API-OD-OPS:
        For CF: It is a self-service, follow steps Configuring Mutual TLS for Default Domain Virtual Host for default domain virtual host and Configuring Mutual TLS for Custom Domain Virtual Host for custom domain virtual host. 

    3. Switch to other supported inbound authentication approaches, like OAuth or Basic Authentication (not recommended for Productive environments).

      See Also

      What is "EKU" and "Client Authentication EKU"?

      As defined in RFC 5280, Extended Key Usage (EKU) is one of the attributes of a Certificate, indicating the specific purpose of the key usage.
      Relevant values are:

      • Server Authentication (or "TLS Server Authentication", "TLS WWW server authentication", "TLS Web Server Authentication", "id-kp-serverAuth", with Unique Object ID: [1.3.6.1.5.5.7.3.1])
        The Certificate is to be used as Server Certificate (a.k.a. One-Way SSL).
      • Client Authentication (or "TLS Client Authentication", "TLS WWW client authentication", "TLS Web Client Authentication", "id-kp-clientAuth", with Unique Object ID: [1.3.6.1.5.5.7.3.2])
        The Certificate is to be used as Client Certificate (a.k.a. Two-Way SSL).

      "Client Authentication" EKU is mandatory for Authentication purpose. See RFC Definition, Extended Key Usage.
      Public Certificate Authorities had been issuing certificates with both EKUs, before this change.

      Why is this change happening?

      In accordance to Google Chrome's root program requirements, major Certificate Authorities (non-SAP) should stop including Client Authentication EKU in the publicly trusted TLS certificates by May 15, 2026*.
      See: https://www.chromium.org/Home/chromium-security/root-ca-policy/#322-pki-hierarchies-included-in-the-chrome-root-store
      *Some CAs are extending this deadline to 2027.

      Consequently, existing CAs listed in SAP Note 2801396 - SAP Global Trust List will not be issuing TLS Certificate with Extended Key Usage "Client Authentication EKU", and hence the existing mTLS process with publicly signed Client Certificates will have to be updated accordingly.

      In the past, Inbound Client Certificate Authentication scenario is supposed to work like:

      1. HTTP Client gets a Key Pair signed by one of the issuers listed in 
        SAP Note 2801396 - SAP Global Trust List
      2. Configure the Client Certificate in SAP Integration Suite
        • for NEO: Certificate to User Mapping
        • for Cloud Foundry: Service Key
        • for API Management: self-service in Virtual Host / Custom Domain

      Going forward, Inbound Client Certificate Authentication scenario on high level will look like:

      1. HTTP Client gets a Key Pair signed by SAP Cloud Root CA (or CDC Cloud Root CA, for China Data centers). 
        • For customer generated Key Pair, use Destination Service API on Cloud Foundry environment.
        • [Only for CPI on Cloud Foundry] For SAP generated Key Pair, follow SAP Blog: Client Certificate Authentication in SAP Cloud Integration: Configuration and Best Practices
      2. [No change] Configure the Client Certificate in SAP Integration Suite
        1. for CPI NEO: Certificate to User Mapping
        2. for CPI Cloud Foundry: Service Key
        3. for API Management: self-service in Virtual Host / Custom Domain

      How do I verify if my client certificate has Client Authentication EKU? 

      Method 1: Using OpenSSL (Command Line)

      1. Run the following command:
        openssl x509 -in <your_certificate>.crt -text -noout
      2. Validation: 
        In the output, locate the "X509v3 Extended Key Usage" section. It must explicitly include: 
        TLS Web Client Authentication or Client Authentication (OID: 1.3.6.1.5.5.7.3.2)
        If this value is not present, the certificate cannot be used for mTLS client authentication.

      Method 2: Using Windows Certificate Viewer

      1. Open the certificate file (.crt).
      2. Navigate to the Details tab.
      3. Locate the "Enhanced Key Usage" field.  It must explicitly include:
        TLS Web Client Authentication or Client Authentication (OID: 1.3.6.1.5.5.7.3.2)
        If this value is not present, the certificate cannot be used for mTLS client authentication.
