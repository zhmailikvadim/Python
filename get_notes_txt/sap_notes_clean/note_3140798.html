Symptom

  • After converting one order in Detailed Scheduling Planning Board and saving it
  • When the same order is changed in the same session (window) it results in an error message: Changes to order XXXX are not transferred
  • The second change in the same order is not saved even pressing refresh button

Environment

Supply Chain Management

Advanced Planning and Optimization

Reproducing the Issue

  1. Open DS Planning Board /SAPAPO/CDPS0
  2. Select one planned order and set Conversion Indicator
  3. Production planning is saved, changes are adopted for planning version 000.
  4. Once the data is saved, Scheduling Board is refreshed. Planned order has been converted.
  5. This first modified order is now fixed.
  6. A second planned order is selected and the Conversion indicator is set
  7. Production planning is saved, changes are adopted for planning version 000.
  8. Once the data is saved, Scheduling Board is refreshed. Planned order is not properly converted.
  9. Message: ‘Changes to order # are not transferred’:

Cause

When you save the results of your planning in the Detailed Scheduling Planning Board (DSPB) for the first time, the current simsession is merged into the active version in LiveCache. Afterwards a new simsession is opened, and directly before the chosen charts are displayed, the simsession will be refreshed. This later step is done to include all the changes which were merged by other, parallel executed simsessions into the active version in LiveCache after the current simsession was opened.

LiveCache knows about the objects which were changed by other simsessions. These objects cannot be changed anymore in the current simsession.

If new changes are made to these objects in the current simsession, LiveCache will discard them. All other changes will be successfully merged during the next save.

When you set the conversion indicator for the first order and press SAVE, the simsession A is merged into the active version in LiveCache. A new simsession B is opened and refreshed. Between opening the simsession B and refreshing the data, the key completion queue for the first order comes back from S4 and changes the order in LiveCache (new order type, new order number).

Therefore, the first order cannot be changed anymore in the current simsession B. Setting the fixed indicator will not be accepted but discarded by LiveCache during the merge of the simsession B after pressing SAVE. This is also what the popup says. The changes on the first order will not be transferred!

If the key completion queue is very fast, this won't happen at all. This is all a matter of timing and therefore also a matter of the hardware, the load on the system at the time when the key completion is processed etc. 

Resolution

One way to avoid this situation is not to touch the objects which were changed in simsession A anymore in simsession B. In the above example, the order can be converted and fixed before you merge the simsession A.

If this is not possible, you can always leave and re-enter the transaction. This would create enough delay for the key completion queues to finish in parallel.

This is the system design.
