Symptom

When replicating B2BUnit data from S/4HANA to SAP Commerce Cloud via OData $batch POST requests, addresses that are removed from the inbound payload (i.e., partner functions deleted in the backend) are not 

  permanently deleted. Instead, after the SapCpiB2BUnitPostPersistenceHook successfully removes the extra addresses, they are recreated with new PKs during the processing of subsequent changesets within the same  $batch request.

Reason and Prerequisites

correctly identifies extra addresses and removes them via modelService.removeAll(). However, removeAll() leaves dirty state in the model service cache. When the next changeset within the same OData $batch request is processed, saveAll() acts on all dirty items in the session — including the removed addresses — and recreates them in the database with new PKs.

Solution

A single modelService.detachAll() after modelService.removeAll(). detachAll() clears the entire model service cache, preventing dirty state from leaking across changesets within the same $batch request.
