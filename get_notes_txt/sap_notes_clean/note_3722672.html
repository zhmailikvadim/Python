Symptom

In SAP ME integrated with SAP EWM, when using asynchronous collaboration for high‑volume assembly (BOM often exceeding 1,000 components), the material consumption quantity reported by SAP ME to SAP EWM can be higher than the actual quantity consumed in SAP ME. This over‑consumption is observed in productive‑like test systems where the production line takt time is approximately 30 seconds, while the system requires several minutes to process assemblies and generate consumption messages for several thousand components. In the affected scenario, notifications for operations OP1 and OP2 are sent after both operations are completed. However, since OP1 is still being processed when OP2 is generated, this overlapping process results in duplicate consumption in the ERP system. The issue does not occur when the line is effectively "synchronized", meaning that operation OP2 is not started until the notification for operation OP1 has been fully processed. However, this workaround is not feasible because it would interrupt production flow and exceed the maximum waiting time permitted by the equipment.

Other Terms

SAP ME, SAP EWM, consumption quantity, duplicate consumption, asynchronous collaboration, Collaboration Directive, Synchronized,EWM_GOODS_ISSUE_POST, ENABLE_ENH_COMPONENT_TRACE, SFC_ASSY_LOG.

Reason and Prerequisites

The issue occurs in scenarios with the following characteristics:

  • SAP ME is integrated with SAP EWM for component consumption posting.
  • Production uses high‑volume assemblies (for example, electronics boards with several thousand components) and asynchronous collaboration mode, because synchronous processing would exceed acceptable response times on the line.
  • The corresponding EWM Goods Issue Posting Collaboration Directive cannot be set to “Synchronized” due to performance and timeout limitations during processing.
  • Machines on the production line cannot wait longer than approximately 10 seconds for completion feedback, while assembly and message generation can take several minutes for large BOMs.
  • In this setup, the consumption notification for a previous operation (OP1) may still be pending when the user or automation completes the next operation (OP2). This timing can cause the same components to be considered and posted again, resulting in higher consumption quantities in EWM than those actually consumed and recorded in ME.

Steps To Reproduce:

1. Configure a routing with at least two consecutive operations (OP1 and OP2) for an electronic board or similar product with a large number of components.

2. Ensure ME–EWM integration is active and collaboration for the relevant directives is configured in asynchronous mode (not Synchronized).
3. Execute production so that:
- Operation OP1 performs assembly and triggers component consumption,
- Due to the large number of components, the generation of the consumption message for operation OP1 takes significantly longer than the line’s takttime.
4. Complete operation OP2 before the notification for operation OP1 has been fully processed and finalized on the EWM side.
5. Check the consumption quantities reported to EWM for the affected SFC or order.

Actual Result:
The consumption quantity reported by SAP ME to SAP EWM is higher than the quantity actually consumed and stored in SAP ME, and duplicate consumption can be observed at subsequent operations.

Expected Result:
The consumption quantity reported by SAP ME to SAP EWM should always correspond exactly to the actual quantity consumed in SAP ME, without duplicates, even when the EWM Goods Issue Posting Collaboration Directive is configured in asynchronous mode and notifications are not yet finalized when subsequent operations are completed.

Solution

A code correction has been implemented in SAP ME to ensure that, in asynchronous collaboration scenarios with high‑volume assemblies, the component consumption logic does not re‑evaluate or repost components that have already been considered for a previous operation. The fix ensures that notifications generated for subsequent operations do not cause duplicate or excessive consumption quantities in SAP EWM, even if the notification for a prior operation is still being processed. In addition, for the EWM Goods Issue Posting Collaboration Directive, the optimisticLock=true (false) parameter has been introduced to prevent overlapping consumption updates. With this setting, when the consumption logic is being processed for operation OP1 and a parallel request is triggered for operation OP2, the SFC_ASSY update is temporarily blocked until OP1 has completed and its consumption message notification has been sent. After applying the correction, the consumption quantity reported from SAP ME to SAP EWM accurately matches the actual consumption recorded in SAP ME for all operations in the routing.

Impact

The code change implemented to address the above symptom is isolated to the ME–EWM consumption posting logic for asynchronous collaboration scenarios. No functional impact is expected outside of the specific consumption calculation and messaging flow between SAP ME and SAP EWM.
