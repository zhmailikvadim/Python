Symptom

Despite parameter enable_full_automatic_unload = true, tables are not fully unloaded after execution of the table consistency check (TCC). Some columns remain loaded in memory, leading to increased memory consumption. Manual unload is required afterwards.

For example, a table with a partitioning criteria using the columns ("MANDT", "RECORD_ID") and 4 partitions. Both columns remain in memory across all 4 partitions after the table consistency check, even if they were unloaded before the TCC command started.

The consistency check is executed using the following command:
CALL CHECK_TABLE_CONSISTENCY('CHECK', '<SCHEMA_NAME>', '<TABLE_NAME>');

Regarding the enable_automatic_unload parameter, SAP Note 2116157 – FAQ: SAP HANA Consistency Checks and Corruptions states: “If set to true, the main storages of columns loaded by the consistency check are unloaded when the check is finished (SAP Note 2962334) whenever possible"

Starting with SAP HANA 2.0 SPS 06, both parameters enable_automatic_unload and enable_full_automatic_unload are set to true by default.

Other Terms

CHECK_TABLE_CONSISTENCY, table unload, consistency check

Reason and Prerequisites

Reason:

The table consistency check (TCC) performs validations at both the table level and the table‑partition level. Automatic unload is supported only for checks executed at the table-partition level; it is not available for table‑level checks.

For partitioned tables, certain validations—such as verifying the assignment of rows to partitions—are performed at the table level. As a result, columns involved in the partitioning criteria, along with the ROWID, remain loaded after the check. Similarly, the cross‑partition unique constraint check is also a table‑level operation, which causes the internal index column "$uc_<UNIQUE_KEY_NAME>$" to remain loaded.

Additionally, the foreign key check is another example of a table‑level check where automatic unload does not apply.

Affected Releases:

  • SAP HANA 2:

    • and higher
  • and higher

Prerequisites:

  • Running a table consistency check on partitioned tables using only the 'CHECK' option of the CHECK_TABLE_CONSISTENCY.

Solution

Option 1: Execute the 'CHECK' action without CHECK_PARTITIONING_DATA, CHECK_UNIQUE_CONSTRAINTS:

  • CALL CHECK_TABLE_CONSISTENCY('CHECK,-CHECK_PARTITIONING_DATA,-CHECK_UNIQUE_CONSTRAINTS', '<SCHEMA_NAME>', '<TABLE_NAME>')

Option 2: If the missing unloads are critical, Execute the CHECK_PARTITIONING_DATA and CHECK_UNIQUE_CONSTRAINTS actions separately followed by an explicit unload. e.g.

  • CALL CHECK_TABLE_CONSISTENCY('CHECK_UNIQUE_CONSTRAINTS', '<SCHEMA_NAME>', '<TABLE_NAME>');

  • CALL CHECK_TABLE_CONSISTENCY('CHECK_PARTITIONING_DATA', '<SCHEMA_NAME>', '<TABLE_NAME>');

  • UNLOAD "<SCHEMA_NAME>","<TABLE_NAME>"; or UNLOAD "<SCHEMA_NAME>","<TABLE_NAME>" PARTITION (PART_ID);
