Symptom

After using the IMPORT_CONTAINER_FOR_COPY SQL API, applications using the target HDI Container no longer work as expected. For example, tables and views deployed in the HDI Container can't be consumed because of missing privileges.

Other Terms

insufficient privileges, RT users, runtime users, IMPORT_CONTAINER_FOR_COPY

Reason and Prerequisites

Reason:

The IMPORT_CONTAINER_FOR_COPY procedure drops all existing objects in the target HDI Container before the actual import. As a consequence, all privileges on these objects are removed from users or external roles. Even if the import re-creates the previously deleted objects, any privileges on them in users or external roles are not restored. From HDI perspective, the assignment of privileges on deployed objects or container roles is managed at application level and falls outside of the scope of IMPORT_CONTAINER_FOR_COPY.

If the import target container is an HDI Container managed by XSA (HANA Platform) or Cloudfoundry (HANA Cloud) and a default access role is used, the technical users generated in service keys or application bindings are also affected. Applications that rely on these users to connect to the HDI Container cannot access it anymore. If no default access role is used, the technical users have a default set of schema level privileges which are not removed during the container import.

Solution

In general, container object privileges and container roles have to be granted again after the execution of IMPORT_CONTAINER_FOR_COPY. System views GRANTED_PRIVILEGES and GRANTED_ROLES can be used to identify relevant assignments before the import.

The default authorizations of technical users from service keys and bindings of HDI Containers managed by XSA or Cloudfoundry can be restored by the HDI Deployer. This includes the default access role, if defined, or a default set of schema privileges. Execute a deployment, even if no files have changed effectively, to trigger the reassignment.

In special cases of XSA or Cloudfoundry managed HDI Containers the technical users' authorizations have to be restored before the next deployment starts. For example, the deployment might execute additional steps before the HDI Deployer is started and the technical users' authorizations are critical for this step to work as expected. Similar to unmanaged HDI Containers the authorizations have to be restored manually by following the steps below. All statements should be executed with SYSTEM (HANA Platform) or DBADMIN (HANA Cloud).

  1. Check for the existence of a default access role:

-- replace <container> with name of import target HDI Container
SELECT role_schema_name, role_name, creator FROM roles WHERE role_schema_name = '<container>' AND role_name = 'default_access_role';

  1. If a default access role exists grant it to the container's global access role:

-- assign container's default access role to the container's global access role
-- replace <container> with the name of the import target HDI Container
GRANT "<container>"."default_access_role" TO "<container>::access_role";

  1. After a successful deployment the default access role should have been granted twice to the container's global access role:

-- roles included in the container's global access role
-- replace <container> with the name of the import target HDI Container
SELECT grantee, role_schema_name, role_name, grantor FROM granted_roles WHERE grantee = '<container>::access_role' order by 1,2,3,4;

  1. The manually granted role can then be revoked again:

-- clean up after first successful deployment
-- replace <container> with the name of the import target HDI Container
-- replace <user> with the name of the user that executed the GRANT above (SYSTEM or DBADMIN)
REVOKE "<container>"."default_access_role" FROM "<container>::access_role" GRANTED BY <user>;

If no default access role exists the technical users should still have a standard set of schema level privileges (SELECT, INSERT, UPDATE, DELETE, EXECUTE, CREATE TEMPORARY TABLE). Since the container's runtime schema is retained during the import these privileges are not removed:

-- replace <container> with the name of the import target HDI Container
SELECT DISTINCT schema_name, object_name, privilege FROM granted_privileges WHERE schema_name = '<container>' AND grantee_schema_name IS NULL AND grantee = schema_name || '::access_role';

General Recommendations:

  • Don't assign HDI Container object level privileges directly to users or external roles.
  • Wrap HDI Container object level privileges in purpose specific container roles (.hdbroles).
  • Grant HDI Container roles first to external roles and then to users.
