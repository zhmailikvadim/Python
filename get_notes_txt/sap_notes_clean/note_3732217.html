Symptom

Customers report poor payroll performance or inconsistent execution times.
Typical issues include:

  • Payroll runs taking significantly longer than expected
  • Difficulty distinguishing whether the issue is:
    • Application-related (Payroll logic, retroactivity, schema)
    • System-related (work processes, system load)
  • Lack of visibility into:
    • Execution time per payroll run
    • Impact of retro calculations, PCC usage, or payslip generation
  • Manual analysis via SM37 and spool logs is time-consuming and inefficient

Cause

The root causes of perceived payroll performance issues are often unclear due to multiple influencing factors:

  1. No standardized analysis tool
    • Performance checks are done manually (SM37 + spool review)
    • Time-consuming and error-prone
  2. Multiple factors impacting runtime
    • Number of payroll periods (main driver of execution time)
    • Retroactivity (Average Retro) multiplying total runtime
    • Payroll Control Center (PCC) usage:
      • Requires population of declustered tables → higher load
    • Payslip generation during payroll:
      • Adds additional processing overhead
    • Log (protocol) activation:
      • Significantly increases runtime
    • Test vs Live runs:
      • Cluster updates in live runs may consume more resources
    • Schema used
      • Different schemas may have different performance characteristics
  3. System-related factors
    • Delay (no available work processes):
      • Causes longer job duration
      • Misinterpreted as poor payroll performance
  4. Misleading indicators
    • Number of employees is less relevant than:
      • Number of periods processed
    • Jobs with very few periods distort performance metrics due to:
      • Fixed startup time of payroll driver

Resolution

1. Use Program RH_PAYR_PERF_CHECK delivered in note http://me.sap.com/notes/3733753

    2. What the Program Does

    The program:

    • Reads all jobs from SM37 for the selected payroll driver
    • Extracts data from:
      • Job logs
      • Spool outputs
      • Execution metadata

    It automatically analyzes:

    • Execution time
    • Delay
    • Number of employees
    • Number of payroll periods
    • Seconds per period (key performance metric)
    • Average retro
    • Execution parameters:
      • PCC usage
      • Log (protocol)
      • Test vs Live
      • Payslip generation
      • Schema used

    3. Key Performance Metric

    Performance is calculated as:

    Seconds per payroll period

    This allows:

    • Fair comparison between jobs
    • Identification of real performance issues independent of volume

    4. Output Interpretation

    The report includes:

    • Delay
      • Indicates system bottlenecks (no available work processes)
      • Not an application performance issue
    • Seconds
      • Total runtime
    • Periods
      • Main driver of execution time
    • Employees
      • Informational only (less relevant than periods)
    • Average Retro
      • Multiplies total runtime
      • Example:
        • Retro = 6 → runtime ~6x higher
    • PCC Indicator
      • Higher load due to declustered tables
    • Payslip Indicator
      • Additional processing overhead
    • Log Indicator
      • Can significantly increase runtime
    • Test / Live
      • Live runs may consume more resources
    • Schema
      • Helps identify schema-related performance differences

    5. Selection Screen Behavior

    • Minimum number of periods
      • Filters out small jobs
      • Avoids distorted metrics due to fixed startup time

    6. Benefits

    • Eliminates manual analysis (SM37 + spool)
    • Provides instant performance diagnostics
    • Helps distinguish:
      • System issues vs application issues
    • Enables:
      • Faster incident resolution
      • Better customer communication
    • Scalable to large customers running payroll in multiple jobs
