Symptom

There is a short dump with exception CX_ODQ_TIMEOUT when trying to get a shared lock.The RSADMIN-parameters TSN_EXCLUSIVE_LOCK_WAIT and TSN_SHARED_LOCK_WAIT which are meant to avoid such time out are already set in the system. 

Other Terms

cl_odq_tsn get_shared_lock

Reason and Prerequisites

The constellation for a shared TSN lock timeout short dump with exception CX_ODQ_TIMEOUT is the following. Usually there are at least 4 processes involved:

  • Process 1 is an application which posts (pushes) records directly into a delta queue via on-commit logic. To guarantee the right sequence of postings for reading and loading delta from the queue, a shared TSN has to be determined during COMMIT-processing, secured by a shared TSN replication lock.  
  • Process 2 and 3 are delta loads from SAP_BW or another subscriber, which require an exclusive TSN secured by setting an exclusive lock to make sure no pending, just now processed delta is lost. These processes have to wait for the shared replication lock of process 1.) , but set intermittently an aspirant lock.
  • Process 4 is the process which short dumps with CX_ODQ_TIMEOUT because the maximum shared lock wait time is exceeded. Process 4 has to wait until there is no aspirant lock, but these are set either by process 2 or process 3, which wait for process 1. 

The problem arises if process 1 holds the replication lock too long while commiting the data. The posting of ODQ-delta data is normally the last process in the COMMIT-logic of a transaction, the lock should be therefore just held some milli-seconds before the DB-update happens. If the lock is held much longer, then this is a result of calling the statistics update not in the intended and designed way. This would be e.g. an application that posts documents within its transaction "on commit" resp. during COMMIT-processing and is also posting into the delta queue directly.

Solution

With the note correction the lock scenario is improved if the RSADMIN-parameter TSN_EXCLUSIVE_LOCK_WAIT is set (see parameter explanation in note 2011814).

Further the RSADMIN-parameter TSN_SHARED_LOCK_WAIT should be set as well (e.g. 60). The parameter is described in note 2966901.

To solve the issue please apply the Support Package as mentioned below. In urgent cases you can use the correction instructions.

Before you use the correction instructions, make sure that you check SAP Note 1668882 and SAP Note 2248091 for transaction SNOTE.
