Symptom

You use SAP Simple Finance 1503 and archive objects that, until this point, previously also handled CO data with the archiving class K_COSTS (such as the archiving object for sales documents SD_VBAK).

With the new data model from SAP Simple Finance 1503, data of the FI document (table BKPF, BSEG, FAGLFLEXA ,…) and of the CO document (COBK, COEP, …) is written to a common persistence (database table ACDOCA). If you archive CO data together with the corresponding CO object (using the archiving class K_COSTS), the CO line items and (implicitly) the totals calculated from ACDOCA remain in the database and are written to the archive only as copies. This is necessary because, for example, the entry in ACDOCA is still required from the FI point of view. (It is then moved to the historical area of the database by the aging object FI_DOCUMENT.) Due to this, a CO line item (or total) can exist in an archive file and in the database at the same time. In the case of a simultaneous access to the archive and the database, only one of the two entries must be displayed.

Other Terms

SAP Simple Finance 1503, K_COSTS, adjustment of customer programs

Reason and Prerequisites

You use archiving objects that previously also archived CO data.

Solution

To find the points in the program in which this problem may occur, proceed as follows:

  • Textual search for archiving object or field catalog:
    • Call transaction DB15 to find all archiving objects that archive one of the following tables: COBK, COEP, COSS, COSP.
    • Determine all field catalogs of the Archive Information System that are defined with regard to these archiving objects. To do this, you can use transaction SE16 to select to the table AIND_STR1, where you specify the archiving objects as a selection criterion for OBJECT (and restrict ITYPE = S).
    • Search textually in your customer-specific programs for the archiving objects and the field catalogs. Note that, in the case of the field catalogs, the string may be set up in the source code. For this reason, use only the first part of the string. You can use a TREX search as a tool; it is integrated into ABAP in Eclipse as "ABAP Source Search". The strings for archiving objects or field catalogs are transferred as input parameters to the function modules AS_API_READ (field catalog), ARCHIVE_READ_OBJECT (archiving object), or ARCHIVE_OPEN_FOR_READ (archiving object), which read from the relevant archives.
    • Using the found locations, it can usually be quickly determined whether COBK, COEP, COSP, or COSS was read from the archive and if, at the same time, a selection was made in the archive and the database.
  • You must also consider the use of the module AS_API_SYSTEM_SELECT (based on an archive index), which is used more rarely and which, for this reason, is more suitable for a search via the where-used list.
  • In addition to direct archive access, consider the calls of the archive read modules CO_DOCUMENT_READ_FROM_ARCHIVE,  CO_DOCUMENT_READ_FROM_ARCH_RW, and RW_ARCHIVE_LINK. Programs that mix COEP, COSP, or COSS from these modules with data from the database are interesting in this regard.

For all locations found, you must make sure that the entries are not further processed twice. To do this, remove the duplicate entries.
