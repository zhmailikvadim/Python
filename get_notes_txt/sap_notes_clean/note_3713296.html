Symptom

When processing a bank statement return file with Brazilian CNAB 400 format, using transaction FF_5 or the corresponding Fiori app, the system fails to clear the intended open item. The post-processing log shows the error message: (F5 263) The difference is too high for clearing.

This error occurs even when you have confirmed that the payment amount in the return file perfectly matches the open document’s amount, meaning there is no actual monetary difference. A key indicator of this issue is that during the import process, the FEBEP table fields PNOTA (Daybook No.), AVKON (Account) and KKREF (Customer-to-Customer Reference) in the bank statement line items is not populated with the expected document reference.

Environment

  • SAP S/4HANA
  • SAP ERP
  • Financial Accounting (FI)
  • Electronic Bank Statement (FI-BL)
  • Brazil Bank Statements

Reproducing the Issue

  1. Upload a bank statement from your bank at FF-5.
  2. Observe the error F5 263 "The difference is too high for clearing"
  3. Verify that the interpretation algorithm in use is 030.
  4. Confirm that one or more of the fields FEBEP-PNOTA, FEBEP-AVKON, FEBEP-KKREF are not being populated during processing.

Cause

The root cause of this issue is not an accounting difference, but a failure in the document search logic during bank statement processing.

The interpretation algorithm used for clearing (e.g., algorithm 030) is designed to find the open item to be cleared by searching for its document number and fiscal year using the KKREF field, line item number using the last 2 digits from the XBLNR field and customer account using the AVKON field at Function Module J_1B_SEARCH_ITEM_EM_COBRANCA. Field PNOTA is not used on this process, but has the same source as KKREF thus it can be used to evaluate the scenario. The bank statement import program (RFEBKA00) populates these fields using reports J_1BBR20 or J_1BBR30 based on information it reads from the bank’s return file.

For the bank to include the SAP document number in the return file, it must first receive it in the original payment (remittance) file. In the CNAB 400 layout, the standard field for carrying this type of company-specific reference is USO DA EMPRESA (Company Use), located in positions 38-62 of the detail record. The USO DA EMPRESA field should be filled as follows:

Positions 1-10: Customer Account Number (left-padded with zeros to fill 10 characters).
Positions 11-20: Document Reference Number.
Positions 21-24: Document Fiscal Year

If the payment file was generated without this data in the USO DA EMPRESA field, the bank will return the file without this reference. Consequently, the AVKON and KKREF fields remain empty. The interpretation algorithm cannot find a specific document to clear, so it treats the entire payment amount as an unassigned difference, which is larger than the configured tolerance, and issues error F5 263.

Resolution

To resolve this issue, you must correct the process used to generate the CNAB 400 payment file.

  1. Identify the tool used for creating the payment file. This is typically done through a classic print program (e.g., RFFOBR_A) or a DMEEX tree format. You can find this information in the payment run details in transaction F110 on the "Printout/data medium" tab.
  2. Modify the mapping logic in your DMEEX tree or the corresponding user exit/BAdI for the classic program.
  3. Ensure that the SAP customer account number (field KUNNR), accounting document number (field BELNR) and accounting document fiscal year (field GJAHR) is correctly mapped to the USO DA EMPRESA field (positions 38-62) in this order in the output CNAB 400 file.

After making this change, future payment files will contain the necessary reference. When the bank sends the return file, the document number will be present, allowing the bank statement import to correctly populate the fields and clear the open item automatically.
