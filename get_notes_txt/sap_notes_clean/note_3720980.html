Symptom

Users working with the SAP Fiori Launchpad in Microsoft Edge experience the following:

  • Native Browser Popup: A "Login" or "Authentication Information" (Authentifizierungsinformationen) window appears unexpectedly when a user returns to a previously inactive Fiori tab.
  • Navigation Failure: After the popup, the Launchpad fails to load its layout, showing "Resource not found" or errors related to Space IDs and Page IDs (e.g., MyHome or custom Spaces show a loading spinner indefinitely).
  • Visual Indicators: The UI may display a "Circle with a Slash" (🚫) / "Do Not Enter" icon on tiles or as a cursor, indicating that background OData requests are being blocked or are unauthorized.
  • Empty SU53: If the user checks transaction SU53 in the backend, no authorization failures are recorded.
  • Time Variance: The issue occurs even if the system's session_timeout is set to 1 hour, sometimes appearing after only 5-10 minutes of tab inactivity.
  • Console Logs: Developer tools show multiple 403 Forbidden or 401 Unauthorized errors for services like /sap/opu/odata4/iwngw/notification_srv.

Environment

SAP S/4HANA 2022 and others.

Reproducing the Issue

  1. Log in to the SAP Fiori Launchpad using Microsoft Edge.

  2. Open multiple browser tabs (e.g., a Fiori App, a news site, and an email tab).

  3. Stay active in the non-SAP tabs for approximately 5 to 10 minutes.

  4. Switch back to the Fiori Launchpad tab.

  5. Observe the "Authentication Information" popup and the subsequent failure of the Page/Space layout to refresh.

Cause

The issue is caused by Chromium-based "Sleeping Tabs" or "Efficiency Mode" interacting with the SAP Session state:

  • Tab Suspension: To save memory, Edge "suspends" background tabs. This stops the Fiori Launchpad's "Keep Alive" heartbeats.
  • Session Handshake Failure: When the tab "wakes up," the UI5 framework attempts to refresh data (like notifications or space layouts). Because the background process was frozen, the security tokens or session cookies may have flickered or been rejected by the Web Dispatcher/ICM, leading to a 401/403 error.
  • Timeout Mismatch: As noted in system logs, a conflict where sessionTimeoutReminderInMinutes is not correctly aligned with the interval can prevent the UI from properly prompting the user before the session actually dies.

Resolution

  • To resolve this, apply a combination of browser and configuration settings:
    • Option 1: Browser-Level Fix (Recommended)
      1. Ensure the Fiori Launchpad is never put to "sleep" by the browser:
      2. Open Edge and go to edge://settings/system.
      3. Locate the section "Never put these sites to sleep."
      4. Add the URL of your Fiori Launchpad to this list.
  • Option 2: Launchpad Configuration

    • Step 1: Ensure the session reminder is properly configured to "wake" the browser tab before the hard timeout occurs:

      • Verify FLP Shell settings to ensure sessionTimeoutReminderInMinutes is higher than the check interval (steps below):

        • Access transaction /UI2/FLP_CUS_CONF (or /UI2/FLP_SYS_CONF). Ensure the following parameters are synchronized:

          • Set sessionTimeoutIntervalInMinutes to match your backend profile parameter rdisp/gui_auto_logout (e.g., 60).
          • Set sessionTimeoutReminderInMinutes to a generous lead time (e.g., 5 or 10 minutes). This forces the UI to attempt a "wake-up" refresh before the critical cutoff.
          • The Session parameters need to check the interval of communication between framework or application from the front and backend (including Web Dispatcher / Load Balancer). The intent should be to check every few seconds to several minutes. 
          • Special Note: The above strategy may work for the "Sleeping Tab" issue: By having a frequent "Check Interval," the Fiori Launchpad attempts to communicate with the server more often. If Edge sees the tab is trying to perform background network activity (XHR), it is less likely to put it into a "Deep Sleep," preventing the 401 error.

    • If users frequently switch tabs, consider forcing critical apps to open in a new window to isolate their session state:

      1. In the Target Mapping for the affected app, add the parameter:

        • Name: sap-ushell-next-navmode

        • Value: explace

      2. Reference KBA 3713752 for details on managing "Dirty Sessions" and soft navigation collisions.
