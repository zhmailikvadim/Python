Symptom

  • No inbound emails to Service Cloud V2 arrive in the Email Monitor.
  • You are using Microsoft Outlook365 as your email provider, but similar issues can also happen with other providers.
  • You have set up a new Email Channel. 
  • Forwarding from the Email Server to the tenant technical email address has been done, but no email arrives in Service Cloud V2.
  • Looking at the rejected Post Master email, you can see the following messages: 
    • "Your email couldn't be forwarded from xxxxxxx@xxxxxx.com to another email address."
    • "Your message wasn't delivered because the recipient's email provider rejected it."
    • "Remote server returned '550 5.7.520 Access denied, Your organization does not allow external forwarding. Please contact your administrator for further assistance. AS(7555)'".

Environment

Timeline for SAP Sales/Service Cloud

Reproducing the Issue

  1. Create and Configure an Email Channel in SAP Service Cloud V2.
  2. Use an external email provider mailbox as the source inbox for the email channel.
  3. Configure the mailbox with an automatic forwarding rule that forwards all incoming emails to the SAP inbound email address
    (for example: inbound_<tenant>@mail.<region>.crm.cloud.sap).
  4. Ensure the email provider has a security policy that disallows automatic external forwarding.
  5. Send a test email to the email address associated with the SAP Service Cloud V2 email channel.
  6. Observe that:
    • The email reaches the source mailbox.
    • The forwarding attempt to SAP is rejected.
    • A non-delivery report (NDR) is generated with error 550 5.7.520.

Expected Result: The email is forwarded successfully from the external mailbox to the SAP inbound email address and processed by SAP Service Cloud V2.
Actual Result: The forwarding attempt is never delivered to SAP Service Cloud V2.

Cause

  • Email Server Policy is not forwarding the emails correctly, or forwarding is not allowed.
  • The policy in Outlook 365 does not allow forwarding emails to an external email address.

Resolution

  • To solve this issue, the customer must allow external forwarding.
  • How to Allow External Forwarding to SAP Service Cloud V2 (Outlook 365)? - These steps must be performed by a Microsoft 365 / Exchange Online administrator:
    • Option 1 (Recommended): Allow External Forwarding via Outbound Spam Policy. This is the official and supported approach in Microsoft 365. 
      1. Sign in to the Microsoft 365 Defender portal.
      2. Go to → Email & collaboration → Policies & rules → Threat policies 
      3. Open Outbound spam filter policy.
      4. Either:
        • Edit the Default outbound spam policy, or
        • Create a custom outbound spam policy scoped to:
          1. A specific mailbox, or
          2. A group used for SAP forwarding
      5. Set Automatic forwarding to -> On – Forwarding is enabled.
      6. Scope the policy only to the mailbox that forwards to SAP (Optional but recommended). 
      7. Save the policy.
        • Result: Automatic forwarding from Outlook 365 to the SAP inbound address (for exampleinbound_<tenant>@mail.<region>.crm.cloud.sap) is allowed. 
    • Option 2: Allow External Forwarding via Remote Domain (Legacy Method). Use this only if outbound spam policies are not available or already restricted. 
      1. Open the Exchange Admin Center.
      2. Go to → Mail flow → Remote domains.
      3. Edit Default or create a new remote domain.
      4. Configure → Allowed domain: crm.cloud.sap.
      5. Enable → Allow automatic forwarding.
      6. Save the configuration.
        • Result: Forwarding to SAP Service Cloud inbound addresses is permitted at the domain level.
  • Security Best Practices (Strongly Recommended).
    • To minimize risk while allowing SAP forwarding:
      • Scope forwarding permissions to required mailboxes only.
      • Avoid enabling global external forwarding
      • Use:
        • Shared mailboxes, or
        • Dedicated integration mailboxes for SAP Service Cloud
    • Periodically audit forwarding rules.
  • Important Notes
    • Error 550 5.7.520 is not caused by SAP
    • SAP Service Cloud V2 inbound addresses are external by design.
    • SPF / DKIM / DMARC passing does not override forwarding restrictions.
