Symptom

In EWA (EarlyWatch Alert) or transaction ST03 / ST03N (Workload Analysis), the following observations may appear:

  • High response times for RFC processing
  • Long-running RFC function modules
  • RFC runtimes of up to two hours
  • Increased number of parallel RFC calls
  • Long execution times without corresponding dumps

Typical observation patterns include:

  • Long-running RFC entries in ST03N → Workload → RFC Client or RFC Server profile
  • EWA alerts indicating high RFC response times
  • RFC function modules running continuously for extended periods
  • Multiple parallel RFC processes per instance

These observations are related to the SAP Cloud ALM Health Monitoring data collection mechanism and are expected by design.

Environment

SAP Cloud ALM 1.0

Resolution

1. Introduction

This document explains how data collection works for SAP Cloud ALM Health Monitoring and clarifies why long-running RFC entries visible in EWA or ST03 are technically expected and harmless.

The focus is on:

  • The role of the Central Collector
  • The behavior of the Health Monitoring Daemons
  • Runtime characteristics
  • The reason long-running RFC function modules appear in workload statistics
  • The correct interpretation of these entries in EWA and ST03

The objective is to provide transparency into the technical architecture and to clarify the relationship between monitoring observations and the underlying design.

2. High-Level Architecture Overview

SAP Cloud ALM Health Monitoring follows a controlled and resilient scheduling concept:

  • A Central Collector acts as a watchdog.
  • It ensures that hourly data collection is properly scheduled.
  • A daemon (RFC Function Module) runs continuously for two hours:
    • First hour: waiting/preparation
    • Second hour: active metric collection

This overlapping design guarantees reliable and precise metric acquisition, especially for time-critical metrics such as:

  • System heartbeat
  • Instance heartbeat

Due to this design, long-running RFC calls are expected and are part of normal operation.



3. The Central Collector (Watchdog Mechanism)

The Central Collector performs a watchdog check every 5 minutes.

Responsibilities include:

  • Verification that a data collection daemon is scheduled for:
    • The current hour
    • The next hour
  • Ensuring uninterrupted hourly metric collection
  • Automatic recovery of scheduling gaps

The Central Collector does not perform the actual metric collection.

Instead, it ensures that the appropriate daemon RFC function module is active and ready.

4. The Health Monitoring Daemon

The daemon is implemented as an RFC-enabled Function Module triggered by SAP Cloud ALM integration.

Each daemon instance runs for two hours:

  1. Waiting phase
  2. Data collection phase
    (In parallel, the next daemon is already in its waiting phase.)

This overlapping concept ensures a seamless handover between two hourly cycles.

The daemons are scheduled per application server instance.

Therefore:

Number of running daemons = Number of instances × 2

During its runtime, the daemon wakes up every 60 seconds to:

  • Check whether metrics are due
  • Collect the required monitoring data

The collection schedule is predefined by SAP and cannot be customized.

5. Appearance in EWA and ST03

Because each daemon runs for two hours, workload statistics may display:

  • RFC calls with very long runtimes (up to 2 hours)
  • High accumulated response times in RFC statistics
  • Multiple parallel RFC calls (2 per instance)
  • Long-running RFC entries without dumps

In ST03/ST03N, this typically appears under:

Workload → RFC Client or RFC Server profile

In EWA, alerts such as the following may be reported:

  • High RFC response time
  • Long-running RFC daemon processes

Important clarification:

These values represent the technical lifecycle of the daemon and do not indicate a performance issue by themselves.

The reported runtime reflects the design (waiting + collection phases), not continuous CPU consumption.

6. System Resource Consumption

Although the daemon shows a long runtime:

  • Most of the time is spent in a waiting state
  • CPU utilization remains minimal
  • Database activity remains minimal
  • The work process is released during idle waiting periods

Long-running RFC function modules in workload statistics are therefore expected behavior and can be considered harmless as long as:

  • No excessive CPU usage is observed
  • No abnormal dumps occur
  • No unusual system-wide performance degradation is present

Only if high CPU consumption, dumps, or system instability accompany the RFC entries should further investigation be performed.

7. Summary

SAP Cloud ALM Health Monitoring uses a robust, overlapping daemon-based architecture:

  • A watchdog (Central Collector) performs scheduling checks every 5 minutes.
  • A two-hour daemon lifecycle ensures seamless hourly data collection.
  • Metrics are evaluated every 60 seconds during collection.
  • Two parallel daemons run per instance.
  • Long runtimes in ST03 and EWA are intentional and technically harmless.

This design ensures:

  • High reliability
  • Accurate metric timing
  • Stability for critical heartbeat monitoring

Long-running RFC entries related to SAP Cloud ALM Health Monitoring do not indicate a system issue when resource consumption remains low.
