Symptom

  • Background job for ACE_PROPOSAL_RUN (transaction ACEPROPOSALRUN) shows excessively long runtime compared to typical runs.
  • Only a small set of purchase order accrual objects take extremely long; most objects complete within seconds.
  • SM37 shows multiple parallel active runs; spool contains the informational message “The accrual amount of object … has been proposed.”
  • Some affected purchase order accrual objects cannot be displayed in transaction POACTREE03.
  • A run was canceled with message “Job canceled due to terminated process or program; see the system log.”
  • No short dump found in ST22 for the canceled run.

Environment

  • SAP S/4HANA Finance
  • SAP S/4HANA
  • Financial Accounting (FI)

Reproducing the Issue

  1. Schedule or monitor the background job running program ACE_PROPOSAL_RUN for purchase order accruals in SM37.
  2. Observe that overall job duration increases to 30h+ while most objects finish quickly.
  3. Identify that excluding a small set of purchase orders from selection leads to normal runtime.
  4. Attempt to display one of the slow accrual objects in POACTREE03 and observe it is not displayable or takes extremely long.
  5. Check the job log/spool and note repeated informational lines “The accrual amount of object … has been proposed.”

Cause

The affected accrual objects have a time-dependent data setup with LIFE_END_DATE set to 31.12.9999 in table ACESOBJ_ITEM. This “end of time” setting causes the system to calculate values for thousands of periods when displaying or processing the object, leading to very poor performance in POACTREE03 and long ACE_PROPOSAL_RUN runtimes. The problematic objects were legacy/migrated entries.

Resolution

  1. Identify affected objects:
    1. From the slow proposal run, list the objects that do not complete or are “processed” for unusually long.
    2. Attempt to open each object in POACTREE03 to confirm display slowness or failure.
  2. Verify time-dependent data:
    1. Check table ACESOBJ_ITEM for each affected object.
    2. Look for entries with an “end of life” set to 31.12.9999 (or an unrealistically distant date) that cause evaluation over thousands of periods.
  3. Correct the end-of-life date:
    1. Determine the appropriate end-of-life date for each object based on business validity.
    2. As no standard transaction exists to change this, perform a controlled data correction (for example, update ACESOBJ_ITEM) following your organization’s change and approval procedures.
  4. Validate in a non-production system first:
    1. Apply the correction in a test environment.
    2. Re-run ACE_PROPOSAL_RUN and verify normal performance and that POACTREE03 displays the object.
  5. Implement in production under change control:
    1. Apply the same correction to the production system.
    2. Re-run ACE_PROPOSAL_RUN to confirm runtime returns to expected levels and POACTREE03 displays correctly.
  6. Interim workaround (optional):
    1. Exclude the identified problematic purchase orders from the proposal run until data is corrected.
