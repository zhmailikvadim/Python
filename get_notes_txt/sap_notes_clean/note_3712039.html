Symptom

  • Primary decision buttons, such as Approve and Reject, remain enabled and clickable
  • This behavior differs from the expected experience in SAP My Inbox, where completed tasks are not actionable
  • Upon opening the completed task, the user interface remains interactive rather than switching to a read-only state
  • Secondary UI elements, such as the Upload button for attachments and the Comments section, remain functional
  • When a workflow task from SAP Build Process Automation (SBPA) is completed, it appears in the Task Center Outbox tab 

Environment

SAP Build Workzone

Reproducing the Issue

  1. Open SAP Task Center.

  2. Complete a workflow task originating from SBPA (e.g., Approve a request).

  3. Navigate to the Outbox tab.

  4. Select the completed task.

  5. Observe that Approve, Reject, and Upload buttons are still active and can be clicked

Cause

  • This is a known product limitation in the integration between SBPA and Task Center
  • Currently, SBPA forms do not support the signaling required to deactivate UI controls or trigger a "read-only" mode specifically for the Task Center Outbox view

Resolution

  • Although the buttons are visually active and clickable, the task cannot be approved or rejected a second time; the underlying workflow engine prevents duplicate processing, ensuring there is no risk to data integrity
  • SAP Development has planned an improvement to address the visual state of these tasks:
    • Planned Improvement: Deactivation of decision buttons/actions when an SBPA form is displayed in the Task Center Outbox
    • Estimated Delivery: Planned for implementation in Q1 2026 (Note: Target delivery can change without further notice) - Please check any newer release notes as this Knowledge Base Article may get dated without any further updates.  The goal of this KBA is simply to state that it is a known limitation as of February 2026.
    • Scope Note: A full "read-only" mode for all form fields is currently not in scope
    • Potential complications or effort on any development tied to this limitation (That may delay any estimated dates):
      • The issue is likely a missing handshake between the Task Center's generic UI and the SBPA Form's specific metadata
        • The SBPA framework likely needs to modify the OData or REST response sent to Task Center when the Outbox is queried wherein when the TaskStatus is COMPLETED, the framework must set a flag (ex: isActionable = false or readOnly = true) in the metadata
        • Then a Backend Decision must stop sending "Positive" and "Negative" decision options (Approve/Reject) in the GET response for tasks retrieved from the Outbox collection
      • Then to Enhance the Task Center UI Controller (Front end)
        • The Task Center shell needs to interpret the new flags coming from SBPA provider 
          • Then a Conditional Rendering must happen where the development team must implement a logic in the FIori XML views or JS Controllers to check the task origin (SBPA) and status
          • Then the Button Visibility logic must be improve wherein if the task is from Outbox, a visible "{=$(Status) !== 'COMPLETED')" or enable="false" property needs to be bound to the footer buttons
      • Then a coordination Form to Shell communication is neeeded
        • Since the "Upload" and "Form" sections are often hosted in an IFrame or as a separate Component, the communication between Task Center Shell and the SBPA needs alignment
          • Then some new message handling is required wherein the Shell must send a "Read-Only" event to the embedded SBPA form so it knows to disable its internal "Upload" and "Comment" button
      • Then Testing
        • Because Task Center aggregates many providers like S/4HANA, Ariba, SuccessFactors and others, the development must ensure that changing the behavior for SBPA does not accidentally hide buttons for other backend systems that do support Outbox actions
