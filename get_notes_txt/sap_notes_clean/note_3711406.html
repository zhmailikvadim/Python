Symptom

In SAP HANA systems with cross or cyclic Smart Data Access (SDA) configurations, PostCommitExecutor threads may become stuck during federation commit operations. This can result in database deadlocks and potentially lead to system unavailability.

Reason and Prerequisites

Issue number 353795

Reason:

Due to a programming error in SAP HANA, PostCommitExecutor threads can deadlock when processing transactions involving federation (Smart Data Access) operations in cross or cyclic remote source configurations. The issue occurs because post-commit operations, both with and without federation involvement, are mixed in the same PostCommitExecutor job queues. Since these queues handle operations sequentially, this mixing can lead to deadlock situations.

Affected Releases:

  • SAP HANA 2:
    • Revisions of SPS00 - SPS06
    • Revisions <= 079.08 (SPS07)
    • Revisions <= 089.01 (SPS08)

Prerequisites:

  1. HANA-to-HANA Connection: The issue affects only HANA-to-HANA SDA scenarios using the hanaodbc adapter.
     
  2. Cross or Cyclic SDA Setup: Two or more HANA systems have remote sources pointing to each other, creating either:
    • Cross setup: System A → System B AND System B → System A
    • Cyclic setup: System A → System B → System C → ... → System A
  1. DML Read/Write Mode: Remote sources are configured with dml_mode=readwrite. 
     
  2. Concurrent Write Operations: Multiple connections perform DML operations (INSERT, UPDATE, DELETE) through Federation simultaneously.

Solution

Apply one of the following SAP HANA Revisions:

  • SAP HANA 2:
    • Revisions >= 079.09 (SPS07)
    • Revisions of SPS08: SAP is currently working on a permanent solution to this issue. This SAP Note will be updated as soon as a revision containing the solution is available
    • or higher

  • or higher

Workaround:

  • Option 1 - Remove Cyclic Dependencies

    Modify the application logic and SDA configuration to eliminate cross or cyclic remote source setups. Review your remote source topology and restructure to avoid circular references.
    To identify potential cross or cyclic dependencies, document remote sources on all systems.
  1. On each system, run:

    SELECT REMOTE_SOURCE_NAME,
    CONNECTION_INFO,
    ADAPTER_NAME
    FROM REMOTE_SOURCES
    WHERE ADAPTER_NAME = 'hanaodbc';

  2. Extract hostname/port from CONNECTION_INFO (typically contains ServerNode=hostname:port)
  3. Create a topology diagram showing System A → System B → System C relationships
  4. Look for cycles where any path leads back to the starting system

Note: If you are using different hostnames or virtual IPs (for example, defined in the hosts file), it is recommended to use the physical hostname or IP address when drawing the topology diagram.

  • Option 2 - Enable Prioritized PostCommit Handling

    This workaround separates Federation-involved post-commit operations into a dedicated PrioritizedPostCommitExecutor queue, preventing circular dependencies.

    Important: Apply this configuration on ALL systems involved in the cross/cyclic SDA setup.

    ALTER SYSTEM ALTER CONFIGURATION ('indexserver.ini', 'SYSTEM')
    SET ('smart_data_access', 'allow_loopback') = 'true'
    WITH RECONFIGURE COMMENT 'SAP Note 3711406. Workaround for Bug 353795. Revert after upgrade to a revision containing the fix';
  • Option 3 - Enable SDA Safe Mode

    Enabling Safe Mode for ODBC connections will not prevent this issue, but it can help minimize business impact if your HANA system encounters such a situation.
    If the issue occurs, you can resolve it by restarting the scriptserver in one of the involved systems.

    To enable Safe Mode, please refer to SAP Note 2602761 and the SAP HANA Administration Guide for SAP HANA Platform for detailed instructions.

 

Disclaimer

The SAP HANA configuration parameter [smart_data_access] allow_loopback is an internal parameter and may only be used in the context of this SAP Note. Using such a parameter requires that the SAP HANA system has the revision specified in the SAP Note. Using internal parameters outside of this given scope is not supported. Please unset this parameter once you have upgraded to a revision containing the permanent solution.
