Symptom

When BOM is being exploded in production orders, change number alternative dates are not being considered. System uses the change number header date instead, leading to BOMs being exploded differently.  

Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental. 

Environment

  • SAP ERP Logistics (LO)  
  • SAP R/3
  • SAP R/3 Enterprise
  • SAP ERP Central Component
  • SAP ERP
  • SAP Enhancement package for SAP ERP
  • SAP Enhancement package for SAP ERP, version for SAP HANA
  • SAP S/4 HANA on premise
  • SAP S/4 HANA Cloud Private Edition

Reproducing the Issue

  1. Create production orders in t-code CO01. 
  2. Input required data and perform BOM explosion. 
  3. Verify the exploded BOM data. System determines base quantity 2 on 01.01.2027. 
  4. Check BOM data in t-code CS03. Base quantity 2 should become valid since 01.01.2028. 
  5. Check change number data in t-code CC03, BOM for material 13270 is assigned to alternative date of change number 500000151769.

Cause

The header material 13270 has an Overall OCM profile maintained in work scheduling view. 

OCM (Order Change Management) functionality relies on only one validity (change number header date) per change number. Alternative dates are not supported. 

Resolution

System behaves as per standard design. 

If OCM functionality is being used, avoid using change number alternative dates in BOM maintenance.
