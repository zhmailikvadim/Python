Symptom

You process a high number of SAP Business Workflow instances or events in your system and have problems distributing the tRFC load especially for workflow processing.
You use S/4 HANA release and want to have similar instructions on how to distribute the load as mentioned in SAP Note 888279 for ERP systems.

Other Terms

WORKFLOW_LOCAL, WORKFLOW_EVENT, WORKFLOW_REFERENCE, SMQS, tRFC, load balancing, MAX_CONN

Reason and Prerequisites

Workflow and tRFC usage

Workflow events and background steps are processed via tRFC in order to parallelize the processing and also change the user to the workflow system user SAP_WFRT.
For this purpose dedicated RFC destinations are used.
Depending on the general system load, number and specifics of workflows and workflow background steps processed, this can lead to a high load on the RFC and delays in tRFC processing.
This note describes the options you have to distribute this load.


RFC Destinations
Starting with S/4 HANA release 1809 and all following S/4 HANA releases the RFC destinations for processing SAP Business Workflows have changed. These destinations are now automatically created via workflow customizing transaction SWU3. In SWU3 you can also find more details under "Edit Runtime Environment -> Configure RFC Destination".

Workflow RFC destinations:
WORKFLOW_LOCAL_<client> - is a logical destination - connection type L - and used for processing workflow background steps
WORKFLOW_EVENT_<client> - is a logical destination - connection type L - and used for processing workflow events
WORKFLOW_REFERENCE_<client> - is a ABAP destination - connection type 3 - and used as a reference destination for both above mentioned logical destinations

We always recommend to use the automatic creation of these destinations via SWU3 (Edit Runtime Environment -> Configure RFC Destinations, then choose "Execute Activity"). In case of issues you can manually delete the destinations in SM59 and recreate them via SWU3, so that they are created with the standard settings that should work.

Never change the name, user or connection type of these RFC destinations!
Otherwise you risk errors during workflow runtime or workflow customizing.

Solution

Implement one or more of the following options to optimize the workflow tRFC processing.

A few prerequisites apply for all of these options:
Always register the WORKFLOW_LOCAL_<client> and WORKFLOW_EVENT_<client> destinations in SMQS, see section 1.
Never change the name, user or connection type of the standard workflow destinations, see also information above. 

1. Registration of logical workflow destinations in SMQS - QOUT Scheduler

Always register the two logical destinations WORKFLOW_LOCAL_<client> and WORKFLOW_EVENT_<client> in transaction SMQS!
Otherwise you risk that all available dialog work processes for tRFC are captured by workflow processing and the processing is done synchronously instead via tRFC. This can cause delays for the dialog users processing tasks or starting workflow events via Apps or even can lead to authorization errors or enqueue errors.

The parameter MAX_CONN specifies the maximum number of dialog work processes that are used for tRFCs running on this destination. It is one of the main regulating factors to optimize tRFC processing.
Finding the optimal value for this parameter per destination can be a challenge:
It needs to fit to the number of dialog work processes available in your system and also depends on other tRFC processing and dialog work process usage. 
If you choose a higher value, processing tRFC units of other RFC destinations might be delayed, because there are no free dialog work processes anymore that can be used for tRFC processing. SAP Note 74141 contains an overview about RFC relevant parameters, that e.g. configure how many percent of resources (dialog work processes) are used for tRFC processing.
The other risk of a too high value for MAX_CONN is on tRFC receiver side: The receiver methods must be able to handle a high number of parallel requests. For example the number of enqueues could overload the enqueue server. See section 5 to restrict processes for such specific events.

If you choose a lower value, entries on this RFC destination can pile up in transaction SM58 and might get delayed.
Monitor the tRFC queue via transaction SM58 especially during high load situations to optimize the value.

In addition, under "AS Group", you have the option to define an RFC server group. If you are using more than one server, it may also be beneficial to use certain servers exclusively for RFC. Use Transaction RZ12 to create an RFC server group that contains only the servers for the RFC processing, and save it in the SMQS qOut Scheduler. Adjust the settings in RZ12 so that a large number of dialog work processes can be used for RFC, since no normal dialog users work on these servers.
Note that this setting applies to all registered RFC destinations! Therefore, if you have registered several destinations, these will now only be carried out on these servers.

2. Restricting the Workflow Processing to separate Servers
By default the WORKFLOW_REFERENCE_<client> destination uses load balancing to distribute the workflow load optimally to the different servers depending on the current situation when the tRFC execution is started.
So from performance perspective it should not really be necessary to change the standard settings.
However if you want to restrict the workflow tRFC processing to separate servers only, you can adjust the settings of WORKFLOW_REFERENCE_<client> destination in SM59.
You can either specify a specific server or you can determine one or more servers for workflow processing by means of a logon group.
If you are using Load Balancing through a logon group, you should set the "ext. RFC enabled" indicator in the attributes of the defined logon group in transaction SMLG (double-click Logon Group => Attributes). After you have set the type of the load distribution to the type R in the table RZLLICLASS, the RFCs are then distributed in the round robin procedure to the individual servers.
Note that these settings can cause problems if you have a very high load. Set the Gateway parameter as described in Note 384971. Extreme loads can cause the gateway to overflow and the RFCs to terminate. If this happens, make sure you have sufficient resources (memory, dialog work processes, CPU) to process the load.

3. Start certain workflows on certain servers

You can execute each workflow on a specific server. To do this, you must first find out which workflows run on your system. If the workflows are triggered by events, there is a corresponding event entry for the workflow in Transaction SWE2, which displays the link between the above "triggered event" and the workflow or other event receiver. In this transaction you can define an RFC Destination as "Destination of Receiver" for each linkage. In SM59, you then create type 3 RFC Destinations that point to individual servers or, by means of Load Balancing, to several servers (set the "ext. RFC enabled" indicator in SMLG). You should use SAP_WFRT as a user. Also see the comments in section 2.
Define these new destinations in the corresponding event type linkages in transaction SWE2.
If the event is now triggered, the workflow is started on the server that is defined in the destination. 
Register the new destination in SMQS and specify a value for MAX_CONN that fits your needs, see section 1.
Make sure that the MAX_CONN value of all workflow destinations summarized does not exceed the number of available dialog work processes meant to be used for tRFC processing.

4. Start certain workflows via a separate logical RFC destination to bypass a possible bottleneck

You do not want to execute a workflow an a specific server or a specific logon group, but you want to avoid that it is delayed by many tRFCs in SM58 that wait to be executed on destinations WORKFLOW_LOCAL_<client> or WORKFLOW_EVENT_<client>.
You already found out which workflow event type linkage you need to adjust as mentioned in section 3.
Copy the WORKFLOW_EVENT_<client> destination in SM59 and choose a new name. You can leave all other settings as they are. It is okay for all logical workflow destinations to use the same workflow reference destination.
Define this new destination in the corresponding event type linkage in transaction SWE2. 
Register the new destination in SMQS and specify a value for MAX_CONN that fits your needs, see section 1.
Make sure that the MAX_CONN value of all workflow destinations summarized does not exceed the number of available dialog work processes meant to be used for tRFC processing.

5. Restrict certain workflows or workflow events via a separate logical RFC destination to avoid that they are flooding the system

You identified problematic workflow events that are flooding the system and delaying all other workflow tRFC processing.
Of course this situation needs to be checked by the responsible application that creates these events. Check to which component the event receiver belongs to and open an incident on that component.
You do not want to deactivate the event type linkage completely, because the events are needed from a business perspective. 
But you observe that the system behaves badly when you increase the MAX_CONN for the general workflow destinations any further, e.g. the enqueue server gets problems when you parallelize the same events too much.

The solution is similar as described in section 4.
Copy the WORKFLOW_EVENT_<client> destination in SM59 and choose a new name. You can leave all other settings as they are. It is okay for all logical workflow destinations to use the same workflow reference destination.
Define this new destination in the corresponding event type linkage in transaction SWE2.
Register the new destination in SMQS and specify a value for MAX_CONN that fits your needs, see section 1. In this case you want to specify a lower value than for WORKFLOW_EVENT_<client> to restrict the parallel processing of these events to a certain number of processes.
Make sure that the MAX_CONN value of all workflow destinations summarized does not exceed the number of available dialog work processes meant to be used for tRFC processing.
These settings will only apply for new events created after the changes, not for events already waiting in tRFC queue.
