Symptom

  • When configuring multiple Corporate Identity Providers (Corp IdPs) that use the same OpenID Connect (OIDC) issuer, you may encounter token assignment issues. If different Corp IdPs share the identical issuer, the system cannot reliably determine which configured IdP an incoming ID token belongs to. As a result, authentication may fail or tokens may not be mapped correctly to the intended application
  • Exception appears: Provided token cannot be matched to a unique corporate IdP

Environment

SAP Cloud Identity Services

Cause

In OpenID Connect, the issuer (iss) claim is typically used to identify the Identity Provider that issued the ID token. However, if multiple Corp IdPs use the same OIDC issuer (iss) and do not provide additional distinguishing claims, then the ID tokens cannot be uniquely associated with the correct configured Corp IdP.

Resolution

Please ensure that your Corporate Identity Provider is configured to issue ID tokens containing an unique aud claim that clearly distinguishes the application.

To ensure correct token assignment, the ID token must contain an aud (Audience) claim. The aud claim must uniquely identify the target application.

The  aud claim should hold the ClientId of the oidc configuration in the target Application. This means that when you have Corporate IdP A, and CorporateIdP B - both sharing the same OIDC issuer - the aud claim should hold the value of the clientId from the config inside the corresponding Corporate IdP. For example, when using Corporate IdP A, the aud should match the clientId of configuration in Corporate IdP A.

Keywords

same issuer, OIDC corporate IdP, unique token
