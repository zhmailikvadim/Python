Symptom

During cleanup, if the truncation of logging tables fails, the cleanup falls back to deleting records individually. This behavior negatively impacts DB performance and resource utilization, especially if a massive quantity of logging table records are to be deleted. In certain database platforms, this is further exacerbated by undo segments.

Other Terms

downtime minimization tool for SAP S/4HANA, NZDT

Reason and Prerequisites

Any of the following add-on support packages is installed in your system landscape:

  • NZDT 1.0 SP03

Solution

  • The fallback logic that individually deletes logging table records has been removed
  • A runtime parameter (CLEANUP_TRUNCATE_BEST_EFFORT) has been introduced to allow suppression of error messages in the event of a failed truncate
    • The runtime parameter is automatically added in the analysis tree and copied to the execution tree when the execution MT_ID is created
    • If the truncation of a logging table fails, processing continues and a warning will be shown in the log after cleanup ends
  • While it's not recommended, it's possible to manually turn off the runtime parameter to enforce table truncate before table drop by adjusting /NZT/RTP4MTOBJ-PARAM_VALUE = space. This would result in the following behavior:
    • If a truncate fails, the entire cleanup will abort with an error
MT_ID PARAM_NAME OBJ_NAME PARAM_VALUE
[EXECUTION_MT_ID] CLEANUP_TRUNCATE_BEST_EFFORT   X

This note is not only relevant for the sender, but also the central and receiver systems because it bundles a a few more cleanup corrections relevant to all 3 systems.

Required Action

In order for this note correction to be effective, the note must be implemented before:

  • Either executing any of the following execution tree activities for the cleanup:
    • NZTR1_DELE_NZDT_OBJ_S
    • NZTR1_DELE_NZDT_OBJ
    • NZTR1_DELE_NZDT_OBJ_R
  • Or executing any of the following cleanup programs locally:
    • /NZT/CLEANUP_SENDER
    • /NZT/CLEANUP_CENTRAL
    • /NZT/CLEANUP_RECEIVER

Implement the correction instructions in transaction SNOTE on the following systems of your system landscape:

  • Sender system
  • Central system
  • Receiver system
