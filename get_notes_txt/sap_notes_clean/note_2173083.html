Symptom

  • For a regular database and an encrypted database with the same size and the same data
  • The dump file size of the encrypted database is larger than the dump file size of the regular database.

  • When dumping an encrypted database, dump compression option is not reducing the size of the dump
    • When dumping a database with lots of encrypted columns using compression, the dumps are somewhat smaller but compression does not seem very effective

Environment

SAP Adaptive Server Enterprise (ASE) - All versions

Reproducing the Issue

The aim is to setup 2 databases with the same size and contents, where one database is encrypted and the other is not.
For encrypted database preparation and details, refer to KBA https://me.sap.com/notes/2129898 - How to create an encrypted database

  1. Create a regular database:

    create database test on test_dat=100 log on test_log=30 

  2. Create an encrypted database

    create database test_en on testen_dat=100 log on testen_log=30 encrypt with dbkey

  3. Populate the same data on these two databases

  4. Dump both databases using with compression = 7

  5. For the regular database:

    dump database test to "/tmp/test_1.dmp'
    stripe on"/tmp/test_2.dmp'
    with compression = 7
    go
    ...
    Backup Server: 3.43.1.1: Dump phase number 3 completed.
    Backup Server: 4.188.1.1: Database test: 1802 kilobytes (100%) DUMPED

  6. For the encrypted database:
     
    dump database test_en to "/tmp/test_en_a.dmp'
    stripe on"/tmp/test_en_b.dmp'
    with compression = 7
    go
    ...
    Backup Server: 3.43.1.1: Dump phase number 3 completed.
    Backup Server: 4.188.1.1: Database test_en: 1804 kilobytes (100%) DUMPED.

  7. On OS level, run ls -al to show the dump file sizes
    This shows the encrypted database is larger than the dump file size of the regular database:

    regular database:
    -rw-r--r-- 1 sybase sybase  73069 May 22 11:43 test_1.dmp
    -rw-r--r-- 1 sybase sybase 152158 May 22 11:43 test_2.dmp

    Total 225227 bytes

    encrypted database:
    -rw-r--r-- 1 sybase sybase 245960 May 22 11:44 test_en_a.dmp
    -rw-r--r-- 1 sybase sybase 389511 May 22 11:44 test_en_b.dmp

    Total 635471 bytes

Cause

This is expected behavior.

  • ASE optimizes the copying of encrypted data by copying ciphertext to make sure the data remains encrypted in disk.
  • Therefore, it is expected that the dump file size of an encrypted database is larger than dump file size of a regular database.

  • Compression will be not be very effective on an encrypted database as the encryption removes the predictable and repeated sequences of bytes in normal data
  • This is the element that compression relies on to reduce size and compression is not effective on relatively random sequences of bytes.
    • This behaviour can be mimicked on OS level
      • Take an uncompressed dump of an encrypted database
      • Use an OS utility (compress, gzip, etc) to compress that dump
      • Observe that file size of the compressed file is not (much) smaller than the original file

Resolution

  • The file size of an encrypted database dump is is documented in ASE Manual page Encrypted Columns and dump database:

    Note
    The file size of a dump of an encrypted database is larger than that of a regular, unencrypted database

  • Measures that can help reduce size of a dump include:
    • For text or image data, use the "LOB compression" feature
    • Take a dump of an unencrypted database and then compress / encrypt on OS level in storage

Keywords

"encrypted columns", "encryption key", ciphertext
