Symptom

Error when trying to use Create Table functionality via SLT to GoogleBigQuery:

 400-client key is not found in /G00G/CLIENT_KEY table

Environment

  • SAP Landscape Transformation Replication Server
  • BigQuery Connector for SAP version 2.6, 

Resolution

 Because the /GOOG/ namespace and its associated tables (like /GOOG/CLIENT_KEY) are part of a Google-developed connector and not a standard SAP product, SAP Technical Support typically does not have authorization or the internal documentation to troubleshoot the contents of those tables. 

Since this is a third-party add-on from Google's perspective within the SAP ecosystem, the responsibility for addressing the "Client key not found" logic error rests with Google Cloud Support. 

The BAdI functions written by Google are programmed to look for a specific identifier in your SLT configuration and match it against an entry in /GOOG/CLIENT_KEY. The "400" error is the connector's way of saying it successfully ran its code but found a data mismatch in your setup: 

Logic Mismatch: The Google Cloud Key Name you entered in transaction /GOOG/SLT_SETTINGS does not exist in the /GOOG/CLIENT_KEY table.
Case Sensitivity: SAP tables are often case-sensitive for these custom fields. If the key is "BigQuery_Key" in one place and "BIGQUERY_KEY" in the other, the Google BAdI will fail to find it.
Missing Entry: The table might simply be empty or missing the specific key required for that Mass Transfer ID. 

As per Google Cloud Documentation:

Check the mapping: Open transaction /GOOG/SLT_SETTINGS. Note the exact string in the Google Cloud Key Name field for your specific Mass Transfer ID.
Verify the table: Open transaction SM30 for table /GOOG/CLIENT_KEY. Ensure a row exists where the Client Key Name matches the string from Step 1 exactly (check for trailing spaces).
Contact Google: If the data is identical and the error persists, you must open a case with Google Cloud Support (under the SAP on Google Cloud category), as they own the code that performs this lookup.
