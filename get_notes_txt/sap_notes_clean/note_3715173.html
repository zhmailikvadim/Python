Symptom

Asymmetric double verification principle is being tried to set up in a system, but it does not work.

"Image/data in this KBA is from SAP internal systems, sample data, or demo systems. Any resemblance to real data is purely coincidental." 

Environment

  • SAP Release 46C and higher
  • Human Capital Management
  • SAP ERP Central Component
  • HR renewal
  • SAP ERP
  • SAP SuccessFactors Employee Central Payroll
  • SAP enhancement package for SAP ERP
  • SAP S/4HANA 

Resolution

The system is set up with a two‑person control mechanism in which both User A and User B must participate whenever data is created or changed. Their roles are clearly separated through distinct authorization levels (P_ORGIN/P_ORGXX). User A has the rights to enqueue (E), read (R), and use matchcodes (M), which allows them to create, modify, or delete data—but only while it is locked. User B, on the other hand, can dequeue (D), read (R), and use matchcodes (M), meaning they can lock or unlock records but cannot directly edit the data itself.

In practice, this creates a structured workflow. When new data needs to be created, User A starts by entering and locking it. Once the entry is ready, User B unlocks and finalizes it. When existing data must be changed, there are two possible approaches. In one, User B first locks the record, enabling User A to make the necessary modifications; afterward, User B unlocks the record to complete the process. Alternatively, User A may create a locked copy of the existing record, perform the modifications, and then depend on User B to unlock it. Deletion follows a similar pattern: User B locks the relevant record, after which User A carries out the deletion.

Across all scenarios, User A is the one responsible for performing the actual data creation or changes, while User B serves as the approver through their control of locking and unlocking actions, ensuring a clear separation of duties and enhancing data governance.
