Symptom

In Service Order ODATA V4 API, when we update service order header or an item which was created from FSM, it leads to an error and the payload gets rejected. In Update case, the user tries to update some field in service header or item, but the user also send the FSMServiceCALL and FSMServiceActivity fields filled.

Other Terms

FSM Service Order ODATA V4 API, validation

Reason and Prerequisites

The prerequisite is the FSMServiceCall or FSMServiceActivity is filled with the correct values and these fields are part of the update payload. The system thinks the user tries to change these fields (although there is no real change). Hence the validation is corrected to allow this case.

Solution

Please apply the attached correction instruction or feature pack.
