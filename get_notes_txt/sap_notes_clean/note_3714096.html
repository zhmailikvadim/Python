Symptom

  • In a SAP HANA scale-out environment using System Replication together with a cluster framework (for example Pacemaker/SAPHanaSR), a failover or takeover attempt does not complete successfully.
  • Both the former primary and secondary systems may become unavailable after the incident.
  • The database services stop without signs of a crash (no SIGSEGV, assertion failure, OOM kill, or kernel panic in HANA traces).
  • HANA traces show internal communication timeouts between hosts or services.
  • The nameserver coordinates topology decisions and interacts with the System Replication hook and the cluster framework.
  • Customers may suspect a database revision defect or operating system issue, although no clear product error signature is recorded in HANA traces.

Environment

SAP HANA DB Version 2.0

Cause

This behavior can occur as part of HA protection mechanisms in SAP HANA high availability design.

In scale-out System Replication environments coordinated with a cluster manager, SAP HANA must ensure that only one consistent primary system exists at any time.
If the system cannot reliably determine the active primary due to temporary communication disruption, network partitioning, quorum uncertainty, or cluster state ambiguity, the takeover may be prevented to avoid conflicting primary state.

This protection avoids a split-brain scenario and potential data divergence.
As a protective outcome, HANA services on both sites may remain stopped or be stopped by cluster coordination logic.

This behavior is consistent with HA protection mechanisms preventing split-brain situations and in such scenarios the behavior is typically not associated with a HANA database crash.

Resolution

No HANA database correction could be identified based on the observed symptoms.

Root cause analysis should include the infrastructure and cluster coordination layer during the incident timeframe, including:

  • cluster quorum and fencing (STONITH/SBD) decisions
  • network interruptions or latency spikes between nodes
  • operating system level communication issues
  • watchdog or storage access delays
  • cluster monitoring timeouts

To analyze the triggering condition, collect and review:

  • operating system logs (for example journal/syslog/messages)
  • Pacemaker and Corosync cluster engine logs
  • fencing / STONITH history
  • SBD or watchdog logs (if configured)
  • System Replication state information around the incident

The temporary unavailability of both sites is a protective consequence of the high availability architecture preventing data inconsistency, not a failure of the high availability mechanism itself.
