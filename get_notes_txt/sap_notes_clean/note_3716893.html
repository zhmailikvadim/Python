Symptom

You are facing a long runtime and data transfer during the RTI integration from S4 to IBP.

You are facing issues Queue Error for STOCKTRANSFERTYPE.

PLEASE NOTE: This correction is valid for SAP S/4HANA version 1709. If you experience the same issue with other version of S/4HANA or SAP ECC, please implement the note 3232488.

Other Terms

select block size, filter block size, long integration, initial transfer

Reason and Prerequisites

The bigger filter and select block size can solve the performance problem but might not be effective for all cases. This depends on the fragmentation of the IM (integration model) vector. IM (integration model) vector is record of all existing objects of specific type with a flag signifying whether the specific object is part of IM selection or not. Simplified explanation of IM vector behavior is that the IM vector is updated when new object of a type is created. New object is added at the end of the vector, so later the object (e.g. material, vendor but also planned order, production order etc.) is created in the system the higher position in the IM vector it receives.
This means that old objects have smaller positions in the vector, new higher positions in the vector.
In the tx. cfc3 you are setting up filter block size, which is saying how many positions system will scan during the IM activation at once and thus how many objects in max. the system would then pass to bgUnit and send/push into IBP together. How many objects are send within one bgUnit (background unit, background LUW - logical unit of work) is specified by the select block size number. Select block size means how many objects from those collected from the vector scanning (specified by filter block number) system put into one bgUnit and send together.

In cfm2 you are selecting the objects, not knowing when they were added to the vector and thus their position. Neither you could know whether they could be found within the one filter block.

So if in the IM you select old objects and a few very new objects, the space which needs to be scanned can be quite large, where many of the filtered blocks won't find any active object or rather just a few. As all these small sets are provided to the select block size mechanism (or bgUnit creation mechanism), the bgUnit is filled with just a few of these objects. For large sets of data, small filter size number and even smaller select size number this leads to a deployment of large number of bgUnits (data transfers) to destination system.
In case of IBP system this is unfavourite situation. IBP is using the HANA DB for persisting the data, the large number of small bgUnits can choke the HANA DB performance as any other type of DB in principle. HANA DB is super strong in persisting large quantity of data. Thus enlarging the filter block number as same as selecting more objects (or rather all filtered objects) to bgUnit improve the performance of the whole integration process.

The Queue Error for STOCKTRANSFERTYPE is caused by the fact that TL (transportation lane) are created during the transfer of material to destination location. The TL creation from special procurement type of the material master in destination location in IBP requires that the material in source location already exists or data of material at source location are available within the same bgUnit. Due to that using the bigger filtered blocks and pushing the bigger filtered blocks into single bgUnit can prevent rising the issue on IBP side.

Functionality provided in this note use different filter block size mechanism if the destination system type is IBP. It scan the IM vector as long as the number of the collected objects is same or bigger than the number of filter block size number. So it uses the filter block size number in two ways-
1. it is the number of possitions scanned within one vector scan,
2. it signify the minimum number of collected objects
This functionality also uses select block size number differently. For master data all objects collected during the IM vector scan are pushed into the one bgUnit and sent at once all together. For transactional data, the select block size is obeyed. This means the collected objects are split into groups, each group containing select block size number of objects.

In principle filter block size should be always bigger that select block size, as you can send only as many objects as you have previously collected. With this new functionality there is small change of this principle as during the filter scanning phase you can in practise collect double the filtersize.

Solution

Apply correction instruction into your S/4HANA 1709 system.
