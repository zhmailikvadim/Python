Symptom

Two splits of wage type /550 are generated in Personnel Calculation Rule Q030. An example is like screenshot below:

This kind of issue is commonly noticed in tax redesigned solution when customers are doing payroll parallel run with tax redesigned solution and old tax solution.

Environment

  • Human Capital Management (HCM)
  • SAP R/3
  • SAP ERP Core Component
  • SAP ERP
  • SAP enhancement package for SAP ERP
  • SAP enhancement package for SAP ERP, version for SAP HANA

Reproducing the Issue

  1. Run Australia Payroll Driver PC00_M13_CALC_SIMU with a tax redesigned solution schema.
  2. Expand payroll log tree to find Personnel Calculation Rule Q030 in 'Formation of Net Pay'.
  3. Check IT table in 'Output'.

Cause

In payroll processing, such kind of splits in wage types are commonly caused by currency difference. If expand the IT table in the 'Output' of Q030, the currency difference can be clearly seen on the two splits of wage type /550:

This currency difference specifically happens to the amount field of wage type /550. Due to this difference the two splits of wage type /550 are not merged together.

Track back the source of these two splits of /550 in Q030, it can be seen that the positive /550 is from wage type /101, and the negative /550 is from /401. The currency difference actually comes from wage types /101 and /401. And this difference can be seen from IT table of 'Input':

In general, wage type /101 is an accumulation of custom payment wage types, and it is initially generated in PCR X023. Additionally, most of customers have requirements to use currency in payroll processing. So the custom payment wage types trend to be populated with currency. As a result, wage type /101 generated in X023 also carries currency.

On the other hand, wage type /401 is generated based on wage type /411. This happens in 'QCUMU POST' in the tax redesigned solution. Looking at wage type /411, it is initially generated by standard function QNTAX in the tax redesigned solution. It must be noted that, in SAP standard delivery for payroll Australia, standard payroll functions don't populate currency for the wage types generated from the functions. For instance, wage types RSAS and RESC generated by standard function SUPER do not have currency. So, the same happens with wage type /411, which is generated by function QNTAX. In consequence, wage type /401 doesn't have currency neither.

Talking about the old tax solution, the reason why /411 is able to be populated with currency is that, wage type /411 is calculated by PCR QTAX in the old tax solution. Different than the processing of a payroll function, in the processing of a PCR, the currency of a wage type being processed, in this case, the currency of wage type /111 is populated to the 'AMT_CURR' field of the header of OT table. After standard tax is calculated by operation QTAXCS, the tax amount as well as wage type /411 must be populated to the header of OT table. Therefore, wage type /411 also has currency. Later, in PCR QCUM wage type /401 is populated based on /411. Hence, /401 can has currency as well.

Resolution

In order to resolve the issue caused by currency difference, customers must populate currency for wage type /401. For this purpose, standard payroll function XDECI can be used. For example:

Fct       Par1  Par2  Par3  Par4
XDECI  FILL   IT     

This call of function XDECI is going to enter country currency in the currency field for all IT table wage types, which have an entry in amount field but no entry in the currency field. 
In the standard sub-schema QNT1 delivered, XDECI has been positioned right after sub-schema QNTX, but it is delivered at the status of inactive ( commented in sub-schema QNT1):

Customers can remove the '*' to get it activated. By this way, XDECI can work to populate currency for IT table wage types. This includes wage type /401. If the currency difference also happens to RT table wage types, customers can activate 'XDECI FILL RT' as well.For more information about function XDECI, please check its documentation via transaction code PE04.
