Symptom

  • In ERP, the redundant storage setting could be toggled via a customizing flag in transaction O1CL; in S/4HANA, this setting is fixed (greyed out) and cannot be changed.

Cause

  • In classic ERP, reference characteristic values were usually calculated at runtime by the business object logic. There was also an option to redundantly store these reference values on the classification side; when enabled (e.g., for transaction CL30N), searches could be performed purely on classification tables.

  • As of S/4HANA OP1809, this option changed: to retrieve all characteristic values via CDS views, redundant storage is mandatory. The related customizing flag is no longer editable and is always set to true. see also SAP notes 3553545 and 3413026

  • Consequently, in S/4HANA OP1809 the classification logic—especially CDS view–based logic—expects all characteristic values for all class types, including reference values, to be persisted in table AUSP. 

  • After migrating from ERP to S/4HANA, you may notice a larger data volume in AUSP. This is expected because, with the redundancy flag enforced, more characteristic values (including reference values) are stored directly in AUSP.

Resolution

  • For materials, equipment, and functional locations, reference characteristic data can be kept synchronized with AUSP even if the classification view is not opened. For other object types, you must open the classification view at least once after changing the master data to trigger synchronization.

  • If the classification view is not accessed for those objects, AUSP will not be updated. As a result, transaction CL30N—which reads directly from the database—may display values that differ from the current master data.
