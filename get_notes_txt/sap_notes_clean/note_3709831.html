Symptom

Error: "Identities were conflicted" when updating providerUID of a provider in identities object using socialize.importIdentities Rest API:

  • "errorCode": 206004
  • "errorDetails": "Identities were conflicted"
  • "errorMessage": "Identity Already Assigned"

Environment

  • SAP Customer Data Cloud
  • Rest API

Reproducing the Issue

  1. Select a user with more than one identity from the Supported providers
  2. Remove one of the identity using socialize.removeConnection endpoint with UID and provider parameters
  3. Add the Identity JSON object with the same provider name and new providerUID
  4. Use socialize.importIdentities with UID and identities parameters.

Example Identity JSON object:

record.identities={};

var test= [
  {
    "provider": "apple",
    "providerUID": "<new_providerUID>", 
    "allowsLogin": true,
    "isLoginIdentity": true
  }
];
record.identities=test;

Cause

This behavior is expected. Without completely removing Login Identity of a provider, it is not supported to update the providerUID.

This will work:
  "provider": "A", "providerUID": "123456789"
  "provider": "B", "providerUID": "123456789"

This will fail with a conflict:
  "provider": "A", "providerUID": "123456789"
  "provider": "A", "providerUID": "555555555"

Resolution

To resolve the 'Identities were Conflicted' error, add the params removeLoginID : true and lastIdentityHandling : remove to the socialize.removeConnection endpoint along with UID and provider.

The steps to update providerUID are the following:

  1. Select the user with UID filter
  2. Using socialize.removeConnection endpoint with above parameters, remove the existing provider
  3. Add the new Identity JSON
  4. Use socialize.importIdentities with UID and identities parameters.

At this point for social identities, the new email added in the identity JSON object will not be added as the login identifier. Either by using accounts.setAccountInfo or accounts.importFullAccount in upsert mode, the new login identifier can be added.
