Symptom

You are using descrapancy handling as of note 2190855. This feature is developed for issuing EWM and receiving EWM. Thats why, a correction delivery with output 'RPOD' can only be created with EWM in issuing plant. However, you would like to get this feature also without EWM in replenishment delivery as otherwise, the STO cannot be closed.

Other Terms

RPOD, 312, correction delivery

Reason and Prerequisites

Feature has not been provided with initial design, but is now enabled.

Solution

Implement attached correction instruction to be able to use this process for a non-EWM supplying warehouse. This allows you to create correction deliveries that are reflected in the purchase order history (instead of creating material documents). You get always a correction delivery that has to be handled and posted then actively within the ERP system. If the correction delivery is relevant for EWM, it will be distributed directly to EWM and EWM can (depending on settings there) directly confirm its goods movement that is done then also in ERP automatically (no manual action needed).

If the outbound delivery contains no warehouse at all (LIKP-LGNUM is initial), the delivery type for the correction delivery cannot be derived from T321-LFART and message RPOD fails. You need then an additional entry in table T156Q for the movement type of the correction delivery, but with KZBEW = 'L' in the key. (You may use view V_156Q_VC in SM30.)

Example: movement type is 312. The entry 312 _ _ _ _ => ... TP exists in T156Q. Then you need an additional entry with 312 _ L _ _ => ... TP. Please set the flag QM check /T156Q-INAKT) as "inactive". The delivery type is then derived from delivery category TP with help of table TVLTZ. The correction delivery has then to be processed and posted goods movement in ERP.
