Symptom

Having multiple registrations of one ID (TP Name) at one SAP RFC Gateway may sometimes lead to sporadic failures indicated by error message:

Function ABC not found

Other Terms

Function not found, Type T destination in SM59, registered server

Reason and Prerequisites

Wrong registration setup: different programs are registered under the same program ID (TP Name) at the RFC Gateway

Solution

Such multiple registrations can be seen in transaction SMGW under Menu -> Goto -> Logged on Clients in the column TP Name. Usually, it is intended to run multiple instances of an external RFC server application and connect each of them with the same unique registration ID (TP Name) at the Gateway. This is the way to achieve load balancing, i.e. high availability for the external RFC server.

Using an SM59 Type T destination referring to that registration ID will then choose, according to the load balancing rules of the Gateway, one of those registrations for an RFC call.

However, it is crucial to make sure that such a registration ID is used uniquely. I.e. one unique ID should only be used by the instances of one and the same application. Following figure sketches an optimal scenario:

SM59 Type T Destination SMGW Logged On Clients
DEST_TO_EXT_SERVER_XYZ EXT_SERV_XYZ (ID / TP Name)
  Functions:
      FUNC_A
      FUNC_B
      FUNC_C

EXT_SERV_XYZ (ID / TP Name)
  Functions:
      FUNC_A
      FUNC_B
      FUNC_C

EXT_SERV_XYZ (ID / TP Name)  
  Functions:
      FUNC_A
      FUNC_B
      FUNC_C




In above example SM59 Type T destination DEST_TO_EXT_SERVER_XYZ refers to an external application/program that should be registered at the Gateway under the ID EXT_SERV_XYZ. Transaction SMGW -> Menu -> Goto -> Logged On Clients would list three occurrences of ID EXT_SERV_XYZ.

Since these are all instances of the same applications, the functions offered are equal.

A broken setup would be if mistakenly different applications are registered under the same ID. This can happen since there is no check in the Gateway that would avoid such registrations.

Next figure illustrates such a situation:


SM59 Type T Destination SMGW Logged On Clients
DEST_TO_EXT_SERVER_XYZ EXT_SERV_XYZ (ID / TP Name)
  Functions:
      FUNC_A
      FUNC_B
      FUNC_C

EXT_SERV_XYZ (ID / TP Name)
  Functions:
      FUNC_001
      FUNC_002
      FUNC_003

EXT_SERV_XYZ (ID / TP Name)
  Functions:
      FUNC_A
      FUNC_B
      FUNC_C




In above example there is one occurrence of EXT_SERV_XYZ which has a different function list than the two other registrations.

If an ABAP program assumes that destination DEST_TO_EXT_SERVER_XYZ offers function FUNC_A and attempts to call it, it will receive an error, if the load balancer happens to choose the instance offering FUNC_001 ... and not FUNC_A ...:

CALL FUNCTION 'FUNC_A' DESTINATION 'DEST_TO_EXT_SERVER_XYZ'

will then return:

Function FUNC_A not found

So it is crucial to ensure a proper and structured concept for registration IDs. One ID can only be used for the instances of one and the same application.

There are two possible tests with which you can check if you have such a situation.

1. Function List Test in the respective destination in transaction SM59

Open the destination and invoke the function list (Shift+F6) several times in sequence. If you see different lists once in a while it's sure that you have different programs registered with the same ID.

But beware if the external program uses the Java Connector (JCo) or the .NET Connector (NCo). External Java or .NET applications hide their actual function list behind a standard API common to JCo and NCo. So, if you have different Java or .NET programs registered with the same ID, then the function list test may show equal lists, since it's the standard list and the actual list is hidden.

Then you need an additional test, i.e. SE37 test.

2. SE37 function invoke test

Open transaction SE37 and enter the function you know your desired external program offers. Provide the destination name. If necessary, provide proper function parameters, run (F8) the function several times in sequence. If once in a while a system failure is returned, then you are likely to have the problem with different programs behind the same ID.
