Symptom

When activating a CDS Analytical Projection View (i.e. a CDS Projection View with provider contract analytical_query) activation errors or unexpected behavior may occur in the following scenarios:

  1. A field in the base cube is annotated with @Aggregation.default:null (or @DefaultAggregation:null) to suppress an inherited aggregation annotation. The syntax check of the Analytical Projection View incorrectly classifies such a field as a Measure instead of falling back to the default behavior (treating it as if no aggregation annotation were set). This can cause false activation errors — for example, when the field is used as a dimension in a WHERE condition or a CASE expression.

  2. A CDS parameter of type DATS or DATN is used in a CDS Analytical Projection View. The activation check incorrectly rejects these date types as invalid numeric parameters, causing a false activation error.

Other Terms

CDS Analytical Projection View, Provider Contract Check, Analytical Query, provider analytical_query, CASE expression, @Aggregation.default, @DefaultAggregation, null annotation, DATS, DATN, numeric parameter, field classification, dimension, measure, activation error, CDS

Reason and Prerequisites

Issue 1 — Null annotation not handled in field classification:

The Provider Contract Check classifies cube fields as Dimension or Measure based on the @Aggregation.default or @DefaultAggregation annotation. The classification logic does not handle the special annotation value null, which in CDS semantics means "no annotation" (i.e., the annotation is suppressed). As a result, a field annotated with @Aggregation.default:null in the base cube is treated as a Measure instead of falling back to the default behavior, leading to incorrect check results.

Issue 2 — DATS/DATN rejected as numeric parameters:

The activation check for in CDS Analytical Projection View validates whether parameters used in numeric contexts have an appropriate type. The check does not include DATS and DATN in the list of accepted types, even though these date types are valid in this context. This can cause the check to report a false error when such parameters are used in certain situations.
