Symptom

Your application experiences connectivity issues with SQL Error Code

-10709 'Connection failed'  (3059060 - How-To Troubleshoot SQL Error -10709 'Connection failed' in an SAP HANA Landscape)

-10807 'Connection down'  (3117294 - How-To Troubleshoot SQL Error -10807 'Connection down' in an SAP HANA Landscape)

 on your SAP HANA landscape.

Other Terms

10709 ,10807

Reason and Prerequisites

The SQL errors are returned by all SAP HANA clients based on the HANA SQLDBC library (e.g. SQLDBC, ADO.NET, Go (golang), Node.js, ODBC, ODBO/MDX, Python, Ruby).

Possible components that can cause the error -10709 and -10807 are:

  • Application server*:

Application (1) --> HANA SQLDBC driver from above (2) --> SAP HANA User Store (3)** --> OS (Linux) (4)

  • Communication layer:

--> Network devices (5): NAT Gateway, Firewall, Load balancer, Proxy, etc.

  • HANA server:

--> OS (Linux) (6) --> HANA instance (7)

 

* A common type of application server using the HANA SQLDBC client is an SAP ABAP Platform (NetWeaver) or S/4 HANA application server

** The use of the SAP HANA User Store (3) is optional

Prerequisites:

  • You are using a SQLDBC library based SAP HANA client to connect your application to SAP HANA database (SAP note 3165810)

Solution

Use tcpdump or Wireshark to to capture TCP/IP and other packets being transmitted or received over a network to which the computer is attached.

The procedure to determine the IP address of the HANA system is only valid if both servers are NOT separated by Network Address Translation (NAT).

For your analysis of IP + TCP headers, the command can be as follows:

sudo tcpdump -s 96 -W 10 -C 50 -n  -i any host a.a.a.a and host h.h.h.h and port <HANA SQL Port> -w /tmp/tcpdump.pcapng

It is in the nature of connection errors that these errors often occur only sporadically. Therefore, it is important to record only the relevant packages and adjust their size to the purpose of the analysis.

s - Snapshot length in byte. If you do not set a value for 's', the entire package is captured.

s 96 is enough for IP + TCP headers (minimal storage & CPU cost)

n - Don’t convert addresses (i.e., host addresses, port numbers, etc.) to names

W - Specifies how many output files tcpdump should keep when used together with

C - rotate files by size in MB

W multiplied by C is the maximum amount of disk space in MB consumed by the tcpdump files. The above command requires up to 500 MB of disk space for the tcpdump files (10 files * 50 MB).

w - Write the raw packets to file rather than parsing and printing them out

You will find the IP addresses and ports of the two affected servers in the error message of the client if both servers are NOT separated by Network Address Translation (NAT).

Connection error between peer {a.a.a.a:<Application Port> -> h.h.h.h:<HANA SQL Port>}

Application server: a.a.a.a
Application port: 5-digit number that can vary with each call

HANA server: h.h.h.h
HANA SQL Port: Static 5-Digit Number: pppp

Example:

Connection error between peer {100.104.244.100:20644 -> 100.104.244.200:30041}

Application server: 100.104.244.100
Application port: 20644

HANA server: 100.104.244.200
HANA SQL Port: 30041

For the example, the command is as follows:

sudo tcpdump -s 96  -W 10 -C 50 -n -i any host 100.104.244.100 and host 100.104.244.200 and port 30041 -w /tmp/tcpdump.pcapng

Collect tcp dumps from application server a.a.a.a and HANA server h.h.h.h at the same time.

Start the analysis

  1. Install Wireshark on your local computer and open the files tcpdump_hana.pcapng.* and tcpdump_app.pcapng.*.

  2. Apply a filter on the application port from the latest error message

Connection error between peer {100.104.244.100:20123 -> 100.104.244.200:30041}

This results in Wireshark filter:

tcp.port == 20123

  1. As a result you get two lists of the same TCP/IP packets sent during the aborted session. One list for the application server and one for the HANA server.

Collect tcpdump on a MS Windows server

On Windows Server, you can capture TCP traffic using pktmon (Packet Monitor), which is the native built-in tool available on Windows Server 2019, 2022, and Windows 10/11.

Here's how to capture traffic between your two hosts:

Step 1: Open an elevated Command Prompt or PowerShell Run as Administrator.

Step 2: Create filters for the traffic

pktmon filter add -i 100.104.244.100 -i 100.104.244.200 -p 30041

Step 3: Start the packet capture of the first 128 bytes

pktmon start --etw

Step 4: Stop the capture

pktmon stop

Step 5: Convert the output to a Wireshark format

pktmon etl2pcap PktMon.etl -o tcpdump.pcapng
