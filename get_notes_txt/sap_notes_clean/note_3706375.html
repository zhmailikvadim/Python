Symptom

If a completed production order has both confirmed goods received and in-procurement scrap,  PMMO Pegging assigns twice the correct quantity of scrap that is still in procurement.

When this happens, Pegging inflates the Total Order Quantity by the real quantity of in-procurement scrap.  Pegging issues application log warning message pmmo_pegging-262 to reflect this:  "Adjusted total order quantity for replen. &1 from &2 to &3 &4".

(For example, if production order 1234567890 should have 2 ea. of in-procurement scrap, out of a total order quantity of 10 ea., the message would read "Adjusted total order quantity for replen. OR001234567890 from 10 to 12 EA".  The quantities of all assignments created for this order will add up to 12 ea., of which 4 ea. will be for in-procurement scrap.  All of these assignments will have 12 ea. in the Total Order Quantity field.)

This happens when database table AFPO contains true ('X') for both fields DNREL and ELIKZ for this order.

Other Terms

PMMO_PEGGING, RPMMOPEG, in process scrap, in-process scrap, in procurement scrap, incorrect total order quantity, scpqty, totalreplqty, pmmo_assignment, TECO, pmmo_pegging262, duplicate in-procurement scrap, confirmed scrap, overassignment, over assignment, overpegged, over-pegged, over-pegging

Reason and Prerequisites

This is due to a programming error.

Solution

Implement this SAP Note, or apply the service pack that contains it, and then re-run Pegging.
