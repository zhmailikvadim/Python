Symptom

  • A cell and a line of a sap.ui.table can be selected as expected.
  • When navigating down the table, the cell marking moves up and down while the line selection remains unchanged.
  • The dark blue indicator shown on a selected cell is a focus indicator, not a cell selection.
  • The focus indicator stays at a fixed position while the data beneath it changes due to virtualization.

Environment

SAP UI5 table, smart table, list and tree controls

Reproducing the Issue

  1. Open the example: https://ui5.sap.com/#/entity/sap.ui.table.Table/sample/sap.ui.table.sample.Selection" sap.ui.table.Table sample selection
  2. Select a cell and a row in the table.
  3. Navigate down the table using the keyboard or scroll.
  4. Observe that the cell marking moves up and down while the line selection remains unchanged.

Cause

Design.

Resolution

This is design, and below is explain of our developer:

1.
In general it is expected that the cell focus and the row selection behave differently. This is due to technical reasons. The sap.ui.table.Table control is designed to render a fixed amount of rows with a virtualised scrollbar. On scroll, the rows don't change, but the contexts do. This allows us to display large data sets with millions or even billions of entries because we only render a small amount of it.
2.
The selection state depends on the identifier of a certain Context that is bound to a row. So scrolling in the data set will move the selection along with the corresponding Context. The focus stays on the same DOM element in the browser and is not affected by scrolling.
3.
The GridTable control virtualizes table rows in order to efficiently handle millions of records. This means that only the rows currently visible in the viewport are rendered. When users scroll, the DOM elements remain the same and only the underlying data bound to those cells is updated.
4.
This behavior is intentional. We explicitly avoid moving the focus indicator during scrolling, as doing so would trigger repeated focus-change announcements by screen readers. In addition, the focus indicator must always remain visible within the viewport. Unlike row selection, which can scroll out of view when the selected row leaves the viewport, the focus outline must never be lost.

A real cell selection (again selecting the underlying data) behaves exactly like row selection and scrolls together with the data. We see this behavior in the following example:
https://ui5.sap.com/#/entity/sap.ui.mdc.Table/sample/sap.ui.mdc.demokit.sample.table.TableCopyPaste

The focus outline is handled differently to ensure it remains visible at all times and to prevent unnecessary screen reader announcements during scrolling.

This intentional behavior is a direct consequence of the GridTable's virtualization mechanism and is therefore not observed in the ResponsiveTable type.
https://ui5.sap.com/#/entity/sap.m.Table/sample/sap.m.sample.TableMultiSelectMode
