Symptom

  • SAP PO application server nodes shutdowns and restarts daily
  • Observed JVM heap memory exhaustion
  • OOM files are generating daily, want to know why the OOM are generating and what is cause of OOM files generation 

Environment

  • SAP Process Integration 
  • SAP NetWeaver
  • SAP NetWeaver Application Server for JAVA

Cause

  • Out‑of‑Memory (OOM) situations in SAP AS Java can arise from multiple different root causes. Proper identification of the issue requires analysis of the generated heap dump using the procedure outlined in SAP KBA
    1883568- How to self-analyze a Heap Dump using MAT 

Resolution

  • The initial investigation should be performed in the BC‑JAS‑COR expert area, as this component is responsible for core AS Java memory behavior.

  • If the Leak Suspect Report indicates that the OOM is triggered by threads/objects/ keywords belonging to a specific application area (like, MII, PI/PO, EP, or BI etc.), the case must be opened in corresponding application support component for further investigation and assist.

Common best Practices from PO application support, to assist with frequent OOM:

  1.   Multiple OOM's are getting generated and causing server restart, to avoid such OOM issues configure Large Message Handling properly. By configuring large message handling, you will define a large message by its size and messages that are of that size and above will be blacklisted. This way the PI system will not try to read/parse/process this message and will avoid the OOM and the followed restart. Blacklisted messages can still be manually started at your discretion, to avoid OutOfMemory error:
    • Refer and implement KBA :1727870- Handling of large messages in the Messaging System
    • SAP Documentation: Properties Related to Large Message Handling. 

      Please refer to the below explained points as also mentioned in KBA.

    • The following properties are related to Large Message Handling

      messaging.largemessage.enabled - Value type: Boolean - Activates on large memory handling in PI messaging system, default value false (deactivated) set it to true.

      messaging.largemessage.permits - Value type: Integer - Defines the total number of permits available for each AS Java server node. Default value: 10.

      messaging.largemessage.queueTypes - Value type: String - Defines the types of queues for which large message handling is activated. Default value is (empty string), it defines the types of queues for which large message handling is activated.  SAP Note 1493502 – Max Receiver Parameter for Integrated Configurations 

      messaging.largemessage.threshold - Value type: Integer - Defines the threshold for large messages in kilobytes. Default value is 10240 or 10 MB set it as per your requirement of largest file in kilobytes.

      messaging.largemessage.blacklistXLMessage - Value type: Boolean - Determines whether a message larger than the threshold multiplied by the total number of permits should be blacklisted. The default value is false, which means that message processing will commence when the message can obtain the total number of permits. 

      Set the values for these parameters as per your need or a little greater than the largest file that you have to resolve the issue. This is the tricky part of the configuration to decide what size is adequate for a threshold. So that not many message are blacklisted, and that can actually be processed, so that messages that cause OOM are blacklisted.
      This decision is based on your system properties, so this is something that you will have to determine based on your system and the messages you are processing. Recommended to perform Volume testing for large message handling in lower environment initially.

    • 3111898 - Large responses of synchronous messages are let through though large message handling is enabled. 

      Note: It is difficult for PI/PO to process this big files ( >=~100MB), We suggested you to break the file in small chunks and then process it. 

  2. To avoid the frequent OOM issues, you can also increase the heap memory size to 8 GB( Minimum 4 GB recommended) following the below steps:
        1. Open the Configtool.
        2. Highlight the instance, where you wish to modify the parameters.
        3. Select the View menu and click on Expert mode.
        4. Select the VM parameters tab.
        5. Highlight the initialHeapSize (this is the Xms parameter) and enter 8192 to the Custom Value field.
        6. Click on Set.
        7. Highlight the maxHeapSize (this is the Xmx parameter) and enter 8192 to the Custom Value field.
        8. Click on Set.
        9. Select the File menu and click on Apply Changes.
        10. Restart the instance .
    Further details can be found in the following SAP KBA:
