Symptom

There are one or more of the following errors in SAP systems, traces, or logs:

  • received a fatal TLS certificate unknown alert message from the peer

  • ssslerr_no_context_yet

  • SSL handshake failures during RFC, HTTP, HTTPS, or ICM communication

  • Connection termination without a clear application-level error

  • SSL-related short dumps or trace entries in dev_icm, dev_w*, SM21, or network traces

These errors often lead to confusion, as similar messages can appear across multiple components and communication scenarios.

Environment

  • SAP NetWeaver
  • SAP NetWeaver Application Server for SAP S/4HANA
  • ABAP PLATFORM - Application Server ABAP
  • Operating System independent
  • Database independent
  • ICM / SAP Web Dispatcher

Cause

  • Certificate trust issues:
    • Server certificate is self-signed or signed by a CA that is not trusted by the peer
    • Required CA certificates are missing from transaction STRUST
    • Certificate chain is incomplete
  • Missing or inconsistent SSL configuration:
    • SSL client or server PSE not correctly assigned
    • Incorrect SSL profile used in communication
  • Cross-component communication issues:
    • The SAP system communicates with another system or via a SAP Web Dispatcher
    • Each component has its own trust store and one side trusts the certificate, the other does not 

Resolution

TLS communication consists of two critical phases:

  1. Handshake phase

    • The client and server negotiate encryption

    • Certificates are exchanged and validated

  2. Data exchange phase

    • Encrypted communication begins

The errors listed above occur before application data is exchanged, meaning:

  • The problem is not caused by business logic

  • It is almost always related to certificate trust or SSL configuration

Before opening a support case, verify the following: 

  • Identify the communication direction

    • Who is the client?

    • Who is the server?

  • Check certificate trust

    • Is the server certificate signed by a CA that is trusted by the client?

    • Is the full certificate chain available?

  • Validate STRUST configuration

    • Correct PSE type used

    • CA certificates imported

    • Changes saved and distributed

  • Review traces

    • dev_icm

    • Other work process traces that are involved in the communication scenario

    • SM21 system log

If the issue persists after verifying:

  • Certificate trust

  • STRUST configuration

  • Correct SSL profile usage

Then open a support case and include:

  • Exact error message

  • Communication direction

  • Relevant trace excerpts

  • Certificate details (without sensitive private key data)
