Symptom

  • Running the OpenRewrite rewriteDryRun recipe with the goal of scoping the run to a single custom extension results in:
    • "Too many open files" errors on macOS/Linux, for example:
      • "java.nio.file.FileSystemException: .../testsrc/.../XTest.java: Too many open files"
      • "Could not update ~/.gradle/caches/.../last-build.bin (Too many open files)"
    • Despite scoping, the dry run still parses files outside the target module.
  • On Windows, the same dry run completes successfully with the same init.gradle and recipe.
  • Attempts to mitigate with higher file descriptor limits and Gradle flags (e.g., -Dorg.gradle.workers.max=1, -Dorg.gradle.parallel=false, -Dorg.gradle.vfs.watch=false) do not resolve the issue.
  • When splitting work into batches and migrating some extensions, the rewritten code replaces javax.* with jakarta.*, causing compilation failures on the older Commerce Suite (JDK 17/Spring 5) due to missing Jakarta dependencies:
    • Errors such as "package jakarta.* does not exist" appear during compilation.

Environment

SAP Commerce Cloud

Reproducing the Issue

  1. Prepare the environment

    • Use an SAP Commerce Cloud project on macOS or Linux with a large number of custom extensions included in Gradle.
    • Ensure OpenRewrite is configured with the provided init.gradle and the JDK 17-based Commerce setup.

  2. Attempt a scoped dry run

    • From hybris/bin/platform, execute the dry run targeting a single custom extension, for example:
      ./gradlew rewriteDryRun --init-script=$PATH_TO_INIT_GRADLE -PcommerceUpdatePaths=<paths_for_target_extension> --no-daemon -Drewrite.activeRecipe=<recipe>
    • Optionally add -PcommerceRewriteExclusion using a fully fledged regex pattern (this intentionally simulates the misconfiguration), and use default or lowered parallelism flags:
      • -PcommerceRewriteExclusion='<regex_not_supported>'
      • -Dorg.gradle.workers.max=1 -Dorg.gradle.parallel=false -Dorg.gradle.vfs.watch=false

  3. Observe that scoping is ineffective due to exclusion pattern misuse

    • Review the Gradle/OpenRewrite output and note that non-target subprojects are still being scanned/parsed:
      • "Scanning sources in project <PROJECT_NAME>" appears for projects that were expected to be excluded.
    • Confirm there are no "Skipping project <PROJECT_NAME> because it is excluded" messages for the intended exclusions.

  4. Trigger file descriptor pressure

    • Ensure the session limit is moderate (for example, default ulimit -n values or even after increasing them).
    • Re-run the dry run to process a larger set of subprojects unintentionally included by the incorrect exclusions.

  5. Encounter the error

    • The build fails with "Too many open files", for example:
      • "java.nio.file.FileSystemException: .../testsrc/.../XTest.java: Too many open files"
      • "Could not update ~/.gradle/caches/.../last-build.bin (Too many open files)"
    • Note that on Windows, running the same command with the same init.gradle may complete successfully.

  6. Attempt batching and observe partial migration side effects

    • Run the recipe on a subset of extensions so that some are migrated (imports rewritten to jakarta.*) while others remain on javax.*.
    • Try to compile the project on the older suite (JDK 17/Spring 5):
      • Observe compilation errors such as "package jakarta.* does not exist", confirming that partial Jakarta migration within a JDK 17 build is incompatible and unsupported.

Cause

  • Scope filtering behavior:
    • The commerceUpdatePaths parameter filters after the project is parsed. Although there is an "early exclusion" mechanism to skip entire Gradle subprojects (extensions) during Logical Source Tree (LST) building, it does not apply to the compilation phase. As a result, large workspaces with many custom subprojects can still hit OS file descriptor limits even when scoping.
    • The -PcommerceRewriteExclusion parameter was used with full regular expressions, which are not supported by OpenRewrite’s exclusion matching. Consequently, targeted subprojects were not excluded and were parsed/compiled, increasing file handle usage.
  • OS differences:
    • Windows typically does not enforce the same strict open-file limits as macOS/Linux; thus the dry run can succeed on Windows under identical inputs.
  • Partial migration side effects:
    • Running the Jakarta-oriented framework update recipes on only a subset of extensions leads to code that imports jakarta.*. The remaining, non-migrated project relies on Java EE (javax.*) and Spring 5. Mixing both in a single codebase causes compilation failures and is unsupported.
    • Adding Jakarta dependencies into an older suite to make partially migrated code compile is not supported and can introduce inconsistencies.

Resolution

Address both the open-file limit and project scoping, and avoid partial Jakarta migration in a single build. The following approach has proven effective and is aligned with internal framework guidance. For additional background on similar symptoms, see KBA 3618495 - Using OpenRewrite recipes to support SAP Commerce Cloud JDK21 Framework Update adoption (Troubleshooting: Step 11 references "Too many open files").

1. Validate and apply proper exclusions

  • Do not use full regular expressions with -PcommerceRewriteExclusion; only supported glob-like patterns are applied.
  • Verify whether exclusions are effective by checking the OpenRewrite logs:
    • Included project: "Scanning sources in project <PROJECT_NAME>"
    • Excluded project: "Skipping project <PROJECT_NAME> because it is excluded"
  • If extensions are outside the package’s custom directory or symlinked, consider relocating them into custom to simplify pattern matching.

2. Use the built-in exclusion generator for precise patterns

  • From hybris/bin/platform, run the helper task to generate exclusion and update-path patterns for a specific subset of extensions:
    ./gradlew generatePatterns --init-script=$PATH_TO_INIT_GRADLE -PcommerceExtensionToMigrate=ext1,ext2
  • Capture the generated variables:

    REWRITE_EXCLUSIONS="<generated_list>"

    COMMERCE_UPDATE_PATHS="<generated_paths>"

  • Export them into the shell that will execute the recipe, and run the dry run with these parameters, for example:

    export REWRITE_EXCLUSIONS="<generated_list>"
    export COMMERCE_UPDATE_PATHS="<generated_paths>"


    ./gradlew rewriteDryRun --init-script=$PATH_TO_INIT_GRADLE -PcommerceUpdatePaths=$COMMERCE_UPDATE_PATHS -PcommerceRewriteExclusion=$REWRITE_EXCLUSIONS --no-daemon -Dorg.gradle.jvmargs=-Xmx16G -Drewrite.activeRecipe=com.sap.cx.rewrite.java.SelectiveCommerceCloudFrameworkUpdate

  • If greater control is required, construct the REWRITE_EXCLUSIONS list manually by enumerating each extension path to exclude, separating entries with commas and without spaces, for example:
    ../custom/<ext_name>,../custom/<ext_name>/**

3. Manage OS file descriptor limits on macOS/Linux

  • Increase allowed open files and verify in the same shell that runs Gradle:
    • Check system maximum and session limits (macOS examples): sysctl kern.maxfiles, ulimit -n
    • Raise ulimit -n substantially (e.g., 81920 or higher), ensuring the shell/session that runs Gradle sees the new value.
  • Disable parallelism and watchers to reduce file handle pressure if needed:
    -Dorg.gradle.workers.max=1 -Dorg.gradle.parallel=false -Dorg.gradle.vfs.watch=false
  • Consider increasing JVM heap (-Xmx) if memory pressure accompanies the issue.

4. Avoid mixing migrated and non-migrated code in one JDK 17 build

  • Do not add Jakarta libraries to the older suite to force compilation of partially migrated code; this is not supported.
  • Instead, use batch execution with isolation between batches to prevent compile-time conflicts:
    • Manual copy/restore approach (proven effective in the reported case):
      • Build the whole project first and create a copy of the extensions as a clean baseline.
      • For a given batch, run the recipe on that batch only. After the batch completes, rename or move the migrated extensions (e.g., append "-jdk21") and restore the original, pre-migration copies into the build so subsequent batches run against the unchanged JDK 17 codebase.
      • Repeat until all target extensions are migrated. Once complete, switch the setup to the Commerce Suite compatible with JDK 21/Spring 6 and proceed with post-migration adjustments.
    • Alternatively, run the entire migration in a single pass on Windows if feasible, where "Too many open files" limits are typically less restrictive.

5. Continue with the standard update process

  • After all extensions have been processed without "Too many open files" errors and the Jakarta migration is complete for the intended set, proceed with the regular update process as described in KBA 3618495 - Using OpenRewrite recipes to support SAP Commerce Cloud JDK21 Framework Update adoption

Notes

  • Exclusions are applied during LST construction; they do not affect the compilation step included in the recipe. Ensure both the exclusion patterns and batch strategy keep the active compilation set within OS limits.
  • The size of each batch should be determined empirically. Start with modest groups (for example, 10–25 extensions) and adjust based on resource constraints; increase -Xmx if necessary.
  • Always confirm that exclusions are recognized by inspecting "Skipping project … because it is excluded" entries in the logs.
