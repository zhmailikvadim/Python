Symptom

  • The database package remains in a "starting" state and fails to reach the "running" state during failover.
  • The database takeover process takes more than 40 minutes, leading to a timeout during failover.
  • The cluster virtual IP (VIP) is not brought online, causing SAP systems to lose access to the database.
  • Error messages observed in logs include "Untracked shutdown" and "Takeover finished successfully."

Environment

  • SAP HANA High Availability (System Replication, DR, etc.).

Reproducing the Issue

  1. Configure SAP HANA database in a two-node cluster with operation mode set to delta_datashipping.
  2. Simulate a failure on the primary node, causing the database package to initiate failover to the secondary node.
  3. Observe that the database package remains in the "starting" state and does not reach the "running" state within the expected timeframe.
  4. Note that the database takeover takes more than 40 minutes and results in a timeout.

Cause

  • The operation mode delta_datashipping causes the secondary system to persist received log entries without replaying them until a takeover is initiated. This results in all logs being replayed from scratch during the takeover process, leading to delays.

Resolution

  1. Change the operation mode from delta_datashipping to log_replay or log_replay_readaccess to decrease the takeover time.
  2. Refer to the SAP HANA Administration Guide for SAP HANA Platform for instructions on changing the operation mode.
  3. Ensure that the redo log is continuously replayed on the secondary system immediately after arrival, making the replay step superfluous during takeover.
  4. Involve the network team to verify network connectivity and ensure there are no issues such as firewalls or throughput limitations between the cluster nodes.
  5. Confirm whether both cluster nodes are hosted in the same region or different regions to rule out regional network latency.
