Symptom

The db2diag.log reports warnings that the OOM - Instance memory request failed. In the following example, it shows errors/warnings which follow the DIA8300C A memory heap error has occurred. 
With STMM (Self Tuning Memory Management) set to ON, Db2 tries to expand a memory heap and fails when the request is more than the available memory limit. This may not be an error but a warning that memory is reaching it's limit.

EXAMPLE:

Below we see that the INSTANCE_MEMORY request failed.  The controller metrics show that MEMORY is at it's fullest peak with no available memory which 
results in the error SQLO_NOMEM "No Memory Available". The area where the out of memory failure occurs is the Application Heap (APPLHEAPSZ)on node 0.
It is not clear what is causing the issue until further investigation is carried out.

Environment

db2 LUW IBM

Reproducing the Issue

This is Db2 STMM Memory allocation failure

Cause

  • Insufficient INSTANCE_MEMORY
  • Insufficient Physical Memory which does not enable INSTANCE_MEMORY allocation.
  • Db2 Parameters not set correctly
  • Long running or poorly optimised SQL statement/job.

Resolution

  1. Check the current memory usage
     'db2pd -d <SID> –dbptnmem'
    The command displays how much memory each part of the database is using. Compare what’s used, what’s allocated, and the peak usage to find memory problem areas like STMTHEAP and APPLHEAP which result in the DIA8300C error.
  2. Check the Db2 configuration in accordance with the Db2 version installed in particular INSTANCE_MEMORY, Database Memory and STMM tuning with AUTOMATIC parameters.
    See SAP notes 
    3518384: Db2 12.1 standard parameter settings
    2751102: Db2 11.5 standard parameter settings
    2303771: Db2 11.1 standard parameter settings 
    1851832: Db2 10.5 standard parameter settings 
    1692571: Db2 10.1 standard parameter settings 
    1329179: Db2 V9.7 standard parameter settings 
  3. Consult KBA 2486074 'How to check instance memory'
    Ensure that memory is sufficiently set in accordance with SAP recommendations.
  4. Check for long running programs/SQL statements and tune where necessary.
    db2 list applications show detail
    Look for SQL statements with high total_act_time, large rows_read vs low rows_returned and statements which are executed frequently that require a lot of resources in the DBAcockpit - Performance - Top SQL Statements.
    EXPLAIN the statement and check the INDEX ADVISOR to determine if the correct index is being used.  Ensure that statistics are current for both table/index.
  5. Check IBM directly for any known APARs related to DIA8300C and the installed Db2 software version.
    Check for any FODC (First Occurrence Diagnostic Check), dumps in the /db2/db2<sid>/ db2dump directory path.
    EXAMPLE:
    FUNCTION: DB2 UDB, SQO Memory Management, sqloMemLogPoolConditions, probe:30
    MESSAGE : ADM14005E  The following error occurred: "Memory".  First Occurrence 
              Data Capture (FODC) has been invoked in the following mode: 
              "Automatic".  Diagnostic information has been recorded in the 
              directory named 
              "/db2/<SID>/db2dump/FODC_Memory_2026-03-18-<XXXXXXXXXXXXXXXX>".
    Look at C-Stacks generated for memory errors which can be checked directly with IBM or SAP support for known APARs/issues. 
  6. Generate the full db2support.zip as per SAP note 83819 before contacting SAP Support and return outputs of the above commands.
