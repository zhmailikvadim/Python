Symptom

You have created a custom field having a Default Mapping of Direct Reference and when changing attribute from source entity, those values are not reflected into target entities.

Environment

  • SAP Sales Cloud Version 2
  • SAP Service Cloud Version 2

Reproducing the Issue

Note that this just an example and it can be applied to different scenarios.

1. Click on the User Icon.
2. Go to Settings > All Settings. 
3. Go to the Extensibility Administration.
4. Select Contact Service > Contact.
5. Go to the Custom Fields tab.
6. Create a Custom Field (e.g. ABC).

7. Select Lead Service > Lead.
8. Go to the Custom Fields tab.
9. Create a Custom Field (e.g. DEF) with Direct Reference to the field created into the Contact service (e.g. ABC).

10. Go to the Contact Object Worklist (OWL).
11. Open a Contact.
12. From this Contact, create a Lead.
13. Observe that the values of the custom field from the Contact are copied to the newly created Lead.

14. Navigate to the Contact.
15. Make a change to the custom field within the Contact (e.g., ABC).
16. Notice that the corresponding values in the Lead custom field do not automatically update to reflect this change. 

Cause

Direct mappings operate by copying values from the parent record to the child record at the moment of mapping. After this one-time copy, the child fields become independent. Subsequent changes made to the original source entity, such as a Contact, do not automatically propagate to the corresponding fields on the target entity, such as a Lead.

Resolution

Subsequent changes made to the original source entity, such as a Contact, do not automatically propagate to the corresponding fields on the target entity, such as a Lead.. The Lead’s fields remain separate and do not follow the parent’s updates.

The only exception to this rule is when the mapped entity (e.g., Contact)  associated with the target entity (e.g., Lead) is changed. For instance, if the Lead’s Contact lookup is switched to a different Contact, the Lead’s mapped fields will be updated to reflect the new Contact’s data. This behavior aligns with the documentation discussion about associated entity references, which pertains to updates occurring when the mapped entity association changes, not when the original parent record is edited after mapping.

A possible alternative to achieve a scenario where changes in the parent entity update the child entity could involve implementing External Hooks in the Account service. These hooks would trigger PATCH calls to the associated target entities (e.g., Leads) according to specific business requirements.
