Symptom

This KBA will suggest tuning steps to improve run times of the Batch Risk Analysis, repository object sync, etc, assuming the user violations table GRACUSERPRMVL is very large and is impacting performance.

Environment

SAP GRC Access Control 10.0 / 10.1 / 12.0

Resolution

1) Table GRACUSERPRMVL

  • Check table GRACUSERPRMVL in tcode SE16. This table holds all risks for all users executed in Batch Risk Analysis. See how long does it take for the screen to return the results after clicking "Number of Entries", the expected return time should be a couple of seconds. Very large number of entries in this table affects negatively and directly the performance of Batch Risk Analysis job.
     
  • Check if this table is impacting performance by executing a Runtime Analysis of report GRAC_BATCH_RISK_ANALYSIS, or GRAC_OBJECT_ROLEREP_USER_SYNC (User Sync), via tcode SAT. Below example shows that we have performance impact when reading from GRACUSERPRMVL:
     

 

2) Risk Analysis parameters in SPRO

  • Parameter 1023 may be set to various types such as 02, 03, 04, 05. Recommendation is to narrow this down to Permission level only (02) - as true representation of risk, and leave other options for end users to make their selections in reporting.
     
  • Parameter 1030 could also be switched to NO, and users can manually triggered this option if they need it in reporting too.
     
  • Parameter 1031 could be set to YES, and navigate to NWBC>Setup>Critical Access Rules to maintain critical roles and profiles such as SAP_ALL and SAP_NEW (plus whatever roles and profiles that contain all authorizations) so they are not analyzed.
     

Suggested action plan to improve performance

After realizing table GRACUSERPRMVL is impacting the performance (1), and adjusting the parameters as recommend (2), you may perform the steps below:

  1. Clear out the violations data by running the new utility GRAC_DELETE_VIOLATION_DATA (the new report to delete violation data is delivered by Note 2440851)
  2. Perform the "Update DB statistics" task with the help of your Database Administrator, and make sure this task is scheduled to run after every batch risk analysis job
  3. Rerun batch risk analysis to repopulate violation tables assuming new parameters values tuned.

In case a dump occurs when executing the deletion report, please see Note 2168653 - Batch Risk Analysis Deadlock Dump DBIF_RSQL_SQL_ERROR.
