Symptom

  • Infopackage loading request REQU_XXX fails or remains in running, yellow status.
  • Program SAPLRS_GENERAL is involved in the failure.
  • Error messages include "TIME_OUT" and "SYSTEM_CORE_DUMPED".
  • Runtime errors dumps such as TIME_OUT and SYSTEM_CORE_DUMPED are logged in the system frequently in ST22 transaction
  • In SM58 from the source system entries as below are presented too:
    RSAR_TRFC_DATA_RECEIVED  Transaction executing/recorded
    IDOC_INBOUND_ASYNCHRONOUS Transaction executing/recorded  
  • The dumps below or one of them occurs:

Environment

  • SAP ERP
  • SAP NetWeaver
  • SAP S/4HANA

Reproducing the Issue

  1. Execute an Infopackage loading request REQU_XXX in the SAP BW system.
  2. Observe the locking issues in SM12.
  3. Check the runtime errors in ST22, including TIME_OUT and SYSTEM_CORE_DUMPED.
  4. Monitor the work processes in SM50, which are busy with SAPLRS_GENERAL.

Cause

  • The infopackage extracts more than 1.000 data packages from the source system. According to note 892513 a request should not contain more than 1.000 data packages, as it can lead to performance and enqueue problems. When more than 1.000 data packages are created, the tRFC queue of the source system has a heavy load and pushes as many packages parallel to BW as defined in SMQS -> Max. Conn. 
  • A single request containing millions of records.

Resolution

  1. Implement solution steps from KBA 1933471.
  2. Points to mainly pain attention as per the KBA Adjust "Max. Runtime" to a value within the range of 120-300 based on business requirements.
  3. Set "Max. Conn." to a reasonable value to prevent connection overload. For more details review KBA 2941940 - Troubleshooting SAP Service API performance issues (Data Packages, Work Processes, RFC Connections, Partner Profiles) 
  4. Avoid creating loading requests with more than 1,000 data packages. For more details refer to SAP Note 892513 - Consulting: Performance: Data loading, number of packages, request size
  5. Limit the number of large requests processed in parallel to prevent system overload.
