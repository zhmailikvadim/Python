Symptom

User has performance issues when opening the My time Events Fiori 2.00 application.

Environment

HCM Fiori 2.0

Reproducing the Issue

Open My Time events 2.0

Cause

  • Several thousands of employees use the app simultaneously
  • Long period for Retroactive processing permitted period in customizing V_PTCOR_TCONSTR
  • Database/table index problem or too many (not necessary) items
  • Customer enhancements

Resolution

Below steps help by analyzing this problem. Please check which can help you.

  • Analyze the backend performance via ST12 transaction. Register the trace, which can be directly checked later also in your system.
    Via the trace check which area/code/table is affected.
  • Change the Retroactive processing permitted period as possible in customizing table V_PTCOR_TCONSTR.
  • Check the performance for the same employee and period via PTCOR/Display Documents. It's not the same test but there can be database/table index problem or too many (not necessary) items, which can be visible here.
    Suggested to contact with database colleagues also, if there is a performance  already here.

Only for the My Time events 2.0 application relevant:

  • To avoid the performance problem by the initial load of the application, an Extension project has to be created enabling the extension hook extHookCalendarMarking. It's available Controllers/Overview/extHookCalendarMarking . It avoids data fetch during initial load and make it happen only on selecting 'Time Event List' tab.
    Code lines merely check if the extension hook has been implemented. Anything inside the implementation does not really count. That means, customer merely leaving a comment inside the implementation would suffice.

See Also
