Symptom

When an employee is replicated (e.g. from SAP SuccessFactors Employee Central or SAP HCM) the automatic HR-BP synchronization delivers the expected values—for example a customer-specific full name format or an additional comment/description (such as an internal tag).

However, if the same employee has to be (re-)processed with the report /SHCM/BUPA_SYNC_S (or by the BP delta sync job that calls the same framework), the Business Partner (BP) is created/updated but the customer-specific adjustments are lost or overwritten by the standard mapping. Typical effects are:

• The BP full name is created with the SAP standard logic instead of the customer-defined format.
• Custom fields or comments (for example a descriptive tag in BP roles, addresses or identification data) are missing after the report is executed.

Environment

• SAP S/4HANA on-premise edition
• SAP S/4HANA private cloud edition

Reproducing the Issue

  1. Ensure an employee already exists in PA infotypes but has no corresponding Business Partner.
  2. Implement customer logic in a classic HR customer exit (e.g. EXIT_SAPFP50M_002 or similar) that changes PA infotype data and calls BAPI_BUPA_CENTRAL_CHANGE to adapt BP data (full name, comments, etc.).
  3. Run report /SHCM/BUPA_SYNC_S (or schedule the HR-BP delta sync job) for the employee.
  4. Check the generated/updated BP.
    Result: the BP exists but shows the standard full-name layout and the custom comment is missing.

Cause

Customer-specific BP changes were implemented inside a PA infotype customer exit and executed by the Employee replication process.
• The HR-BP synchronization framework (used by /SHCM/BUPA_SYNC_S and the delta sync job) does not call PA infotype exits.
• Therefore the custom logic is skipped when the BP is created or updated by /SHCM/BUPA_SYNC_S, and the standard mapping overwrites the previously adjusted fields.

Resolution

Move all Business-Partner-related enhancement logic from PA infotype customer exits to the BAdI delivered for HR-BP synchronization.
These BAdIs are invoked in every scenario where employee data is transferred to the BP layer: • Automatic BP creation during employee replication
• Report /SHCM/BUPA_SYNC_S
• HR-BP delta synchronization job

Determine which data model your system uses:

  1. In transaction SFW5 open node ENTERPRISE_BUSINESS_FUNCTIONS.
  2. Check business function /SHCM/EE_BP_1.

If /SHCM/EE_BP_1 is inactive (classic model)
• Implement BAdI: BADI_FITV_VENDOR_SYNC
• Use method: MODIFY_COMPLETE_DATA
• The method provides the complete BP / Vendor / Customer structures. Adapt any field (e.g. full name, comments) before they are written to the database.

If /SHCM/EE_BP_1 is active (new model)
• Implement BAdI: /SHCM/B_EE_BP_SYNC
• Use method: MODIFY_ALL
• The method offers the same structures and timing as above.

After moving the code to the appropriate BAdI, execute /SHCM/BUPA_SYNC_S again or let the delta job run. The modified name format and additional fields are now retained for all processing modes.
