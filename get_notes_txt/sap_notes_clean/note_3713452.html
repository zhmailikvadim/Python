Symptom

When running an allocation cycle using Universal Allocation with many segments, you receive error messages indicating that the number of generated messages has exceeded the limit of 999,999.

Other Terms

XMESG, Messages, universal allocations, F3548, run allocations, F4363, allocation results, UA.

Reason and Prerequisites

Duplicate messages from segments, Technical limitation in processing that volume of messages.

Solution

The best approach is to split the cycle and/or run it over fewer periods. However, if this is not desirable, please implement the attached correction and set the flag as described below.

The system will issue messages for the entire cycle rather than separately for each segment.

As a result, messages are issued for the entire cycle, eliminating the duplicate records in the XMESG internal table and prevent the error for the allocations.

The trade-off is that you may not be able to link messages to a specific segment.

After the note is implemented, please check transaction FIN_FLAGS for the following: Application = FIN, Sub. application = UA, Parameter = UA_MESG_COMPOSE.

In case the above parameter is not available, please add the Application, Sub-Application and Parameter manually. In transactions FIN_FLAGS_CUST1, FIN_FLAGS_CUST2 you can follow the same instructions as in SAP note 3361820.

For transaction FIN_FLAGS_CUST3, please add the following:

Application Subapplication Parameters Data element  SAP Note Number Chk Values Medium Field Label Long Field Label Description
FIN UA UA_MESG_COMPOSE FIN_FLAGS_ON_OFF 3713452 X Switch Switch for parameter UA_MESG_COMPOSE Parameter to compose messages at the cycle header instead of segments

Once done, please go to transaction FIN_FLAGS and find the parameter.

By setting the parameter value to 'X', you will activate the message composer. If the parameter value is left blank, the system will be in the standard state.

The note resolves uncommon issue, so this parameter should not be kept permanently. After the problematic cycle has been executed, you need to reset it to a blank value in FIN_FLAGS to ensure the process returns to the standard logic.
