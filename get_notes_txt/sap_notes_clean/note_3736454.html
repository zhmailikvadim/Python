Symptom

The scheduled stock task DeltaStockRequest stops running at the expected frequency or stops executing entirely.

When reviewing the transaction logs in Managed Gateway for Spend & Network (Transaction Tracker), reservation‑related transactions — such as DeltaStockRequest and QueryStockRequest — do not run according to the frequency defined in the parameter Application.Catalog.IS.TenantConfig.

Environment

SAP Ariba Catalogs Management

Reproducing the Issue

  1. Configure Application.Catalog.IS.TenantConfig with a defined freq value. 
  2. Open Managed Gateway for Spend & Network and monitor DeltaStockRequest (and optionally IngestStockRequest) using Transaction Tracker.
  3. Observe that the tasks run intermittently (for example, only a few times per day) instead of running at the frequency defined in the parameter.

Cause

This issue is typically related to misconfigurations in Application.Catalog.IS.TenantConfig, often caused by parameter inconsistencies between environments. Review the audit logs for this parameter to identify recent changes and potential mismatches between environments.

Several scenarios may lead to this behavior:

  1. Incorrect parameter configuration in the affected environment: The parameter may not contain the correct structure, catalog qualifiers, or supplier identifiers required for stock update tasks to run.
  2. Parameter inconsistencies between environments: Values from another environment (for example, Test) may have been copied into Production or vice‑versa, resulting in mismatched configurations and invalid catalogQualifier entries.
  3. Invalid catalogQualifier values: For example, missing environment‑specific suffixes in supplier identifiers, or catalog qualifiers that do not correspond to active inventory catalogs. Such inconsistencies may result from parameter edits, overwrites, or unintended propagation between environments.

Resolution

Correct the parameter by ensuring the proper structure and valid values are configured for each environment where the stock task must run.

You must verify that:

  • The parameter structure follows the expected format.
  • All catalogQualifier entries are valid and correspond to the correct catalogs.
  • Supplier identifiers include any required environment suffixes (e.g., Test suffixes).
  • Each environment (Production and Test) has the appropriate values and is not unintentionally sharing or overwriting configurations.

Once the parameter is corrected, the scheduled tasks should resume running according to the defined frequency.
