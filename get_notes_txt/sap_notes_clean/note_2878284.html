Symptom

Symptom

One of the following:

﻿(Symptom 1)

You migrated your system to SAP HANA with the DMO option of SUM. As part of this procedure, pooled and clustered tables are converted to transparent tables. After the procedure, the HANA MiniChecks (see SAP Note 1969700) report that physical ABAP pool and cluster tables are not in the correct state (i.e. not empty).

﻿(Symptom 2)

During a Pool-to-Pool conversion for a table TAB, Step 3 fails with runtime error ITAB_DUPLICATE_KEY. The problem occurs when an inactive version of the pool table TAB exists, which can be verified in SE11.

Other Terms

ABAP DDIC depooling pool table

Reason and Prerequisites

Program Error.

The following applies to (Symptom 1) only:

Physical table pools are database tables that contain multiple logical tables. An example would be table ATAB (physical pool) that contains (logical) tables like SCIPE, SCIPT,... The ABAP application operates just on the logical tables and sees them as 'normal' tables.

The concept of tables pools (and table clusters) is outdated and is removed from the ABAP DDIC beginning with SAP_BASIS 7.53. All logical tables are converted to 'normal' transparent tables. The same happens even in lower releases when a migration to SAP HANA via DMO (Database Migration Option) takes place.

The process of converting a physical table pool into transparent tables is also called sometimes 'depooling'. Once the transparent tables are created and filled, the data must be deleted from the physical table pool because there it is not needed anymore. This deletion in the physical table pool does not happen if the depooling is performed with the DMO option of SUM (there are other methods for depooling).

Solution

Corrected with the support packages mentioned below. As an alternative, apply the attached correction instructions.

The following applies to (Symptom 2) only: after applying the correction, go back to SE14 and display the table. Consider selecting "Background" Processing type if the table is very large. Click on "Continue Adjustment".

Other Terms

R_MIGOBJCLASSREL_PARA, KMAT_MIGR/60, conversion of generic articles and variants

Reason and Prerequisites

For a certain number of the work processes available in the system for parallel processing of the report, the number of work processes available to the report for parallel processing is 0. As a result, no server group is transferred to the Framework for Parallel Processing, which ultimately leads to the error.

Solution

Implement this SAP Note in your system.
