Symptom

You experience performance degradation after upgrade from HANA1.0 to HANA2.0, especially in function / procedure written by SQLScript.

Other Terms

SAP HANA, Performance Degradation, upgrade, sqlscript, procedure, function

Reason and Prerequisites

Reason:

SQLScript engine has been optimized in HANA 2.0 to support new features and improve performance. From optimization point of view, parameter handling is changed between HANA 1.0 and HANA2.0.

In HANA1.0, all input parameters will be binding as constants, SQL statements in function / procedure are compiled at every execution and plan cache might be flooded with different constant values.

In HANA2.0, all input parameters will be parameterized as variables, it can be expected to avoid the re-compilation and reduce plan cache size as comparing to the behavior of the constant values in HANA1.0.

Due to behavior difference of parameter handling in SQLScript engine, in HANA2.0 SQL optimizer might choose un-optimal plan compared to HANA 1.0. SQL optimizer needs to defer the decision when the input parameters arrive during the execution of functions / procedures. So it might lead to un-optimal execution plan especially for non-equivalent expressions like CASE WHEN expression.

Please see the example below which describes what SQL statements and plans are generated on each version.

Example:

CREATE COLUMN TABLE TAB_TEST_01 (MANDT NVARCHAR(3), C1 NVARCHAR(10), C2 NVARCHAR(10));

CREATE COLUMN TABLE TAB_TEST_02 (MANDT NVARCHAR(3), C1 NVARCHAR(10), C2 NVARCHAR(10));

CREATE TYPE TT_RESULT AS TABLE(MANDT NVARCHAR(3), C1 NVARCHAR(10), C2 NVARCHAR(10), C1_2 NVARCHAR(10), C2_2 NVARCHAR(10));

INSERT INTO TAB_TEST_01 VALUES('100','TEST1','TEST2');
INSERT INTO TAB_TEST_01 VALUES('100','TEST11','TEST21');
INSERT INTO TAB_TEST_01 VALUES('100','TEST12','TEST22');
INSERT INTO TAB_TEST_02 VALUES('100','TEST1','TEST2');
INSERT INTO TAB_TEST_02 VALUES('100','TEST11','TEST21');
INSERT INTO TAB_TEST_02 VALUES('100','TEST12','TEST22');


CREATE PROCEDURE PROC_TEST_01
(
  IN    "IV_MANDT" NVARCHAR (000003),
  IN    "IV_CASE_WHEN" NVARCHAR (000020),
  OUT   "ET_RESULT" TT_RESULT
)
language sqlscript sql security invoker reads sql data  as begin
    lt_temp1 =
        SELECT
            tab1.mandt,
            CASE WHEN :IV_CASE_WHEN like '%TEST1%' THEN tab1.c1 ELSE null END c1,
            CASE WHEN :IV_CASE_WHEN like '%TEST2%' THEN tab1.c2 ELSE null END c2
        FROM
            TAB_TEST_01 as tab1
       WHERE tab1.mandt      = :IV_MANDT;
      
    lt_temp2 =
        SELECT
            tab2.mandt,
            CASE WHEN :IV_CASE_WHEN like '%TEST1%' THEN tab2.c1 ELSE null END c1,
            CASE WHEN :IV_CASE_WHEN like '%TEST2%' THEN tab2.c2 ELSE null END c2
        FROM
            TAB_TEST_01 as tab2
       WHERE tab2.mandt      = :IV_MANDT;

    ET_RESULT =
         SELECT temp1.*, temp2.c1 c1_2, temp2.c2 c2_2
           FROM :lt_temp1 as temp1
           LEFT OUTER JOIN :lt_temp2 as temp2
           ON temp1.c1 = temp2.c1
           AND temp1.c2 = temp2.c2
           ;
END;

From the generated SQL statements,

In HANA 1.0, constant value __TEST1__ is binding in the CASE WHEN expression so null as the result is determined.

CASE WHEN CAST('__TEST1__' AS NVARCHAR(20)) like '%TEST2%' THEN tab2.c1 ELSE null END c1

In HANA 2.0, variable __typed_NString__ is parameterized in the CASE WHEN expression so the result is not determined.

CASE WHEN __typed_NString__($1, 20) like '%TEST2%' THEN tab2.c1 ELSE null END c1

As a result, the plan in HANA2.0 gets more complex, especially when the input parameters are used in non-equivalent expressions.

 

Prerequisites:

  • Upgrade from HANA 1.0 to HANA 2.0

Solution

Use bind_as_value in expressions. The built-in function "bind_as_value" will materialize the variable as a constant.

With the above example, you can use bind_as_value as below:
  CASE WHEN :IV_CASE_WHEN like '%TEST1%' THEN tab1.c1 ELSE null END c1,
  =>
  CASE WHEN bind_as_value(:IV_CASE_WHEN) like '%TEST1%' THEN tab1.c1 ELSE null END c1,

 

For the compatibility reason, we do support both hint and parameter. You can apply them for test purpose.

  • with hint (SQLSCRIPT_VERSION(2))
    • Only available as of 2.00.042 (SPS04)
  • ALTER SYSTEM ALTER CONFIGURATION ('indexserver.ini', 'system') SET ('sqlscript', 'kernel_mode') = '2' WITH RECONFIGURE;
    • It configures to switch back to the behavior of SQLScript engine same as HANA 1.0 in a system widely.  

Please note that the hint and parameter should be used ONLY in very limited/urgent issues on while customer's system upgrade and it doesn't guarantee the same execution plan as before because new execution engines(i.e. esx, hex) are added since HANA 2.0.
The side effects of the old kernel mode setting on query or system.

  • Not available to use new features of HANA 2
  • Potential issues occurred at HANA 1, but resolved in HANA 2
