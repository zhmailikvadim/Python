Symptom

  • What is the Resolution for blank screen issues in Fiori apps occurring during back-navigation and re-initialization of OData models?
  • When a user opens a custom/standard Fiori application (e.g., Manage Consolidated Billing or any other Fiori app or custom app), the application loads correctly the first time. However, after the user clicks the "Back" button to return to the Fiori Launchpad home and attempts to open the same application a second time, the screen remains blank.
  • The browser's URL updates to the correct App Hash, but the UI fails to render. Investigating the browser's Console/Developer Tools reveals the following error:
    • Assertion failed: oModel must be an instance of sap.ui.model.Model, null or undefined

Environment

S/4HANA

Reproducing the Issue

  1. Log in to the Fiori Launchpad.

  2. Click the tile for the custom application (e.g., ManageConsolBilling-display or any Fiori app or custom application).

  3. The app loads and displays data successfully.

  4. Click the Back (<) button in the Fiori Shell header.

  5. Click the same application tile again.

  6. Result: The application shows a blank screen.

Cause

  • The issue is caused by a collision in the browser's memory during "Soft Navigation."
  • When the "Back" button is pressed, the Fiori Launchpad does not always fully destroy the previous instance of the application to improve performance. If the custom application's Component.js does not properly clean up or "destroy" its OData models upon exiting, a "dirty session" or "ghost instance" remains in the background.
  • Upon the second launch, the application attempts to initialize the OData model again. Because a reference to the previous (but now invalid or null) model still exists in the Document Object Model (DOM) cache, the UI5 framework fails the assertion check. This causes a silent crash during the binding process, preventing the View from rendering.

Resolution

  • There are three primary strategies to resolve this issue, ranging from configuration changes to code-level fixes:
    • Option 1: Force App to Open in a New Tab (Recommended for quick fix)
      • By forcing the app to open in a new browser tab, the "Back" button event is avoided entirely. Closing the tab ensures a clean state for the next session.
        1. Open the Fiori Launchpad Designer or Content Manager.
        2. Locate the Target Mapping for the affected app.
        3. In the Parameters section, add:
          • Parameter Name: sap-ushell-next-navmodeValue: explace

    • Option 2: Clear App State via Target Mapping
      • You can instruct the Launchpad to treat every navigation as a fresh start.
        1. In the Target Mapping, add the following parameter:
          • Parameter Name: sap-keep-alive Value: false
    • Option 2A: (Optional) If multiple mappings exist, ensure priority by adding:
      • Parameter Name: sap-priority Value: 1 (This ensures the custom mapping is favored over standard ones).

Option 3: Implement Lifecycle Clean-up (Code-level fix) - This solution is for the SAPUI5 custom developer who developed the code.

Ask the custom UI5 developer to ensure the application explicitly destroys models when the component is destroyed (The example below is theoretical or a general response for a Javascript and/or UI5 expert to review tied to the custom coding). 

Update the exit function in Component.js:

JavaScript (Theoretical Example): 
exit: function() {
    if (this.getModel("yourModelName")) {
        this.getModel("yourModelName").destroy();
    }
}


In manifest.json, ensure OData models are set to load asynchronously to avoid synchronization conflicts:

JSON
"models": {
    "": {
        "dataSource": "mainService",
        "settings": {
            "preload": true,
            "synchronizationMode": "None"
        }
    }
}
