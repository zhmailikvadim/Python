Symptom

When I run the Active Worker Download, the number of records processed is much higher than the actual number of active workers in my system. For example, the processed count may show over a million records even though there are only a few thousand unique workers. When I attempt to download the resulting file, I may receive an error stating the file is too large.

Cause

This issue occurs because the Active Worker Download is not designed to filter for unique rows by default. The record count is often amplified (sometimes by a factor of 5 or more) when the Segmented Object Detail (SOD) field is added to the download.

If a worker has multiple cost centers, task codes, or expense codes associated with them, the standard Active Worker Download logic creates a separate row for every possible combination of these attributes for each worker, leading to extreme record duplication.

Resolution

To avoid high duplicate counts and record amplification, you should switch to a connector designed to handle unique entries:

  1. Use the Active Worker Single Line Download connector instead of the standard Active Worker Download.
  2. The Single Line version is specifically designed to return only unique entries and will not generate duplicate rows for the same worker.
  3. You can still include Segmented Object Detail data in this connector by adding it to the defined download fields.
  4. Test the Active Worker Single Line Download to verify that the record count now accurately reflects your unique worker population.
