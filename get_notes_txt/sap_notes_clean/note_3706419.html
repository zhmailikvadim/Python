Resolution

Why Do We Need a Trust Store for HTTPS Calls?

A trust store, such as a Java KeyStore (JKS), is a crucial component in establishing secure HTTPS connections. It serves as a repository for trusted root Certificate Authority (CA) certificates. The primary purposes of a trust store are:

  1. Verification of Server Authenticity: It allows the client to verify that it's communicating with a legitimate server.
  2. Establishment of Secure Connections: It enables the creation of encrypted connections, ensuring data privacy and integrity.
  3. Prevention of Man-in-the-Middle Attacks: It helps prevent unauthorized interception of communications.

Common Error: XCE108 - No trusted certificate found

When a trust store is not properly configured or is missing the necessary certificates, you may encounter the following error:

com.sap.b1i.xcellerator.XcelleratorException: XCE108 Caller returned HTTP error: HTTP-code: -1; HTTP-message: ; Exception: javax.net.ssl.SSLHandshakeException: sun.security.validator.ValidatorException: No trusted certificate found

This error indicates that the SSL/TLS handshake failed because the client (B1i in this case) could not find a trusted certificate in its trust store to validate the server's certificate. This typically means that the certificate presented by the server is not trusted by the client's trust store.

To resolve this issue:

  1. Verify that your trust store contains the correct root CA certificate or the server's certificate.
  2. Ensure that the trust store is correctly configured in your B1i setup.
  3. Check if the certificate in your trust store has expired. If so, you need to rotate the certificate following the steps outlined in this document.
  4. If you're using a self-signed certificate, make sure it's properly imported into your trust store.
  5. Double-check the <trustStoreURI> in your B1i configuration to ensure it points to the correct trust store file.

Remember that certificate rotation is an essential part of maintaining secure connections. Regularly update your trust store with current certificates to prevent such errors and ensure smooth, secure communications.

How to get the New Trust Certificate

When obtaining a new trust certificate, it's important to follow best practices to ensure the security and longevity of your trust store.

Option 1: Official Sources for Certificates

Some organizations, especially government agencies and financial institutions, provide specific links or instructions for downloading the correct certificates. For example, the Spanish Tax Agency might provide a link like:

https://sede.agenciatributaria.gob.es/Sede/en_gb/todas-noticias/2025/noviembre/17/renovacion-certificado-electronico-sede-electronica-tributaria.html

When available, always use these official sources to obtain certificates. This ensures you're getting the correct and most up-to-date certificate directly from the authoritative source.

Best Practice: Import only the CA certificate

It is recommended to import only the CA (Certificate Authority) certificate into your trust store, rather than individual server certificates. This approach offers several advantages:

  1. Long-term validity: CA certificates are typically valid for extended periods (e.g., 10 years), reducing the frequency of updates to your trust store.
  2. Multiple server support: A single CA certificate can sign many server certificates, allowing your trust store to validate multiple servers with just one CA certificate.
  3. Simplified management: When server certificates expire, you only need to obtain new ones from the server without modifying your trust store, as long as they're signed by the trusted CA.
  4. Efficient handshake: As long as the server certificate is valid and signed by a trusted CA in your trust store, the SSL/TLS handshake will succeed.

In contrast, when using self-signed certificates, you must import the server certificate (or its CA) into the trust store, which requires more frequent updates and management.

Option 2: Using OpenSSL to obtain the CA and full certificate chain

To get the CA certificate and full certificate chain of a target website using OpenSSL, you can use the following command-line example:

openssl s_client -showcerts -connect example.com:443 </dev/null 

This command does the following:

  • Establishes a connection to the specified host (replace example.com with your target website)
  • Retrieves and displays the full certificate chain, including the server certificate and any intermediate certificates
  • The root CA certificate will typically be the last certificate in the chain

To save the certificates to separate files, you can redirect the output and manually extract each certificate:

openssl s_client -showcerts -connect example.com:443 </dev/null | awk '/-----BEGIN CERTIFICATE-----/,/-----END CERTIFICATE-----/' > certificate_chain.pem

This command saves the entire certificate chain to a file named certificate_chain.pem. You can then split this file into individual certificate files as needed.

Remember to verify the authenticity of any certificates before adding them to your trust store.

Option 3: Using a Web Browser (Chrome) to obtain certificates

Another method to obtain certificates is by using a web browser like Google Chrome. This approach can be more user-friendly for those who prefer a graphical interface. Here's how to do it:

  1. Go to the website you want to obtain the certificate from (e.g., https://example.com)
  2. Click on the padlock icon in the address bar
  3. Click on "Certificate" in the dropdown menu
  4. In the certificate information window, click on "View Certificate"
  5. Navigate to the "Details" tab
  6. Look at the "Certification Path" to see the certificate chain
  7. To save each certificate in the chain:
  • Select the certificate you want to save in the certification path
  • Click on "Copy to File" or look for an "Export" option
  • Follow the wizard to save the certificate (typically in .cer or .pem format)
  • Repeat this process for each certificate in the chain you want to save

This method allows you to visually inspect the certificate details and save individual certificates in the chain. Remember to save the root CA certificate, as it's the one you'll typically want to import into your trust store.

As with the OpenSSL method, always verify the authenticity of certificates before adding them to your trust store.

How to Create a Trust Store JKS (KeyStore)

There are several methods to create a Trust Store JKS (KeyStore). Here are three recommended options:

Option 1: Using B1i Control Center

  1. Navigate to the B1i TOOLS - Control Center.
  2. Go to CONFIGURATION > Certificate Admin in the side menu.
  3. Input the BizStore URI (e.g., /com.sap.b1i.vplatform.scenarios.design/vPac.sap.ARG.eCom/trusted.jks).
  4. Enter a store password.
  5. Click submit. The system will create the keystore if it doesn't exist.
  6. After submission, you will be directed to the detailed Certificate Admin page.
  7. Click on "Add Entry" to open a modal window.
  8. In the modal, import the proper alias for your certificate.
  9. Enter the keystore password. Make sure this password is properly saved and secured.
  10. Click "Submit" to add the entry to your keystore.

Note: We recommend creating a keystore with a .jks extension. While .p12 or .pfx are optional, not all adapters currently support trust store passwords for these formats.

Important: Always ensure that the keystore password is properly saved and kept secure. This password is crucial for accessing and managing your trust store.

Option 2: Using JDK Keytool

You can use the JDK keytool command-line utility to create a keystore and import certificates. Here's an example that explicitly uses the JKS storetype:

# Create a new keystore
keytool -genkey -alias mykey -keyalg RSA -keystore mytruststore.jks -storetype JKS -storepass mystorepass
 
# Import a certificate
keytool -import -trustcacerts -alias myalias -file mycertificate.cer -keystore mytruststore.jks -storetype JKS -storepass mystorepass

Note the addition of -storetype JKS in both commands to ensure the Java KeyStore (JKS) format is used. Same reason mentioned above.

Option 3: Using Keystore Explorer GUI Tool

Keystore Explorer is a user-friendly GUI tool for managing keystores. Here's how to use it:

  1. Download and install Keystore Explorer from https://keystore-explorer.org/.
  2. Launch Keystore Explorer.
  3. Go to File > New > JKS.
  4. Set a password for your new keystore.
  5. To import a certificate, go to Tools > Import Trusted Certificate.
  6. Select your certificate file and provide an alias.
  7. Save your keystore.

Whichever method you choose, ensure you follow security best practices when creating and managing your trust store. Keep your keystore and its password secure, and regularly review and update the certificates in your trust store.

How to upload the trust store to B1i

There are two ways to upload the trust store to B1i:

Option 1: Using MAINTENANCE > Zip Import

  1. Navigate to MAINTENANCE in the side menu.
  2. Select Zip Import.
  3. Prepare a zip file with three layers of files: dataset/group/filename.
  4. Upload the prepared zip file.

Option 2: Using TOOLS > Control Center > MAINTENANCE > BizStore Upload

  1. Navigate to TOOLS in the side menu.
  2. Select Control Center.
  3. In the Control Center, go to MAINTENANCE.
  4. Choose BizStore Upload.
  5. Select the keystore file you want to upload.
  6. Input the full bizstore URI, for example: /com.sap.b1i.vplatform.scenarios.design/vPac.sap.ARG.eCom/trusted.jks(unified)

How to use trustStore in B1i

The trustStore is primarily used for two important atoms in B1i: HttpCall (com.sap.b1i.vplatform.runtime/callhttp.bfd) and WebService Call (com.sap.b1i.vplatform.runtime/callsoap.bfd). Both of these atoms use a similar payload structure for configuring the connection, including the trustStore settings. 

Important Notice: Currently, only the HTTP Call adapter supports trust store passwords. Support for trust store passwords in the WebService Call adapter will be added in a future update. An SAP note will be released when this feature becomes available for WebService Call.

Here's an example of how to use the trustStore in the payload:

<call xmlns="">
  <connect>
    <destProtocol>https</destProtocol>
    <destHost>example.com</destHost>
    <destPort>443</destPort>
    <destPath>/</destPath>
    <query/>
    <proxyHost></proxyHost>
    <proxyPort></proxyPort>
    <method>POST</method>
    <authentification>none</authentification>
    <user/>
    <password/>
    <user2query/>
    <password2query/>
    <trustStoreURI></trustStoreURI>
    <trustStorePassword></trustStorePassword>
  </connect>
</call>

To use the trustStore in your B1i configuration:

  1. Set the <trustStoreURI> to the path of your uploaded JKS file. For example:
   <trustStoreURI>/com.sap.b1i.vplatform.scenarios.design/vPac.sap.ARG.eCom/trusted.jks</trustStoreURI>
  1. If you're using a JKS (Java KeyStore) format for your trustStore, you don't need to provide a password. Leave the <trustStorePassword> element empty:
   <trustStorePassword></trustStorePassword>

Note: 

  • For the HTTP Call adapter: If you're using a format that requires a password (e.g., PKCS12), you would include it in this element.
  • For the WebService Call adapter: The trust store password is not currently supported. Leave this element empty regardless of the trust store format.
  1. Ensure that the rest of the connection details (destProtocol, destHost, destPort, etc.) are correctly set for your specific use case.

By configuring the trustStore in this way, you enable B1i to use the certificates in your trustStore for establishing secure HTTPS connections when making HTTP calls or invoking web services. This is crucial for verifying the authenticity of the servers you're connecting to and ensuring secure, encrypted communications.

Remember to keep your trustStore up-to-date by regularly reviewing and rotating certificates as needed, following the procedures outlined in the earlier sections of this document.

Importing CA Certificate and Certificate Chain without Keystore

There is another way to import CA certificates and certificate chains without using a keystore. This method involves importing the certificates directly into the JRE's cacerts file. Here's how you can do it:

  1. Open Windows Explorer and navigate to the cacerts file, which is located in the jre\lib\security subfolder where B1i is installed. The default location is typically:
   C:\<B1i installation path>\sapjre_64\lib\security
  1. Before making any changes, create a backup copy of the cacerts file.
  2. Depending on the certificates you receive from the Certificate Authority you are using, you may need to import an intermediate certificate and/or root certificate into the cacerts file. Use the following syntax to import one or both of these certificates:
   keytool -import -alias <alias> -keystore <cacerts_file> -trustcacerts -file <certificate_filename>

Replace <alias> with a unique name for the certificate, <cacerts_file> with the full path to the cacerts file, and <certificate_filename> with the name of the certificate file you want to import.

  1. You will be prompted for the keystore password. The default password for the cacerts file is "changeit" (without quotes). You should change this password after the import for security reasons.
  2. Repeat this process for each certificate you need to import.

Note: This method affects the JRE's global trust store, which means it will impact all Java applications using this JRE, not just B1i. Use this method with caution and ensure you have proper backups before proceeding.

When using this method, it's important to leave the <trustStoreURI> element empty in your B1i configuration. This is because you're not using a separate trust store file, but rather the JRE's built-in cacerts file. Your configuration should look like this:

<trustStoreURI></trustStoreURI>
<trustStorePassword></trustStorePassword>

By leaving these elements empty, B1i will default to using the JRE's cacerts file for certificate validation.

Remember to restart your B1i service after making these changes for them to take effect.
