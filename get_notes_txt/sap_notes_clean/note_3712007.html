Symptom

  • Fiori tiles are missing or show "Cannot load tile" in the Quality (QA) or Production (PROD) environments.

  • Standard transport of the catalog did not resolve the issue, or the catalog appears empty in the target system.

  • Initial troubleshooting via /UI2/FLC or /UI2/FLIA indicates the Catalog ID is missing from the backend or the Page Cache.

Environment

S/4HANA

Cause

The Catalog definition in the target system has become desynchronized or was not fully captured in the original Transport Request (TR). A "Fresh" transport is required to overwrite the corrupted or missing metadata in the target system.

Resolution

I. Create and Prepare a New Transport

  1. Access transaction SE01 (Transport Organizer) or SE10 (Transport Organizer: Display).

  2. Click Create and select Customizing Request.

  3. Provide a descriptive Short Text (e.g., "Repair Transport for Catalog [Catalog_ID]").

  4. Note the generated Transport Number - May be in format: K (e.g., SIDK900XXXor DEVK900123)

II. Assign Transport in Fiori Launchpad Designer

  1. Open the Fiori Launchpad Designer (Customization scope):

    • URL: https://<host>:<port>/sap/bc/ui5_ui5/sap/arsrvc_upb_admn/main.html?scope=CUST

    • Or Transaction: /UI2/FLPD_CUST

  2. Click the Settings (gear icon) in the top-right corner.

  3. In the Assign Transport Request dialog, select the Request created in Step I.

  4. Click OK (to set as default transport). All subsequent changes will now be recorded in this  transport TR.  Additional references (Setting Default Transport Request)

II. Perform a "Dummy" Edit to Force Object Capture

  1. Locate the problematic Catalog and the specific Tile.

  2. Click on the Tile to enter Edit mode.

  3. Modify the Title field slightly (e.g., add a "1" or a hyphen to the end).

  4. Click Save. This action forces the system to record the entire Catalog/Tile object into the assigned Transport Request (from the earlier step).

  5. (Optional) Change the Title back to the original name and Save again to maintain naming standards.  Note: This is optional but for testing purposes and guarantee, it is better to show that you have an actual change and cause a delta for debugging purposes.  Sometimes the platform needs to see some differences. 

IV. Release and Import (Release Transport Request)

  1. Go to SE10 (Transport Organizer).

  2. Expand your Request, right-click the Task (sub-entry), and select Release.

  3. Right-click the Main Request and select Release.

  4. Coordinate with your BASIS team to import this TR into the target system (QA/PROD) via STMS (Transport Management System).

Note: At this point, try to see if the changes were applied to confirm the fix is done.  Sometimes, cache clearing may be needed (See extra steps below).

V. Post-Import Cache Clearing (Mandatory - If Step I to IV is limited)

After the import, run the following steps in the target system to ensure the new metadata is active:

----Checklist-----

Typical workflow for cache clearing (in sequential order):

  1. Clear HTTP Server cache -  Tcode = SMICM → HTTP Plugin → Server Cache → Invalidate Globally 
  2. Clear Metadata Cache - Report/Tcode =  /IWFND/CACHE_CLEANUP 
  3. Clear Metadata Cache (backend) - Report/Tcode =  /IWBEP/CACHE_CLEANUP 
  4. Invalidate Global UI2 caches - Report/Tcode =  /UI2/INVALIDATE_GLOBAL_CACHES 
  5. Recalculate Application Index - Report/Tcode =  /UI5/APP_INDEX_CALCULATE 
  6. If needed: Sychronize Chip cache - Report/Tcode =  /UI2/CHIP_SYNCHRONIZE_CACHE 
  7. If needed: Invalidate Client Cache - Report/Tcode =  /UI2/INVALIDATE_CLIENT_CACHES 
  8. If needed: /UI2/PAGE_CACHE_SYNCHRONIZE  <-- On newer systems this is mandatory (A popular customer missed step). 

Note: The order of execution was provided above (1 to 5) . Step 6, 7, and 8 are additional steps ran if the prior sequential steps are limited to resolve the cache issue.
