Symptom

  1. In long‑running applications that use RFC compression, memory usage grows continuously and may lead to performance degradation or crashes.
  2. When using a PasswordChangeHandler for expired passwords, the handler does not receive the expected details in all scenarios, so the application cannot handle the password change properly.
  3. Reading SAPUILandscape.xml fails with an “Invalid format of landscape XML file” error although the file is valid.
  4. WebSocket RFC connections fail to establish in environments where the upgrade handshake response contains many or long headers.
  5. When an RFC call is cancelled by the ABAP backend (rc 10002), a registered RFC server connection does not always terminate cleanly and may leave the connection in an unexpected state.
  6. When using WebSocket RFC server connections with client certificate authentication, logon can fail or behave inconsistently (e.g., optional certificates are rejected or required certificates are not handled as expected).

 

Reason and Prerequisites

The symptoms above have been caused by the following issues in previous patch levels:

  1. Native compression resources were not released reliably by the .NET finalizers, causing a memory leak in compression usage.
  2. Password change details were not consistently propagated because the internal ChangePassword() method did not take the username as an explicit parameter. When rfc/reject_expired_passwd=1 is configured on the ABAP system and a user logs on with an expired password, the ABAP system returns largely initial SystemAttributes, including an empty user name. As a consequence, the actual logon user name was not available when the internal password-change function module SUSR_USER_CHANGE_PASSWORD_RFC was invoked, causing the password change to fail. The username is now captured at logon time and passed explicitly to the password change logic, independent of the SystemAttributes returned by the server.
  3. The XML parser relied on a fixed node navigation pattern and failed on valid XML with different structure.
  4. WebSocket upgrade handshake buffers were too small for large responses.
  5. CPIC cancellation (CANCEL_REQUEST / rc 10002) was not treated as a normal termination path in registered RFC server connections. When the ABAP backend cancelled an active conversation, the server-side receive loop did not handle the cancel return code explicitly. Instead of cleanly waiting for the subsequent deallocation and then continuing to serve the next request, the connection could end up in an unexpected state.
  6. WebSocket server logic did not correctly distinguish between client certificate REQUEST vs REQUIRED and did not check certificate availability consistently.

 

In addition, this patch level provides the following enhancements:

-Enhanced diagnostic trace output for WebSocket connections and RFC operations, improving troubleshooting capabilities.

Solution

 Install and use the latest patch level of NCo 3.1, at least NCo 3.1.7, which is available on the download page under https://support.sap.com/nco#downloads.
