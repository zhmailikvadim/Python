Symptom

You execute the „Change Move-In Date” process on the CRM system and change the move-in date of an existing contract to an earlier day. Once you save the data, the CRM system replicates the contract data to the ERP system where the replication get stuck due to whatever reason. This issue is registered in ECRMREPL thus the ERP contract is not updated yet on the ERP system. After that, a user starts a new contract change process. These changes are replicated to the ERP system, too. However, the replication fails as there is already a pending operation for the same contract and this issue gets registered in ECRMREPL as well. When these ECRMREPL entries on the ERP system are triggered for a reprocessing, the data of both ECRMREPL entries get incorrectly merged leading eventually to a data inconsistency comparing the contract data on both ERP and CRM system.

Under normal circumstances, it is not expected to have pending operations in ECRMREPL related to the same contract before performing the next change of this contract on the CRM system. You can, for example, prevent a change to a contract on the CRM system while a previous replication attempt has not yet finished. For this, you need to implement a related check using via Utilities Process Check Framework (see attached document with a sample implementation of such a check).

Other Terms

ECRMREPL

Reason and Prerequisites

Program error

Solution

After implementing this SAP Note, the issue will not recur. Technically speaking, the change of the move-in date will be applied to all pending ECRMREPL entries relating to the same contract while other changes, such as product relevant changes, remain within each ECRMREPL entry. Please keep in mind that a full generation of the contract will remediate any data inconsistency anyway.
