Symptom

Symptom

Performance enhancements have been implemented in DO BDOSOC postings to improve system efficiency. Optimizations were applied specifically during the Approve Line action for Revenue Allocation Lines for FI Approval and while executing BDOSOC for the RALL-04 activity. These changes reduce processing time and improve overall posting performance.

Other Terms

DO BDOSOC Performance, BDOSOC Postings, Approve Line Issue, Revenue Allocation Lines for FI Approval, RALL-04 Activity

Reason and Prerequisites

This is caused by program error

Solution

Please implement the correction instructions attached. These corrections are only valid if you have applied the Support Packs.

Release V/4 - 2021, 2022, 2022 V2, 2023, 2023 V2, 2023 V3 and 2023 V3 Support Pack 1

Use transaction SNOTE to implement the correction instructions specified below and activate the affected program object.

FLP notifications were not enabled for workflow actions such as Submit, Approve, Reassign, Partial Approval, or Reject.

Solution

FLP notification functionality has been enabled for AdjustmentHub and ApprovalHub applications. The system now automatically sends notifications to Adjusters and Approvers based on workflow actions and approval status changes.

Notification Scenarios

Submit Action

Case 1 – Auto Approve
Notification To: Adjuster
Message: Item(s) created on <date> are auto approved.

Case 2 – Pending Approver Assignment
Notification To: Adjuster
Message: Item(s) created on <date> are pending approver assignment.

Case 3 – Auto Approver Assignment
Notification To: Approver
Message: Item(s) created on <date> are pending your approval.

Reassign

AdjustmentHub Reassign
Case 1 – 1st Level Approver
Notification To: Approver
Message: Item(s) created on <date> are pending your approval.

ApprovalHub Reassign
Case 1 – Assigned Approver
Notification To: Approver
Message: Item(s) created on <date> are pending your approval.

Approval

Case 1 – Next Level Approval
Notification To: Approver
Message: Item(s) created on <date> are pending your approval.

Case 2 – All Level Approved (Skip Auto Approval)
Notification To: Adjuster
Message: Item created on <date> has been approved.

Partial Approval

Case 4 – Document Level Next Level Approval
Notification To: Approver
Message: Item(s) created on <date> are pending your approval at the document level.

Case 3 – Document Level All Level Approved
Notification To: Adjuster
Message: Item created on <date> has been approved at the document level.

Reject

Case 1 – Reject at Group Level
Notification To: Adjuster
Message: Item(s) created on <date> has been rejected.

Case 2 – Reject at Document Level
Notification To: Adjuster
Message: Item(s) created on <date> has been rejected at the document level.

BADI Details for Email Trigger.

  • Email notification functionality is not delivered as standard and must be implemented using the provided BADI.

To enable email notifications, implement the following BADI:

  • BADI Name: /FC1/BADI_EMAIL_TRIGGER
  • Method: TRIGGER_EMAIL

This method is triggered during workflow actions and can be used to send email notifications to relevant users.

Method Signature

Parameter

Description

IV_CINS_ID

CaseIns ID of the request

IV_GROUP_ID

Group ID of the request

IV_ACTION

UI action name (e.g., 'ADJH-SUBMIT', 'ADJH-REASSIGN', 'APPH-REASSIGN', 'APPH-APPROVE', 'REJECT')

IV_PARTIAL

Indicator for partial approval scenario

IT_APPR_CHECK

Contains approver details, approval level, auto-approve flag, and item status

IT_DATA

Contains request item details such as Posting Date, Document ID, current approval level

IT_DOCID

Document ID (used in partial approval scenarios)

CT_NOTIF_PARAMETERS

Deep internal table containing finalized notification data

CV_SKIP_NOTIFICATION

Flag to skip FLP notification trigger

Implementation Details

  • Implement custom logic inside method TRIGGER_EMAIL to send emails based on:
    • Action (IV_ACTION)
    • Approval level
    • Item status
  • Use available data in IT_APPR_CHECK and IT_DATA to construct email content dynamically.
  • Handle partial approval scenarios using IV_PARTIAL and IT_DOCID.

Important Note

  • Parameter CT_NOTIF_PARAMETERS contains all the finalized recipient and message details used for FLP notifications.
  • It is recommended to reuse this parameter to trigger email notifications for the same set of users.

. Enhancement of Notification Logic via Class /FC1/CL_AH_NOTIFICATIONS

  • Standard notification texts and values generated in FLP Notifications may not meet specific customer requirements (e.g., custom text, additional fields).

Solution

Customers can enhance the notification logic by post-Method enhancement of the following methods of class:

Class: /FC1/CL_AH_NOTIFICATIONS

Methods for Enhancement

  1. /IWNGW/IF_NOTIF_PROVIDER~GET_NOTIFICATION_TYPE_TEXT
    • Used to determine and modify the notification text displayed in FLP.
    • Can be enhanced to provide custom message content based on:
      • Action (Submit, Approve, Reject, etc.)
  2. SET_NOTIF_VALUE
    • Used to populate and control notification parameters/values.
    • Can be enhanced to:
      • Add additional data to notifications
      • Modify existing values
