Symptom

You want to install SAP Data Intelligence 3 on SUSE Container as a Service Platform 4.2 on premise.

Other Terms

Data Hub, SLES, Kubernetes, SUSE CaaSP, SUSE, Linux, Container, cri-o, SAP Data Hub, CaaSP, SLES for SAP, Data Intelligence, SAP Data Intelligence, SAP DI

Reason and Prerequisites

Installation of SAP Data Intelligence 3.0.4 (or greater) on SUSE Container as a Service Platform 4.2

Solution

Planning SAP Data Intelligence 3 on SUSE Container as a Service Platform 4.2

For a basic installation of SAP Data Intelligence 3 on SUSE CaaS Platform 4.2 (CaaSP 4.2), you will have to have a minimum

    3 servers as Kubernetes master nodes, 64 GiB RAM, 150 GiB for root file system
    4 servers as Kubernetes worker nodes, 64 GiB RAM, 150 GiB for root file system
    1 management host (jump host), 16 GiB RAM, 200 GiB for root file system
    The installation of SUSE CaaSP 4.2 requires a load balancer. A setup of a simple loadbalancer can be found in the SUSE CaaSP 4.2 deployment documentation.
    You will need as well access to a local registry for storing the Docker images of SAP Data Intelligence.
    You also need the access to a SUSE Enterprise Storage 5 (or higher) to provide the data store for SAP Data Intelligence 3.

 

Preparation for SAP Data Intelligence 3 on SUSE Container as a Service Platform 4.2

Make sure you have your SUSE subscription keys at hand for registering your installation of SUSE CaaS Platform 4.2.

Registering your systems entitles you to get SUSE software updates and support.

It is highly recommended to have a static IP setup for your SUSE CaaS Platform 4.2 installation.

You have to provide a time source for your installation (ntp server).

 

Installation of SUSE Container as a Service Platform 4.2

1) Install SUSE Linux Enterprise Server 15 SP1 on all the nodes that should join the Kubernetes cluster

    Follow the SUSE documentation for installing SUSE CaaS Platform 4.2: https://documentation.suse.com/suse-caasp/4.2/
    Especially follow https://documentation.suse.com/suse-caasp/4.2/html/caasp-deployment/_deployment_instructions.html#deployment.preparations
    Here the general preparations are described for deploying SUSE CaaS Platfom 4.2.
    For the on-premise/bare-metal installation check out
        https://documentation.suse.com/suse-caasp/4.2/html/caasp-deployment/_deployment_instructions.html#deployment_bare_metal
        https://documentation.suse.com/suse-caasp/4.2/html/caasp-deployment/_deployment_instructions.html#_deployment_on_existing_sles_installation

    Register your installations either during installation workflow or after the installation.

        To do a registration it is necessary for your landscape to be able to connect to the internet in order to reach SUSE's Customer Center (SCC) https://scc.suse.com

2) Install a management host with SUSE Linux Enterprise Server 15 SP1

    Install SUSE Linux Enterprise Server 15 SP1 by following the product documentation.
    Register your installation and do not forget to add these software repositories:

        sudo SUSEConnect -r CAASP_REGISTRATION_CODE
        sudo SUSEConnect -p sle-module-containers/15.1/x86_64
        sudo SUSEConnect -p caasp/4.0/x86_64 -r CAASP_REGISTRATION_CODE
        sudo SUSEConnect -p sle-module-python2/15.1/x86_64

 

    Install the required tool set for managing SUSE CaaS Platform 4
        sudo zypper in -t pattern SUSE-CaaSP-Management

    Install Docker, this is needed for installing SAP Data Intelligence 3
        sudo zypper in docker

    Install python-2.7 components needed for installing SAP Data Intelligence 3
        sudo zypper in python2-pyOpenSSL
        sudo zypper in python2-PyYAML

3) If not already available, install a machine with SLES 15 SP1 as loadbalancer

    Follow the instructions given in https://documentation.suse.com/suse-caasp/4.2/html/caasp-deployment/_deployment_instructions.html#deployment.preparations to configure a machine as loadbalancer.

4) Create the Kubernetes cluster:

    Log into your management host as root and execute the commands to create the SUSE CaaSP 4 Kubernetes cluster.
    Make sure to have ssh-agent running with respective keys to access the cluster nodes.
    Initialise the cluster:
    skuba cluster init --control-plane <LB IP/FQDN> my-cluster
    First master node and cluster bootstrap:
    cd my-cluster
    skuba node bootstrap --target <IP/FQDN> <NODE NAME>
    Additonal master nodes:
    skuba node join --role master  --target <IP/FQDN> <NODE NAME>
    Worker nodes:
    skuba node join --role worker--target <IP/FQDN> <NODE NAME>
    Check status of the newly created kubernetes cluster:
    skuba cluster status

 For more information see the official SUSE CaaS Platform 4.2 documentation: https://documentation.suse.com/suse-caasp/4.2/

5) Preparations for SAP Data Intelligence 3

     If not stated otherwise the commands and actions below have to be performed on the management host ("jump host")

 5.1)   Install ceph-common and xfsprogs on all SUSE CaaS Platform 4.2 Kubernetes nodes to make use of storage classes based on SUSE Enterprise Storage
        zypper in ceph-common xfsprogs
    
  5.2)   Edit /etc/crio/crio.conf on all Kubernetes nodes:
           set
           pids_limit = 4096
           Edit /etc/containers/registry.conf on all Kubernetes nodes:
          Create entry for your secure local registry, e.g.:
          [["FQDN_of_secure_registry:5000"]]
          insecure = false
          prefix = "FQDN_of_secure_registry:5000"
          location = "FQDN_of_secure_registry:5000"

 5.3) Re-load crio on all Kubernetes nodes:
        systemctl restart crio
    
 5.4) Install python2 and python2-PyYAML on the management host:
        If not already done, add the python2 module
        SUSEConnect -p sle-module-python2/15.1/x86_64
        zypper in python2-PyYAML
        zypper in python2-pyOpenSSL
        This installs python2-base, python2-PyYAML and python2-pyOpenSSL these are requirements for the installation of SAP Data Intelligence 3.

 5.5) To easily use kubectl and helm on the management host, copy the cluster configuration file ~/my-cluster/admin.conf to ~/.kube/config
        cp -av  ~/my-cluster/admin.conf  ~/.kube/config

 5.6) Create namespace for installation of SAP Data Intelligence 3, e.g. run on the management host:
        kubectl create namespace di304

 5.7)  Create CEPH based storageClass and secrets for use with SAP Data Intelligence 3

         On the management host create a file named storageclass.yaml, for example:
        cat <<EOF > storageclass.yaml
        apiVersion: storage.k8s.io/v1
        kind: StorageClass
        metadata:
        annotations:
          storageclass.kubernetes.io/is-default-class: "true"
         name: <NAME OF STORAGECLASS>
         namespace: default
       parameters:
         adminId: <ADMINID>
         adminSecretName: ceph-admin-secret
         adminSecretNamespace:  default
         imageFeatures: layering
         imageFormat: "2"
         monitors: <IP OF CEPH MONITOR 1 >:6789, <IP OF CEPH MONITOR 2>:6789, <IP OF CEPH MONITOR 3>:6789
         pool: <CHANGE TO YOUR CONVENIENCE>
         userId: <USERID to access SES >
         userSecretName: ceph-user-secret
       provisioner: kubernetes.io/rbd
       reclaimPolicy: Delete
       volumeBindingMode: Immediate
    EOF

   Run
    kubectl create -f storageclass.yaml
    to create the storage class.

   Create the secret files for accessing the storage

   Get the necessary userids and credentials from your storage administrator and create the secret files as shown by example:
    Create the base64 encoded credentials:
    export ADMINKEY=$(echo "<userkey>"  | base64)
    export USERKEY=$(echo "<adminkey>" | base64)

    cat <<EOF >ceph-admin-secret.yaml
    apiVersion: v1
    kind: Secret
    metadata:
        name: ceph-admin-secret
        namespace: <NAMESPACE FOR DATAHUB>
    type: "kubernetes.io/rbd"
    data:
       key: $ADMINKEY
    EOF

    cat <<EOF>ceph-user-secret.yaml
    apiVersion: v1
    kind: Secret
    metadata:
        name: ceph-user-secret
        namespace: <NAMESPACE FOR DATAHUB>
    type: "kubernetes.io/rbd"
    data:
       key: $USERKEY
    EOF

    kubectl create -f ceph-admin-secret.yaml
    kubectl create -f ceph-user-secret.yaml

5.8)  Adapt Pod Security Policy and Hostpath Mount Configurations

        To edit the default pod security policy of your SUSE CaaS Platform cluster so that pods are able to mount to the "/" directory of the host, run the following command:
        kubectl edit psp suse.caasp.psp.privileged
        Change the pathPrefix in allowedHostPaths to /

5.9) To let the service account create pods using the privileged mode, create the clusterrolebinding.yaml file using the following commands:

cat <<EOF > clusterrolebinding.yaml

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: suse:caasp:psp:priviliged:default
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: suse:caasp:psp:privileged
subjects:
- kind: ServiceAccount
  name: default
  namespace: XXX
- kind: ServiceAccount
  name: vora-vsystem-XXX
  namespace: XXX
- kind: ServiceAccount
  name: vora-vsystem-XXX-vrep
  namespace: XXX
- kind: ServiceAccount
  name: XXX-elasticsearch
  namespace: XXX
- kind: ServiceAccount
  name: XXX-fluentd
  namespace: XXX
- kind: ServiceAccount
  name: XXX-nodeexporter
  namespace: XXX
- kind: ServiceAccount
  name: vora-vflow-server
  namespace: XXX

- kind: ServiceAccount
  name: mlf-deployment-api
  namespace: XXX

EOF

5.10) To insert the Kubernetes namespace that you are installing SAP Data Intelligence 3 into in the clusterrolebinding.yaml file, run the following commands:

Note: Change <your_namespace> to the Kubernetes namespace.

export NAMESPACE=<your_namespace>
sed -i "s/XXX/${NAMESPACE}/g" clusterrolebinding.yaml

5.11) In the same directory where you created the clusterrolebinding.yaml file, run the following command:

kubectl apply -f clusterrolebinding.yaml

        Run the following command:

        kubectl edit clusterrolebinding system:node

        Insert the following at the end of the file:

        subjects:
        - apiGroup: rbac.authorization.k8s.io
          kind: Group
          name: system:nodes

    If the registry certificate is untrusted or self-signed, follow the steps below in order to create necessary Kubernetes secret with the certificates so that SAP Data Intelligence installation procedure can import it into SAP Data Intelligence Connection Manager.
        Certificate chain should be saved into file named "cert" in PEM format.
        export DI installation namespace into NAMESPACE environment variable.
        Ensure that your file doesn't contain DOS-type line endings. To convert from DOS-type line endings, you may use the following commmands:

                mv cert cert_with_carriage_return

                tr -d '\r' < cert_with_carriage_return > cert

        run "kubectl create secret generic cmcertificates --from-file=cert -n $NAMESPACE" command.

 NOTE: File name should be cert.

6.    Installation of SAP Data Intelligence

6.1.) Install SAP Data Intelligence according to Installation Guide for SAP Data Intelligence. For CaaSP 4 specific configurations, check the items below.

        Installation/Upgrade with SLC Bridge (SL Plugin):
        Choose "Advanced Installation" for "Installation Type"
        Choose "Configure container log path" for "Docker Container Log Path Configuration"
        Set "Container Log Path" as "/var/log/containers"
        Set "Enable Kaniko Usage" as yes
        "-e diagnostic.fluentd.logDriverFormat=regexp -e vsystem.vRep.exportsMask=true" into "Additional Installation Parameters"

6.2.) Expose SAP Data Intelligence System Management Externally

After following the steps in the "Expose SAP Data Intelligence System Management Externally in On-Premises" section in the Installation Guide for SAP Data Intelligence, and installing ingress-nginx, the service account of nginx should be authorized.

1. Get the name of the service account that is used by nginx-ingress:

          kubectl -n <your_namespace> get serviceaccounts -lapp=nginx-ingress -o name

2. Edit the clusterrolebinding that you created above, and add a new item into the list using the following command:

         kubectl edit clusterrolebinding  suse:caasp:psp:priviliged:default

           - kind: ServiceAccount
             name: <service_account_of_nginx_ingress>
             namespace: <your_namespace>

7.) Uprgading from existing SAP Data Hub 2.7.x Installation on SUSE CaaSP 4.0 to SAP Data Intelligence 3.0 patch 4 and greater.

First you need to upgrade to the latest version of SAP Data Intelligence 3 that is released for SUSE CaaSP 4.0, see SAP Note 2898735 and SAP Note 2871970 for details.

Make sure to read SAP Note https://launchpad.support.sap.com/#/notes/2908055

The upgrade can only be done with SLC Bridge

Choose "Expert Mode" during setup of the SLC Bridge to be able to select "nodePort" for the SLCBridge service.

Add the ML ServiceAccount to ClusterRoleBinding suse:caasp:psp:priviliged:default
- kind: ServiceAccount
  name: mlf-deployment-api
  namespace: XXX

After successfull upgrade of SAP Data Hub 2.7.x to SAP Data Intelligence 3.0 patch 4, proceed with the upgrade of SUSE CaaSP 4.0 to SUSE CaaSP 4.2.

Follow the update procedure described in SUSE CaaSP 4.2 Administration Guide.
