Symptom

I notice that the counts of records created, updated, or deleted in my Import task summary do not match the number of rows in my source file.
I see that the task shows Completed with specific counts, but these totals differ from the total entries I provided in my CSV or Excel file.

Cause

This behavior is expected because the import task summary reflects the exact number of objects committed to the database, not the number of rows processed from your file. The discrepancy can be attributed to the following scenarios.

Why the Record Count is Lower than the Number of File Rows

  • Import Operation: The selected import operation (Load, Create, Update Only, Deactivate) restricts the system's actions. For example, if you choose Update Only, the system will skip any new records in your file, resulting in a Created Records count of zero. The Load operation is required to both create new records and update existing ones in the same task.
  • Header Row: The system ignores the header row of your import file.
  • Invalid Records: The system skips any records with formatting errors or missing mandatory fields, so they are not counted.
  • Duplicate Entries: If your file contains multiple rows for the same unique identifier (e.g., UniqueName), the system consolidates them into a single update.
  • Object-Level Mapping: In tasks like Import User to Group Mapping Data (Consolidated File), the system counts the target object being modified (the group), not the number of users being mapped to it.
  • Deactivated Objects: If you import an object that was previously deactivated (e.g., a Cost Center), the system updates the existing record instead of creating a new one.

Why the Record Count is Higher than the Number of File Rows

  • Hierarchical Object Creation: When importing data with parent-child relationships (like Cost Centers or Purchasing Units), a single row can trigger the creation of multiple objects. If you import a child object and its specified parent objects do not yet exist, the system will automatically create the entire parent hierarchy needed to support the new child object. This results in a higher count for Created Records.
  • Cascading Updates in a Hierarchy: When you update a parent object, the system may cascade changes to its child objects. For example, reactivating a parent Purchasing Unit can trigger updates to all its child units, resulting in a higher count for Updated Records than the single row in your file that initiated the change.

Resolution

To investigate the record counts and understand why the numbers differ, follow this complete troubleshooting workflow:

  1. Review the Task Summary:

    • Navigate to Manage > Core Administration > Data Import/Export and click the Import tab.
    • Locate the import task and click the status link (e.g., Completed, Completed (!)) to open the Details page and review the final record counts.

  2. Verify the Import Operation:

    • The task details page in Data Import/Export does not show which import operation was used. This is a common reason for count discrepancies.
    • To find this information, review the audit logs stored in the Intelligent Configuration Manager (ICM):
      • Navigate to Manage > Intelligent Configuration Manager > Manage Configurations.
      • Search the audit log integration event history for the task to confirm which operation (Load, Update Only, etc.) was used. This will explain why certain records may not have been created or updated.

  3. Check for Row-Level Errors:

    • Return to the task details page in Data Import/Export.
    • If the task status is Completed(!), click the Error details tab. This tab lists the specific rows from your file that failed to process and provides the reason for each error.
    • Note: The Error details tab only displays rows with explicit failures. It will not explain discrepancies caused by system logic, like record consolidation or hierarchical updates.

  4. Investigate Logical Discrepancies in the File:

    • To check for consolidation, open the source file and filter by the unique identifier column (e.g., UniqueName) to see whether multiple rows were correctly processed as a single update to a single object.
