Symptom

Initializing High Availability Disaster Recovery (HADR)
If you are setting up HADR in a Db2 pureScale environment or if you want to use multiple standby databases, you need to set the hadr_target_list configuration parameter on all participating databases. This parameter lists the standbys in the scenario when the database becomes a primary.
It is required even on a standby. Mutual inclusion is required (that is, if A has B in its target list, B must have A in its target list). This ensures that after a takeover from any standby, the new primary can always keep the old primary as its standby.
After adding the additional serve in hadr_target_list, db2 prompts to deactivate and activate the database.


 

Environment

Db2 LUW IBM

Reproducing the Issue

  • Update hadr_target_list online. 
  • db2 prompts to deactivate and activate the database.

Cause

Updating hadr_target_list online is possible, however there are restrictions 

Resolution

  1. The hadr_target_list has the following restrictions when updating the parameter online
    The full restrictions are listed in IBM document hadr_target_list - HADR target list database configuration parameter - IBM Documentation

    You cannot change the principal standby of the primary while HADR is active; you must first stop HADR on the primary.
    You cannot remove a standby from the list if it is currently connected to the primary; the standby must be deactivated first.
    You cannot change the parameter online for a standby database unless reads on standby are enabled.
    You cannot remove the primary from the target list of a standby if that standby is currently connected to the primary.
    You cannot change the parameter online in a DB2 Pure Scale environment.
    All IP addresses in the target list must be either all IPv4 or all IPv6 — you cannot mix address types.

  2. Although there are restrictions to change the hadr_target_list configuration parameter online, you do not need to take an outage of the primary database in order for the change to take effect.
    In all situations, changes to hadr_target_list can take effect after STOP HADR and START HADR command on the primary database.
