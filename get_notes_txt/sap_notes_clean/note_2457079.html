Symptom

While trying to remove a certain JVM parameter in Config Tool, following error popup appeares:

"Cannot validate jvm parameters - java process cannot start; Exit code: [1]; output of the process [Error...]"

Certain JVM parameter should be removed due to emergency case (eg AS Java startup issue).

Steps 1-3 of KBA 1775485 have been executed without success. Due to some reasons JVM upgrade is not possible at that time either. Workaround is needed.

Environment

  • SAP NetWeaver
  • Release Indepedent

Reproducing the Issue

Open Config Tool -> Navigate to "VM Parameters" -> try to remove desired parameter

Cause

When the java VM parameter is added/modified in the Configtool, the tool tries to validate this new/modified parameter by running a java process and using the new/modified parameter on it.

Resolution

  • As a workaround you can disable the relevant parameter:

    Open Oflline Config Editor -> press "Edit" (pencil button with glasses) -> press "Yes" -> navigate to the affected Porpertysheet where the JVM parameter is present (e.g. cluster_config -> instances - ID<XXXXX> -> cfg -> jvm_params -> Propertysheet additional) -> double click on the Propertysheet -> choose the rlevant JVM parameter and double click on it -> check "Show Meta Attributes" -> check "Disabled" -> Apply custom:
      

    Then restart the application server to validate changes.

  • If it is required the relevant parameter to be removed try the following workaround:
    1. Disable the bootstrap configuration as per the SAP Note 710663. Use the "skip" option. This will prevent your changes to be overwritten.
    2. Restart AS Java in order to validate boostrap.properties settings.
    3. Remove the desired parameters from the ..LocalDrive\usr\sap\<SID>\<instnr>\j2ee\cluster\instance.properties file. Before you save the file make a backup from this instance.properties file. Then save the file.
    4. Instead of disabling them now remove the JVM parameters from Configtool and save the settings.
    5. Restart the Java cluster again in order to validate instance.properties file and Configtool.

Important note: 

If this workaround is used then none of the AS Java files in the filesystem will be updated to the fresh ones that are in the DB. This could lead to inconsistences. For example, when a patch is applied to the main instance, it will not be replicated to the other instances. Even if another patch is applied, and after some time boostrap is enabled. In this case the applications will be working with a different logic in the cluster. This could cause a series of failures, including startup, or p2p communication between the server nodes. Knowing these vulnerabilities do not use this as permanent solution. 

  • Permanent solution:

After setting back the bottstrap.properties to default value the problem might reoccure again. Therefore the JVM has to be updated to the latest available patch.
