Symptom

You are replicating Contracts to C4C and observing the following error messages: 

"Saved consistent active contract cannot be saved inconsistent active".
"Saved replicated contract cannot be saved inconsistent".
"Location for incoterms missing".

Environment

SAP Cloud for Customer

Reproducing the Issue

  1. Replicate contract data from SAP S/4HANA to  C4C.
  2. Go to the Administrator work center.
  3. Go to the General Setting work center view.
  4. Open the Web Service Message Monitoring under System Administration
  5. Select query All
  6. Search for message with errors:

"Saved consistent active contract cannot be saved inconsistent active".
"Saved replicated contract cannot be saved inconsistent".
"Location for incoterms missing".

Cause

Payload is missing an Incoterms location:

      <DeliveryTerms>
        <Incoterms>
          <ClassificationCode>ABC</ClassificationCode>
          <TransferLocationName/>
        </Incoterms>
      </DeliveryTerms>

(ABC means a generic ClassificationCode).

This absence is causing the error message: "Location for incoterms missing".

The contract with C4C ID 123 ("123" represents a generic ID), (External ID 456 ("456" represents a generic ID)) already exists in C4C with an Incoterms Location set to "Destination". It's important to note that deleting the Incoterms Location from an active contract is not permitted in C4C system. If you open Contract 789 ("789" represents a generic contract), in C4C UI and enter Edit mode, you’ll see that this field cannot be changed or removed.

As a result, attempting to delete the Incoterms Location via the replication message would render the contract inconsistent, which is not permitted once contract has been saved in an active (and consistent) state. This leads to the following error messages:

"Saved consistent active contract cannot be saved inconsistent active".
"Saved replicated contract cannot be saved inconsistent".
 

Resolution

When compare the message payload with the message payloads of other ContractReplicationFromExtIn messages, then there the Incoterms Location is filled like this:

      <DeliveryTerms>
        <Incoterms>
          <ClassificationCode>ABC</ClassificationCode>
          <TransferLocationName>Destination</TransferLocationName>
        </Incoterms>
      </DeliveryTerms>

To resolve this issue, you must update the payload to include the Incoterms Location.
