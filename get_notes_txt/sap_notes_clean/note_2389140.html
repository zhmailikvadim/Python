Symptom

The correspondent business scenario, from the technical point of view, requires a Lumira Designer Application to be built. 

Conditional formatting is needed to be used in order to define threshold values and visualize based on those.

Many differences are recognized in terms of the dialogs, depends on whether the rules are defined in Design Time/Runtime, or Conditional formatting technical component/BEx Exceptions used.

Environment

  • Lumira Designer 2.x

Reproducing the Issue

  1. Lumira Designer Application is needed to be created.
  2. Conditional formatting Rules are wanted to be defined either with, or without Conditional formatting technical component.
  3. Define Rules either in Design Time, or Runtime.

Cause

In Lumira Designer, there is no common dialog in order to define Conditional formatting rules. To be able have the Conditional formatting visible (regardless whether it is BEx Exception, or not ), it is needed to set 'Conditional Formatting Visible' flag of the visualization components to be set to 'true'.   

 First of all it is needed to be decided whether Conditional formatting technical component is wanted to be used, or not. As soon Conditional formatting technical component in place, the dialog of the component will be used, which is slightly different from Conditional formatting dialog which is used, if no technical component added to the Application. For example, previously defined BEx exceptions will not be visible in the Conditional formatting dialog, even if the BEx Exception thresholds are applied. Without the technical component, BEx Exceptions are visible Conditional formatting dialog, those can be modified and copied. It is possible to set 'Enable BEx Exception' flag of the technical component  to 'true', but it does not change the above mentioned fact, it just allows the application designer to create BEx level exceptions.

Even if Conditional formatting technical component is used, the dialog can be still slightly different. First lets talk about Design Time. By double-clicking on the technical component a generic rule manager dialog appears, where it is possible to define rules, but not possible to bind those directly to visualization components like Chart, or Crosstab, hence no 'Apply' checkbox visible. When the 'Conditional Formatting Settings' property of the visualization components pressed, the dialog which is opened, is the dialog of the technical component, but related to the visualization component itself. It means that it is possible to bind the previously defined rules, hence there is an 'Apply' checkbox available.

Similarly to Design Time, also during runtime we differentiate in between generic rule manager dialog of the technical component and component specific rule dialog of the technical component.
 - openRulesDialog() scripting API triggers the generic rule manager dialog, hence no 'Apply' checkbox.
 - openRulesDialog(<"DataBoundComponent">) triggers the component specific rule dialog, hence hence 'Apply' checkbox available.
   Same stands for the workflow if the Condition formatting dialog triggered from the context menu by clicking on the visualization components.

Resolution

The above mentioned technical facts are needed to be considered by the application designer.
