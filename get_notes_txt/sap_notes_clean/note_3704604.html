Symptom

  • When creating a model 55 NFe (Brazilian electronic invoice) debit note, type 04 (fine/interest), the XML tag for the tax field ICMS is automatically set to 40 when the law field (J_1BDYLIN-TAXLW1) is not filled.
  • This behavior is incorrect for NFe with document type D, as ICMS information should not be included in the XML.
  • Similarly, if the IPI law field (J_1BDYLIN-TAXLW2) is not filled, the XML tag for cEnq is populated, which is incorrect for this type of document.
  • The error message encountered is: "[Error] - cvc-type.3.1.3: The value '' of element 'cEnq' is not valid."

Environment

SAP S/4HANA PRIV. CL. ED.
SAP/R3
Outbound

Reproducing the Issue

  1. Create a model 55 NFe (Brazilian electronic invoice) debit note, type 04 (fine/interest), using BAPI_J_1B_NF_CREATEFROMDATA.
  2. Assign debit NF type 04 (fine/interest) to the nota fiscal type ND with document type D for outgoing transactions.
  3. Leave the law field (J_1BDYLIN-TAXLW1) blank.
  4. Observe that the XML tag for ICMS is automatically set to 40.
  5. Leave the IPI law field (J_1BDYLIN-TAXLW2) blank.
  6. Observe that the XML tag for cEnq is populated incorrectly.
  7. Encounter the error message: "[Error] - cvc-type.3.1.3: The value '' of element 'cEnq' is not valid."

Cause

  • The SAP programs SAPLJ_1B_NFE / LJ_1B_NFEF80 in the FORM MAP_TAX automatically set the XML tag for ICMS to 40 when ls_rfc_tax_icms-cst is blank.
  • Similarly, the IPI law field (J_1BDYLIN-TAXLW2) is incorrectly populated in the XML when left blank, which is not appropriate for document type D.

Resolution

  • The legacy NF-e item tax mapping in J_1B (SAPLJ_1B_NFE, include LJ_1B_NFEF80) is defensive: when TAXLW1 (ICMS law) isn’t provided, it defaults to an “exempt/other” CST (40) to keep the ICMS group syntactically valid for typical merchandise flows.
  • A similar pattern wrongly emits the IPI node with an empty cEnq when TAXLW2 is blank.
  • For a debit note of multa/juros (D / type 04), that fallback is incorrect, as the document is not merchandise taxation—only CBS/IBSS must be present.

What “good” should look like for D/04

  • Item-level XML:
    • No ICMS group at all
    • No IPI group at all (hence no cEnq)
    • Presence of only CBS/IBSS nodes, as applicable to the debit (fine/interest)
  • CFOP should reflect the financial nature (e.g., “other” operations) and avoid merchandise triggers.
  • J_1B settings and tax code must not calculate ICMS/IPI.

Configuration approach:

  • CFOP and operation type
    • Use a CFOP appropriate for fines/interest (financial/other), not a merchandise sale. This helps prevent merchandise tax routines.
  • Nota fiscal item type as Service/Other
    • If your debit note line is not “goods,” classify the NF item type as Service/Other so the ICMS routine is not triggered. In practice:
      • Ensure the item category and J_1B item attributes mark it as service/other (ISSQN/CBS/IBSS relevant) and not ICMS/IPI relevant.
  • Tax procedure and tax code
    • Use a tax code that calculates only CBS/IBSS; no ICMS/IPI condition types should be active.
    • In J1BTAX, remove ICMS/IPI conditions from this scenario. If any ICMS/IPI base is non-zero, the mapping tends to build groups.
  • Maintain both “ICMS law” and “IPI law” empty
    • This is desired for D/04. Combined with the service/other classification and proper tax code, the system should not enter MAP_TAX logic for ICMS/IPI.
