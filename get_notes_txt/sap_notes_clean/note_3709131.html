Symptom

In T-code CWBQM, users can enter the single‑object detail screen and edit for a while, but the lock conflict is only shown when pressing Save.

Environment

  • SAP ERP Quality Management (QM)
  • SAP ERP
  • SAP R/3
  • SAP R/3 Enterprise
  • SAP ERP Central Component
  • SAP Enhancement package for SAP ERP
  • SAP Enhancement package for SAP ERP, version for SAP HANA
  • SAP S/4HANA, on-premise
  • SAP S/4HANA Cloud Private Edition

Reproducing the Issue

  1. Open T-code CWBQM and select the working area. 
  2. Click on Load Task Lists after entering selection criteria.(This simulates the situation where an user is locking the object.)
  3. Open a new session (do not close previous window) and repeat steps 1–2. 
  4. Select one line to edit(for example: select group 3 & counter 2, and click the "Material assigned" icon.
  5. Add information including material, plant etc. on the Assigned Materials overview screen.
  6. Lock conflicts only at Save

Cause

It is by standard design. 

CWBQM built for list-based and multi-object editing uses a late/optimistic locking approach.

The system stores changes in a session buffer and only requests the object lock upon saving, instead of immediately placing an SAP enqueue lock on the task list header. 

By deferring locks until saving, the framework is to minimize overlocking multiple objects during browsing or editing.

Conversely, T-code QP02 uses an early or pessimistic locking mechanism. It requests the `PLKO lock` as soon as the Group + Counter is entered. 

Resolution

As a workaround, to help mitigate potential wasted time during editing, the following is recommended:

  1. Use QP02 when an immediate lock check is required. 
  2. Monitor the lock indicators in the list (e.g., the padlock icon/column) for early signs that an object may already be in use.
