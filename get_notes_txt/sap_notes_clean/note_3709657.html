Symptom

Large S/4HANA orders (900+ items) sometimes trigger duplicate API requests to EMS, causing duplicate entitlements to be generated.

Environment

  • SAP Entitlement Management
  • SAP S/4Hana

Cause

EMS sequential queues only serialize/wait for prior messages to finish; they do not inspect content to suppress or discard duplicates. Relying on sequential mode alone will not prevent duplicate entitlements when S/4 sends near‑simultaneous or repeated messages for large orders.

Resolution

Implement a middleware idempotency filter that checks a stable idempotency key (e.g., SalesOrderGUID_ItemNumber_ChangeVersion) against a lightweight datastore with a TTL (24–48h); if the key exists discard/log the message, otherwise insert the key and forward to EMS. Configure EMS inbound queue = Sequential with one worker so processing is ordered and single‑threaded.

Limitations: there is no native content-based duplicate suppression inside EMS queues; queues only serialize processing. If you require a native EMS feature (content-based suppression or a supported pre-create hook), please open an incident with sample payloads so we can escalate to Development.
