Symptom

  • You have enabled the 'Dead-Letter Queue' parameter in the configuration of the JMS sender adapter but messages are not moved to the Dead-Letter Queue after a specific number of failed processing attempts
  • You do not see the 'Dead-Letter Queue' parameter available in the configuration of the JMS sender adapter

Environment

  • Cloud Integration
  • SAP Integration Suite

Cause

The option to enable 'Dead-Letter Queue' in the configuration of the JMS sender adapter is only available for 'Non-Exclusive' JMS queues. Ie. if the Access Type parameter in the adapter is configured as 'non-exclusive'.

Furthermore, the Dead-Letter Queue is not applicable for messages which are failing due to some error during processing, eg. due to an error being thrown at some processing step, or the unavailability of a receiver system during an outbound call from the integration flow. It is only relevant for messages where the processing has been 'hard-interrupted', ie. due to a node restart/Out Of Memory event. 

Resolution

For a detailed description of how the 'Dead-Letter Queue' operates, we would recommend checking the SAP community blog at Cloud Integration – Configure Dead Letter Handling in JMS Adapter.

As per this blog, if a message fails processing due to an Out Of Memory (OOM) event on the tenant, it will only be tried twice more. If both of these runs also fail due to an OOM event, it can be determined that most likely the cause of the Out Of Memory is due to this specific message. In this case, if the 'Dead Letter' queue setting is enabled, no further retries will occur, and the message will instead be 'blocked'. This is to prevent the message from causing further node restarts due to OOM. 

If a message is failing due to an error being thrown during the processing of the message, this will not result in the message being moved to the dead-letter queue. Instead, they would be retried over and over until the processing is successful.  

If you wish to set a limit on the number of retries which are performed for such messages, you would need to implement an 'Explicit Retry Configuration'. This is also mentioned in the developer blog - 

Dead Letter Queue Versus Explicit Retry Configuration

The dead letter queue configuration is not connected to the configuration of explicit retry handling described in the blog Configure Asynchronous Messaging with Retry Using JMS Adapter. The dead letter queue is only used for messages that could no longer be processed anymore because of a node crash.

Messages that fail because of an error during processing, do not go to the dead letter queue, but either stay in the normal processing (with a defined retry interval) or are removed from JMS (if configured in explicit retry modeling).

You can find further details on how to configure an explicit retry configuration in this Configure Asynchronous Messaging with Retry Using JMS Adapter blog, see the section 'Optional: Explicit Retry Configuration'.

The implementation of an explicit retry configuration will allow you to apply a 'Maximum number of retries', should you wish to do so.
