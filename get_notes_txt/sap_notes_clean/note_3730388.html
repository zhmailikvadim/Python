Symptom

Your data includes control characters, and you are using the methods CL_ABAP_LIST_UTILITIES=>DISPLAY_TO_STRUCTURE and CL_ABAP_LIST_UTILITIES=>DYNAMIC_OUTPUT_LENGTH. When you display the string, you notice that the visual length is not compatible with what is actually displayed: you see control characters being represented by '#', but they are not counted in the visual length. Additionally, when splitting the string, you may experience unexpected misalignments resulting from this behavior.

Example of the issue:

You have a string with a dynamic display length of 10 units that should be split into two substrings, each has display length of 5 units.

Input text: aあ#a12345

Whereas # represents a control character.

Expected result:

FIELD1   aあ#a
FIELD2   12345

Actual result:

FIELD1   aあ#a1
FIELD2   2345

Other Terms

CL_ABAP_LIST_UTILITIES, Control characters, horizontal_tab, tab, dynamic output length, display length, visual length, string spliting based on visual length

Reason and Prerequisites

Control characters are normally non-printable. However, in ABAP List, they are replaced by the displayable '#' character (which should have a visual width of 1). Currently, the kernel module's calculation returns a visual width of 0 for these characters. This leads to incorrect substring splits and misalignment in the structured list output.

Solution

Development is in-progress.
