Symptom

  • Running sp_configure 'o/s file descriptors', the output shows a wrong value for Config Value compared to the Run Value:
    1> sp_configure 'o/s'
2> go
 Parameter Name       Default     Memory Used   Config Value   Run Value    Unit   Type
 -------------------- ----------- ------------- -------------- ------------ ------ ---------
 o/s file descriptors           0           0           8192         100000 number read-only
  • The ASE error log correctly shows it is booting with the Run Value:
    kernel  Using 100000 file descriptors.

Environment

  • SAP Adaptive Server Enterprise (ASE) 16.0
  • SAP Adaptive Server Enterprise (ASE) 16.1

Reproducing the Issue

High level sequence that creates this issue:

  • ASE Server is born with file descriptors = XX, Config Value is 0, Run Value is XX.
  • ASE Server is upgraded to a new version, Config Value is now XX Run Value is XX. 
  • Host reconfigures file descriptors for ZZ file descriptors.
  • After reboot, ASE Server shows Config Value as XX, Run Value is ZZ.

Cause

  • This can happen to an ASE that has undergone an upgrade and later changed the available number of file descriptors since the upgrade.
  • The Config value is updated during upgrade time only, which leads to a stale value when the number of descriptors has changed.
  • The issue has been logged under SAP CR 831766. Its description reads, "sp_configure "o/s file descriptors" can report a different value for Config Value compared to Run Value if the ASE has been upgraded and the Run Value has changed since the last upgrade.".

Resolution

    • There is currently no target SAP ASE Service Pack scheduled for this issue.
    • It has been assigned to our Product Group. 
    • Please mark this KBA as a Favorite to be automatically notified when the CR status is changed.
    • Workaround:
      • Manually update the correct value in sysconfigures.
        *******DISCLAIMER*********
        Direct updates to system tables are inherently dangerous. Take all appropriate precautions (including a master dump) to ensure the system is recoverable.
        The information in this article provides scripts "as is" without warranty of any kind and run at own risk.
        *******DISCLAIMER********* 
        use master
        go
        begin tran
        go
        update sysconfigures set value = <value from Run Value> where config=158
        go
        <if 1 row affected>:
        commit tran
        go
        <if more than 1 row affected>:
        rollback tran
        go
      • Force the upgrade to occur again by running sp_downgrade_esd and reboot.

        1. Start ASE in single user mode.
        2. execute 'sp_downgrade_esd "master","SPxx PLyy" '
          Here, SPxx PLyy is a version older than the current version.
          For example, if the current version is SP04 PL08, it would look like this:
          sp_downgrade_esd "master","SP04 PL07"
        3. Reboot ASE in multi-user mode, which will rerun the upgrade process on the master, and as a result, resync Config Value with the Run Value.
