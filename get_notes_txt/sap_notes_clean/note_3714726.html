Symptom

  • Partitioning of a table containing over 2 billion rows is stuck or fails in SAP HANA.
  • The process halts specifically at "RepartitionDeltaAndMVCC"
  • You encounter the following errors during the process:
    Error: "statement_allocation_limit error"
    Error: "transaction rolled back by lock wait timeout: Lock-wait time out exceeded"
  • The issue persists even after re-execution attempts.

Environment

SAP HANA, platform edition

Reproducing the Issue

  1. Attempt to partition a table with more than 2 billion rows using SAP HANA Studio or SQL command
  2. Monitor the progress and observe the process hanging at the mentioned steps (e.g., RepartitionDeltaAndMVCC)
  3. Check the trace files or console for memory allocation limits or lock wait timeouts

Cause

The partitioning process fails due to two primary technical constraints:

  • Memory Limits: The statement_allocation_limit is reached because the repartitioning of extremely large tables (2 billion+ rows) requires significant memory resources for the delta merge and MVCC (Multi-Version Concurrency Control) updates
  • Resource Contention: Object locks (INTENTIONAL_EXCLUSIVE) prevent the process from completing because other processes or internal locks are not released within the timeout period

Resolution

The following manual table recreation approach is recommended when standard repartitioning fails:

  1. Stop all application access to the HANA database to prevent data changes during the migration
  2. Ensure a full backup of the SAP HANA database is available before proceeding
  3. Create a new, empty table with the identical structure as the original table, but include the desired partition specifications
  4. Copy data from the original table to the new partitioned table using an INSERT ... SELECT statement.
    Note: For tables of this size, perform the copy in chunks/steps to stay within memory limits
  5. Verify that the record count in the new table matches the original table
  6. Rename the original (old) table to a backup name (e.g., TABLE_NAME_OLD)
  7. Rename the new partitioned table to the original table name

Note: Test the process in a non-production environment with a smaller dataset first to ensure familiarity with the steps
