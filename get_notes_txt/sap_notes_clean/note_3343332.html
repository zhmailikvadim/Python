Symptom

 

With SAP S/4HANA release 2023 the newest release of the product SAP S/4HANA Supply Chain for Extended service parts planning (eSPP) is shipped.
Extended Service Parts Planning is available in PCE (Private Cloud Edition).

It allows planning and execution processes for service parts in one single logical system (embedded eSPP). You can also use Extended Service Parts Planning (eSPP) on a separate decentralized SAP S/4HANA system (side-by-side eSPP) with a connection to one or multiple backend systems (ECC systems and/or SAP S/4HANA systems).

The major differences between the previous eSPP release (as part of SAP S/4HANA 2022) and the two deployment modes (embedded and decentralized eSPP) are described below.

From a license perspective, eSPP differs from the Service Parts Planning (SPP) product on SAP SCM because it was set up as a completely new product.
Functionally, both products are comparable, though. Innovations including for example new algorithms in Forecasting, Inventory Optimization enhancements, SAC integration, UI renovations and architectural simplifications are available in eSPP only.

This note will list (major) differences between the two products, as well as new capabilities. Additionally, some general information on the product is given here as well. This note does not intend to explain any functions in detail. Please refer to the online documentation of the SAP S/4HANA system for more details.

All statements in this note refer only to release S/4HANA 2023, not to any other release. In case SAP lifts restrictions described here with a later release, those changes will then be documented with the new release.

Installation of SAP S/4HANA 2023

Solution

Availability
eSPP is available in the PCE (Private Cloud Edition) version of SAP S/4HANA 2023 (Technical component S4COREOP, Release 108).

Integration of eSPP with other components

  • Integration with SAP S/4HANA Logistics master data and transactional data
    eSPP is integrated with SAP S/4HANA master data and can access transactional data within the SAP S/4HANA system in transparent tables or in Live-Cache.
    No integration with external systems is required. It is possible to integrate data (e.g., Material Master data) from external systems via APIs.

  • Instead of gATP, SAP S/4HANA contains advanced ATP (aATP)
    eSPP was adopted to using aATP instead of gATP. Please see below for descriptions of changes this brought for eSPP processes.
    For limitations and differences between gATP and aATP, check the application help of both solutions.

  • Integration with SD
    You can automatically extract the relevant demand from SD tables to build the demand history for eSPP.
    As an option, you can also upload demand from other sources via Excel files or a SOAP service.

  • Integration with SAP Analytics Cloud (SAC)
    As an optional component, you can use SAC to perform analytics in order to provide deep insights in eSPP requirements planning.
    Find more information on stories and models provided here: Architecture and Abstract - SAP Help Portal

  • Integration with the Supply Chain Algorithm Library (SCAL) to perform Multistage Inventory Planning
    As an optional component to perform multi-echelon safety stock optimization you can use the integration of eSPP with the SCAL library.
    This multi-echelon safety stock optimization algorithm is embedded in eSPP as of SAP S/4HANA release 2022. In this scenario, a new planning service that is tightly integrated with other eSPP processes performs the multi-echelon safety stock optimization. 
    To install the SCAL library, follow the instructions described in note 3307389.
    As an alternative, you can use the integration of eSPP with IBP-I.
    Find more documentation on that scenario in note 2940185. As this option results in a higher integration effort and less coupled processes, we strongly recommend using the embedded approach which also has a larger scope of functionality.

  • Integration with PAL (Predictive Analytics Library) procedures (to use additional Forecasting algorithms)
    eSPP uses Forecasting models in PAL procedures, which are not contained in the previous solution SPP.
    To use PAL procedures, you have to
    • Install SAP HANA Platform 2.0 SPS 05
    • Install the Application Function Library (AFL), which includes the PAL.


Conceptual Differences of Decentralized eSPP on SAP S/4HANA compared to embedded eSPP in SAP S/4HANA

Some aspects of the simplified integration of embedded eSPP with other SAP S/4HANA applications of the same system & client, as described below, are different in case of the decentralized eSPP on SAP S/4HANA due to the nature of a two-system landscape. Thus, regarding interfaces the decentralized eSPP on SAP S/4HANA is more similar to the Service Parts Planning (SPP) product on SAP SCM 7.0 (or lower versions thereof) than the embedded eSPP in SAP S/4HANA.

The functional scope of the embedded version of eSPP does not differ from the decentralized eSPP.
There is one exception to this: In the decentralized (side-by-side) eSPP, the ability to link to inbound deliveries in the shortage monitor is not yet available.

There is a technical difference as well: In the decentralized scenario, eSPP reads sales order data from the Live-Cache. In the embedded scenario, eSPP reads sales order data directly from SD tables.

Details on the side-by-side implementation of Extended Service Parts Planning are provided in SAP note 3362987.


Conceptual Differences of embedded eSPP on SAP S/4HANA 2023 compared to SPP based on SCM 7.0 (EHP 4)

Integration with BW:

As a major simplification, eSPP no longer requires an integration with BW.
BW is technically still available internally in an SAP S/4HANA system, however SAP no longer ships content for it.
Therefore, any SPP functionality relying on an internal BW system is either discontinued or is provided on a new technical foundation. See the following sections for details.

Integration via CIF:

SPP on SCM uses the Core Interface (CIF) for exchanging data (both master and operational data) with a connected (SAP) ERP system.
eSPP follows the goal of reducing CIF usage as much as possible. For most ERP objects no CIF integration models are needed anymore.
Instead the data is automatically made available for eSPP (i.e., represented in the LiveCache) if a material/product is flagged as relevant for eSPP usage.

For a few object types this was not possible yet, though. For those objects integration models are still necessary.

In the embedded eSPP, no integration models are needed for these operational objects:

  • Stocks
  • Sales Orders
  • Planned Orders
  • Production Orders
  • Purchasing requisitions and Purchasing orders
  • Schedule lines
  • Inbound deliveries

Instead, you only need to activate the relevant object types in Customizing (SPRO -> Advanced Planning -> Baisc Settings -> Settings for Data Transfer).

For master data objects in the embedded eSPP, the following integration models are not or no longer required:

  • Product
  • Product substitution master data + Customizing
  • Plant
  • Customer
  • Vendor
  • MRP Area
  • BOM

In the embedded eSPP, integration models are still needed for these operational objects:

  • Info record
  • Scheduling agreement
  • Contract

Master data

Material Master respectively Product Master

  • Using a product in eSPP
    Materials that were created in SAP S/4HANA releases prior to 1610 have to be enabled for usage in eSPP in SAP S/4HANA. To do so, run report PRD_SCM_GUID_CONVERSION. For more information, check the report documentation.
    To enable eSPP-specific fields to be shown in SCM product master, run report /SAPAPO/SWITCH_ON_SPP_MD_FLD.
    In order to mark a product as 'to be used within eSPP', it is necessary to maintain the flag 'Extended SPP' in the maintenance transactions (transaction MM01 or MM02) on the tab 'Extended SPP'. Alternatively, this flag can be set in the Fiori App  “Manage Product Master Data”.
    If this flag is set, the product is available for eSPP, and any operational data (like orders) relating to this product will automatically start to integrate with the eSPP data stores, i.e. with LiveCache.
    In order to prevent that the MRP run will plan such products, set the field 'MRP Type' (on tab 'MRP 1') to e.g. 'ND' ('No Planning') or a similarly customized MRP type. Failing to do so will result in both MRP and eSPP planning the procurement of this product, with one always cancelling out the results of the other.
    Please pay attention to the fact that a product in a particular location can either be flagged for usage with eSPP, or for usage with Advanced Planning.
    To deactivate products for Extended SPP, use report /SAPAPO/ESPP_PLSTA_RESET. Note that it only considers products at plants with planning status "Marked for deactivation". You can use transaction MM02 or report /SAPAPO/ESPP_PLSTA_PHASEOUT to mark eSPP products for deactivation. Check the application help of eSPP for further details. Be aware that report /SAPAPO/ESPP_PLSTA_PHASEOUT is only available as of S/4HANA 2023FPS01 or via SAP note.
  • Maintaining Master Data for an eSPP product
    Since release SAP S/4HANA 2021 the leading transaction for maintaining product master data is the Fiori app “Manage Product Master Data”.
    Alternatively, with the exception of most fields on MRP area level, product master data fields can be maintained as well in transactions MM01/MM02. Transaction /SAPAPO/MAT1 is no longer supported for maintaining eSPP-specific fields in the active version. You can still use transaction /SAPAPO/MAT1 to display details of existing products.

Please pay attention to the following hints:

  • The eSPP-specific fields for such location products can now be maintained via the standard mass maintenance transaction MASSD as well. Choose object type "ESPP_PROD"-eSPP product in TA MASSD.
  • For maintaining (eSPP)-related product master data for MRP areas, you can use the existing mass maintenance reports for MRP areas (reports RMMDDIBE and RMMDDIBE02).

Plant/Storage Location respectively Location

  • New locations need to be created in the eSPP master data using report /SAPAPO/CREATE_LOCATION instead of CIFing them from the ERP side.
  • MRP area
    MRP areas are made available for eSPP via the report /SAPAPO/CREATE_LOCATION as well.

Customer and Supplier (Vendor)

  • New customers and suppliers have to be created as Business Partners (location type 1021) with the corresponding role ‘Supplier’ respectively ‘Customer’. For this as well the report /SAPAPO/CREATE_LOCATION is used.

Bill of Material (BOM)

  • BOMs for eSPP are created based on Product Data Structures (PDS) via transaction CURTOADV_CREATE. It is not possible to create a BOM alone, instead a production version must be created as the basis.

Interchangeability / Supersession

  • With release SAP S/4HANA 2020, eSPP switched to using the new Interchangeability solution developed by the aATP team. This new solution is based on a new and completely different data model and brings along a new UI. Therefore, any existing Interchangeability chains need to be created again, using the new solution and its UI. For more details on the new solution, please see its online documentation.
  • Available Substitution types are:
    • 1:1
    • 1:1 Partial (1:N with one set, where Product=Substitute)
    • 1:N (And/Or)
    • N:1 (Or)
  • The new solution does not support the types N:1 (AND) and M:N.
  • You can use transaction /SAPAPO/SPPSUSEXPL and report /SAPAPO/SPP_PRODSUB_EXPL to get an eSPP view on product substitution master data. Since SAP S/4HANA 2023, you can also run consistency checks (e.g., indicating loops/ redundancies in supersessions) on the screen of report /SAPAPO/SPP_PRODSUB_EXPL.
  • Find more details on the new eSPP product substitution in note 2912293.

eSPP planning processes

Capture and Manage Demand (CPD)

Capture and Manage Demand is re-implemented without the usage of an internal BW system. Instead of using BW Extraction for new or changed demands, demand data collection is now provided by new PSM services and reports.

Besides storing the aggregated historical demands on the database, CPD supports with release SAP S/4HANA 2021 a virtual mode. If this mode is switched on, each demand history read access will go to the raw (individual) demands and perform the aggregation and (possibly necessary) realignments on the fly. In this scenario no (or fewer) realignments are necessary.
The demand storage in transparent tables also allows easier maintenance of the demand history with upload and deletion reports. Since the SAP S/4HANA 2023 release, an option to mass adjust raw demand is available as well.

Identifying the stockholding location for an order is performed via the Bill of Distribution (BOD), or via customer specific code in a method prepared for a customer enhancement.
eSPP considers the original "Ship to" location in SD Sales Orders, but also takes into account aATP location substitution to find the "Ship to" location.

In order to realign demand, there's only the (new) PSM Service SPP_DEMAND_ALIGNMENT_SRV needed, which takes care of all relevant recalculations. The new CPD architecture works without the administration of realignment steps.

See notes 2878051 and 2912255 for details on the simplified Capture and Manage Demand process.

With release SAP S/4HANA 2021 a new SOAP interface is provided, which can be used to feed historical demands into the CPD module. A corresponding sending interface is provided with this release as well. You can use this interface to import demand data from an ECC or SAP S/4HANA Execution system into a side-by-side Planning system. Please check note 3362987 for more details.
The same interface has the ability to load demand data from external sources (e.g. customers) into the standalone or side-by-side planning system (primarily to support retail-level planning in upcoming releases). For current information on planned innovations, see roadmaps.sap.com.

 

Forecasting

Since release 2021, the historical forecast is recalculated during the normal forecast execution. Therefore, the planning service for recalculation of forecast in the past is not needed anymore and is deactivated. The service can be activated again, if the flag ‘Use Forecast in the Past’ is set (Customizing: Extended Service Parts Planning --> Forecasting --> General Settings for Forecast).

Forecasting in general supports the same models and algorithms SPP supports with the additions stated in the next section.
Leading Indicator forecast can be used on single parts level (removing the need to create an equipment BOM).

You can use the new ability to upload and visualize any number of external time series and improve forecast quality by considering one or multiple external time series as an additional external predictor/ as additional predictors.
On top of the SPP forecast models, there are the following additional new forecast models in eSPP:

  • ARIMA (autoregressive integrated moving average) model / ARIMA seasonal model
  • ARIMAX (autoregressive integrated moving average with external data) model / ARIMAX seasonal model
    (a machine learning model that takes one or multiple external time series into consideration which you can upload in addition to the historical usage of parts)
  • GBDT (gradient boosting of decision trees) model
    (a machine learning model that takes one or multiple external time series into consideration which you can upload in addition to the historical usage of parts)

Inventory Planning

Inventory Planning in general works the same way as in SPP.
Pricing scales in inventory planning are supported with a new concept.

Release SAP S/4HANA 2022 offers the option to perform an embedded multi-echelon (multilevel) safety stock calculation (which includes the Reorder Point (ROP), the Economic Order Quantity (EOQ) and the Period of Demand (POD) or EOQ Period)), using the algorithm from Integrated Business Planning for Inventory (IBP-I). This can lead to lower safety stocks while retaining the same service level.
In release SAP S/4HANA 2023, users get additional explanation on the composition of the safety stock.

With SAP S/4HANA 2021 the option to perform a multi-echelon safety stock calculation using the corresponding functionality from Integrated Business Planning for Inventory (IBP-I) initially became available. It requires to exchange data between eSPP in SAP S/4HANA and IBP-I via the CPI-DS interface. Check note 2940185 for details on the configuration and data flows of the IBP-I integration.
Note:
SAP recommends to use the "embedded" approach, which contains enhanced development scope items.

Please pay attention to the fact that the multi-stage safety stock calculation cannot be used if the product to be planned has a future BOD assigned in eSPP.

Distribution Requirements Planning (DRP)

The DRP planning algorithm was not changed.
A new option to run DRP without consideration of "DS copy" data was added as part of SAP S/4HANA 2021.

Deployment

Deployment works the same way as in SPP for SCM.
It performs an aATP check in eSPP instead of a gATP check.

Inventory Balancing

Inventory Balancing works the same way as in SPP for SCM.

Analytics

Since release SAP S/4HANA 2020, eSPP can be easily connected to the SAP Analytics Cloud (SAC). Analytics with SAP Analytics Cloud (SAC) can provide deep insights in eSPP requirements planning. The stories and models delivered with eSPP SAC content provide both overview and detailed data of eSPP raw demand and demand history, all eSPP forecasting key figures with their quantities, a comparison of raw demand with forecast quantities, and DRP results.

The displayed key figures are easily comparable and support eSPP planners in choosing effective planning parameters and meeting target service levels.

Please be aware, that SAP only provides the data connection (as a live data connection, not via an export into SAC), while the customer should create any dashboards they require based on this data. The provided stories serve as templates for customer specific stories.

As neither the internal BW system is supported anymore, nor SNC is available, the area of Analytics sees major restrictions.

  • Available are these tools:
    • Inbound Monitor
      Results can be displayed in an external BW or in the Fiori Query Designer using the CDS query C_SRVCPARTSINBOUNDMONITORQ.
    • Shortage Overview/Shortage Monitor
      Functionality is restricted due to the missing CRM and SNC systems. These transactions are now available in Fiori and in the classical SAPGUI (no longer in NWBC).
    • Planners Worklist
      This transaction is now available in Fiori and in the classical SAPGUI (no longer in NWBC).
    • SPP Cockpit
      This transaction is now available in Fiori and on the classical SAPGUI (no longer in NWBC).
    • SPP Alert Monitor
      This transaction is now available in Fiori and on the classical SAPGUI (no longer in NWBC).
      Since the SAP S/4HANA 2023 release, a new option to "snooze" Alerts is available.

Fiori Launchpad

The SAP Fiori Launchpad can be used as a starting point to access the SAP S/4HANA eSPP application.

All the main eSPP transactions including the Planning Service Manager can be called from the Fiori Launchpad. They are displayed with SAP GUI for HTML. The available transactions are based on standard eSPP business roles.

The SAP Fiori Launchpad integration is available since SAP S/4HANA 2020, FPS02, which became available for customers on May 25th, 2021.

  • Supplier Worklist
  • Supplier Delivery Performance Rating (SDPR)

Simulation

  • The Simulation functions and transactions based on Models and Versions are technically available but are subject to major restrictions. Because the ERP side master data in SAP S/4HANA (like the Material Master) is not version dependent, and due to some other differences in the system architecture, a fully working Simulation based on Models and Versions (i.e., without side-effects into the operatively used data) cannot be guaranteed. Though some usage scenarios may work, SAP advises therefore against using this functionality.
  • Simulation in dialog transactions DRP matrix, Deployment, Forecast, and Inventory Planning is still working unchanged. 

OEM-managed inventory (OEMMI)

  • OEM-managed inventory is not available.

Upgrading from older Releases to eSPP on SAP S/4HANA 2023

  • Upgrading from SAP S/4HANA 2021 to SAP S/4HANA 2023 release
    Processing of manual demand adjustments has been modified and enhanced in the 2022 release (and subsequent releases).
    To upgrade demand adjustment data, follow the instructions described in SAP note 3170314.

  • Upgrading from SAP S/4HANA 2020 to SAP S/4HANA 2023 release
    In release SAP S/4HANA 2020, raw demand was stored in table /SAPAPO/RAWDEM.
    Since release SAP S/4HANA 2021, table /SAPAPO/RAWDEM is no longer used. Raw demand is instead stored in table /SAPAPO/RAWDML.

  • Upgrading from SAP S/4HANA 1909 to any later release
    You can productively use eSPP since the SAP S/4HANA 2020 release.
    While most of eSPP´s application and customizing transactions are already part of the SAP S/4HANA 1909 system, the SAP S/4HANA 1909 system was only used for eSPP pilot projects and is not in general released for eSPP.
    There is no licensing material for this release, and there is no path for migration of eSPP data from a 1909 system to a SAP S/4HANA 2020 (or later) system. 
    Therefore, we strongly recommend upgrading to SAP S/4HANA 2020 (or later) first before creating eSPP data in a system.

General Information

If you are upgrading from eSPP in SAP S/4HANA 2020 or from eSPP in SAP S/4HANA 2021, pay attention to the changes described above, and adjust your processes accordingly.

If you are implementing eSPP coming from an SPP on SCM installation, pay attention to the differences described above, and adjust your processes accordingly.

If you are implementing eSPP newly, pay attention to the restrictions described above when designing your processes.

In order to avoid incorrect behavior of eSPP functionality in SAP S/4HANA create the following entry in database table /SAPAPO/PUB_CHK:

  • MANDT: current client
  • ACTIVE: - (Hyphen)

Make sure that this entry is the only entry in the database table. Delete all other entries.

Background information on the purpose of database table /SAPAPO/PUB_CHK is provided in the notes 2609363 and 2643483. The performance optimization referred to there has to be switched off, which is done via the step described above.
