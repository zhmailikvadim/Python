Symptom

In an SAP HANA System Replication landscape, the secondary (DR) system cannot be started independently.

A standalone database startup fails with: FAIL: process hdbindexserver not running

The indexserver stops during startup although no service crash is visible.

Typical indexserver trace entries:

PersistenceLayer startup failed
exception 3020027
retryPageLoadAfterFailedChecksumCheck
retrieved invalid page again
assign failed with persistence startup error

However:

  • the database starts successfully when System Replication is enabled
  • the system operates normally as a secondary site
  • no service termination occurs while replication is active

Environment

SAP HANA DB Version 2.0

Cause

During a standalone startup SAP HANA performs a local database recovery using the data and log volumes of the host.

If the local persistence recovery cannot be completed successfully, the indexserver startup is prevented.

When the system starts as a System Replication secondary, the database state is synchronized with the primary system through log replay. The redo logs received from the primary system allow the secondary system to reconstruct a consistent database state even if the local persistence recovery cannot be completed independently.

Therefore, the database may start successfully in replication mode while a standalone startup fails.

This behavior is consistent with the SAP HANA System Replication architecture, where the secondary system state is derived from the primary system during synchronization and log replay.

  • SAP HANA System Replication FAQ
  • SAP HANA Database Backup & Recovery in a System Replication Landscape

Resolution

Attempting to repair the persistence of the secondary system independently is generally not recommended in a System Replication setup.

To restore an operational state, the secondary system must be re-initialized from the primary system using System Replication.

Rebuild the secondary site using a fresh System Replication initialization:

  1. Ensure the secondary database is fully stopped.
  2. If necessary, remove the existing replication configuration.
  3. Register the secondary system again to the primary system using:
  4. hdbnsutil -sr_register
  5. Start the secondary database and allow the initial data synchronization (data shipping) to complete.
  6. The reinitialization rebuilds the persistence on the secondary system and restores a consistent database state. 

After synchronization, the system operates normally again as a secondary site.
