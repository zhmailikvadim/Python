Symptom

  • Communication between FSM and ECC or S/4 HANA is not functioning as expected.
  • On the FSM side, objects remain blocked/locked for an extended period of time.
  • Additionally, objects that are created or modified in ECC or S/4 HANA are not being synchronized and therefore do not appear in FSM.

Environment

  • SAP Field Service Management (FSM)
  • Proaxia Connector (PACG)
  • ECC, S/4 HANA

Reproducing the Issue

Expected Behavior
Communication between FSM and ECC or S/4 HANA operates correctly. Objects created or modified on either system are successfully synchronized and propagated to the other system via the proaxia connector.

Actual Behavior
On the FSM side, objects remain blocked/locked for an extended period. Additionally, objects created or modified in ECC or S/4 HANA are not synchronized and therefore do not appear in FSM.

Cause

Inbound and/or Outbound IDocs in ECC or S/4 HANA are not being fully processed. As a result, communication between FSM and ECC or S/4 HANA is disrupted.

Resolution

A common initial approach when encountering communication issues between ECC or S/4 HANA and FSM is to restart the Message Broker. However, this action can have significant side effects and should only be performed after careful evaluation.

In many cases, restarting the Message Broker is neither necessary nor effective in resolving the underlying communication issue. Therefore, it is recommended to first analyze the root cause (e.g., IDoc processing status, queue backlogs, or connector errors) before considering a restart.

After a restart of the Message Broker, all outbound IDoc queues with the prefix /PACG* are initially locked. At the same time, a dedicated IDoc queue for Resend mode is created.

Resend mode is designed to reprocess all unconfirmed outbound IDocs from a configurable number of past days, as defined in transaction /PACG/ECM_CPROP ( /PACG/ECM_CPROP - General Properties of FSM Cloud Connector - FSM Cloud Connector - Confluence ).

The start of Resend mode is indicated by the message type /PACG/ECM_RESENDSTART, which is generated for each company configured in the restarted Message Broker. This message type was introduced in SP12 ( PACG 200 SP 12 / S4PACG 200 SP 12 - FSM Cloud Connector - Confluence ).

Once all relevant IDocs have been reprocessed, the Message Broker automatically switches back to standard processing mode. At this point:

  • All /PACG* IDoc queues are unlocked

  • Normal communication between systems resumes

The end of Resend mode is indicated by the message type /PACG/ECM_RESENDEND, which can be monitored in transaction WE02 for each configured FSM company.

For a correctly executed Message Broker restart, the number of /PACG/ECM_RESENDSTART and /PACG/ECM_RESENDEND messages should be equal for each company.

Important Considerations for Message Broker Restart

Restarting the Message Broker during Resend mode causes the entire resend process to restart from the beginning. Therefore, this activity should not be interrupted unless no alternative solution is available.

The same behavior applies to the Fullsync process. If a Fullsync is interrupted due to a Message Broker restart, all Fullsync-relevant objects are resent, including those that were already successfully processed.

For additional details about the Resend process, refer to:  Resend - FSM Cloud Connector - Confluence 

Connection Verification Between ECC or S/4 HANA and FSM

To verify the connectivity between ECC/S/4 and the Message Broker, perform the following steps:

  1. Execute transaction /PACG/ECM_PING_MB.

  2. Default Logical Port configuration:

    • Leave the selection parameter empty.

    • A successful result should display:

      • “The connection between SAP and the Message Broker works properly”

      • “Message Broker engine state 'Started'”

  3. If the result is negative:

    • Run the transaction again, this time specifying the Logical Port.

    • If the result is then positive:

      • Maintain the Logical Port as the default in transaction SOAMANAGER, as described in:
         Outgoing Service (SAP to FSM) - Windows/Linux - FSM Cloud Connector - Confluence 

  4. Multiple Logical Ports scenario:

    • Execute the transaction for each Logical Port available in the selection list.

  5. If the result remains negative in all cases (with and without Logical Port):

    • This indicates that there is no connection between ECC/S/4 and the Message Broker.

    • Further investigation should be performed on the platform hosting the Message Broker (e.g., SAP BTP, Windows server, or Linux server).

Check Outbound (SMQ1) and Inbound (SMQ2) IDoc Queues

If the connection to the Message Broker has been successfully verified but communication between ECC/S/4 and FSM is still not functioning, the issue is likely related to IDoc queue processing.

In most FSM Cloud Connector implementations, the outbound qRFC queue (SMQ1) is used for both outbound and inbound communication. Therefore, analysis should typically begin with transaction SMQ1.

1. Check for Resend Mode Queues

  • Look for queues with the pattern /PACG_ECM_RESEND*.

  • If such a queue exists, the Message Broker is currently in Resend mode and is processing previously unconfirmed IDocs.

Actions:

  • Verify the queue status:

    • If the queue is locked (STOP) → manually activate it.

  • The /PACG_ECM_RESEND* queue should contain only one LUW.

    • If multiple LUWs are present, implement the solution described in SAP Note 3635258 ( 3635258 - No communication between ECC/S4 and FSM after restart of Message Broker - SAP for Me )

2. Analyze Standard /PACG* Queues

If no /PACG_ECM_RESEND* queue is present:

  • Check all /PACG* queues in SMQ1.

  • Refresh the queue list multiple times and observe the number of LUWs.

Healthy behavior:

  • The number of LUWs decreases over time.

  • Queues disappear and reappear dynamically.

Problematic behavior (queues stuck):

  • Queue status is STOP (locked), or

  • No change in LUWs for an extended period (READY or RUNNING), or

  • LUWs continuously increase

Actions:

  • Manually activate the queue.

  • If activation fails:

    • Lock and unlock the queue, then activate again.

3. Handling Problematic LUWs

If queues repeatedly stop or re-lock:

  • Check the status of the LUWs inside the queue.

  • If a LUW has status VBERROR:

    • Remove the LUW from the queue.

    • Reactivate the queue.

Important:
Removing a LUW from the queue does not delete the underlying IDocs.

  • The IDocs can still be found in transaction WE02, typically in status:

    • 64 (ready for dispatch)

    • 63 (ready for processing)

    • 03 (data passed to port OK)

Reprocessing notes:

  • IDocs in status 03 must first be changed to status 30 before reprocessing.

  • Use transaction BD87 to reprocess IDocs.

  • If reprocessing fails, further analysis is required.

Root cause analysis should include:

  • SM13 – Update errors

  • ST22 – ABAP dumps

  • SM58 – tRFC errors

4. Inbound Queue Verification (SMQ2)

If your system is configured to use both outbound and inbound queues (check transaction /PACG/ECM_CPROP):

  • Repeat all the above checks in transaction SMQ2 (inbound queues).

5. Empty Queues Scenario

If all /PACG* queues are empty:

  • The issue is likely not related to outbound processing.

  • The problem most likely lies in inbound processing, such as:

    • Missing IDoc creation

    • Errors in Message Broker delivery

    • Issues on the FSM side or connector layer

Further investigation should then focus on inbound processing logs and the Message Broker environment.

 

No Inbound Communication

If no data is being received from FSM, the issue is likely related to inbound processing.

Check IDoc Reception (WE02)

  • Run transaction WE02 and filter for IDoc types with prefix /PACG*.

  • Focus on IDocs with the following statuses:

    • 53 – Successfully processed

    • 51 – Application error

    • 41 – IDoc ready for application

  • Identify the timestamp of the most recent inbound IDoc.

Interpretation:

    • If no IDocs with these statuses have been received for an extended period, this indicates that inbound communication from FSM is not reaching ECC/S/4.

Analyze Message Broker and Transporter Logs

  • Review logs from:

    • Message Broker

    • Transporter service

  • Focus on the timeframe starting from when the last inbound IDoc was received.

Typical issues to look for:

  • Connection failures between FSM and Message Broker

  • Errors in message delivery or transformation

  • Authentication or authorization issues

  • Transporter service interruptions

In most cases, the root cause of the communication failure can be identified in these logs.

Next Steps

  • After completing the log analysis, evaluate whether a Message Broker restart is necessary.

  • A restart should only be considered if:

    • The root cause has been identified, and

    • Restarting is expected to resolve the issue without causing additional side effects (e.g., triggering Resend mode or Fullsync repetition).

Recommendation:
Avoid restarting the Message Broker without prior analysis, as it may not resolve the issue and can introduce additional processing overhead.
