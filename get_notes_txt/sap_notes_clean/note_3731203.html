Symptom

This article explains the concept of Number of Processing Packages in the SAP IBP Copy Operator, its impact on performance and memory usage, common warning messages, and recommendations for determining the optimal configuration.

Environment

  • Copy Operator

Reproducing the Issue

Use and execute the Copy Operator in IBP

Resolution

1. Overview of Packaged Processing

To optimize memory consumption, the Copy Operator in SAP IBP supports packaged processing, where data is split and processed in smaller chunks.

How it works

  • Data is divided period-wise (e.g., months).
  • The total number of periods is split into N packages, based on the configured parameter “Number of Processing Packages”.
  • Each package is processed sequentially.
  • For Details see Processing of Key Figures in the application help.

Example

If a copy operator processes 12 months, it can be split into 1 to 12 packages:

Number of PackagesBehaviorImpact
1All 12 months processed in a single packageFaster runtime, higher memory usage
12Each month processed individuallyLower memory usage, longer runtime

Key Trade-off

  • Fewer packages → Better performance, higher memory consumption
  • More packages → Lower memory consumption, slower execution

2. Warning Messages Related to Packaged Processing

During execution, the Copy Operator may generate warnings indicating potential inefficiencies or risks.

2.1 Warning: “Please check the settings for packaged execution”

Cause

  • Triggered when a source key figure contains a filter block, especially on PERIODID.

Background

  • If filtering occurs late in processing, the system may still read the entire time horizon before applying the filter.
  • In such cases, splitting into multiple packages may not provide performance benefits.
  • Processing everything in one package could be more efficient.

Important Limitation

  • The system detects the presence of a filter block, but cannot determine when it is applied (early vs. late in processing).

Recommendations

  • Remove the filter block if possible.
  • If not:
    • Test with Number of Processing Packages = 1
    • Compare runtime performance.

2.2 Warning: “Increase the value of parameter NUMBER_OF_PROCESSING_PACKAGES”

Cause

  • Triggered when processed records exceed the threshold defined by:
    • MIN_RECORDS_FOR_MEMORY_WARNING (global configuration parameter)

Interpretation

  • Indicates potential high memory usage, but:
    • Memory consumption depends on multiple factors:
      • Data volume
      • Key figure complexity
      • Calculation logic

Recommendations

  • Analyze actual memory usage using the Monitor System Tasks app.
  • If memory usage is acceptable:
    • Consider increasing MIN_RECORDS_FOR_MEMORY_WARNING to suppress unnecessary warnings.
  • If memory usage is high:
    • Increase the Number of Processing Packages.

3. Determining the Optimal Number of Processing Packages

There is no universal setting—the optimal value depends on data volume and usage scenario.

General Guidelines

Low Data Volume (e.g., interactive use)

  • Example: Excel-triggered planning runs
  • Recommendation:
    • Use fewer packages (1–2)
    • Benefit from faster execution

High Data Volume (e.g., batch jobs)

  • Example: Nightly planning runs
  • Recommendation:
    • Use more packages
    • Prevent memory bottlenecks

Performance vs. Memory Trade-off

ScenarioRecommended Setting
Fast execution requiredFewer packages
Memory constraintsMore packages
Complex key figure logic (If no Filter block on PERIODID exists)More packages
Simple calculationsFewer packages

4. Monitoring and Analysis

To make informed decisions:

  • Use the Monitor System Tasks app to:
    • Track memory usage
    • Analyze runtime behavior
    • Identify bottlenecks in disaggregation and copy operations

  • Use the Execution History of the copy Operator to see 
    • Memory usage Information
    • Runtime information

5. Best Practices

  • Start with a balanced configuration (e.g., 3–6 packages).
  • Adjust based on:
    • Runtime performance
    • Memory consumption

  • For small jobs, use package size one. This applies especially for jobs that are scheduled from Excel during the day for ad-hoc copy operations

  • If a source key figures has a filter block on any PERIODID, use package size one.
