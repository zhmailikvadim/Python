Symptom

  • After a system refresh or copy, certain Fiori Catalogs or Tiles are missing.
  • The administrator needs a quick, precise way to identify which specific Transport Requests (TRs) contain the missing content without importing the entire transport queue.

Environment

S/4HANA 2023 and other versions

Cause

The requirement to manually identify and surgically transport Fiori objects after a system copy or refresh typically stems from three primary architectural factors:

  • 1. Metadata Fragmentation in the SUI (Software User Interface) Framework
    In S/4HANA 2023 and higher, Fiori catalog metadata is no longer stored in a single table. It is fragmented across the SUI (Software User Interface) framework, specifically:
    • SUI_TM_MM_CAT (Customizing/Header Data)
    • SUI_TM_MM_APP (Workbench/Tile & Descriptor Data)
    • Standard client copy profiles (such as SAP_CUST) may successfully move the "Header" (Type C) but skip the "Content" (Type W/A), leading to "empty shell" catalogs in the target system.
    • Note: Always query the physical SUI tables in SE16 to avoid stale data from the application cache.
  • 2. Transport Buffer Complexity ("The Shotgun Risk")
    Following a Homogeneous System Copy, the target system's transport buffer may contain hundreds of legacy requests. Without a surgical identification workflow, administrators often attempt to "Import All" to resolve missing data. This introduces the risk of Version Regression, where older technical settings from the source system overwrite stabilized, newer configurations in the target environment.
  • 3. Delivery Class Mismatches
    Fiori artifacts do not share a uniform Delivery Class. While some tables are classified as Customizing, others (like those containing App Descriptors) are treated as Workbench or Local data. Standard Post-Copy Automation (PCA) or SCC1N tools may bypass these "Type W" or "Type L" objects silently, leaving the database inconsistent.

Resolution

Disclaimer & Scope of Workflow: The methods outlined below are provided as strategic guidelines for SAP Basis administrators. Due to the evolving nature of the S/4HANA architecture—specifically the transition to the SUI (Software User Interface) metadata framework in version 2023 and beyond—the physical storage layer and object dependencies are subject to change.

These techniques should be treated as a flexible toolkit. Users must account for their specific landscape versions, client-specific constraints, and the ongoing evolution of SAP metadata persistence. What constitutes a "Standard" table or "Object Type" today may be superseded by future SAP Innovation or Feature Pack Stacks (FPS). Always validate these steps in a non-production test client prior to execution in a Development or Quality environment.

Method A: The "Object-to-TR" Reverse Lookup (Transaction SE03)

This is the gold standard for mapping metadata to its delivery vehicle.

  1. Execute Transaction: SE03.
  2. Navigate to: Search for Objects in Requests/Tasks.
  3. Object Selection:
    • PGMID: R3TR
    • Type: UIAC (Catalogs) or UIAD (App Descriptors/Tiles).
    • Object Name: ZBC* (Wildcards are recommended for broader discovery).
  4. Selection Limits: Ensure both "Modifiable" and "Released" are checked to capture the full lifecycle of the object.
  5. Result: The system returns all Transport Requests (TRs) containing these specific Fiori artifacts.

Method B: The Cross-System Table Comparison (Transaction SCMP)

Best used for identifying "Data Gaps" where headers exist but content does not.

  1. Execute Transaction: SCMP in the Target system.
  2. Target Table: Use SUI_TM_MM_CAT (SUI: Software User Interface - Task Manager Master Metadata for Catalogs) for 2023+ or /UI2/C_CAT (Legacy).
  3. Remote Comparison: Enter the RFC Destination for the Source system (e.g., Sandbox).
  4. Interpretation: * Entries marked "L" (Left): Exist only in the Source system.
    • Next Step: Extract these Catalog IDs and proceed to Method A to find the missing Transport.

Method C: Fiori Launchpad Content Manager (Transaction /UI2/FLPCM_CUST)

The "Surgical" UI tool for identifying the parent TR of a specific UI object.

  1. Execute Transaction: /UI2/FLPCM_CUST.
  2. Locate Catalog: Search for the missing or "empty shell" catalog ID.
  3. Audit Transport: Select the catalog and navigate to the Transport Research or Object Directory Entry tab at the bottom.
  4. Identify: The tool identifies the last TR used to persist that specific catalog's metadata within the SUI (Software User Interface) framework.
