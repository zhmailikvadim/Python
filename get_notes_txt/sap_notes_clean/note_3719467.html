Symptom

If remuneration is recalled from the period subject to SP in the previous year after the end of the SP obligation, a negative inflow of the remuneration subject to SP occurs in the period after the end of the SP obligation. In contrast to positive inflows, contributions and contributions are calculated for these negative inflows. 
To calculate these contribution amounts or contributions in the inflow month, an artificial SP liability is generated for this month in the payroll program. The calculated recovery amounts are assigned to the last payroll period with SP liability when the notification is created using the program RPCZVMD2.

This procedure is not always desired. If a supplementary pension institute stipulates that recalls are not taken into account in the same way as arrears payments after the end of the SP obligation, this requirement cannot yet be implemented in the SAP system.

Solution

This SAP Note adds the new field No Recovery After End of SP Liability in the view Payroll Control (V_T5DPBSZVPY).

If the procedure described in the symptom is not desired for a supplementary pension employer, use the Customizing activity Maintain Payroll Control  to select the No Reclamation After End of SP Liability field. Negative inflows of the remuneration subject to SP are therefore not taken into account after the end of the SP liability in the same way as positive inflows.

To access the IMG activity Maintain Payroll Control, choose the following path:
Payroll: Germany -> Industry -> Public Services -> Supplementary Pension -> SP Notification Program 2006 -> Payroll -> Maintain Payroll Control.

Depending on whether you want to make this change for all SP employers in your system or only for individual SP employers, delimit the existing entry to the required start date of the change or create a new entry and assign it to the relevant SP employers in the view Employer-Specific Settings (V_T5DPBZSZVAG). The validity of the table entry in the inflow month is decisive for the changed behavior. The setting has no effect on the origin of the outflow wage types in the month of origin of the recovery.

If you only want to make the change in individual cases, this is possible using the technical wage type /2F4. If wage type /2F4 exists in table RT when function DOZV ZFL is called, negative inflows of the remuneration subject to SP are not taken into account in the inflow month. This wage type can also be used to correct previous payroll periods in which contribution amounts and contributions to negative inflows were calculated. The correction in payroll then also leads to the correction of SP notifications that have already been created.

Activities to be performed by you

If you want to correct individual cases by entering a wage type, you require a customer-specific primary wage type that can be entered in the infotype Additional Payments (0015) and stores the wage type /2F4 in the RT. You can use model wage type O380 as a copy template. Instead of the statement wage type /2F0, enter the wage type /2F4  in the table view V_512W_O.
