Symptom

  • When simulations and executions are run in the Migrate Your Data Fiori app, jobs in SM37 are generated with the Job owner as the ID maintained in SJOBREPO_STEPUSER.
  • The job steps are executed under the ID of the dialog user who initiated the simulation/migration.
  • The user is seeking a setting or configuration to define a different ID for the job steps.

Environment

  • S/4HANA Migration Cockpit
  • Migrate your data Fiori app
  • S/4HANA private cloud and on premise

Reproducing the Issue

  1. Run simulations and executions in the Migrate Your Data Fiori app.
  2. Observe that jobs in SM37 are generated with the Job owner as the ID maintained in SJOBREPO_STEPUSER.
  3. Note that the job steps such as /LTB/MC_LOAD_PREPARE and DMC_MT_STARTER_BATCH are executed under the ID of the dialog user who initiated the simulation/migration.

Cause

The issue arises because the load jobs in the Migrate Your Data Fiori app are executed under the ID of the dialog user who initiates the activity, rather than the step user. This behavior is intentional to ensure that specific migration objects requiring particular authorizations are executed securely. Allowing the step user, which has broader authority, to execute the job could bypass critical authorization checks and lead to security vulnerabilities.

Resolution

  1. The current behavior aligns with the standard system response and the migration cockpit job framework.
  2. This behavior cannot be modified or adjusted through configuration settings.
  3. It is recommended to use a specific business user with correct authority to run the load job.
