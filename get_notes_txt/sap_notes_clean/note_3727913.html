Symptom

Whenever a revenue contract is created—whether from subscription billing, service contracts, customer projects or sales order, revenue contracts and performance obligations (POBs) as business objects have their own attributes, such as:

  • Revenue Contract Description
  • POB name
  • POB Type
  • Fulfil Type: Time‑based or event‑based recognition
  • Defer method when fulfill type is time-based.
  • Start and end dates
  • Standard selling price

These attributes aren’t hardcoded. Instead, they’re determined through Decision tables in BRF+ Rules, giving customers flexibility and transparency

Other Terms

BRF+, POB Attributes, URR

Reason and Prerequisites

You need to use BRF+ Rules to determine attributes of the POB

Solution

1. Key Configuration Steps

Accessing the CBRR Setup

Configuration begins in Manage Your Solution, under:

Finance → Revenue and Cost Accounting → Contract-Based Revenue Recognition

Here, you maintain the configuration steps: Contract Attribute Business Rules (ID 105700). For standard URR, you don’t need to maintain account determination rules—those are only relevant for third‑party integrations.

Important: in your browser settings, you must allow pop-up as the system will open a new tab from the current screen.

Once you start the configuration, a BRF+ application ZFARR_AP_INB_PROCESS_TEMPLATE is created.

 2. What You Configure as rules based BRF+

When you click Configure, SAP opens the Contract Attribute Rules in a new browser tab (so pop‑ups must be enabled).

In URR, only the following rules are relevant.

  • Contract Attributes
  • POB attributes
  • Standalone Selling Price

Contract Attributes

It defines description for the revenue contract.  It is currently not possible to define additional contract category.

POB Attributes

This is the most important part. SAP delivers a template decision table where you define how POBs behave depending on the integration scenario.

Standalone Selling Price

SAP recommends using pricing procedures in sales order, service management and subscription billing. If you cannot define the SSP using the pricing condition, you can optionally define a rule for SSP here

3. Technical Notes 

Important: System evaluates the rules according to the sequence of the rules in the table. And once a matching rule is found, it will return the result and stop looking for further matching rules.  Therefore please make sure that a more specific rule shall be always placed before a more generic rule. For example, you have 2 lines in the table:

#

Company code (as condition)

POB Name (as output)

1

Empty

POB General

2

1000

POB 1000

If you want to give a POB name as POB 1000 when company code is 1000, as the first line has more generic rule (“company code = empty” means that for any company code, the condition is met), it will always return POB General, therefore you need to change the sequence of the above 2 rules.

BRF+ rules are transportable objects, so they must be included in a transport request.

  • You can view and manage these rules in the Simplified BRF+ Workbench under Extensibility based on role SAP_BR_EXTENSIBILITY_SPEC.
  • You may need specific authorizations to display or edit BRF+ objects. 

4. POB Attribute Rules — The Most Critical Part

SAP delivers a template decision table for POB attributes. In the sample template, you will find the following sample rules, and it can be directly used.

  • Subscription Billing
    • POB name = Product ID
    • Fulfill Type = Always time‑based
    • Default defer method = 1
  • Service Contracts
    • POB name = Product ID
    • Fulfill Type = Always time‑based
    • Default defer method = 1
  • Sales Orders
    • POB name = Material number
    • Fulfill type
      • For Standard Item(TAD):→ event‑based
      • For Customer Project with Fixed Price->Event-based
      • For Customer Project with Periodic Service->Time-based

Important: Event types are automatically controlled by the reference key in Universal Journal scenarios, so no manual configuration is needed.

List of Relevant Fields as output

  • POB Name

This field must be filled. And the template has provided default POB Name as follows:

  • Subscription Billing: Product ID
  • Service Contract: Product ID
  • Sales Order and Customer Project:  Material Number
  • POB Type

You can assign the POB Type which you already define in configuration steps “Define Performance Obligation Types” (103886).

  • Fulfill Type

You must specify the fulfill type either as ‘E(Event-based)’ or T(Time-based)’.  Fulfill type ‘O(PoC-based)’ is not yet supported.

Depending on the integration scenarios, the following fulfill type are allowed

  • Subscription Billing: always time-based
  • Service Contract:  always time-based
  • Standard Item in Sales Order:  always time-based
  • Sales Orders from Project: 
    • Fixed-price items: always event-based
    • Service items: always time-based
  • Deferral Method

This is relevant if the fulfill type is T(Time-based)’. If you don’t define a value in the BRF+ rule, system will default to method 1.

You can override this to methods 2, 3, or 4 depending on your revenue policy.

  • Event Type

For universal revenue recognition, event type is not configured manually — it is derived automatically from the Revenue Recognition Key which is defined in Event-Based Revenue Recognition.

  • Residual Method

Determines how residual allocation is handled.  You can set the ‘X’ if the POB uses residual allocation method.

  • Exclude from Allocation

Defines whether the POB participates in price allocation or. You can set an ‘X’.

  • Start and End Date Logic

Usually you don’t need to define the start date or end date, if the value is empty, then it will take the value from the relevant object per default

  • Subscription Billing: It is the start date and end date of the billing plan
  • Service Contract: It is the start date and end date of the service contract item.
  • Periodic Service of Customer Project: It is the start date and end date defined in the Project 

You can optionally define them here in the BRF+ rule with more complex logic using formular feature of the BRF+.

  • Start Date type

The start date type is always fixed to ‘1’ for time-based POB. If you configure other values,

  • Invoice Effective Type

Controls whether how and when invoiced amount and credits change the contractual price.

Important:  Once the POB is created and the values are derived for the POB, it cannot be changed anymore for this POB!

This is essential for handling cancellations and credit memos.

List of Irrelevant Fields

The following fields are not yet relevant for POBs in URR. Please don’t configure any value for those fields

  • Unit Distinct
  • Duration
  • No Cost Recognition
  • Value-Based Fulfillments

5.Standard Selling Price (SSP) Rules

If you cannot define the SSP using the pricing condition, you can optionally define a rule for SSP here.

You can define SSP directly in BRF+, but it is very limited for example

  • You cannot define a validity of the SSP. 
  • You cannot define SSP based on other pricing information. 
  • You cannot define scaled SSP

SAP recommends using pricing procedures in sales order, service management and subscription billing.

You may use the SSP definition here for simply use case, for example, if there is no SSP pricing condition define, then the SSP is defined as 0 value here.

Important: A POB must have a SSP unless it is defined as ‘Excluded from Allocation’ or ‘Residual Method’!
