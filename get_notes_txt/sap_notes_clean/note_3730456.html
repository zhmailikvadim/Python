Symptom

After an account customizing change, when running the payroll posting process for a period with retroactive calculations to earlier periods, the report RPCIPE00/RPCIPE01 displays the following error:
  • Account with the following search key was not found: **********
    Message no. 3G203
  • Internal error when defining the account for search key **********
    Message no. 3G373
  • Required field GL_ACCOUNT was not transferred in parameter ACCOUNTGL
    Message no. RW602

Environment

  • HR renewal
  • SAP S/4HANA
  • SAP ERP
  • SAP SuccessFactors Employee Central Payroll

Reproducing the Issue

  1. Run the payroll posting including a determined wage type (****)  for a specific period (A)
  2. Adjust the symbolic account assigned to that wage type (****)  in the customizing.
  3. Execute a retroactive payroll run in the current period (B) that goes back to that period (A).
  4. Run the posting for the current period (B).
  5. Observe the error displayed in report RPCIPE00/RPCIPE01.

* You might be able to run the posting without errors for periods in which there is no retroactivity or in which there is no retroactivity to periods that were posted before the customizing change involving the affected wage type (period A).

Cause

  • In case of retroactive calculation, the previous posting runs from table PPOIX is read.
  • The system detects that the wage type was originally posted using symbolic account ****.
  • Since symbolic account **** no longer has a valid account assignment, the system identifies a customizing inconsistency.
  • This triggers the error.

Resolution

  1. Review and adjust your customizing as per the KBA 1869773 - Error 3G203 Account with the following search key was not found.
  2. Refer to SAP note 119865 - Q&A concerning posting from payroll in Release 4.x in case of account customizing changes:

    Alternative for reversal in case of incorrect Customizing:

                  If a document was posted incorrectly, most errors can be solved by retroactively correcting the incorrect Customizing (assignment "Wage type <-> Symbolic account") and performing forced retroactive accounting for the affected employees to the following period.
    Note: This is possible only together with adjustment postings which are often difficult to determine if the errors affect payroll results from retroactive accountings whose original posting was performed before Release 4.0 and which have not yet been evaluated productively in Release 4.0.
    In general, you should not change the assignment "Symbolic account <-> Account in Accounting". You may have to define new symbolic accounts, if necessary. Old assignments "Symbolic account <-> Account in Accounting" are still required for retroactive accountings and the reversal of the original payroll result entry required in this process.
