Symptom

In SAP MII, you have developed transactions that you have previously called using the MII destination system in SAP Plant Connectivity. However, the MII destination system is marked as obsolete. You have therefore replaced the MII destination system with a REST Web service call via the universal Web service destination system in accordance with the documentation in the application help. You fill the inbound parameter of the Web service from the special output expression ~NotificationOutputInXmlFormat.

However, at runtime, the transaction call fails. In the log of the agent instance whose notification you want to call the Web service, the system issues the following error message: ArgumentException: Cannot have ']]>' inside an XML CDATA block.

Other Terms

MII transaction, XML, GET, POST, REST

Reason and Prerequisites

Restrictions of the Web service protocol used.

The documentation describes the REST service call as a GET method. The parameters of the Web service are written to the URL at runtime. For large notifications, the size limit for the URL can be exceeded.

Solution

For the Web service call in the universal Web service destination system, use the POST method instead of the GET method proposed in the documentation. To call MII transactions using a Web service, proceed as follows:

  • When creating the universal Web service destination system, select the description type No Service Description.

  • In the Endpoint URI field, insert the URI as follows:

    https://<server name>:<port>/XMII/Runner

  • On the Security Settings tab, select the appropriate authentication parameters.

  • On the Operation Configuration -> Request Message Configuration tab page, change the method from GET to POST.

    IsAsync=false&Persistence=NEVER&Transaction=<folder path>/<transaction name>&inXML={inXML}&Session=false&content-type=application/x-www-form-urlencoded

    Here, the assumption was made that the inbound parameter of the transaction is called inXML and is filled from the Web service input variable with the same name. Names in curly brackets are interpreted as input variables of the Web service and can be filled dynamically at runtime. If, for example, you do not want to determine the transaction name until runtime, you must write ...&Transaction={inTransaction}&... in the template. The system then generates the inTransactioninput variable, which you must fill with the transaction name at runtime.
  • On the Operation Configuration -> Extended Configuration tab page, in the Request Settings or Response settings screen area, change the MIME format to xml and the content type to application/x-www-form-urlencoded.

After these changes, the parameters become part of the user data of the Web service call. As a result, they are no longer subject to the size restrictions for the URL.

It is possible that the service call in MII creates an error message in the system log because the content type application/x-www-form-urlencoded is not supported. In this case, you must allow this content type using the following command:

https://<server>:<port>/XMII/Illuminator?Service=Admin&Mode=AddContentType&ResourceContentType=application/x-www-form-urlencoded

If you run multiple MII servers in a cluster with load balancing, this command must be executed for each server in the cluster. If the error message still appears in the system log, clear the cache for the content types:

https://<server>:<port>/XMII/Illuminator?Service=Admin&Mode=ResetContentTypeCache
