Symptom

In anticipation of the upcoming migration to common super domain, to mitigate potential issues by major browsers’ deprecation of 3rd party cookies (refer to help and blog for more details), starting from 2H 2023 release, integration between SuccessFactors HCM Suite (aka BizX) and SAP Identity Authentication Services (IAS) will be using settings with IAS based on the common super domain (either cloud.sap or sapcloud.cn).
This would happen when the integration between your SuccessFactors HCM tenant and IAS tenant is established via either of the methods below after 2H 2023 release:

  • As a result of running the Initiate IAS Upgrade or Change IAS tasks in Upgrade Center of SuccessFactors HCM Suite
  • As a result of new SuccessFactors HCM tenant provisioning with IAS/IPS preconfigured

Environment

SAP SuccessFactors HCM Suite

Reproducing the Issue

SSO login failing for the customer using IAS as proxy with corporate IDP setup

Resolution

To find out whether IAS is configured on common super domain or old ondemand.com domain for the SuccessFactors tenant: 

Option 1: Check SuccessFactors tenant configuration in IAS 

  1. Log into IAS Admin console;

  2. Click on Applications;

  3. Locate the entry for the BizX tenant under Bundled Applications e.g. CQARCMRMK;

  4. Under the Trust tab, click on SAML 2.0 Configuration link;

  5. Review the entries in Assertion Consumer Services endpoint, and Single Logout Endpoint, you should notice that the entry would be using IAS domain ending in cloud.sap:

 

This would indicate that the SuccessFactors integration to IAS is using IAS under common super domain.  

Option 2: Check IAS configuration in SuccessFactors:  

  1. Log into provisioning
  2. Click on Single Sign On (SSO) Settings link 
  3. From the SAML Assertion Parties (IdP) drop down list, select the entry for the IAS currently integrated with this BizX tenant (indicated by Enable SAML Flag param with value of Enabled),
  4. Under ISP-Initiate Logout section, check the value for Global Logout Service URL (Logout Request destination) would be similar to:  https://xxxxx.accounts400.cloud.sap/saml2/idp/slo/xxxxx.accounts400.ondemand.com 
  5. Under IDP-Initiate Global Logout section, the value for Global Logout Service URL (Logout Response destination) would be similar to:  https://xxxxx.accounts400.cloud.sap/saml2/idp/slo/xxxxx.accounts400.ondemand.com 
  6. The value for Single Sign-On Redirect Service Location (To be provided by IDP) would be similar to:  https://xxxx.accounts400.cloud.sap/saml2/idp/sso/xxxxx.accounts400.ondemand.com 

In summary, the first part of the URLs should be in the format of ias tenantname.accounts.cloud.sap, which would indicate that the SuccessFactors integration to IAS is using IAS under common super domain.

Corresponding configurations with IAS under common super domain:

If IAS is a proxy to a corporate Identity Provider (IdP), customer will need to add a new Assertion Consumer Service endpoint for IAS within the corporate IdP:

  • If you are using a corporate identity provider such as Microsoft Azure AD, Google, Okta, etc, with IAS acting as proxy to the corporate identity provider, you likely will also need to make changes to the setting related to IAS on the corporate identity provider. Configurations are different for different IDPs, please contact the admin of the corporate IDPs in your organization and/or SIs experienced in the corresponding IDPs to identify, plan and perform the changes needed. Below is an example of changes made on Microsoft Azure AD:  

    • Configuration of additional Reply URL to reflect the common super domain supported by IAS 

    • Follow the steps described in the tutorial to integrate Microsoft Azure AD with IAS, step 5.b, add an additional Reply URL that should contact IAS ACS endpoint based on the new SAP common super domain (extracted from the corresponding SAML metadata), so, at the end the configuration should have 2 URLs as Reply URL – one with the ondemand.com domain and one with the new cloud.sap or sapcloud.cn as shown on the screenshot below. 

 

The additional URL value to be added can be found in the IAS metadata, retrievable via IAS metadata API call, for example: https://best-run.accounts.cloud.sap/saml2/metadata/, then locate the parameter “AssertionConsumerService”, the URL is the location value of this param, e.g: 

IAS supports both the new cloud.sap domain and the old ondemand.com domain at the same time, so you can retrieve the metadata using the API for both domains, to review, compare the metadata, e.g.  

  • https://best-run.accounts.cloud.sap/saml2/metadata/ 
  • https://best-run.accounts.ondemand.com/saml2/metadata/ 

After you add the additional Assertion Consumer Service endpoint in your corporate IDP, you can test to ensure that it works with IAS with the following steps:  

  1. Navigate to your IAS under the new domain, such as https://best-run.accounts.cloud.sap/ - It will load the IAS login page to Profile Management;
  2. Login with a username that is registered on the corporate IDP;
  3. You should be redirected to the corp IDP enter the username and password for login;
  4. After successful login on corp IDP, you should be redirected back to IAS’s user profile page. This indicates that corp IDP is now integrated with IAS on cloud.sap. 

Construct IDP-Initiated Login URLs for SuccessFactors 

  1. When IAS is the real IDP, the IDP initiated login URL should be in the format of: https://<tenant_ID>.accounts.cloud.sap/saml2/idp/sso?sp=<sp_name>[&RelayState=<sp_specific_value>&index=<index_number>] or https://<tenant_ID>.accounts.sapcloud.cn/saml2/idp/sso?sp=<sp_name>[&RelayState=<sp_specific_value>&index=<index_number>]

    Help Doc here

    To see the pattern of IDP initiated login URLs under common super domain, you can log into IAS admin console using the new domain, for example: https://best-run.accounts.cloud.sap/admin, then navigate to Applications & Resources ==> Tenet Settings ==> Single Sign-On, under IdP-Initiated SSO row
