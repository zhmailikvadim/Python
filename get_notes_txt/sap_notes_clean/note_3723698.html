Symptom

CPU spikes and long commit times

You see very long running calls in Local Secure Store (LSS) that call into the external key management system (KMS)

lss_ANL0123456789.<SID>.nnn.trc

i RequestHandler RequestHandler.cpp(00055) : reportBackupCompleted() took 108288920 us

From LSS protection traces you can see that the delay is coming from the external Data Custodian (DC) KMS service:

lssprotection_ANL023456789.<SID>.nnn.trc

01:29:48.440691 i Protection ProtectionBase.cpp(00084) : Running lssprotection, action=decrypt
# call to DC KMS
01:29:48.440714 i Protection lssprotectionDataCustodianKMS.cpp(00033) : decrypt()
# wait 54 seconds for DC KMS response
01:30:42.779244 i Protection lssprotectionDataCustodianKMS.cpp(00071) : Parsed properties from decrypt endpoint

You will find information about LSS runtime dumps being created for the long lasing DC KMS response

lss_ANL0123456789.<SID>.nnn.trc

01:30:42.767546 i RuntimeDumpTrigg RuntimeDumpGuard.cpp(00052) : Creating runtime dump

You will find the created LSS runtime dump in the trace folder

lss_ANL0123456789.<SID>.rtedump.yyyymmdd-013042.767555

Other Terms

 DC KMS, CPU spikes, LSS

Reason and Prerequisites

Issue number 348099

Reason:

Due to a programming error in SAP HANA, the exchange of keys between coordinator and worker hosts is slow when the validation call to the external key management system itself is slow.

Affected Releases:

  • SAP HANA 2:
    • Revisions of SPS04 - SPS06
    • Revisions <= 079.08 (SPS07)
    • Revisions <= 089.01 (SPS08)

Prerequisites:

  • You are using the Local Secure Store (LSS), in combination with an external key management system (KMS) on a multi host scenario. Single hosts are not affected.

Solution

Apply one of the following SAP HANA Revisions:

  • SAP HANA 2:
    • Revisions >= 079.09 (SPS07)
    • Revisions of SPS08: SAP is currently working on a permanent solution to this issue. This SAP Note will be updated as soon as a revision containing the solution is available
    • or higher

  • or higher

Workaround:

Disable the external key management system

  1. Log on to the coordinator node as <sid>crypt user on OS level.

  2. Reset the active key management configuration to disable external key management for LSS:

    lsscfg -t <tenant_name> activateKeyManagementConfiguration DEFAULT
