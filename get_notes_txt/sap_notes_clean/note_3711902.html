Symptom

You are using an SAP product based on SAP NetWeaver ABAP on IBM i. During a system upgrade or SP update using Software Update Manager (SUM), you encounter a problem in phase PREP_INSTALL/SHDINST_OS. The shadow instance cannot be created. SHDINST_OS.LOG shows the following error:

1 ETQ200 Executing actual phase 'PREP_INSTALL/SHDINST_OS'.
3 ETQ399 Executing command: CALL PGM(SAP<sid>UPG/RUNLIBCMD) PARM('SAP<sid>UPG' 'CRTR3INST SID(<sid>) INST(20) ROLE(*SHADOW) PROFILE(''/usr/sap/<sid>/SUM/abap/system/<sid>/SYS').
1EETQ399 Last error code set is: SHADOW Instance could not be created. Check joblog for details.

In the job log you see entries like the following:

The user profile <sid>adm has been changed.
The value for the SUPGRPPRF parameter is not correct.
CRTR3INST: Error updating user <sid>OFR
CRTR3INST: Failed to create instance <nn> for system <sid>

User profile <sid>ADM has been changed to USRCLS(*PGMR) INLMNU(*SIGNOFF) LMTCPB(*YES).

Other Terms

IBM i, AS/400, CRTSAPUSR, iSeries

Reason and Prerequisites

The problem occurs only if the following conditions are met:

  • The SAP system is configured to use the new user concept (see SAP Note 1123501), but user profile <sid>OFR still exists.
  • The system update or upgrade procedure using SUM creates a shadow instance.
  • The upgrade kernel (in library SAP<sid>UPG) has a lower patch level than the kernel in the source system. OR: SAP Host Agent 7.22 patch level 69 or higher is installed on the system.
  • The upgrade kernel (in library SAP<sid>UPG) suffers from the problem described in SAP Note 3701638 (aka. SAP Note 3628734, TOP 8). The upgrade kernel is lower than the one mentioned in the Support Packages & Patches section of SAP Note 3701638, but greater than the one in SAP Note 3627373:
    • Kernel 7.54: greater than PL 600, but lower than PL 633 
    • Kernel 7.53: greater than PL 1500, but lower than PL 1535
    • Kernel 7.22: greater than PL 1500, but lower than PL 1514

If all of the above is true, SUM fails to create the shadow instance. This happens, because SUM runs as user <sid>ADM and calls SAPINLPGM from the upgrade kernel. Due to the problem described in SAP Note 3628734, TOP 8, this disables the current user, which is <sid>ADM.

The problem cannot be worked around by re-enabling <sid>ADM as long as user profile <sid>OFR still exist.

Solution

Select an upgrade kernel which includes the ILE patch for your kernel version mentioned in SAP Note 3701638:

  • Kernel 7.54: ILE.SAR PL 633
  • Kernel 7.53: ILE.SAR PL 1535
  • Kernel 7.22: ILE.SAR PL 1514

If you haven't started the update procedure yet, add the ILE patch matching your upgrade kernel version mentioned in SAP Note 3701638 to the download directory. Once started, SUM will automatically recognize the patch and add it to the upgrade kernel.

If you have started the update already, we recommend to reset SUM to add the patch.

If resetting SUM is not an option, you have two options to work around the problem:

  • Workaround 1: Follow SAP Note 177123 to manually update the upgrade kernel.
  • Workaround 2: Delete user profile <sid>OFR. (Since running SAPINLPGM will not disable the current user profile if <sid>OFR does not exist anymore.) Then enable user <sid>ADM again perform the following two commands:
    • CHGOBJOWN OBJ(<sid>ADM) OBJTYPE(*USRPRF) NEWOWN(<sid>ADM) CUROWNAUT(*REVOKE)
    • CHGUSRPRF USRPRF(<sid>ADM) INLPGM(SAP<sid>IND/SAPINLPGM) LMTCPB(*NO)

Workaround 1 (Manually update upgrade kernel)

Follow SAP Note 1177123 - "IBM i: Reconstructing an ILE kernel library from SAR archive" to extract the ILE patch for your upgrade kernel version to library SAP_TOOLS.

Replace the following objects in SAP<sid>UPG by the objects in SAP_TOOLS:

  • CRTSAPUSR *CMD
  • CRTSAPUSR *PGM

Workaround 2 (Delete <sid>OFR):

If the SAP system has been switched from to old to new user concept according to SAP Note 1123501, user profile <sid>OFR might still exist, although it is technically not needed anymore to run SAP. If there are no other reasons to keep it, you can delete it to work around the problem described above.

Such reasons might be (amongst others) that the user still owns objects. Use the following command to check this:

WRKOBJOWN USRPRF(<sid>OFR) OBJTYPE(*ALL)

If so, use CHGOBJOWN to transfer ownership and CHGPGP to assign a new primary group if possible.

Verify also if the user profile is still used as a logon user or to run scheduled jobs.

Once user profile <sid>OFR has been deleted, user <sid>ADM can be enabled again using the following commands:

CHGOBJOWN OBJ(<sid>ADM) OBJTYPE(*USRPRF) NEWOWN(<sid>ADM) CUROWNAUT(*REVOKE)

CHGUSRPRF USRPRF(<sid>ADM) INLPGM(SAP<sid>IND/SAPINLPGM) LMTCPB(*NO)
