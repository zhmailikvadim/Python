Symptom

This document discusses some common understandings about SAP HANA that are not completely true when you have a closer look.

Environment

SAP HANA, platform edition

Resolution

The following table lists typical misconceptions in the SAP HANA environment:

MisconceptionDetails

All SAP HANA tables are processed column-oriented.

SAP HANA allows to put a table either in column store or in row store. Tables in row store are accessed record-wise like traditional relational databases. Only tables in column store are processed column-wise. Based on the access pattern it is possible to individually decide which table works better in row and in column store.

The column store generally provides better performance than the row store.

In most cases the column store provides good performance, but there are still situations where the row store has advantages. As a general trend SAP tries to locate as many tables in column store as possible.

No indexes are required.

It is true that SAP HANA can scan large amounts of data even without the existence of an index. Nevertheless there are still situations where indexes are useful:

  • Support of a primary key or unique key constraint (e.g. primary indexes in SAP ABAP scenarios)
  • Optimization of central, performance-critical queries
  • Optimization of queries consuming a lot of CPU resources due to highly parallelized operations
  • Tables in row store need indexes similarly like tables in classic relational databases

For more information on SQL statement optimization and indexes see SAP Notes 2000002 and 2160391.

Indexes are not persisted to disk.

This is not generally true. For example, the main storage of column store concat attributes related to indexes is generally persisted to disk.

See SAP Note 2160391 ("Are indexes persisted to disk?") for more details.

SAP HANA is a pure in-memory database.

Disk I/O doesn't matter.

SAP HANA is an in-memory database, but nevertheless there are different areas where disk I/O impacts the performance, e.g.:

  • Row store load during startup
  • Column store load
  • Hybrid LOB accesses
  • Savepoints and snapshots
  • Commits
  • Backups

For more information of SAP HANA I/O analysis see SAP Note 1999930.

All table data is permanently in memory.

For some reasons not all SAP HANA tables are permanently and completely kept in memory:

  • Both in row store and column store large hybrid LOB column values are never loaded into memory (SAP Note 1994962).
  • In column store tables aren't necessarily in memory if they aren't used or if they are unloaded due to memory issues.
  • Historic, rarely accessed data can be stored on extended storage and nearline storage outside of the actual SAP HANA database.
Row store and column store functionality is strictly separated.Historically row store and column store came from different databases, but during the integration in SAP HANA functionality was adjusted and consolidated. Therefore it is not unusual to see e.g. row store modules involved in the processing of column store table accesses.
MOTIVATION = 'AUTO' in M_DELTA_MERGE_STATISTICS always indicates an auto merge.

Up to SAP HANA Rev. 1.00.121 the 'AUTO' entry in M_DELTA_MERGE_STATISTICS is not only used by auto merges, it is also used by critical merges which follow completely different rules. As a consequence you can see 'AUTO' entries even for tables with an explicitly disabled auto merge.

Starting with SAP HANA Rev. 1.00.122 critical merges are recorded with MOTIVATION = 'CRITICAL' in M_DELTA_MERGE_STATISTICS.

See SAP Note 2057046 for more information regarding delta merges.

M_DELTA_MERGE_STATISTICS contains only delta merge information.

No, M_DELTA_MERGE_STATISTICS contains also information for other operations like compression optimizations (TYPE = 'SPARSE') or delta storage garbage collection (TYPE = 'RECLAIM').

See SAP Note 2057046 for more information.

The duration of a delta merge highly depends on the merged records.

A delta merge (SAP Note 2057046) recreates the complete table, so it can be considered as a table reorganization. As a consequence the runtime of a delta merge depends much more on the size of the table (partition) in total and not so much on the amount of merged records. In order to avoid long delta merges (and optimize compressions) and related issues like high resource consumption or locking, you should make sure that large tables are set up with a sufficiently high amount of partitions (SAP Note 2044468), e.g. making sure that a single partition doesn't exceed a size of 20 to 50 GB.

Monitoring view information is generally lost during a restart.

Many SAP HANA monitoring views (having names starting with 'M_') contain runtime information that is lost when a restart happens. Nevertheless this is not true for all monitoring views, e.g.:

  • M_CS_UNLOADS is based on the unload trace files which can contain data of several weeks and months regardless of restarts (see SAP Note 2088971). The same applies to M_CS_LOADS.
  • M_EXPENSIVE_STATEMENTS (SAP Note 2180165) and M_EXECUTED_STATEMENTS (SAP Note 2366291) are also based on trace files located on operating system level.
  • M_LOAD_HISTORY_SERVICE and M_LOAD_HISTORY_HOST are based on the nameserver load history trace files (SAP Note 2222110).
  • M_MERGED_TRACES is based on the existing SAP HANA trace files on file system level.

As a side-effect the content of these monitoring views can change when a system replication takeover to a different host is performed (SAP Note 1999880).

TOTAL_EXECUTION_TIME in M_SQL_PLAN_CACHE contains the total elapsed time of a SQL statement.

The total elapsed time consists of both the preparation time (i.e. parse time) and the actual execution time. The preparation time can be significant in some cases. So the following formula is correct:

"Total elapsed time of SQL statement" = TOTAL_PREPARATION_TIME + TOTAL_EXECUTION_TIME
The unload trace files contain only information about column unloads.

This is not always true. With SAP HANA 1.0 SPS 08 both unloads and loads are recorded in the unload trace files. As a consequence it can happen that large unload trace files exist but at the same time the monitoring view M_CS_UNLOADS doesn't contain any record. See SAP Note 2058284 for more information.

Fragmentation is no issue.

No reorganizations are required.

SAP HANA suffers from less fragmentation because the merges in the central column store are implicitly recreating the tables. Therefore reorganizations of the column store in memory aren't required. Nevertheless there is the risk of fragmentation in other areas:

  • Row store
  • Data volumes
  • Heap allocators

See SAP Note 2000003 ("Where can fragmentation occur in SAP HANA environments?") for more information.

The finest granularity of a column store load is a column

Per default this is correct, but it is possible to defined paged attributes (SAP Note 1871386) so that columns can be loaded in smaller chunks.

There is no SQL plan cache.

There is no impact of parsing for the first execution of a SQL statement.

SAP HANA takes advantage of a SQL cache (see SAP Note 2124112). This means that parsing / preparing of a SQL statement is only required when it is executed the first time. If parsing consumes significant time, subsequent executions of the SQL statement can be significantly quicker than the first execution.

In scale-out scenarios no column store tables are located on the coordinator* node.

It is normal that several tables are still located in the column store of the coordinator* node in a scale-out system. It is mainly important that large and critical tables are moved to the worker* nodes (e.g. infocube, DSO and PSA tables in BW environments).

In scale-out scenarios the row store is only on the coordinator* node.

From an application table perspective it is correct that only the coordinator* node is populated with row store tables. Nevertheless also the other nodes contain a minimal technical implementation of the row store, so that row store dictionary tables also exist on the slave nodes.

The join engine only processes join requests.

The join engine is not only used for joins. It is also used for certain queries on single tables, e.g. if an aggregation like SUM or GROUP BY is performed.

UPDATE statements are always DML operations

That's not generally true. SAP HANA uses UPDATE operations also as DDL operations. As an example, the following UPDATE triggers an optimization of the column store compression:

UPDATE "<table_name>" WITH PARAMETERS ('OPTIMIZE_COMPRESSION' = 'YES')
There are no popular misconceptions for other databases.There are popular misconceptions also for other databases. See SAP Note 825653 for common misconceptions in Oracle environments.
The amount of records per table is limited to 2 billion.

Tables can have a nearly unlimited amount of records:

  • In the column store each table partition can hold 2 billion records. If a non-partitioned table is about to exceed 2 billion records, you can activate partitioning (SAP Note 2044468) in order to eliminate the limitation.
  • In the row store there is generally no 2 billion record limit.
Table partitioning is only useful in scale-out scenarios.

SAP HANA table partitioning is useful for various purposes, e.g.:

  • Load distribution in scale-out scenarios
  • Elimination of the 2 billion record limit
  • SQL statement optimization (e.g. due to partition pruning)
  • Optimization of merge operations

See SAP Note 2044468 for more information.

Restore and complete recovery can never eliminate existing corruptions.

It depends on the root cause of the corruption:

  • If corruptions are caused by SAP HANA bugs, a restore and complete recovery will typically re-introduce them, because the bug is part of both the data and the log area.
  • If corruptions are caused by hardware issues (e.g. "wrong checksum"), a restore and complete recovery will typically fix them.

See SAP Note 2116157 for more information.

Tables with a partition specification have PART_IDs greater than 0

Normally the assumption is true that a non-partitioned table is display with PART_ID 0 in views like M_CS_TABLES while table partitions have PART_ID values greater than 0. Nevertheless it can happen that a table has only PART_ID 0 although a partition specification is used: A partitioning specification of HASH 1 or ROUNDROBIN 1 will result in a non-partitioned table and so the PART_ID is 0. See SAP Note 2044468 for more information.

The lock mode INTENTIONAL EXCLUSIVE indicates an exclusive lock.

The name is misleading: INTENTIONAL EXCLUSIVE can be considered as a shared lock. This means that e.g. multiple concurrent changes of different records of the same table can happen in different transactions, although an INTENTIONAL EXCLUSIVE object lock can be seen on table level. Real exclusive locks, that don't allow concurrent changes, have the lock type EXCLUSIVE. See SAP Note 1999998 for more information.

The unload priority of row store tables is generally 0.

Tables in row store are generally kept in memory, so one could assume that the unload priority is always 0, i.e. never unload the table (SAP Note 2127458). If you check the UNLOAD_PRIORITY information in TABLES you can instead typically see the default value of 5 for application tables in row store, so a normal unload behavior.

In fact unloads are linked to the column store and the unload priority isn't evaluated for the row store. The unload priority is still useful, e.g. in case the table is moved to column store in the future.

When all table columns (from M_CS_COLUMNS) are loaded, the table will have LOADED = 'TRUE' in M_CS_TABLES.

Even if all normal table columns visible in M_CS_COLUMNS are loaded, the table itself is not necessarily fully loaded. It can still happen that it shows up with LOADED = 'PARTIALLY'. This constellation indicates that at least one internal column (e.g. $trex_udiv$ or a concat attribute) is not loaded into memory. You can find information about internal columns and their load state in M_CS_ALL_COLUMNS.

For the same reason a "SELECT *" doesn't necessarily result in a fully loaded table.

Query optimization only happens during plan generation

During prepare a generic execution plan is created, but during actual execution some more optimizations (e.g. column access order) happen. As a consequence you can see activities of the column store query optimizer (qo3 / trex_qo) or other SAP HANA optimizers also during the actual execution of a SQL statement.

The table attribute NO LOGGING on SAP HANA can be compared with NOLOGGING options of other relational databases.

NOLOGGING on databases like Oracle indicates that no redo log information is generated. The table data itself is properly persisted. If a table in SAP HANA is created with NO LOGGING, neither log nor data is persisted to disk. Therefore NO LOGGING can't be used for efficient data loads without generating log information. Instead you have to use ALTER TABLE ... DISABLE DELTA LOG for that purpose.

See SAP Note 2000003 ("What kind of temporary and non-persisted tables can be created with SAP HANA?") for more information.

M_TABLE_VIRTUAL_FILES contains disk size information for all tables.

Virtual files only exist for column store tables, so row store table are not covered in this view. If you are interested in global disk size information, you can check M_TABLE_PERSISTENCE_STATISTICS.

The column MODIFY_TIME of M_CS_TABLES indicates the timestamp of the last DDL operation on the table.

MODIFY_TIME of M_CS_TABLES neither corresponds to the last DDL time nor to the last DML time. Instead it is only updated if a CONCAT attribute (e.g. a multi column index) is adjusted (e.g. created, modified or dropped). See SAP Note 1986747 for more information related to CONCAT attributes.

As of SAP HANA SPS 10 you can use the column LAST_MODIFY_TIME of M_TABLE_STATISTICS to check when the last DML operation has happened on a table.

The SAP HANA Studio load graph and the information in M_LOAD_HISTORY_HOST and M_LOAD_HISTORY_SERVICE is collected by the statistics server

Although the load graph information is a history, it is not collected by the statistics server. Instead the nameserver process is in charge of collecting this information and persisting it to the nameserver_history.trc files. See SAP Note 2222110 ("Does the load history require certain statistics server levels?") for more details.

The statement_memory_limit parameter controls the maximum memory a single SQL statement can allocate.

The statement_memory_limit parameter (SAP Note 2222250) defines the maximum memory a SQL statement can allocate on a single host (in a single service). In scale-out systems a SQL statement can allocate this memory on various hosts and so the overall memory allocation of the SQL statement can be much higher.

SELECT statements can't hold SAP HANA locks or be blocked by SAP HANA locks.

This statement is correct for transactional locks, but it is not necessarily true for internal locks. See SAP Note 1999998 where you can e.g. see that SELECTs may hold or wait for "BTree GuardContainer" locks in certain scenarios.

The parameter indexserver.ini -> [transaction] -> idle_cursor_lifetime controls the maximum acceptable idle time of cursors.

The idle_cursor_lifetime parameter is not limited to idle cursors, it also covers active SQL statements. See SAP Note 2169283 ("When does SAP HANA automatically terminate a connection for garbage collection reasons?") for more information.

The EXCLUSIVE_ALLOCATED_SIZE in M_HEAP_MEMORY indicates how much memory is currently allocated by a heap allocator.

The column EXCLUSIVE_ALLOCATED_SIZE in monitoring view M_HEAP_MEMORY (respectively HOST_HEAP_ALLOCATORS) contains the sum of all allocations in this heap allocator since the last startup. Normally also a lot of deallocations happen, so the EXCLUSIVE_ALLOCATED_SIZE can be much higher than the currently allocated size. For example, if over time 100 MB are allocated and deallocated 10 times, the actual allocated size is 0, but EXCLUSIVE_ALLOCATED_SIZE would show 1 GB (10 * 100 MB). See SAP Note 1999997 ("Why is the allocated memory in some heap allocators very large?") for more information.

All tables approaching 2 billion records should be partitioned.

There are important exceptions from the rule to partition tables that are coming close to the 2 billion record limit. For example, SID tables in BW (/BI*/S*) should never be partitioned because they can never exceed 2 billion records and at the same time the two unique indexes can cause trouble if the table is partitioned. See SAP Note 2044468 for more information.

The first argument of ALTER SYSTEM ALTER CONFIGURATION command always specifies the parameter file.

Usually the first argument of ALTER SYSTEM ALTER CONFIGURATION is the parameter file, but there is one exception: If topology.ini is specified as first argument, the change is done in the topology within the nameserver persistence. There is no parameter file topology.ini.

Delta merges are either triggered by the mergedog or manually / by the application.

This statement is valid most of the time, but it is important to understand that merges can also be triggered SAP HANA internally in specific contexts, e.g. during recovery or log replay. These implicit merges can't be controlled by mergedog adjustments.

A recovery related merge can e.g. be determined by the following modules in the call stack: DataRecovery::RecoveryQueue::run, UnifiedTable::RedoHandler::redo and UnifiedTable::MergeJob::execute.

SAP HANA automatically returns records in the intended sort order.

As long as you don't specify an ORDER BY, the order of the result set of a database request is undefined. You can't assume a specific order, it can change at any time. SAP HANA indexes don't support the sort order (SAP Note 2160391), so e.g. accessing a table via the primary key doesn't mean that also the result is returned in the order of the primary key.

Therefore it is essential that you specify an explicit ORDER BY whenever you need a specific sort order.

See SAP Note 2347758 for more information.

Columns are only recompressed during optimize compression runs.

In general columns are recompressed during optimize compression runs, but in specific scenarios it can happen that the compression of a column changes during a delta merge. See SAP Note 2112604 ("14. Can a delta merge also include compression activities?") for more information.

The [internal_hostname_resolution] settings are only used when listeninterface is set to .internal.

As described in SAP Note 2183363 you can set the listeninterface parameter to .internal in order to make sure that SAP HANA only listens for incoming requests related to IP addresses configured in the [internal_hostname_resolution] section. So having proper values configured in [internal_hostname_resolution] is a prerequisite for using the .internal value for listeninterface.

The opposite is not true: If you configure an [internal_hostname_resolution] but you leave the listeninterface on .global, the settings in [internal_hostname_resolution] are still preferably used for SAP HANA server communication. On top of it SAP HANA also listens for other incoming requests and - in case a host is not maintained in the [internal_hostname_resolution] - it will use another IP address for communication.

See SAP Note 2222200 for more information related to SAP HANA network configuration.

The executed SQL statements can be found in monitoring view M_EXECUTED_STATEMENTS.

M_EXECUTED_STATEMENTS only contains DDL operations like CREATE or TRUNCATE, normal SQL types like SELECT, INSERT, UPDATE and DELETE aren't shown in this view. Instead you can check M_SQL_PLAN_CACHE for example. See SAP Note 2366291 for more information related to the executed statements trace and SAP Note 2000002 for more information related to SQL statement analysis.

A dedicated statement preparation is only used in context of bind variables.

No, using statement preparation (SAP Note 2124112) is independent of the use of bind variables, it can also be used in context of SQL statements without bind variables. Standard tools like SAP HANA Studio (SAP Note 2073112), SAP HANA Cockpit (SAP Note 2800006) or SAP ABAP per default use a dedicate prepare step before executing the database request. Among others this dedicated prepare phase is important for proper statement routing (SAP Note 2200772).

There is a 1:1 relationship between statement text and statement hash.

In general SAP HANA generates the statement hash based on the text of a SQL statement (SAP Note 2000002), so usually the same statement hash represents the same SQL text. However, there is one exception in context of statement hints (SAP Note 2400006): Up to SAP HANA 2.0 SPS 03 the statement hint is internally added to the statement text and the statement hash is calculated for this extended statement. So it appears that the identical SQL statement has two different statement hashes, dependent on the existence of a statement hint. This behavior is changed with SAP HANA >= 2.0 SPS 04 where the statement hash is no longer impacted by the defined statement hints.

A high memory allocation of SAP HANA indicates a problem.

SAP HANA is in general lazy in terms of deallocating memory, so the allocated memory can be quite high compared to the used memory. Only values too close to the allocation limit can sometimes cause trouble due to memory reclaim performance issues so that the parameters gc_unused_memory_threshold_abs and gc_unused_memory_threshold_rel should be set for proactive memory garbage collection (SAP Note 2600030).

What actually matters more is the memory used by SAP HANA, not the memory allocated by SAP HANA.

The operating system may allocate additional memory for the Linux page cache if available. This also doesn't indicate a problem because in case of a higher memory requirement the page cache will reduce its size flexibly. Typically a potentially unlimited page cache is configured (vm.pagecache_limit_mb = 0).

As a consequence high values for the following sizes are common and on their own don't indicate a problem (because they are closely correlated to the allocated size on SAP HANA side):

  • Resident size on operating system level
  • Used size on operating system level
  • Allocated size on SAP HANA level

See SAP Note 2081473 for more details.

The SAP HANA Mini Checks (SAP Note 1999993) consequently don't report high values for these memory sizes as potentially critical.

See SAP Note 1999997 for more information related to SAP HANA memory analysis.

A record lock is only held during the time the modification command is running.

No, record locks are held until the next COMMIT happens. Between the actual modification and the COMMIT a lot of other activities can take place, e.g.:

  • Access to different SAP HANA tables
  • Communication with the client
  • Client-side activities (e.g. ABAP processing)

As a consequence it is normal that the lock holder of a record lock is shown as inactive (not executing any database request) or with database requests that aren't directly linked to the modified table.

SET TRANSACTION commands like SET TRANSACTION READ ONLY only apply to the current transaction.

Although the command name indicates that it only influences the current transaction, the SET TRANSACTION properties are valid for the connection and not just for the transaction. So even after a commit the setting is still in place and it is also used for the subsequent transactions on the same connection. This can have unexpected impact - in the worst case even on different secondary ABAP connections that share a database connection pool - and so SET TRANSACTION commands must not be used in ABAP environments (SAP Note 2560888).

The SAP HANA parameter settings are perfect out of the box and no manual adjustments are required.

This is wrong, there are several parameters that should be set based on the individual system and load scenario or as a workaround for known SAP HANA issues. See SAP Note 2600030 for details about parameter best practices on SAP HANA, SAP ABAP and operating system level.

ABAP table buffering isn't required for SAP HANA because it is already an in-memory database.

The main advantage of the ABAP table buffer (SAP Note 2103827) is that network roundtrips to the database are eliminated. It is of minor importance if the requested data is actually buffered on database side or not. Therefore the ABAP table buffer is also of high importance in context of SAP HANA databases.

The Linux file system cache isn't required for SAP HANA because it is already an in-memory database.

Even though HANA doesn’t use Linux filesystem cache for its persistence, caching provides advantages in terms of accessing executables, dumps, traces, configuration files and Linux logs. The Linux file system cache only uses available memory and is freed in case memory is required elsewhere, being also used in SAP HANA contexts.

A CANCEL SESSION operation always rolls back the related transaction.

CANCEL SESSION (SAP Note 2092196) only performs a transaction rollback when a database request is currently running on the connection. In case the connection is currently idle, the transaction is not rolled back. This can result in unexpected scenarios like locks of previous modifications still held after having executed CANCEL SESSION. If you want to ensure that the transaction is rolled back, you have to use DISCONNECT SESSION.

The views TABLES and M_TABLES contain the same number of records.

Although both views are supposed to list the existing permanent tables in the SAP HANA database, there can be differences caused by different filtering in the view definition. For example, M_TABLES displays internal tables with names starting with _SYS while TABLES filters these tables:

... T1.NAME NOT LIKE '\_SYS%' ESCAPE '\' ...
The RECORD_COUNT in views like M_TABLES or M_CS_TABLES is always transactionally correct.

Overall high efforts are put in the transactional correctness of the displayed number of records (RECORD_COUNT) that includes a shared index handle lock (SAP Note 1999998) on the underlying table. Nevertheless there are scenarios where a wrong RECORD_COUNT value is possible, e.g. when a table is currently not loaded (issue number 253016). SAP Note 2955698 describes a problem in context of ABAP client copies where tables weren't properly copied because the procedure erroneously relied on RECORD_COUNT values. For an existence check you can run a SELECT COUNT or SELECT TOP 1 on the underlying table instead.

Removing the previous value in the "Configuration" -> "Change Configuration Value" section of SAP HANA Studio restores the default value

No, if you remove the previous value, i.e. leave "New Value:" empty, an empty value is stored for the parameter. An empty value is usually outside of the parameter specification and can lead to a high amount of unexpected effects. For example, when global.ini -> [multidb] -> mode is set to an empty value, the database will no longer come up when persistence encryption is used because the related key is no longer found in the secure store. Empty values can also result in terminations of analysis commands available via SAP Note 1969700 in case a number value is expected that is used for calculations.

If you want to restore the default value of a parameter, you have to chose the "Restore Default" button instead.

See SAP Note 2073112 for more information related to SAP HANA Studio and SAP Note 2186744 for more information to SAP HANA parameters.

If you search for table entries that either fulfill a condition or do not fulfill a condition, all table entries are returned.

The following statements, which, at first glance, appear to be the same, do not necessarily return the same number of entries:

SELECT COUNT(*) FROM <table>;
SELECT COUNT(*) FROM <table> WHERE <column> = <condition> OR <column> != <condition>;

Instead, the second statement may return a much lower number of entries. This is the case if <column> contains NULL values. For databases the value NULL is not a value that is comparable with any condition. Therefore, the second statement only returns entries that do not contain the value NULL in <column>.

The SAP HANA Mini Checks detect any kind of database issue.

The SAP HANA Mini Checks (SAP Note 1999993, SQL: "HANA_Configuration_MiniChecks") perform comprehensive monitoring and alerting, but you must not rely too much on them:

  • Certain problem scenarios can't be detected automatically or with reasonable efforts, so they are not included.
  • Important checks are executed by other analysis commands available via SAP Note 1969700, e.g. SQL: "HANA_Configuration_Parameters", SQL: "HANA_SQLCache_TopLists", SQL: "HANA_TraceFiles_MiniChecks" or SQL: "HANA_Callstacks_MiniChecks".
  • The Mini Checks provide some generic results without criticality rating. In any system they will report potentially critical issues (C = 'X'). Thus, they are helpful to improve the overall database health, but they don't provide targeted recommendations for a specific critical issue that you want to solve.
Online database operations must be executed when the application runs, offline operations must be executed when the application is stopped.

This is not generally true. Online in context of DDL operations basically means that the amount of locking is minimal so that concurrent application load can be executed without delays. Offline means that a transactional lock is held during the execution and as a consequence modification operations are blocked. It is no problem to execute an online operation while the application is stopped and an offline operation at the same time can sometimes also be executed in parallel to concurrent business load (e.g. when the runtime is limited or the business load doesn't touch the processed tables).

See SAP Note 1999998 -> "Can DDL commands be executed online?" for more information related to online and offline operations.

The disk size and number of records of a table are rather correlated.

Most of the times this is true, but there can be a temporarily increased table disk size that can't be explained with the number of records:

  • Active table optimization (SAP Note 2057046)
  • Active standard repartitioning (SAP Note 2044468)
  • Increased delta storage size
  • Blocked garbage collection (SAP Note 2169283)
An out-of-memory (OOM) situation always results in a termination

No, there are specific scenarios where running into an OOM situation doesn't mean that the activity is cancelled with an error. One example is the data shipment on secondary system replication site as explained in SAP Note 1999880 -> "How can pending savepoints and large page cache sizes on secondary site be explained?". Instead of terminating, it will wait for more memory to be available after data being flushed to disk.

The more distinct values a column has, the more even is the distribution of records with hash partitioning based on this column

A high number of distinct values is not sufficient for a rather even record distribution, also the existence of popular values influences the distribution. Instead, the best data distribution is achieved with the column having the lowest inverted individual costs. See SAP Note 2044468 -> "HASH partitioning: Selective partitioning column(s) without popular values" for more details.

The load history ping time is an indicator for communication overhead and database health.

The load history ping time indicates how long it takes the nameserver to collect the load history details from other services (SAP Note 2222110). The runtime can heavily depend on factors like the number of existing internal locks. Thus, the load history ping time can't be used to check for communication issues and average runtimes between 1 and 400 ms are in a normal range. Also, temporary variations and spikes are possible for various reasons without indicating a severe SAP HANA problem. Hence, load history ping times shouldn't be used for performance benchmarking.

As an exception, episodes of ping times of several seconds can indicate SAP HANA health issues, e.g. blockages due to memory reclaims, locking or resource overload. In this case, a more detailed SAP HANA performance analysis (SAP Note 2000000) can be performed, focusing on the time of the load history ping time spikes.
