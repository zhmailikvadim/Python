Symptom

When processing a Direct Debit Authorization (DDA) return file (CNAB 240 format) using transaction FF_5 or the Fiori app 'Manage Incoming Payment Files', it is observed that some entries from the bank file are missing in the table J1B_ERROR_DDA.

This leads to an incomplete update of supplier invoices, as the DDA_BR transaction cannot find all the necessary records to apply the barcodes. Upon inspection, it is noted that for transactions with a duplicated reference number, only the last item is saved in the J1B_ERROR_DDA table.

Environment

  • SAP ERP
  • SAP S/4HANA
  • Brazil Payment Files and Bank Statements

Reproducing the Issue

  1. Prepare a Direct Debit Authorization (DDA) return file in CNAB 240 format. The file must contain at least two transactions (i.e., at least two Segment G records) where field 16.3G (Número do Documento de Cobrança), located at positions 148 to 162, is empty. Other transactions in the file may have this field populated.
  2. Process the prepared DDA file using transaction FF_5 or the Fiori app 'Manage Incoming Payment Files'.
  3. After the file is processed, check the contents of the table J1B_ERROR_DDA (for example, using transaction SE16 or SE16N).
  4. Observe that the number of entries in the table J1B_ERROR_DDA is less than the number of transactions present in the imported DDA file. This occurs because the record for the first transaction with the empty reference field is overwritten by the subsequent one during import.

Cause

The root cause of the issue is that the Direct Debit Authorization (DDA) return file provided by the bank contains an empty value in field 16.3G (Número do Documento de Cobrança), located at positions 148 to 162 within Segment G.

The standard SAP solution relies on this field (which is mapped to the internal field FEBEP-XBLNR) to uniquely identify each transaction. When field 16.3G is empty, the standard logic in report RFEBBU00 attempts to generate a reference number using the following fallback logic:

*      Bildung der Referenzbelegnummer
      IF febep-xblnr IS INITIAL.
        febep-xblnr+0(5)  = febko-hbkid.
        febep-xblnr+5(5)  = febko-hktid.
        febep-xblnr+10(2) = febko-azdat+2(2).
        febep-xblnr+12(3) = febko-aznum+15(3).
      ENDIF.

This automatically generated reference may not be unique for all items within the same bank statement file, leading to duplicates. When the system processes the file, these duplicate references cause previous entries in the J1B_ERROR_DDA table to be overwritten, resulting in only the last item with a given reference being saved.

Resolution

To resolve this issue and ensure all records from the Direct Debit Authorization (DDA) file are processed correctly, two approaches can be taken. The recommended approach is to correct the data at the source.

1. Recommended Solution: Correct the Outgoing Payment File Process

The most robust solution is to ensure the DDA return file from the bank contains the necessary unique identifier.

  • Review the process for generating the original payment file sent to the bank (e.g., using DMEEX tree BR_FEBRABAN_A or report RFFOBR_U).
  • Ensure that a unique document identifier (such as the document number - BELNR or reference - XBLNR) is correctly populated in the payment file sent to the bank.
  • By providing this unique reference in the outgoing file, the bank can then return it in the DDA file within field 16.3G (Número do Documento de Cobrança) of Segment G. This will prevent the generation of duplicate references during import.

2. Alternative Solution: Implement an Enhancement

In scenarios where correcting the incoming bank file is not possible, implement a custom logic enhancement to generate a unique reference number during the import process.

  • Implement the enhancement spot DDA_BR_POINT.
  • Within this enhancement, develop logic to create a unique value for the FEBEP-XBLNR field for each record processed from the DDA file. This will ensure that no entries are overwritten in the J1B_ERROR_DDA table, allowing the DDA_BR transaction to process all items correctly.
