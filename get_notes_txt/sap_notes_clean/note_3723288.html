Symptom

  • After reversing a Preliminary DME run with PRDX_DEL, the system creates a new run.
  • Entries for the old run remain visible in the payment program cluster tables, especially REGUH (and potentially REGUV/REGUP).
  • You would like to delete the REGUH entries for the old run and are considering using program SAPF110R.

Environment

  • SAP R/3
  • SAP ERP
  • SAP ERP Central Component
  • SAP enhancement package for SAP ERP
  • SAP enhancement package for SAP ERP, version for SAP HANA
  • SAP S/4HANA

Reproducing the Issue

  1. Execute PRDX_DEL – Reverse Preliminary DME Run.
  2. Observe that a new run is created by the system.
  3. Check REGUH (and related cluster tables) via SE16/SE16N and see entries still exist for the first run reversed.
  4. Consider manual deletion or look for a standard cleanup option.

Cause

  • Expected system behavior. 
  • PRDX_DEL reverses the preliminary DME processing logic but does not purge all technical/payment program data already written to the cluster tables.
  • By design, REGUH (and related cluster data such as REGUV/REGUP) can remain for consistency and auditability, even if the business run is no longer valid.
  • REGUH is a cluster table; direct deletion (SE16/SM30/custom ABAP) is not supported and risks corrupting payment program data.

Resolution

  • Do not delete REGUH entries manually. Direct deletion or custom programs are not supported.
  • Use SAPF110R as the supported tool to clean up obsolete payment run data (including REGUH/REGUV/REGUP) for the specific run.

  • Recommended approach:
    1. Preparation and safeguards:
      • First validate the behavior in a non-production system with a comparable run.
      • Ensure a recent database backup/restore point is available; inform/align with your Basis team.
    2.  Execution of SAPF110R:
      • Restrict the selection strictly to the obsolete run.
      • Run in test mode initially to review exactly which objects will be deleted (e.g., REGUH/REGUP/REGUV entries and related logs).
      • After verifying the test log, execute in update mode with the same, narrow selection.
    3.  Post-activities and verification:
      • Confirm in SE16/SE16N that REGUH entries of the obsolete run are no longer present.
      • Verify that the new run remains intact and fully usable.
      • Perform functional checks for payment processing and reporting to ensure no side effects occurred.

  • From a support and audit perspective, SAPF110R is the only recommended approach for this scenario. Direct deletion or custom solutions are not supported and should be avoided.
