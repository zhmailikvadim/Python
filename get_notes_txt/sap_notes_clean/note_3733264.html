Symptom

Upon installing the ASCS instance, SWPM got stuck at the NW_GetUserParameterWindows step, sapstartsrv returned error -1

Environment

  • SAP NetWeaver on Windows

Reproducing the Issue

  1. Start SWPM/SAPinst (e.g., 753.0.14) to install Application Server > Distributed System > SCS Instance on Windows.
  2. Proceed until the step that creates and starts the SCS Windows service, step Execute Service in SWPM.
  3. Observe the step fails with sapstartsrv returning -1 and Windows service logon errors.

Cause

  • The sidadm account lacks the "Allow log on locally" (interactive logon) user right on the ASCS host, causing Windows to deny LOGON32_LOGON_INTERACTIVE and raise error 1385. This is typically enforced by Local Security Policy or a domain Group Policy (for domain-joined workstations).

    Logon failed. DETAILS: The system could not log on user DOMAIN\sidadm because the user has not been granted the right to log on interactively to this computer. SOLUTION: Check the user rights assignment (Local Security Policy). Make sure that the user has been assigned the right "Log on locally" has not been assigned the right "Deny logon locally", the rights are effective and there is no domain policy that overrides these right assignments. You can try whether the user has the necessary rights using the runas command (runas /profile /user:DOMAIN\sidadm).

Resolution

  1. Sign in to the SCS host with an account that can modify local security policy (for domain-joined systems, use a domain admin or an account permitted to change effective policies).
  2. Open the Local Security Policy (Windows + R, run: secpol.msc).
  3. Verify the sidadm account is not listed in "Deny log on locally". If domain Group Policies apply, ensure they do not override these settings.
  4. Navigate to Security Settings > Local Policies > User Rights Assignment > Allow log on locally, and add the sidadm account.
  5. Proceed with the SWPM phase.
