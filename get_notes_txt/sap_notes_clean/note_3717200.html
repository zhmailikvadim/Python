Symptom

SSC(Solution Sales Configuration) External Configuration API incorrectly defines multiple mandatory parameters (id, kbbuild, kbid, rootItem, and configXML) in the request payload schema. For CPS integration, only rootItem should be mandatory. The current implementation creates unnecessary redundancy by requiring these many parameters. This misalignment between API specification and CPS requirements increases integration complexity and forces API consumers to provide duplicate/unnecessary data.

Other Terms

MANDATORY_PARAMETERS, REQUIRED_FIELDS, EXTERNAL_CONFIGURATION_API, SSC_API, CPS_INTEGRATION, ROOTITEM, CONFIGXML, API_VALIDATION, REQUEST_SCHEMA, PARAMETER_REDUNDANCY, API_CONTRACT, SCHEMA_DEFINITION, OPENAPI, KBBUILD, KBID, BACKEND_ALIGNMENT

Reason and Prerequisites

Prerequisites:

- Understanding of SSC External Configuration API and CPS integration
- Knowledge of configuration creation methods:

  • recreateConfigFromXmlWithContext() (when configXML provided)
  • createConfiguration() (when configXML is null/absent)

- Access to OpenAPI YAML/JSON files for parameter schema updates
- Test data: valid rootItem values and sample configXML
- Access to API validation and routing logic code
- Testing tools to verify both configuration creation paths
- Understanding of parameter validation requirements

Reason:

SSC External Configuration API incorrectly enforces five mandatory parameters (id, kbbuild, kbid, rootItem, configXML). Only rootItem is required for CPS integration. The current implementation creates unnecessary redundancy (both rootItem and configXML required) and prevents flexible configuration creation patterns. API contract must be updated to align with CPS backend requirements - making rootItem the only mandatory field and implementing conditional routing based on configXML presence.

Solution

Solution: Changes as part of API Parameter Optimization:

1. Updated Mandatory Parameters:
- Changed from: id, kbbuild, kbid, rootItem, configXML (all mandatory)
- Changed to: rootItem (only mandatory), all others optional

2. Implemented Conditional Configuration Logic:

Validation:
- Added rootItem presence check
- Reject request if rootItem is missing
- Proceed if rootItem is present

Configuration Creation Routes:
- IF configXML provided WITH rootItem:

  • Call recreateConfigFromXmlWithContext()
  • Generate configId from XML context


- IF configXML is null/absent:

  • Call createConfiguration()
  • Generate new configId from rootItem

3. Code Modifications:

Parameter Validation:
- Removed mandatory checks for: id, kbbuild, kbid, configXML
- Retained mandatory check for: rootItem only

Configuration Routing:
- Added conditional logic to route to appropriate creation method
- Handles both recreation (with XML) and new creation (without XML)

4. Documentation Updates:
- Updated OpenAPI YAML 'required' array to rootItem
- Marked id, kbbuild, kbid, configXML as optional
- Added usage pattern documentation

Files Modified:
- API validation logic
- Configuration routing/creation logic
- OpenAPI specification YAML/JSON
