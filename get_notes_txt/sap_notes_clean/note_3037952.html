Symptom

You want to use the 'Mass Change' functionality of one of the Line Item Browsers (FAGLL03H, FBL5H, FBL1H) and note 2450963 is already installed. However, applying changes is unfeasible for a high number of records due to poor perfomance.

This note describes how to set up parallel processing of the mass change using asynchronous RFC modules.

Other Terms

FI_ITEMS_MASS_CHANGE FI_ITEMS_MASS_CHANGE_CALL FI_ITEMS_MASS_CHANGE_RFC_CALL FBL5H FAGLL03H FBL1H MASS_CHANGE FAGL_LIB_PIVB

Reason and Prerequisites

Poor performance of mass change call.

Solution

Requirements

  • 2450963 - Line Item Browsers: PIVB
  • 3025355 - FI Line Item Browser: DDIC Objects for Mass Change Call
  • 3025989 - FI Line Item Browser: Code Changes for Mass Change Call
  • 3034497 - CALL TRANSACTION USING is aborted abnormally in asynchronous RFC sessions triggered from WebGUI

Note 3034497 is only required when the mass change aRFC call is used via WebGui frontend applications. If the note is not installed aRFC calls fail without prompting any error message. The user is informed that the mass change was run successful despite no changes being appplied.

Setup

The aRFC processing of the mass change call can be adjusted with the following parameters in table FAGL_SETTINGS:

  • MASS_CHANGE_NUM_TASK: Determines the maximum number of dialog work processes that can be used for a single mass change. When MASS_CHANGE_NUM_TASK is less than 1 usual sequential processing instead of parallel aRFC is performed.
  • MASS_CHANGE_SERV_GRP: Defines the server group destination. When no server group is specified the default server group parallel_generators (delivered by SAP) is used.   
  • MASS_CHANGE_PAC_SIZE: Controls the number of records processed in each task. See section 'Package Size Calculation' for more information.

Caution: When configuring the server group and maximum number of work processes keep in mind that ill-conceived settings can lead to performance issues in the whole system. These parameters are not user dependent. With multiple users at the same time the number of work processes/system resources used up by the mass change might become unreasonable. You have to thouroughly test these settings in an adequate test scenario before applying this in a production environment.

Package Size Calculation

The parameter MASS_CHANGE_PAC_SIZE controls the number of records that is send to each individual task. But single documents must not be split into different work packages. Thus, work packages are created on basis of the number of records per company code/fiscal year/document number instead of single line items. An optimization is performed to distribute all line items as evenly as possible onto the number of tasks provided via MASS_CHANGE_NUM_TASK. However, there might be single documents with more line items than defined in MASS_CHANGE_PAC_SIZE causing poor performance. When MASS_CHANGE_PAC_SIZE is not defined (value in FAGL_SETTINGS is initial) a default value of 100 is used.

Application Log

If any number of records wasn't changed successfully an application log is created. This log directly comes up after the mass change call has finished. For each record that wasn't changed two lines are created in the log:

  1. The first line specifies the exact document in the form of 'company code/fiscal year/document number'.
  2. The second line shows the error message for the specified document.

For later review it's also saved and can be found via transaction SLG1. To search for application logs created by the mass change call you can use object 'FI_LIB' and/or subobject 'FI_LIB_MC'.
