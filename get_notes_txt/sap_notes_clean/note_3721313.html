Symptom

In certain scenarios, when the RFC function module CDESK_SRV_DISPLAY_DMUDATA is called with variant‑specific information, the input parameter IV_VARIANT_GUID is ignored.

As a result, the function may return DMU data for all variants. The caller is not able to identify which data belongs to the requested variant.

Other Terms

PLMECTR-1122, CL_CDESK_SRV_DMUDATA_DISPL, ECTR

Reason and Prerequisites

The behavior may occur when variant‑dependent DMU data is maintained and the caller expects IV_VARIANT_GUID to control the selection.

Solution

The handling of the variant GUID was adjusted to ensure consistent application during DMU data processing.
The following changes were implemented:

  • The input parameter IV_VARIANT_GUID was removed from method IF_CDESK_SRV_DMUDATA_DISPL~GET_DMUPB_BY_STPO of class CL_CDESK_SRV_DMUDATA_DISPL.
  • The method now exclusively uses the instance attribute MS_CDESK_VARIANT-VARIANT_GUID, which is initialized by the constructor.
  • The selection logic was aligned so that the variant GUID stored in the instance is consistently applied.

Existing behavior remains unchanged for other scenarios.
