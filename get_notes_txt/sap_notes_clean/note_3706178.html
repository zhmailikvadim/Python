Symptom

The choice between Full Load DTP, Delta, or Delta Initialization with ODP (Operational Data Provisioning) in SAP BW, BW/4HANA is a challenge and you need to know what is the best approach to use according to your business scenario.

Environment

  • SAP S/4HANA
  • SAP ERP
  • SAP NetWeaver
  • SAP BW/4HANA

Resolution

Overview

The choice between Full Load DTP, Delta, or Delta Initialization with ODP (Operational Data Provisioning) in SAP BW depends on data volume, source system capabilities, business requirements, and performance considerations. Each approach serves specific scenarios in the data loading process.

DTP Types Comparison

DTP TypeUse CaseData VolumeFrequencyPerformance Impact
Full LoadComplete data refresh, data repair, small datasetsSmall to mediumInfrequentHigh resource usage
DeltaIncremental updates, ongoing operationsAny volumeRegular/scheduledOptimized performance
Delta InitializationHistorical data load, delta setupLarge volumesOne-time setupVariable based on approach

Decision Criteria

Full Load DTP
  • When to Use:
    • Small data volumes where complete refresh is acceptable 
    • Data repair scenarios for specific semantic filters with wrong transformation rules
    • Source systems that don't provide delta mechanisms 
    • When existing records need to be overwritten in the target
  • Advantages:
    • Simple implementation and execution
    • Complete data consistency guaranteed
    • Can repair data with specific filters 
  • Disadvantages:
    • High resource consumption for large datasets
    • Overwrites existing data completely
    • Not suitable for high-volume scenarios
Delta DTP
  • When to Use:
    • Source system supports delta extraction 
    • Regular incremental data updates are required
    • Ongoing operational data loading
    • When only changed/new records need processing
  • Prerequisites:
    • Source DataSource must support delta mechanism
    • Delta initialization must be completed first 
    • Filter conditions for different Delta DTPs must be non-overlapping
  • Advantages:
    • Optimal performance for incremental loads
    • Preserves system resources
    • Maintains data consistency with proper sequencing 
Delta Initialization
  • When to Use:
    • First-time historical data load from source systems
    • High-volume scenarios where full DTP would be too slow
    • When different filter settings are needed for historical vs. ongoing data 
    • Setting up delta processes without transferring data
  • Process Steps:
    1. Create DTP with Extraction Mode set to Full (for historical data)
    2. Create second DTP with Extraction Mode set to Delta and "Delta Init w/o Data" checkbox selected 
    3. Execute the Delta DTP again to start actual delta loading
  • Considerations:
    • Risk of data changes between Full load and Delta Initialization 
    • Requires careful timing to avoid missing changes
    • May involve system downtime for some applications

ODP-Specific Considerations

  • Source System Requirements:
    • DataSource must be ODP-enabled (ROOSATTR table with EXPOSE_EXTERNAL = 'X') 
    • Source system connection under ODP-SAP (Extractors) must be established
  • Delta Capability:
    • ODP supports delta extraction through Operational Delta Queue (ODQ)
    • Delta framework guarantees records are uploaded only once and in correct order
  • Extraction Logic:
    • Direct extraction from source system bypassing PSA when using ODP 
    • Data stored in different ODQ tables: ODQDATA_F (Full), ODQDATA_C (Init), ODQDATA (Delta) 

Best Practices

  1. Use Delta DTPs for regular operational loads to optimize performance
  2. Implement Full DTP before Delta approach for high-volume historical data scenarios 
  3. Ensure proper filter design - overlapping conditions not allowed for Delta DTPs
  4. Monitor ODQ using ODQMON transaction for delta queue management 
  5. Use Standard DataStore Objects when loading from different sources to control update sequence 

Common Scenarios

  • Initial Load: Delta DTP for low volume, Full DTP + Delta Init for high volume
  • Regular Updates: Delta DTP with scheduled execution
  • Data Repair: Full DTP with specific semantic filters
  • Historical Data: Full DTP before Delta approach with different filter settings

IMPORTANT: Always keep in mind that for some of SAP delivered extractors/datasources this can vary as they have their unique logic, so it is require to check with the responsible application and obey its logic accordingly.
