Symptom

  • In a Java program, the following exception may occur when a temporary table is created and accessed across separate java.sql.PreparedStatement objects.

        at com.sybase.jdbc42.tds.Tds.nextResult(Tds.java:3347)
        at com.sybase.jdbc42.jdbc.ResultGetter.nextResult(ResultGetter.java:78)
        at com.sybase.jdbc42.jdbc.SybStatement.nextResult(SybStatement.java:308)
        at com.sybase.jdbc42.jdbc.SybStatement.nextResult(SybStatement.java:290)

Environment

  • SAP Sybase Software Developer Kit (SDK) 15.7
  • SAP Adaptive Server Enterprise (ASE) Software Developer Kit (SDK) 16.0
  • jConnect
    • JDBC 7.0
    • JDBC 7.07
    • JDBC 16 ( is version JDBC 7.07 )

Cause

  • This is expected behavior where a temporary table cannot be accessed across multiple PreparedStatement objects if DYNAMIC_PREPARE connection property is set to true.

Resolution

1. Background of ASE behavior

In SAP ASE, temporary tables created within a stored procedure have a scope that is strictly limited to the lifetime of that stored procedure.
According to the ASE documentation "Manipulate Temporary Tables in Stored Procedures" :

Temporary tables whose names begin with # and are created inside a stored procedure are automatically dropped when the procedure finishes execution.

2. jConnect implementation

When the jConnect DYNAMIC_PREPARE connection property is set to true, jConnect internally generates a stored procedure — referred to as a Lightweight Procedure (LWP) — for SQL statements defined in a java.sql.PreparedStatement. 
As a consequence, any temporary tables created by using a PreparedStatement object exist only within the scope of the generated LWP. Once the LWP completes, those temporary tables are no longer available.

Therefore, attempting to reference a temporary table created via a PreparedStatement object from outside the LWP will result in an error.
This behavior is expected and by design, as it aligns with ASE's stored procedure scoping rules.

For this reason, SQL statements that create temporary tables should not be executed using java.sql.PreparedStatement when DYNAMIC_PREPARE is enabled.
Instead:
   (1) Use java.sql.Statement to create temporary tables, or
   (2) Set DYNAMIC_PREPARE=false to prevent jConnect from generating Lightweight Procedures.

3. jConnect special handling

On the other hand, jConnect has a special handling.
If the first command in a PreparedStatement is a CREATE statement, jConnect avoids creating a Lightweight Procedure, even when DYNAMIC_PREPARE is true.
As a result, the following code works because the CREATE TABLE statement is the first command in the PreparedStatement:

-----------
    pstmt = conn.prepareStatement("create table #test1 (col1 int, col2 char(10))");
    rtn = pstmt.executeUpdate();
    pstmt.close() ;

    rtn = 0 ;

    sqlstr = "insert into #test1 values ( ?, ? ) " ;

    System.out.println( "prepareStatement runs." ) ;
    pstmt = conn.prepareStatement(sqlstr);

    pstmt.setInt(1, 1);
    pstmt.setString(2, "TEST");
    rtn = pstmt.executeUpdate();
-----------

However, this special handling applies only when the CREATE statement is the first command in a PreparedStatement. This is because checking all commands within a  PreparedStatement to determine whether each command is a CREATE statement would introduce additional overhead in jConnect, which could potentially accumulate and impact overall processing performance when processing a large number of PreparedStatement objects.

Therefore, if the CREATE TABLE statement is preceded by another SQL command within the same PreparedStatement, this special handling is not applied, a Lightweight Procedure is generated, and the temporary table becomes inaccessible across multiple PreparedStatement objects.

For example, the following code fails as expected:

-----------
    pstmt = conn.prepareStatement("insert into real_table values (1) \n create table #test1 (col1 int, col2 char(10))");
    rtn = pstmt.executeUpdate();
    pstmt.close() ;

    rtn = 0 ;

    sqlstr = "insert into #test1 values ( ?, ? ) " ;

    System.out.println( "prepareStatement runs." ) ;
    pstmt = conn.prepareStatement(sqlstr);

    pstmt.setInt(1, 1);
    pstmt.setString(2, "TEST");
    rtn = pstmt.executeUpdate(); /* An exception is raised */
-----------

To ensure that this special handing is applied, CREATE TABLE #temp_table_name statement must be the first command in a java.sql.PreparedStatement.
If this requirement cannot be met, java.sql.Statement should be used instead, or DYNAMIC_PREPARE should be disabled.
