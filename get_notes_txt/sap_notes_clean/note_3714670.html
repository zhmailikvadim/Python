Symptom

  • For time-based POBs, revenue schedule is not getting changed upon revenue date changes.
  • FARR_D_FULFILLMT table is being updated, but the reason for this update is unclear.
  • This is applies for both OCM and CCM version, but with some differences.

Environment

  • SAP ERP
  • SAP S/4HANA
  • SAP S/4HANA Finance

Cause

For time-based POB, system works as follows:

  • Time-based POBs generate entries in the FARR_D_FULFILLMT table as part of standard functionality.
  • For time-based POBs, fulfillment does not rely only Fulfillment RAIs. The system creates a sequence of fulfillment events over the service period based on the POB’s Start/End Dates, Start Date Type, and Deferral Method.

This applies for OCM and CCM version, but there are some differences for them:

CCM (Classic Contract Management):

  • Time-based POBs (Start Date Type 1 and 2; SDT 3 when triggered by CI) generate time-based fulfillments that are persisted in FARR_D_FULFILLMT. Deferral methods supported: 1, 2, S, F, L.
  • The quantity representation for time-based fulfillments uses fractions (nominator/denominator). These fulfillments are “RAR-self” generated based on the start/end dates and deferral method once the start date is known.

OCM (Optimized Contract Management):

  • Time-based POBs also create entries in FARR_D_FULFILLMT. OCM supports SDT 1 and 2; SDT 3 is supported only with CI as the trigger. Time-based quantities are day-based (no fraction fields for the schedule logic; deferral method “3” merges CCM’s 3 and 4). Fulfillments are generated by RAR when the start date is determined and persisted to FARR_D_FULFILLMT.
  • OCM introduces differences in deferral method handling (new BAdI FARR_BADI_DEFERRAL_METHOD_V2), and uses days to present quantities per period. Start Date Type 3 is limited to CI trigger; groups of fulfillments are created upon the triggering invoice.

Edge cases and clarifications:

  • For SDT 3, entries in FARR_D_FULFILLMT are created only after the trigger event (in CCM and OCM the supported trigger is CI; OCM explicitly limits SDT 3 to CI).
  • Event-based fulfillments from external RAIs do not populate FARR_D_FULFILLMT for a time-based POB unless the POB is configured with a start-date-type that expects an event trigger (SDT 3) and the event matches the configured type (e.g., CI).
  • Value-based fulfillment behavior differs: OCM has a “value_based_fulfillment” concept; however, time-based POBs still write their period fulfillments to FARR_D_FULFILLMT.

Resolution

This is not an issue but a standard system behavior for time-based POBs.
