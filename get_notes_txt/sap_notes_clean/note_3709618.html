Symptom

There is a desire to implement Subresource Integrity (SRI) for the SAPUI5 bootstrap script (id="sap-ui-bootstrap") by adding the integrity attribute when fetching the framework from SAP-managed CDN hostnames such as ui5.sap.com or sapui5.hana.ondemand.com

Environment

SAPUI5 framework

Resolution

While SRI is a valuable mechanism for assets loaded from untrusted third-party sources, its implementation in the context of SAPUI5 resources involves significant technical constraints and a different security profile.

Architectural Limitations & Dynamic Loading

The SAPUI5 framework is a modular system. The bootstrap script (sap-ui-core.js) is only the entry point: once initialized, it dynamically creates script tags to load additional libraries and components based on your application's needs. While it is technically possible to provide a hash for the initial bootstrap script, the framework does not currently have a mechanism to provide or verify SRI hashes for the dozens (or hundreds) of bundles or modules loaded dynamically afterwards. Applying SRI only to the bootstrap would provide an incomplete security chain, as the majority of the framework code would still be loaded without integrity verification.

Incompatibility with "Evergreen" Versions and Cache-Busting

SRI is fundamentally designed for immutable resources. The SAPUI5 "evergreen" concept (e.g. the long-term SAPUI5 release 1.136 being redirected to the URL that contains the latest patch number) and SAPUI5 cache buster mechanics pose a challenge to SRI. If SAP releases a critical security patch or bug fix, the file content changes and the hash changes. If you had an SRI hash in your HTML, the browser would block the updated SAPUI5 script, causing your application to break immediately upon the release of the patch. SRI can only be used with "pinned" versions (e.g. 1.136.14), which places a significant maintenance burden on you to update hashes manually for every patch.

SAP Security Assurance and Infrastructure

SAPUI5 resources are served from SAP-managed infrastructure, which operates under a comprehensive security framework that ensures end-to-end integrity:

  • Managed Infrastructure: Assets are served through secure internal systems with strict role-based access controls and immutable deployment processes. The delivery pipeline is fully self-contained, ensuring traceability without third-party dependencies.
  • Continuous Validation: SAP maintains a rigorous security program, including regular static/dynamic code analysis (SAST/DAST), periodic penetration testing by internal and external experts, and continuous vulnerability monitoring.
  • Trusted Source Profile: SRI's primary purpose is to protect against supply-chain tampering from uncontrolled external CDNs. Because SAPUI5 resources are delivered from SAP’s own trusted environment, the risk of unauthorized modification is mitigated by these platform-level controls.

Recommendation

SAP strongly recommends following the documented security guidelines which provide comprehensive protection for SAPUI5 applications. Specifically:

  • Strict Content Security Policy (CSP): Ensure your application is ready for a strict CSP. This is a more effective defence-in-depth mechanism for modular frameworks than SRI.
  • Security Best Practices: Please review our documentation on securing SAPUI5 applications, which covers protection against XSS, clickjacking, and other attack vectors:  Securing Apps - SAPUI5 Documentation

By implementing a robust CSP and following these guidelines, one can achieve a high level of security that aligns with how modern modular frameworks operate.
