Symptom

  • An error occurs when refreshing the webpage in any infotype mashup, displaying the message: "HRSFEC_SERVICE245 - You do not have the authorization to view this pay."
  • The issue persists despite clearing cookies, using different web browsers, verifying certificates in TCode STRUST, activating services, and checking SU53 for authorization errors.

Environment

Employee Data Maintenance via Mashups

Reproducing the Issue

  1. Open the EC mashup portlets and open one of them.
  2. Refresh the webpage when in infotype mashup.

Cause

The issue arises due to the interaction between the secure pay statement functionality and the configuration in table T77S0. The URL nonce supplied by the EC URL is compared with the last sequence timestamp stored in HRSFEC_D_REPLSEQ, leading to the error when the timestamp comparison fails.

Resolution

The problem is interaction between the secure pay statement functionality and the configuration in table T77S0. Specifically, the security switch for pay statements (GRPID = 'SFEC', SEMID = 'SPYSL') is in effect, and the URL nonce (timestamp) supplied by the EC URL is being compared with the last sequence timestamp stored in HRSFEC_D_REPLSEQ. When the nonce in the URL is not newer than the stored LAST_SEQ_TIME (or matches it while iv_ec_1 is not '1'), the routine sets ev_is_ok to false and returns with ev_subrc = 2, which produces the error you are seeing.

The relevant code section is shown below:

34        METHOD    CHECK_URL    CL_HRSFEC_WD_EC_ID_MAPPING====CP        CL_HRSFEC_WD_EC_ID_MAPPING====CM003    60

  ev_is_ok = abap_true.
* Check for the switch for enabling secure payslip logic
  SELECT SINGLE gsval FROM t77s0 INTO lv_gsval
  WHERE grpid = 'SFEC' AND semid = 'SPYSL'.
  IF sy-subrc EQ 0 AND lv_gsval IS INITIAL.
    ev_is_ok = abap_true.
    ev_subrc = 0. 
.....
    lv_date_time = iv_ec_nonce.
*   Check if the time stamp is new
    SELECT SINGLE * FROM hrsfec_d_replseq BYPASSING BUFFER INTO ls_time
      WHERE content_type = 'SU'  "Content Type - Secure URL
      AND employee_id = iv_ec_objid AND country_iso = lv_intca ##WARN_OK.
    IF sy-subrc = 0 AND ( lv_date_time LT ls_time-last_seq_time OR
    ( lv_date_time EQ ls_time-last_seq_time AND iv_ec_1 NE '1' ) ).
      ev_is_ok = abap_false.
      ev_subrc = 2.
      RETURN.
    ENDIF. 

In practical terms, there are two viable approaches to resolve the situation. The first is to modify the T77S0 setting for the SFEC/SPYSL entry, but this should only be done if it aligns with your security policy and after appropriate change control review. The second is to ensure that each generated EC URL includes a fresh, unique timestamp that will always be newer than the value stored in HRSFEC_D_REPLSEQ; this prevents the timestamp comparison from failing and allows the mashup to load normally.
