Symptom

  • Errors in application log:
    Application error in called function module
    An exception was raised
    Persistence engine defined for // is not supported by XML enabler
  • Invalid characters in documents in source system

Environment

SAP Landscape Transformation Replication Server


Cause

Enable more detailed error reporting:

SAP Note 3620378 SLT encounters 'Persistence engine defined for &1/&2/&3 is not supported by XML enabler' error - SLT

SAP Note 3422997 XML Engine: more specific exception when persisting payload raised an error.

Invalid characters in SLT documents often arise due to incorrect non-Unicode codepage settings in the RFC connection or the source data being stored with the wrong codepage.

Resolution

Steps to Resolve Invalid Characters in SLT Documents
  1. Check RFC Connection Settings:

     

    • Verify the non-Unicode codepage settings in the RFC connection.
    • Ensure that the codepage specified matches the codepage used for the source data.
  2. Examine Source Data:

     

    • Confirm that the source data is stored with the correct codepage.
    • If a record is saved with a specific codepage (e.g., 8000 for Japanese), ensure it is not mistakenly saved with a different codepage (e.g., 8400 for Chinese Simplified).
  3. Language Key Verification:

     

    • Ensure that the language key used in the source data is correct.
    • Refer to note 2633548 for single character language key mapping to ensure proper configuration.
SAP Landscape Transformation Replication Server (SLT) has methods to handle records with invalid characters, primarily by using transformation rules to clean the data or by managing the error logs to prevent them from stopping the replication process. 
Here is how to deal with invalid characters in SLT and skip loading them:
 
Using Transformation Rules to Clean/Skip Data
The most effective way to handle invalid characters in SLT is by creating a transformation rule to check and clean the data before it is loaded into the target system. 
  • Create a Rule in LTRC: Go to transaction LTRC, navigate to "Expert Functions" -> "Define Transformation Rules".
  • Field-Related Rule: Choose a field-related rule to identify the invalid characters (e.g., in long text fields).
  • ABAP Logic: Use ABAP code to replace or remove the invalid characters. You can use CL_ABAP_CHAR_UTILITIES to identify characters between HEX00 and HEX1F, which are often the cause of errors.
  • Skip Option: To skip a specific record, you can write a rule that deletes or masks the data in that record, or in complex scenarios, handle it via custom ABAP coding. 
Alternative: Pre-Processing or Data Source Filtering 
  • Source System Cleanup: It is often more efficient to identify the invalid characters (using PATINDEX or CHARINDEX in SQL, for example) and clean them in the source system before they reach SLT.
  • Filtering in SLT: Use the filtering capabilities in LTRC (under "Advanced Replication Settings") to exclude specific rows or datasets that contain known invalid data.
