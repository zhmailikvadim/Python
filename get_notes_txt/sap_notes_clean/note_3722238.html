Symptom

When a Cloud Integration IFlow contains a Splitter step and fails with an exception afterward, multiple alert emails are generated for what appears to be the same failed message processing. Instead of a single alert for the failed MPL, several alerts are triggered. This occurs because each split message instance results in its own exception handling and MPL creation. As a result, users observe multiple alert notifications related to the same original IFlow execution.

Environment

  • SAP Integration Suite
  • Cloud Integration

Cause

The issue occurs because the IFlow uses a Splitter step that processes multiple split messages independently. When an exception (such as an authentication failure) happens after the Splitter, the error is triggered separately for each split message. Each failing split instance enters the exception subprocess individually. As part of this exception handling, the defined integration flow in the exception subprocess is called. Every time this happens, a separate Message Processing Log (MPL) is created for that specific split instance.

Since the Splitter is configured without selecting the “Stop on Exception” option, the remaining split messages continue to be processed even after the first failure occurs. If the same error condition exists for the other split messages, each of them will also trigger the exception subprocess. Consequently, multiple calls to the exception handling IFlow are executed. This results in several MPLs being generated for what functionally relates to a single original message. The multiple MPLs then lead to multiple alert emails being sent.

Resolution

To prevent multiple alert emails and MPLs from being generated for the same original message, enable the “Stop on Exception” option in the Splitter step of the IFlow. When this checkbox is selected, the processing of further split messages is immediately stopped as soon as the first exception occurs. This ensures that only one split instance triggers the exception subprocess and the corresponding error handling integration flow. As a result, only a single Message Processing Log (MPL) is created for the failure scenario.

You can configure this setting in the Splitter step under the General tab of the IFlow design in SAP Cloud Integration. By activating this option, you prevent successive split messages from executing after an error, thereby avoiding multiple calls to the exception subprocess and eliminating duplicate alert notifications.
