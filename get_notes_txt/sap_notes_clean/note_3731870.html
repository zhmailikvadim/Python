Symptom

SAP Service and Asset Manager incorrectly allows entering and saving decimal values during Measuring Point evaluation, even if the corresponding backend Measuring Point is configured with "Decimal places = 0"

Other Terms

  •  SAP Service and Asset Manager
  • SAM
  • SSAM
  • Decimal
  • Take Reading
  • Measurement Point
  • ICMTANGOAMF10-43433

Reason and Prerequisites

Reason:

The evalPrecision() logic relies on JavaScript Number() conversion for determining decimal precision. In comma‑based locales, numeric conversion fails (Number("1,5") → NaN), resulting in evalPrecision() returning 0. This makes the application incorrectly assume no decimal places were entered, bypassing integer validation for Measuring Points with Decimal = 0.

Prerequisites:

  •  BAS or Visual Studio Code with SAM MDK project
  • `patch` command (available on BAS and macOS by default) via terminal or other patching utility
  • Read the guidelines on how to apply the patch and read the appendix section if you receive any error

Solution

The issue was fixed by replacing the decimal-precision check with a direct integer-validation step whenever the Measuring Point has the decimal value set to 0. The prevents locale-related parsing errors and ensures that only whole numbers are accepted. The logic for Decimal > 0 was also updated.

To apply the patch, download the attached 3731870.patch file and save it in a known location, preferably adjacent to your SAM.mdkproject.

If you are on Windows, see SAP Note 3533879 for how to install a .patch file in Windows. If you are on a Mac, follow the steps below:

  1. Open a terminal in the root of your SAM metadata (i.e. SAM.mdkproject/metadata or SAM.mdkproject/metadata/base.definitions)
  2. Run the terminal command `patch --no-backup-if-mismatch -p4 < path/to/3731870.patch`. 
  3. Verify in the terminal window that the following files have been modified and listed as patched, as indicated by the console output:
  • metadata/definitions/Rules/Measurements/MeasuringPointLibrary.js
