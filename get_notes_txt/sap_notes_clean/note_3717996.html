Removing HA/DR provider configuration sections from the HANA DB configuration files manually fixes the issue and the database starts up. Example:

[ha_dr_provider_SAPHanaSR]
path = /usr/share/SAPHanaSR
execution_order = 1
provider = SAPHanaSR

Note, that "SAPHanaSR" is the provider name in the examples above but it can have any other name.

Environment

SAP HANA, platform edition 2.0

Reproducing the Issue

  1. Stop the secondary site HANA DB.
  2. Unregister it from its primary site.
  3. Start the former secondary site as a standalone system.

Resolution

SAP HANA does not start if an HA/DR provider is configured but cannot be loaded. In this case, the nameserver terminates with messages such as "Could not load HA/DR Provider" (for example, due to a missing provider module or an invalid provider path).

HA/DR provider configuration is maintained by the customer. System Replication operations executed with hdbnsutil (for example, -sr_register / -sr_unregister) do not create, adjust, or remove HA/DR provider entries in the HANA configuration.

For this reason, after performing System Replication changes (including hdbnsutil -sr_unregister), the customer has to verify whether the HA/DR provider entries that are present in the configuration are still required:

  • If they are still relevant for your setup, keep them and ensure the provider can be loaded (correct provider name/path, provider present).
  • If they are no longer relevant (for example, the system should run without that provider), remove the corresponding HA/DR provider entries from the HANA configuration (e.g. the [ha_dr_provider_<provider>] section). Then start the system again.
