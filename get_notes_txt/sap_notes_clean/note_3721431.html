Symptom

In certain scenarios, calling the function module /DSCSAG/DOC_COMPONENTS_BY_DATE may return document structures that do not match the requested timestamp.
This may occur when no historical Rule Header is available or when iteration‑accurate structure persistence is not active.
In these cases, the function silently uses the current “as‑saved” structure, without informing the caller about the fallback.
This can result in inconsistent snapshot resolutions or unexpected behavior in consumer applications.

Other Terms

 PLM-42160, ECTR, /DSCSAG/DOC_COMPONENTS_BY_DATE, /DSCSAG/CL_CONTENT_VERSION_ITR, CDESK_SRV_GET_EXPL_RULESET

Reason and Prerequisites

The previous implementation did not provide explicit return messages when:

  • No Rule Header was found for the requested timestamp.
  • The preference SAP.PLM.CONTENT_VERSION_ITR.SWITCH was missing or set to false, meaning iteration‑specific structure data is not being persisted.

This lack of messaging may lead to callers assuming that a historical structure was resolved successfully, even though the system had internally fallen back to resolving the latest “as saved” structure.

Solution

The handling in /DSCSAG/DOC_COMPONENTS_BY_DATE and the related class /DSCSAG/CL_CONTENT_VERSION_ITR was enhanced.
