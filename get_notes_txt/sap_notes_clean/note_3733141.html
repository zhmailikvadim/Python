Symptom

While setting the warehouse loading stop to "Loading Completed" from the RF-UI screen in Advanced Shipping and Receiving processes, users encounter the locking error:
"TO XXXXXXXXXX is already locked by another user and cannot be processed" (Message /SCMTMS/BATCH_80 187)

This error prevents the loading stop from being completed and blocks the goods issue process for outbound deliveries. Users will have to wait for couple of seconds and retry the action to set loading completed to proceed.

Reason and Prerequisites

When a handling unit is scanned and set to "loaded" via RF-UI, a bgRFC unit is created to process the status change asynchronously. This bgRFC process acquires an optimistic lock ('O') on lock object /SCMTMS/E_SYLDEN for the transportation order.

According to the design, when the user triggers "Loading Completed", the system should:

  1. Acquire an exclusive lock ('E') on /SCMTMS/E_SYLDEN to signal bgRFC units to suspend processing
  2. Process remaining LINC activities synchronously

However, the optimistic lock ('O') acquired by the bgRFC process cannot be promoted to an exclusive lock ('E') by the "Loading Completed" process because they operate in different Logical Units of Work (LUWs). This results in a foreign lock error (message /SCMTMS/BATCH_80 187: "TO XXXXXXXXXX is already locked by another user and cannot be processed"), preventing the loading stop from being completed.

Solution

Instead of acquiring an optimistic lock ('O') on /SCMTMS/E_SYLDEN in the bgRFC process, only conflict check with exclusive lock ('V') will be performed. The 'V' lock mode for bgRFC process allows the "Loading Completed" process later on to successfully acquire an exclusive lock ('E') on /SCMTMS/E_SYLDEN without encountering foreign lock errors, even when the bgRFC and "Loading Completed" processes operate in different LUWs.

This change enables the synchronization mechanism to function as designed, allowing the "Loading Completed" process to properly signal bgRFC units to suspend and subsequently process all pending LINC activities synchronously within one LUW.

Implement this note to correct the lock mode used during bgRFC processing of HU loading status updates.
