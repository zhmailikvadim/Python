Symptom

You have BW scenarios (ADSO, External HANA View, modeled HANA views based on BW content) which are using SAP HANA database for hot data and Near- Line Storage (NLS) for cold data. In the corresponding HANA Calculation Scenario, HANA and NLS data are combined via Union or MultiProvider node(s). Parts of the HANA data are Unicode-enabled character strings (NVARCHAR(M)). On the top level Union, MultiProvider, Aggregation or Semantic Node for such columns, you see datatype VARCHAR(3*M) instead.

Other Terms

varchar nvarchar mismatch, union datatype inconsistency, NLS varchar problem, calculation view column type wrong, multiprovider datatype issue, BW archiving type mismatch, HANA upgrade datatype change, cold hot data type difference, sybase IQ varchar length, query fails different datatypes

Reason and Prerequisites

Issue number 355266

Reason:

In the affected releases of SAP HANA, the data type of VARCHAR(M) columns in Union or MultiProvider nodes is derived differently. When a Union or MultiProvider node needs to determine the data type (because no explicit data type definition is available) from multiple sources, HANA now selects the largest data type among all corresponding child nodes. This creates issues because NLS database (Sybase IQ) handles NVARCHAR(M) strings differently than regular HANA storage — these strings are exposed as VARCHAR(3*M) in the corresponding HANA virtual table and then propagated into the HANA Calculation Views built on top.

The old behavior was that in the same situation (Union or MultiProvider node) where the data type had to be derived, HANA used the data type of the first corresponding child node, thus avoiding the issue.
This old behavior is the expected behavior for BW NLS scenarios.

Affected Releases:

  • SAP HANA 2:
    • Revisions 079.07 and 079.08 (SPS07)
    • Revisions >= 088.00 (SPS08)
  • SAP HANA Cloud Edition:
    • Revisions >= 2025.40 (2025 QRC4)

Prerequisites:

  • You are (re)deploying BW scenarios (ADSO, External HANA View, modeled HANA views based on BW content) containing NVARCHAR(M) columns.

Workaround:

The steps 1.-4. we document below as "Manual Workaround" can be executed programmatically. Download ABAP program "ZSAP_TYPES_IN_VIRT_TABLE.txt" attached to this Note and run it in your affected system. The program has entry fields for "DSO name" which supports wildcards. Also there are flags "show statements" and "simulate only" which are set by default.

After running the program, make sure to also run step 5. from the "Manual Workaround" section.

5. Redeploy the (ADSO) Scenario views by running ABAP Program RSDDB_LOGINDEX_CREATE.

Manual Workaround:

As an alternative to using the program in the previous section "Workaround", you can manually modify the existing HANA Virtual Table for the NLS data source and set the required datatypes manually. On a high level, proceed as follows:

  1. Take note of all table columns and their data types.
  2. Drop all columns of the Virtual Table except for one (e.g. let's say you decide keep the last column).
  3. Add all dropped columns again. If the dropped column has had datatype VARCHAR(3*M), add it as NVARCHAR(M). Otherwise, keep the previous datatype.
  4. Drop and recreate the remaining colum (in our example, the last column), again changing the datatype if it was of the form VARCHAR(3*M).
  5. Redeploy the (ADSO) Scenario views by running ABAP Program RSDDB_LOGINDEX_CREATE.

Example:

Usually the Virtual Table for BW ADSO <ADSONAME> follows the naming convention "/BIC/OU<ADSONAME>".

1. As an example, ADSO ZRKADSO3 has the Virtual Table "<SCHEMA>"."/BIC/OUZRKADSO3". Now assume the Virtual Table has the following columns and datatypes:

(
"ARCHREQTSN" DECIMAL(23)
,"ARCHDPAKID_SID" INTEGER
,"ARCHRECNO_SID" INTEGER
,"ARCHDOBJID_SID" INTEGER
,"CALMONTH" VARCHAR(18)
,"CALMONTH2" VARCHAR(6)
,"CALQUARTER" VARCHAR(15)
,"CALQUART1" VARCHAR(3)
,"CALWEEK" VARCHAR(18)
,"CALYEAR" VARCHAR(12)
,"FISCVARNT" VARCHAR(6)
,"FISCPER" VARCHAR(21)
,"FISCPER3" VARCHAR(9)
,"FISCYEAR" VARCHAR(12)
,"HALFYEAR1" VARCHAR(3)
,"WEEKDAY1" VARCHAR(3)
,"UNIT" VARCHAR(9)
,"RECORDMODE" VARCHAR(3)
)

2. In this case, the SQL statement for step 1. (Drop all columns of the Virtual Table except one) above is

ALTER TABLE "<SCHEMA>"."/BIC/OUZRKADSO3" DROP
(
"ARCHREQTSN"
,"ARCHDPAKID_SID"
,"ARCHRECNO_SID"
,"ARCHDOBJID_SID"
,"CALMONTH"
,"CALMONTH2"
,"CALQUARTER"
,"CALQUART1"
,"CALWEEK"
,"CALYEAR"
,"FISCVARNT"
,"FISCPER"
,"FISCPER3"
,"FISCYEAR"
,"HALFYEAR1"
,"WEEKDAY1"
,"UNIT"
)

3. Add the dropped columns again, if the dropped column has had datatype VARCHAR(3*M), add it as NVARCHAR(M). Otherwise keep the datatype.

 ALTER TABLE "<SCHEMA>"."/BIC/OUZRKADSO3" ADD

(
"ARCHREQTSN" DECIMAL(23)
,"ARCHDPAKID_SID" INTEGER
,"ARCHRECNO_SID" INTEGER
,"ARCHDOBJID_SID" INTEGER
,"CALMONTH" NVARCHAR(6)
,"CALMONTH2" NVARCHAR(2)
,"CALQUARTER" NVARCHAR(5)
,"CALQUART1" NVARCHAR(1)
,"CALWEEK" NVARCHAR(6)
,"CALYEAR" NVARCHAR(4)
,"FISCVARNT" NVARCHAR(2)
,"FISCPER" NVARCHAR(7)
,"FISCPER3" NVARCHAR(3)
,"FISCYEAR" NVARCHAR(4)
,"HALFYEAR1" NVARCHAR(1)
,"WEEKDAY1" NVARCHAR(1)
,"UNIT" NVARCHAR(3)
)

4. Repeat step 2. and 3. for the last remaining column of the Virtual Table.

ALTER TABLE "<SCHEMA>"."/BIC/OUZRKADSO3" DROP
(
"RECORDMODE"
)

ALTER TABLE "<SCHEMA>"."/BIC/OUZRKADSO3" ADD
(
"RECORDMODE" NVARCHAR(1)
)

5. Redeploy the (ADSO) Scenario views by running ABAP Program RSDDB_LOGINDEX_CREATE.
